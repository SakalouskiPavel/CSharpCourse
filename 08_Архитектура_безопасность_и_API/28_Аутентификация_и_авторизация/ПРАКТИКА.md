# Практика — урок 28

## Цель

Защитить API заказов из урока 27, не создавая собственный небезопасный сервер паролей. Для локальной работы допустим тестовый authentication handler; в README объясните, чем он должен быть заменён в реальной системе.

## Сценарий

Есть пользователи:

- покупатель видит только свои заказы;
- сотрудник с permission `orders:read:any` видит любой заказ;
- отменять заказ может владелец со `orders:cancel:own` либо сотрудник с `orders:cancel:any`;
- анонимному пользователю доступен только `/health`.

## Обязательная часть

1. Создайте ASP.NET Core API или продолжите проект урока 27.
2. Подключите authentication scheme. Для тестов handler принимает заранее заданные test identities; никаких настоящих паролей и ключей в Git.
3. Настройте fallback policy, требующую аутентификацию. `/health` явно сделайте анонимным.
4. Создайте policy `orders.read` с подходящим permission.
5. Реализуйте resource-based requirement/handler для чтения собственного или любого заказа.
6. Реализуйте аналогичное правило отмены.
7. Не принимайте user id/role из тела запроса как доверенное доказательство; subject берите из проверенного principal.
8. Возвращайте согласованные `401`, `403` и `404`, не раскрывая существование скрытого заказа.
9. Записывайте audit отмены: subject id, order id, действие, результат, UTC-время и correlation id. Не логируйте credential или token.
10. Добавьте `SECURITY.md` с моделью угроз и ограничениями учебной реализации.

## Автоматические тесты

Через `WebApplicationFactory` или эквивалент проверьте минимум:

- анонимный `/health` успешен;
- анонимный запрос заказа получает `401`;
- владелец читает свой заказ;
- покупатель не узнаёт о чужом заказе;
- сотрудник с `orders:read:any` читает чужой заказ;
- владелец с permission отменяет свой заказ;
- аутентифицированный пользователь без permission получает отказ;
- user id в теле запроса не позволяет выдать себя за владельца;
- audit не содержит тестового токена.

## Критерии готовности

- решение собирается и тесты проходят;
- policies названы по намерению, а не по контроллеру;
- проверка владельца находится на сервере;
- отрицательные сценарии проверены;
- секретов и персональных данных в репозитории/логах нет;
- README объясняет `401`/`403`, cookie/token и ограничение тестовой схемы.

## Дополнительное задание

Добавьте две схемы (cookie и bearer) только как конфигурационный эксперимент. Опишите, какая подходит браузерному BFF, какая — отдельному API, где возникает CSRF и почему наличие двух схем требует явного выбора default/challenge scheme.
