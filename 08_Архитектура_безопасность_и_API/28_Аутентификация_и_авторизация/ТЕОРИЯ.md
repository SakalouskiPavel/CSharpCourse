# Урок 28. Аутентификация и авторизация

> Безопасность — не кнопка «защитить приложение». Система должна установить, кто обращается к ней, проверить право на конкретное действие и сделать это так, чтобы секреты и личные данные не утекли.

## Результаты обучения

После урока вы сможете:

- отличать аутентификацию, авторизацию и идентификацию;
- объяснять роли credential, identity, claim и principal;
- выбирать между cookie и bearer token для понятного сценария;
- читать устройство JWT, не считая его содержимое секретным;
- различать OAuth 2.0 и OpenID Connect;
- реализовать policy- и resource-based authorization в ASP.NET Core;
- объяснять ответы `401`, `403` и защиту от CSRF;
- проектировать хранение паролей, токенов, ключей и audit-событий.

## 1. Модель угроз начинается раньше кода

Защищать «всё от всего» невозможно. Сначала называют активы: учётные записи, заказы, деньги, персональные данные, ключи. Затем — возможного нарушителя, точку входа и ущерб. Например: злоумышленник похищает cookie менеджера и отменяет чужой заказ.

Такой конкретный сценарий помогает выбрать меры: короткая сессия, MFA, повторная проверка для опасной операции, audit и отзыв сессии. HTTPS необходим, но он защищает канал, а не исправляет слабый пароль или ошибочное разрешение.

## 2. Четыре близких понятия

| Понятие | Вопрос | Пример |
|---|---|---|
| идентификатор | как объект называется? | `user-42` |
| credential | чем субъект доказывает личность? | пароль, ключ, код MFA |
| аутентификация | кто действует? | пароль проверен, создана identity |
| авторизация | разрешено ли действие? | владелец может читать свой заказ |

Имя пользователя — не доказательство личности. Роль `Admin` — не аутентификация. Вход в систему ещё не разрешает каждую операцию.

## 3. Identity, claim и principal

После успешной проверки приложение получает **identity** — представление установленной личности. **Claim** — утверждение, которому система доверяет: идентификатор пользователя, отдел, роль или permission. У утверждения есть issuer — тот, кто его выдал.

`ClaimsPrincipal` объединяет одну или несколько identities. В ASP.NET Core он доступен как `HttpContext.User`.

```csharp
var userId = httpContext.User.FindFirst("sub")?.Value;
var isAuthenticated = httpContext.User.Identity?.IsAuthenticated == true;
```

Claim — вход для решения, а не готовое разрешение. Даже claim `role=manager` не отвечает, может ли этот менеджер видеть заказ другого филиала.

## 4. Пароль нельзя хранить как текст

Серверу не нужно узнавать исходный пароль. При регистрации он сохраняет результат специально медленного **password hashing** с уникальной случайной солью. При входе повторяет проверку подходящим библиотечным API.

Обычный быстрый SHA-256 для паролей не подходит: атакующий способен перебирать варианты слишком быстро. Используют адаптивные алгоритмы и готовые реализации платформы, например `PasswordHasher<TUser>`. Параметры со временем усиливают, а hash обновляют после успешного входа. Пароль нельзя логировать, отправлять по почте или шифровать обратимо «для восстановления».

Ограничение частоты попыток и MFA уменьшают риск, но не заменяют правильное хранение.

## 5. Сессия и cookie

После входа неудобно передавать пароль с каждым запросом. Сервер выдаёт доказательство состоявшейся аутентификации. В браузерном приложении им часто служит cookie.

Варианты:

- cookie содержит защищённый authentication ticket;
- cookie содержит случайный session id, а состояние хранится на сервере.

Полезные свойства cookie:

- `Secure` — отправлять только через HTTPS;
- `HttpOnly` — JavaScript не может прочитать cookie;
- `SameSite` — ограничивает межсайтовую отправку;
- узкие `Domain` и `Path`;
- разумный срок жизни и механизм отзыва.

`HttpOnly` снижает ущерб некоторых XSS-атак, но не делает уязвимую страницу безопасной: вредный скрипт всё ещё может отправить запрос от лица пользователя.

## 6. CSRF, XSS и CORS — разные задачи

Браузер автоматически прикладывает cookie к подходящему адресу. Поэтому чужой сайт может попытаться инициировать запрос к вашему приложению — это **CSRF**. Защита включает anti-forgery token, корректный `SameSite`, проверку origin и отказ от изменяющих операций через `GET`.

**XSS** — выполнение чужого скрипта в доверенной странице. Нужны контекстное кодирование вывода, безопасная работа с HTML и Content Security Policy.

**CORS** говорит браузеру, разрешено ли JavaScript-коду одного origin читать ответ другого. Это не система входа и не защита API от `curl` или серверного клиента.

## 7. Bearer access token

API часто принимает заголовок:

```http
Authorization: Bearer eyJ...
```

Слово bearer означает: предъявитель считается владельцем. Похищенный токен можно использовать без знания пароля, поэтому access token делают короткоживущим, передают только по TLS, не пишут в логи и выдают с минимальными полномочиями.

Токен может быть ссылочным (opaque), тогда сервер обращается к хранилищу или introspection endpoint, либо самодостаточным, например JWT.

## 8. Что такое JWT

JWT обычно состоит из трёх Base64Url-частей: header, payload, signature. Подпись позволяет обнаружить изменение header/payload. Она **не шифрует** содержимое: payload легко прочитать.

При проверке важны:

- подпись и разрешённый алгоритм;
- доверенный issuer (`iss`);
- предназначенная аудитория (`aud`);
- срок действия (`exp`, при необходимости `nbf`);
- назначение токена и необходимые claims.

Нельзя просто декодировать payload и поверить ему. Нельзя принимать алгоритм только потому, что он написан в недоверенном header. Ключи требуется ротировать.

## 9. OAuth 2.0 и OpenID Connect

**OAuth 2.0** — протокол делегированного доступа: клиент получает ограниченное право обращаться к resource server. Он не является протоколом входа сам по себе.

**OpenID Connect (OIDC)** добавляет поверх OAuth слой аутентификации: ID token и стандартизированные сведения о пользователе. Упрощённо: OAuth отвечает «к чему клиенту дали доступ», OIDC — «кто вошёл».

Для интерактивных приложений типичен Authorization Code flow с PKCE:

1. клиент создаёт случайный verifier и производный challenge;
2. браузер переходит к authorization server;
3. пользователь проходит вход и подтверждение;
4. клиент получает короткий code;
5. code обменивается на токены вместе с verifier.

PKCE связывает перехваченный code с исходным клиентом. Пароль пользователя клиенту не передаётся. Устаревшие implicit и Resource Owner Password flows не следует выбирать для нового дизайна.

## 10. Access token, ID token и refresh token

| Артефакт | Для кого | Назначение |
|---|---|---|
| access token | API | разрешить доступ к ресурсу |
| ID token | клиент | сообщить результат входа |
| refresh token | authorization server | получить новый access token |

ID token нельзя отправлять API вместо access token. Refresh token живёт дольше и потому защищается строже: безопасное хранилище, rotation, обнаружение повторного использования, отзыв. Logout клиента не обязательно мгновенно лишает силы уже выданный stateless access token.

## 11. Authentication scheme в ASP.NET Core

Scheme связывает имя с handler: cookie handler читает cookie, JWT bearer handler проверяет токен. Handler может:

- `Authenticate` — попытаться создать principal;
- `Challenge` — потребовать вход;
- `Forbid` — сообщить, что установленной личности не хватает прав.

Middleware располагают до защищённых endpoints:

```csharp
builder.Services
    .AddAuthentication("Bearer")
    .AddJwtBearer("Bearer", options =>
    {
        options.Authority = configuration["Identity:Authority"];
        options.Audience = "orders-api";
    });

builder.Services.AddAuthorization();

var app = builder.Build();
app.UseAuthentication();
app.UseAuthorization();
```

Ключи и адрес authority берут из конфигурации/secret store, не вшивают в исходники.

## 12. `401` и `403`

- `401 Unauthorized` исторически назван неудачно: запрос не получил приемлемой аутентификации; клиенту предлагают challenge.
- `403 Forbidden`: личность установлена, но операция ей запрещена.

`404` иногда сознательно возвращают вместо `403`, чтобы не подтверждать существование скрытого ресурса. Такая политика должна быть последовательной.

## 13. От ролей к policies

Роль объединяет людей организационным ярлыком. Permission описывает возможность: `orders:read`, `orders:cancel`. Policy формулирует проверяемое правило.

```csharp
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("orders.read", policy =>
        policy.RequireAuthenticatedUser()
              .RequireClaim("permission", "orders:read"));
});

app.MapGet("/orders/{id:guid}", GetOrder)
   .RequireAuthorization("orders.read");
```

Запрет по умолчанию безопаснее списка случайных исключений. Fallback policy может потребовать аутентификацию для всех endpoints, а публичные точки явно отмечают `AllowAnonymous`.

## 14. Авторизация над ресурсом

Одного permission недостаточно для правила «пользователь читает только собственный заказ». Сначала загружают ресурс, затем передают его в authorization service.

```csharp
var result = await authorization.AuthorizeAsync(
    User, order, "CanReadOrder");

if (!result.Succeeded)
    return Results.Forbid();
```

Handler сравнивает `order.CustomerId` с надёжным subject id и учитывает особые полномочия сотрудника. Проверка находится на сервере рядом с use case; скрытая кнопка в интерфейсе — лишь удобство, не защита.

## 15. Границы доверия и least privilege

Данные извне недоверенные, включая claims от неизвестного issuer. Каждый компонент получает минимальные права и минимальный срок. API не должен использовать учётную запись администратора БД, если ему достаточно чтения пары таблиц.

Service-to-service доступ обычно представляет приложение, а не человека. Client Credentials подходит конфиденциальному сервису; разрешения такого токена нельзя автоматически смешивать с пользовательскими.

## 16. Секреты и криптографические ключи

Секрет — пароль БД, client secret, private key. Репозиторий, образ контейнера и лог — плохие хранилища. Для разработки применяют user secrets/переменные окружения, для среды — управляемый secret store и, где возможно, workload identity.

Нужны владелец, контроль доступа, срок, rotation и план отзыва. Удаление секрета из последнего коммита не удаляет его из истории: скомпрометированное значение всё равно меняют.

## 17. Audit без утечки данных

Обычный диагностический лог помогает отладке; audit отвечает, кто, когда и над чем совершил значимое действие. Полезны subject id, действие, resource id, результат, время, correlation id и источник — но не пароль, полный token или лишние персональные данные.

Audit защищают от незаметного изменения, ограничивают доступ и задают срок хранения. Сообщение «login failed» не должно раскрывать, существует ли такой пользователь.

## 18. Типичные ошибки

- собственный алгоритм паролей или выдачи токенов;
- JWT с долгим сроком и без проверки issuer/audience;
- использование ID token как access token;
- доверие роли из тела запроса;
- авторизация только на уровне UI;
- `AllowAnonymous`, оставленный после отладки;
- токены, пароли и персональные данные в логах;
- разрешительный CORS как попытка защитить API;
- проверка лишь счастливого сценария.

## Итог

Аутентификация создаёт доверенную identity, авторизация принимает решение для конкретного действия и ресурса. Cookie и token — переносчики результата, а не волшебные щиты. Надёжная система проверяет всё на сервере, выдаёт минимум полномочий, умеет отзывать доступ и не превращает диагностику в новую утечку.
