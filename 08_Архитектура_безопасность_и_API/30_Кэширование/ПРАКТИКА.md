# Практика — урок 30

## Цель

Добавить измеряемый и безопасный cache к чтению заказов из урока 29, не меняя источник истины и HTTP-контракт.

## Перед кодом

Создайте `CACHE_DESIGN.md` и зафиксируйте:

- какая операция медленная и как это измерено;
- source of truth;
- допустимый возраст данных;
- формат ключа и максимальную cardinality;
- local или distributed cache и почему;
- positive/negative TTL;
- момент invalidation;
- поведение при stampede и отказе cache;
- риск tenant/user leakage;
- метрики успеха.

## Обязательная реализация

1. Оберните query чтения заказа декоратором cache-aside.
2. Ключ включает environment, tenant id, schema version и order id.
3. Храните immutable `OrderDto`, не domain entity.
4. Задайте absolute TTL с небольшим jitter. Если применяете sliding, добавьте абсолютный предел.
5. После успешного commit изменения/отмены инвалидируйте запись. Ошибка транзакции не должна удалять корректную копию преждевременно.
6. Реализуйте single-flight на ключ, не глобальную блокировку.
7. Если кэшируете not-found, используйте отдельный короткий TTL и объясните последствия.
8. Cache timeout/error переводит чтение к источнику с ограничением concurrency и метрикой fallback.
9. Добавьте счётчики hit, miss, source load, cache error, invalidation и loader wait.
10. Не логируйте cache value, token и персональные данные.

## HTTP-уровень

Для публичного учебного endpoint каталога настройте output caching с ограниченным сроком и явным vary по pagination-параметрам. Для защищённого заказа оставьте безопасную политику (`private`/`no-store` либо обоснованный персональный ключ). Сохраните ETag revalidation из урока 29.

## Тесты корректности

Проверьте:

- первый запрос даёт miss, второй — hit;
- разные tenants с одинаковым order id не смешиваются;
- изменение заказа после commit инвалидирует запись;
- неуспешное изменение не портит cache;
- concurrent misses вызывают один source load на ключ;
- разные ключи загружаются параллельно;
- cache exception приводит к контролируемому source fallback;
- negative entry истекает раньше positive;
- изменение схемы ключа не читает старый формат;
- приватный ответ не попадает в общий output cache.

## Нагрузочный эксперимент

Запустите одинаковый read workload:

1. без cache;
2. с cache на прогретых данных;
3. после одновременного expiration;
4. при недоступном cache.

Запишите p50/p95 latency, число source calls, hit ratio, errors и loader concurrency. Сделайте вывод: стало ли лучше и какой новый риск появился.

## Критерии готовности

- решение собирается, тесты воспроизводимы;
- cache остаётся копией, а не скрытым источником истины;
- ключ изолирует tenants;
- expiration/invalidation обоснованы требованиями;
- stampede и cache failure имеют определённое поведение;
- метрики позволяют доказать эффект;
- `CACHE_DESIGN.md` честно называет окно stale data.

## Дополнительное задание

Замените `IMemoryCache` на реализацию `IDistributedCache`. Версионируйте сериализованный DTO, измерьте дополнительную latency и опишите, какие тесты перестали быть только внутрипроцессными.
