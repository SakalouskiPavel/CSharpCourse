# Урок 30. Кэширование без волшебства

> Кэш хранит копию данных ближе к потребителю, чтобы следующий запрос был быстрее. Цена ускорения — память, устаревание данных и новые сценарии отказа.

## Результаты обучения

После урока вы сможете:

- объяснять hit, miss, eviction, TTL и freshness;
- отличать источник истины от его кэшированной копии;
- реализовать cache-aside для `IMemoryCache` и distributed cache;
- проектировать ключи с учётом tenant, пользователя и версии;
- выбирать expiration и invalidation по требованиям бизнеса;
- защищать источник от stampede и опасного negative caching;
- применять ETag, `Cache-Control`, output caching и `Vary`;
- измерять пользу кэша и безопасно переживать его отказ.

## 1. Мысленная модель

Представьте библиотечную книгу и заметку с её кратким содержанием. Заметку читать быстрее, но после нового издания она может устареть. База данных или внешний сервис обычно остаётся **source of truth**, а cache — восстанавливаемой копией.

Основные события:

- **hit** — запись найдена;
- **miss** — записи нет, нужно читать источник;
- **set/populate** — результат помещён в cache;
- **eviction** — запись удалена из-за срока, размера или давления памяти;
- **invalidation** — приложение объявляет копию недействительной.

Cache не гарантирует наличие значения. Код обязан корректно работать после miss и eviction.

## 2. Когда кэш действительно полезен

Кандидат обычно часто читается, редко меняется, дорого вычисляется/загружается и допускает некоторую задержку обновления. Кэшировать дешёвую операцию с уникальным ключом бесполезно: hit почти не будет, а сложность появится.

До реализации измеряют latency, частоту, стоимость источника и допустимую stale-данность. После — hit ratio, latency hit/miss, нагрузку источника и расход памяти. Высокий hit ratio не доказывает пользу, если выигрыш микроскопический.

## 3. Cache-aside

Наиболее понятная схема:

1. построить ключ;
2. попытаться прочитать cache;
3. при miss прочитать source of truth;
4. положить копию с политикой expiration;
5. вернуть результат.

```csharp
public async Task<OrderDto?> FindAsync(Guid id, CancellationToken ct)
{
    var key = $"orders:v1:{id:N}";
    if (_cache.TryGetValue(key, out OrderDto? cached))
        return cached;

    var value = await _repository.FindAsync(id, ct);
    if (value is not null)
        _cache.Set(key, value, TimeSpan.FromMinutes(2));

    return value;
}
```

Между чтением источника и `Set` данные могут измениться. Несколько клиентов могут одновременно загрузить одно значение. Это часть модели, а не дефект конкретной библиотеки.

## 4. Другие стратегии

- **read-through**: cache сам загружает значение через настроенный loader;
- **write-through**: запись проходит через cache и синхронно попадает в источник;
- **write-behind/write-back**: cache подтверждает запись, а источник обновляется позже;
- **refresh-ahead**: популярное значение обновляется до истечения срока.

Write-behind уменьшает задержку записи, но повышает риск потери, переупорядочивания и сложность восстановления. Названия не создают гарантий сами по себе — их определяет конкретная система.

## 5. In-process cache

`IMemoryCache` живёт в памяти одного процесса. Он очень быстр и не требует сети, но:

- исчезает при перезапуске;
- каждый экземпляр приложения имеет свою копию;
- запись, удалённая на node A, может остаться на node B;
- неограниченные записи способны вытеснить полезную память.

Он подходит для локальных вычислений, справочников и данных, где различие экземпляров допустимо. Следует задавать size limit/size там, где cardinality не ограничена.

## 6. Distributed cache

Распределённый cache — отдельное сетевое хранилище, общее для экземпляров приложения. Это улучшает согласованность копий и переживает рестарт app node, но добавляет сеть, сериализацию, задержку, стоимость и собственные отказы.

`IDistributedCache` работает с bytes/string, поэтому формат и версия сериализации становятся частью дизайна. Большие объекты увеличивают трафик. Distributed cache быстрее БД не автоматически — это проверяют измерением.

## 7. Проектирование ключа

Ключ должен включать все входы, меняющие результат:

```text
orders:{environment}:{tenantId}:v3:{orderId}:{culture}
```

Если результат зависит от tenant или permission, а ключ их не учитывает, возможна утечка чужих данных. Если включить случайные query-параметры без нормализации, возникает неограниченная cardinality.

У ключей полезны namespace, версия схемы и стабильный формат. Нельзя помещать в ключ пароль, token или лишние персональные данные: ключи видны в диагностике.

## 8. Absolute и sliding expiration

**Absolute expiration** удаляет запись не позднее заданного момента. **Sliding expiration** продлевает жизнь после использования. Только sliding способен держать популярную запись бесконечно; часто его сочетают с абсолютным пределом.

TTL отвечает на бизнес-вопрос: насколько старый ответ приемлем? Остаток товара может терпеть секунды, исторический справочник — часы. «Поставим пять минут» без требования — не архитектурное решение.

К небольшому случайному **jitter** добавляют разброс, чтобы тысячи записей не истекли одновременно.

## 9. Invalidation после записи

Распространённый cache-aside write:

1. изменить source of truth;
2. после успешного commit удалить cache key.

Удаление до commit опасно: другой запрос может прочитать старую БД и снова заполнить cache. Но и удаление после commit может не выполниться из-за падения процесса. Возможные меры: короткий TTL, retry/outbox-событие invalidation, versioned keys или смена версии данных.

Идеальной атомарности между БД и отдельным cache обычно нет. Нужно явно назвать окно несогласованности и допустимое поведение.

## 10. Versioned key и event invalidation

Versioned key включает номер версии ресурса. Новая версия пишет/читает новый ключ, а старая запись исчезает по TTL. Это уменьшает риск гонки, но вызывающий код должен знать актуальную версию.

Событие `OrderChanged` может удалить запись во всех потребителях. Доставка события бывает задержана или повторена, поэтому handler делают идемпотентным, а TTL оставляют страховкой.

## 11. Cache stampede

После истечения популярной записи тысяча запросов одновременно получает miss и идёт в БД. Это **stampede** (thundering herd).

Способы смягчения:

- single-flight/keyed lock — один loader на конкретный ключ;
- jitter для expiration;
- refresh-ahead;
- stale-while-revalidate — временно отдать старую копию, пока одна задача обновляет;
- ограничение concurrency и backpressure.

Один глобальный lock сериализует все ключи и создаёт bottleneck. Lock должен освобождаться при исключении, а ожидание — поддерживать cancellation/timeout.

## 12. Negative caching

Можно кратко сохранить «не найдено», чтобы повторные запросы не били источник. Но если объект создадут через секунду, cache продолжит отвечать `404`. Поэтому negative TTL обычно короче положительного.

Нельзя одинаково кэшировать not-found, timeout и `403`. Timeout временный, а caching authorization response без identity может утечь между пользователями. Сначала классифицируют результат.

## 13. Ошибка cache и graceful degradation

Cache обычно оптимизация, поэтому при его недоступности чтение может пойти в источник. Но если все nodes одновременно переключатся на БД, получится лавина. Нужны timeout, circuit breaker/ограничение параллелизма, метрика fallback и capacity источника.

Для данных, где cache выполняет роль обязательной session/idempotency инфраструктуры, «просто пропустить» его нельзя: семантика отказа должна быть другой. Важно определить, является ли компонент оптимизацией или частью корректности.

## 14. Serialization и мутабельность

В memory cache ссылка указывает на тот же изменяемый объект. Один потребитель может незаметно изменить значение для остальных. Безопаснее хранить immutable DTO или создавать копию.

В distributed cache объект сериализуется. При изменении DTO старые записи могут перестать читаться; помогают schema version в ключе/данных и tolerant deserialization. Не используйте небезопасную полиморфную десериализацию недоверенных данных.

## 15. HTTP caching

HTTP имеет собственный cache contract для browser, proxy и CDN:

- `Cache-Control: max-age=60` — срок свежести;
- `public`/`private` — может ли shared cache хранить ответ;
- `no-store` — не сохранять;
- `ETag` и `If-None-Match` — проверка версии;
- `Last-Modified` и `If-Modified-Since` — проверка времени;
- `Vary` — какие request headers меняют представление.

`304 Not Modified` экономит тело, но всё равно обращается к серверу для revalidation. Персональный ответ нельзя случайно объявить `public`. `Vary: Authorization` не всегда достаточная и эффективная защита — чувствительные ответы часто делают `private`/`no-store`.

## 16. Output caching в ASP.NET Core

Output cache сохраняет готовый HTTP-ответ по policy. Это не то же самое, что `IMemoryCache`: первый работает на границе HTTP, второй хранит выбранные приложением объекты.

```csharp
builder.Services.AddOutputCache(options =>
{
    options.AddPolicy("catalog", policy =>
        policy.Expire(TimeSpan.FromSeconds(30))
              .SetVaryByQuery("page", "pageSize"));
});

app.UseOutputCache();
app.MapGet("/catalog", GetCatalog).CacheOutput("catalog");
```

Перед включением определяют методы, статусы, query/headers, identity и invalidation. Ответы, зависящие от пользователя, нельзя смешивать одним ключом.

## 17. Кэширование и безопасность

Основные риски:

- cross-tenant/user leakage из-за неполного ключа;
- сохранение token, пароля или лишних personal data;
- shared cache для приватного HTTP-ответа;
- устаревшее authorization decision после отзыва права;
- cache poisoning недоверенным ключом/ответом;
- DoS через безграничное множество ключей или огромные значения.

Cache получает минимальные сетевые права, шифрование канала и ограниченный доступ. Чувствительные данные либо не кэшируют, либо защищают на уровне, соответствующем их риску.

## 18. Наблюдаемость

Минимальные метрики:

- hits, misses и hit ratio по операции;
- latency hit, miss и source load;
- количество/причины eviction;
- размер и cardinality;
- возраст/freshness выдаваемых данных;
- ошибки cache и fallback;
- число concurrent loaders/stampede protection waits.

Логи не должны содержать значение cache целиком. Метрики сравнивают с baseline без кэша: иначе невозможно доказать улучшение.

## 19. Как принимать решение

1. Измерить исходную проблему.
2. Назвать source of truth.
3. Определить допустимый возраст и ущерб stale data.
4. Спроектировать key и максимальную cardinality.
5. Выбрать local/distributed/HTTP уровень.
6. Определить populate, expiration и invalidation.
7. Разобрать гонки, stampede и отказ cache.
8. Добавить security boundaries и наблюдаемость.
9. Нагрузочно проверить и сравнить с baseline.

## 20. Типичные ошибки

- кэшировать до измерения;
- считать cache источником истины без соответствующего дизайна;
- забывать tenant, culture или permission в ключе;
- использовать sliding expiration без верхней границы;
- удалять cache до commit БД;
- ставить глобальный lock;
- кэшировать все ошибки одинаково;
- считать distributed cache бесплатной памятью;
- объявлять приватный ответ `public`;
- измерять только hit ratio.

## Итог

Кэш — управляемая копия с ограниченной свежестью. Хороший дизайн заранее отвечает: что является истиной, почему копия выгодна, кто её видит, когда она устаревает, как удаляется и что произойдёт при отказе. Если эти ответы не записаны, ускорение легко превращается в трудноуловимую ошибку данных.
