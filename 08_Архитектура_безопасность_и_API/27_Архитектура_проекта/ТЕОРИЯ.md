# Урок 27. Архитектура проекта

> Архитектура — набор решений о границах, зависимостях, данных и изменениях. Хорошая структура делает важные правила видимыми, а стоимость типового изменения — локальной и проверяемой.

## Результаты обучения

- проектировать границу и её контракт;
- называть эксплуатационные и security-компромиссы;
- проверять негативные сценарии и совместимость;
- документировать решение и его ограничения.

## Карта темы

### 1. Архитектурные характеристики

Надёжность, безопасность, latency, throughput, сопровождаемость и стоимость конфликтуют. Требование должно иметь сценарий и измеримый порог, иначе им нельзя руководить решением.

### 2. Модули и cohesion

Модуль объединяет данные и поведение вокруг одной предметной способности. Высокая cohesion и явный публичный API уменьшают случайные зависимости.

### 3. Coupling

Зависимость существует не только через ссылку проекта, но и через общую БД, формат сообщения, порядок вызовов и deployment. Stable dependencies направляют более изменчивый код к устойчивым контрактам.

### 4. Layered architecture

Presentation, application, domain и infrastructure разделяют причины изменения. Слой — логическая ответственность, а не обязательный отдельный проект для каждого типа.

### 5. Ports and adapters

Ядро задаёт ports для нужных внешних возможностей, adapters реализуют HTTP, БД или CLI. Dependencies направлены внутрь, composition root связывает реализации.

### 6. Vertical slices

Группируют код по use case, уменьшая прыжки между техническими папками. Общие abstractions извлекаются после появления подтверждённого повторения.

### 7. DDD boundaries

Bounded context задаёт собственный язык и модель. Entity имеет identity, value object — значение, aggregate защищает транзакционный инвариант и является границей записи.

### 8. CQRS

Разделение моделей команд и чтения полезно при существенно разных требованиях, но не требует двух БД. Оно добавляет модели и eventual consistency, поэтому не является стартовым шаблоном.

### 9. Dependency injection

Constructor injection делает обязательные зависимости явными; lifetime singleton/scoped/transient должен согласоваться с потокобезопасностью и владением.

### 10. Architecture tests

Project references, forbidden namespaces и conventions можно проверять автоматически. Decision records сохраняют контекст и последствия необратимых решений.

## Минимальный пример

```csharp
public sealed class PlaceOrderHandler(IOrderRepository orders, IClock clock)
{
    public async Task<OrderId> Handle(PlaceOrder command, CancellationToken ct)
    {
        var order = Order.Place(command.CustomerId, command.Lines, clock.UtcNow);
        await orders.AddAsync(order, ct);
        return order.Id;
    }
}
```

Минимальный пример показывает форму API, но production-границе также нужны валидация, наблюдаемость, timeout/cancellation, защита данных и согласованная модель ошибок.

## Типичные ошибки

- выбирать технологию до формулировки качественных требований;
- скрывать важные зависимости и состояния за «универсальной» абстракцией;
- проверять только happy path;
- считать локальную успешность доказательством эксплуатационной надёжности;
- хранить секреты или персональные данные в коде и логах.

## Итог

Архитектурное решение оценивается через сценарии изменения и отказа. Фиксируйте контекст, последствия и критерий пересмотра, чтобы решение можно было проверить, а не защищать авторитетом.
