# Урок 27. Архитектура проекта: границы, зависимости и решения

> Архитектура — не количество проектов и не картинка из окружностей. Это набор важных решений о границах системы, данных, зависимостях, качестве и способе изменений. Хорошая архитектура делает типовое изменение локальным и отказ понятным.

## 1. Система, приложение и модуль

**Система** включает код, базы, внешние сервисы, пользователей, deployment и процессы поддержки. **Приложение** — одна запускаемая часть. **Модуль** — логическая граница связанной предметной способности внутри приложения.

Например, интернет-магазин:

```text
Orders | Catalog | Payments | Delivery
```

Это не обязательно четыре микросервиса. Они могут быть модулями одного deployable приложения — modular monolith.

Архитектура начинается с контекста: кто использует систему, какие соседние системы существуют и где проходят trust/ownership boundaries.

## 2. Функциональные и качественные требования

Функциональное: «пользователь оформляет заказ».

Качественное/architectural characteristic:

- 95% запросов быстрее 300 ms;
- заказ не теряется после подтверждения;
- персональные данные доступны только владельцу;
- новая payment implementation подключается за один модуль;
- сервис восстанавливается за 30 минут;
- 1000 заказов/минуту.

Слова «быстро», «масштабируемо», «надёжно» без сценария и порога не помогают выбрать решение. Характеристики конфликтуют: строгая consistency может увеличить latency, изоляция процессов — operational cost.

## 3. Constraints

Ограничения не выбираются архитектурой:

- обязательная платформа/БД;
- законодательство и хранение данных;
- budget/team skill;
- legacy integration;
- deadline;
- offline environment.

Записывайте constraint отдельно от preference. «Команда знает PostgreSQL» и «регулятор требует хранение в стране» имеют разную жёсткость.

## 4. Cohesion и coupling

**Cohesion** — насколько элементы модуля служат одной связанной ответственности. **Coupling** — насколько изменение одного вынуждает менять/координировать другое.

Coupling бывает:

- compile-time project reference;
- runtime synchronous call;
- общая database/schema;
- message contract;
- deployment order;
- shared library;
- организационная зависимость команды.

Отсутствие reference в C# не означает слабую связь, если два сервиса обязаны одновременно менять одну таблицу.

Цель не «нулевая связанность»: компоненты должны сотрудничать. Нужна явная, стабильная связь по маленькому contract.

## 5. Причины изменения

Разделяйте код по причинам изменения:

- предметные правила заказа;
- orchestration use case;
- HTTP presentation;
- SQL persistence;
- внешняя payment integration.

Если смена JSON naming ломает расчёт скидки, границы смешаны. Если каждая простая feature требует править десять «слоёв», разделение стало церемониальным.

Архитектура оценивается change scenario: «добавить новый способ скидки», «заменить provider», «добавить поле response».

## 6. Layered architecture

Классические логические слои:

```text
Presentation
    ↓
Application
    ↓
Domain
    ↓
Infrastructure/Data (в традиционном варианте)
```

- Presentation переводит внешний input/output;
- Application координирует use case/transaction;
- Domain содержит бизнес-правила и model;
- Infrastructure работает с DB, network, filesystem.

Layer — ответственность, не обязательно отдельный `.csproj`. Маленькое приложение может иметь папки/namespaces; project boundary нужен для compile-time enforcement/reuse/deployment reason.

Проблема строгого horizontal layering: feature размазана по папкам Controllers/Services/Repositories, изменения требуют прыжков, layers превращаются в pass-through.

## 7. Dependency direction

Domain policy не должна зависеть от детали SQL/HTTP:

```text
Infrastructure ──implements──> Domain/Application port
                   dependency points inward
```

High-level code объявляет нужный interface:

```csharp
public interface IOrderRepository
{
    Task<Order?> FindAsync(OrderId id, CancellationToken ct);
    Task AddAsync(Order order, CancellationToken ct);
}
```

Infrastructure реализует EF repository. Composition root связывает.

Но interface не нужно создавать для каждого class. Port появляется у значимой внешней capability клиента. Domain не обязан зависеть от абстракции, которую придумала infrastructure для себя.

## 8. Hexagonal / Ports and Adapters

Application core имеет **ports**:

- inbound: use cases, которые вызывает внешний мир;
- outbound: capabilities, нужные core.

Adapters:

- inbound: HTTP controller, CLI, message consumer;
- outbound: EF Core, SMTP, payment client.

```text
HTTP adapter → PlaceOrder use case → Payment port ← Provider adapter
```

Преимущество — core тестируется без infrastructure. Цена — mapping/contracts/composition. Для CRUD без предметных правил полная церемония может быть избыточной.

Clean/Onion architecture используют сходный принцип направления dependencies, различаясь терминологией/слоями. Не спорьте о названии, проверяйте references и ownership.

## 9. Vertical Slice Architecture

Код группируется по use case:

```text
Features/
  PlaceOrder/
    Endpoint.cs
    Command.cs
    Handler.cs
    Validator.cs
  GetOrder/
    Endpoint.cs
    Query.cs
```

Slice содержит путь от входа до результата, уменьшая coupling между features и навигацию по техническим папкам. Общий domain/infrastructure всё равно существует там, где действительно shared.

Vertical slices не означают дублировать всё. Сначала локальность, затем извлечение устойчивого повторения. Не создавайте universal base handler раньше второго реального случая.

Layers и slices можно сочетать: module/slice с внутренними domain/application/infrastructure boundaries.

## 10. Domain-Driven Design: стратегический уровень

**Domain** — область деятельности и правил. **Ubiquitous Language** — согласованный язык экспертов и кода.

**Bounded Context** — граница, внутри которой термины имеют одну модель. `Customer` в Sales и Support может означать разные данные/правила. Не создавайте один enterprise Customer с сотней полей.

Contexts интегрируются явными contracts/translation; Anti-Corruption Layer защищает внутреннюю модель от чужой.

DDD не равно entities/repositories. Стратегическая ценность — правильные границы и язык. Для простого CRUD tactical patterns могут не окупиться.

## 11. Entity, Value Object, Aggregate

- **Entity** имеет identity и lifecycle: Order с OrderId.
- **Value Object** определяется значениями, обычно immutable: Money+Currency, Address.
- **Aggregate** — cluster объектов с root, защищающий consistency invariant в одной transaction boundary.

Внешний код меняет aggregate через root. Не загружайте/сохраняйте огромный graph только потому, что объекты связаны в реальности.

Aggregate boundary выбирают по invariants, а не по navigation convenience. Между aggregates обычно eventual consistency/reference by id.

## 12. Application service и Domain service

Application service:

1. получает command;
2. загружает aggregate;
3. вызывает domain behavior;
4. сохраняет/commit;
5. публикует результат/events согласно policy.

Domain service содержит предметную операцию, не принадлежащую естественно одной entity/value object. Он не должен становиться свалкой логики с DB/email.

```csharp
public sealed class PlaceOrderHandler(
    IOrderRepository orders,
    IUnitOfWork unitOfWork)
{
    public async Task<OrderId> Handle(PlaceOrder command, CancellationToken ct)
    {
        Order order = Order.Place(command.CustomerId, command.Lines);
        await orders.AddAsync(order, ct);
        await unitOfWork.CommitAsync(ct);
        return order.Id;
    }
}
```

## 13. CQRS

Command Query Responsibility Segregation разделяет модели изменения и чтения.

- Command меняет state и возвращает outcome/id;
- Query не меняет observable state и возвращает read model.

CQRS не требует двух databases, broker или event sourcing. Начальный вариант — отдельные handlers/models в одном process/DB.

Усложнение (отдельный read store) оправдано, если read/write requirements сильно различаются. Цена: eventual consistency, synchronization, больше моделей/operations.

## 14. Event-driven и Outbox

Domain event фиксирует факт внутри domain. Integration event — versioned message другим boundaries.

Нельзя безопасно сделать:

```text
commit DB → publish broker
```

без анализа сбоя между шагами. Transactional Outbox сохраняет business change и outgoing message в одной local transaction, relay публикует позже. Подробно — урок 31.

Не публикуйте external event до commit: subscribers увидят факт, который может откатиться.

## 15. Modular monolith и microservices

Modular monolith:

- один deployment/process;
- модули с API/ownership;
- часто одна DB server, но schema ownership разделён;
- прямые in-process calls/messages;
- проще transactions/operations.

Microservices дают independent deployment/scaling/team ownership ценой network, partial failures, data duplication, observability и DevOps.

Начинать с modular monolith часто разумнее. Если boundaries нечётки внутри процесса, сеть их не исправит.

## 16. Dependency Injection и lifetimes

- Transient — новый instance при resolve;
- Scoped — один на scope (например request/unit of work);
- Singleton — один container lifetime.

Captive dependency: singleton удерживает scoped/transient state, нарушая lifetime/thread safety. DbContext обычно scoped/short-lived, не singleton.

Composition root — место registrations/graph. Business code не получает IServiceProvider для произвольного Resolve.

DI container проверяет construction, но не архитектурную корректность interface boundaries.

## 17. Transactions и boundaries

Одна application operation определяет transaction boundary. Не распределяйте transaction случайно по repository methods, каждый из которых SaveChanges самостоятельно: use case теряет atomicity.

Внешний HTTP/broker call не входит в local SQL transaction. Длинные distributed workflows требуют idempotency, outbox/saga/compensation, а не lock базы на минуты.

## 18. Errors at boundaries

Domain errors — ожидаемые outcomes (insufficient stock, invalid transition). Infrastructure exceptions преобразуются application boundary в устойчивые categories. HTTP adapter выбирает status/problem details.

Не позволяйте SQL exception просачиваться как public API и не превращайте любой defect в `Result.Failure`. Programming errors должны оставаться заметны.

## 19. Architecture fitness functions

Часть правил автоматизируется:

- Domain project не ссылается на Infrastructure;
- module internals не доступны соседям;
- naming/namespace conventions;
- handlers не используют forbidden types;
- public contracts не содержат persistence entities;
- dependency graph не имеет циклов.

Architecture tests не доказывают хорошую модель, но не дают случайно разрушить принятое направление.

## 20. ADR

Architecture Decision Record:

- контекст/проблема;
- decision drivers;
- рассмотренные варианты;
- решение;
- последствия/риски;
- status/date/owner;
- критерий пересмотра.

ADR не оправдывает решение задним числом. Он сохраняет причины, которые Git diff кода не показывает. Короткий живой ADR полезнее огромного документа, который никто не обновляет.

## 21. Диаграммы

C4 levels:

- Context — система, пользователи, внешние системы;
- Container — приложения/DB/broker, не Docker container обязательно;
- Component — крупные части внутри container;
- Code — детали, редко поддерживаются вручную.

Диаграмма должна отвечать вопросу и иметь legend/границы. Стрелка подписывается протоколом/намерением. Картинка без data ownership и failure mode создаёт ложную уверенность.

## 22. Observability и operability

Архитектура включает:

- structured logs;
- metrics/SLO;
- distributed traces/correlation;
- health/readiness;
- configuration/secrets;
- migrations/backups;
- deployment/rollback;
- ownership/on-call.

Если систему нельзя безопасно обновить и диагностировать, кодовые layers не спасают architecture.

## Частые ошибки

- architecture = число projects;
- quality attribute не имеет сценария/порога;
- interface создаётся для каждого class;
- layers только передают вызов;
- shared database считается отсутствием coupling;
- DDD сводится к repository/entity;
- aggregate становится всем object graph;
- CQRS автоматически означает две DB;
- microservices выбираются до boundaries/operations;
- singleton захватывает scoped dependency;
- service locator прячет graph;
- diagrams/ADR не отражают реальность.

## Итог

```text
Контекст + quality attributes + constraints
  ↓
Boundaries и ownership
  ↓
Dependency direction и contracts
  ↓
Transactions, failures, operations
  ↓
Fitness tests + ADR + observability
```
