# Практика — урок 27

## Цель

Спроектировать и реализовать один сквозной use case сервиса заказов с явными boundaries, inward dependencies, transaction ownership и архитектурными проверками.

## До кода

Создайте:

- C4 Context и Container diagrams;
- список actors/external systems;
- минимум пять quality scenarios с числовыми/проверяемыми условиями;
- constraints/preferences;
- bounded contexts Orders, Catalog, Payments (границы можно скорректировать с обоснованием);
- data ownership table.

## Use case `PlaceOrder`

Command содержит customer id, lines и idempotency key. Application handler:

1. validates shape;
2. получает catalog snapshot/port;
3. создаёт Order aggregate через domain behavior;
4. сохраняет через outbound port;
5. commit выполняет один Unit of Work;
6. возвращает OrderId/ожидаемую domain error.

HTTP, EF types и configuration не попадают в domain.

## Структура

Выберите layered или vertical-slice layout. Для каждого project/folder напишите responsibility и allowed dependencies. Не создавайте project без enforcement/value.

Composition root связывает adapters. Business code не использует IServiceProvider.

## Архитектурные тесты

Автоматически проверьте минимум:

- Domain не ссылается на Infrastructure/Web;
- persistence entities/DbContext не в public application contracts;
- module internals не используются напрямую соседями;
- composition root — единственное место registrations;
- project dependency graph не имеет запрещённого цикла.

## ADR

Напишите два ADR:

1. modular monolith vs microservices;
2. vertical slices vs horizontal layers.

Включите alternatives, consequences и revisit trigger.

## Failure walkthrough

Письменно пройдите:

- catalog недоступен;
- DB commit упал;
- duplicate idempotency key;
- application process упал после commit до response;
- integration event нужно отправить после commit.

Для последнего опишите Outbox boundary, не обязательно реализуя broker.

## Критерии приёмки

- diagrams соответствуют коду;
- quality attributes измеримы;
- dependency direction enforced;
- domain invariants не находятся в endpoint/DB adapter;
- transaction принадлежит use case;
- failures имеют outcome;
- ADR объясняют цену решений;
- solution собирается, use case проверен integration/domain tests.
