# Урок 29. WCF, ASMX, Web API и REST

> Сервис — программа, к которой обращается другая программа через явно описанную границу. ASMX, WCF и ASP.NET Core Web API решают эту задачу в разные эпохи и с разными моделями.

## Результаты обучения

Вы научитесь:

- различать RPC, SOAP и REST;
- читать назначение WSDL, XSD, endpoint, binding и contract;
- понимать место ASMX, WCF и CoreWCF в современном .NET;
- проектировать ресурсы, URI, методы, статусы и ошибки HTTP API;
- применять идемпотентность, conditional requests и пагинацию;
- описывать API через OpenAPI и планировать совместимые изменения;
- выбирать стратегию миграции legacy SOAP-сервиса.

## 1. Зачем приложениям контракт

Процессы не могут вызвать метод друг друга как обычный метод в памяти. Запрос нужно сериализовать, передать по сети, распознать на другой стороне и вернуть ответ. Между сторонами возникают задержка, недоступность, дублирование, несовместимые версии и частичный отказ.

**Контракт** определяет доступные операции или ресурсы, формат сообщений, ошибки и правила совместимости. Сетевой вызов внешне может напоминать локальный, но его нельзя считать таким же дешёвым и надёжным.

## 2. RPC, message и resource style

- RPC описывает действия: `CancelOrder(orderId)`.
- Message style делает главным документ/событие: `OrderCancellationRequested`.
- Resource style моделирует состояние доступных сущностей: `DELETE /orders/{id}` или `POST /orders/{id}/cancellation`.

Это способы мышления, не только форматы. JSON не превращает RPC endpoint `/doEverything` в REST, а XML не обязывает систему быть SOAP.

## 3. XML, XSD, SOAP и WSDL

XML — текстовый формат с элементами и пространствами имён. **XSD** формально описывает допустимую структуру XML.

SOAP задаёт envelope сообщения: `Header` для метаданных, `Body` для полезной нагрузки и `Fault` для ошибки. SOAP может работать поверх HTTP и других transports.

**WSDL** описывает операции, сообщения, типы, binding и адрес сервиса. По нему инструменты генерируют клиентский proxy. Формальность удобна для корпоративной интеграции, но изменение схемы и совместимость требуют дисциплины.

## 4. ASMX

ASP.NET XML Web Services (ASMX) — технология .NET Framework, где класс и методы отмечались атрибутами вроде `[WebService]` и `[WebMethod]`, а платформа публиковала SOAP/WSDL.

ASMX встречается в legacy-системах. Для нового сервиса на современном .NET его обычно не выбирают: модель расширения и протокольные возможности ограничены, технология привязана к старой платформе. Поддерживать ASMX иногда разумнее, чем немедленно переписывать стабильную интеграцию.

## 5. WCF: ABC

Windows Communication Foundation объединил разные способы service communication. Удобная мнемоника:

- **Address** — где находится endpoint;
- **Binding** — как передавать: transport, encoding, security и другие параметры;
- **Contract** — что сервис умеет.

Service contract задаёт операции, data contract — передаваемые данные, message contract — точную форму сообщения. Behaviors настраивают runtime-функции. Один service contract мог публиковаться через несколько endpoints с разными bindings.

## 6. WCF сегодня и CoreWCF

Полноценный серверный WCF относится к .NET Framework. В современном .NET доступны WCF client libraries для обращения к существующим сервисам. Для переноса серверных WCF-сценариев существует CoreWCF — отдельный open-source проект, совместимый с частью модели WCF.

Это не означает автоматическую совместимость каждого binding и behavior. Перед миграцией составляют инвентаризацию contracts, bindings, WS-* функций, hosting и клиентов, затем делают proof of concept.

## 7. Когда SOAP всё ещё уместен

SOAP может быть оправдан, если уже существует обязательный WSDL-контракт, партнёры используют code generation, нужны конкретные WS-* стандарты или цена изменения всех клиентов слишком высока. «Старое» не равно «срочно удалить».

Для публичного HTTP API, browser/mobile clients и простых CRUD-сценариев ASP.NET Core с JSON обычно проще. Выбор делают по требованиям и экосистеме потребителей.

## 8. ASP.NET Core request pipeline

Входящий HTTP-запрос проходит middleware: обработку ошибок, HTTPS, routing, authentication, authorization и endpoint. Endpoint может быть minimal API или controller. Model binding превращает route/query/header/body в .NET-значения; validation проверяет форму данных; use case выполняет бизнес-операцию.

```csharp
app.MapGet("/orders/{id:guid}", async (
    Guid id, IOrderQueries queries, CancellationToken ct) =>
{
    var order = await queries.FindAsync(id, ct);
    return order is null ? Results.NotFound() : Results.Ok(order);
});
```

`CancellationToken` сообщает об отмене запроса, но бизнес-операция должна отдельно определить, в какой момент отменяться уже поздно.

## 9. REST как архитектурный стиль

REST описывает ограничения: client-server, stateless, cacheable, uniform interface, layered system и необязательный code-on-demand. На практике важнее всего единообразно использовать ресурсы и HTTP.

**Stateless** означает, что каждый запрос содержит необходимый контекст; это не запрет БД или состояния ресурса. Сервер не обязан помнить скрытый диалог «какой шаг клиент выполнял до этого».

## 10. Ресурс и URI

Ресурс — значимое для клиента понятие: заказ, клиент, коллекция заказов. URI обычно строят из существительных:

```text
/orders
/orders/7f...
/customers/42/orders
```

Не стоит механически отражать таблицы БД. API — контракт продукта, а не удалённый SQL. Для сложного действия допустим подресурс или команда, например `POST /orders/{id}/cancellation`; ясность важнее религиозной чистоты.

## 11. Методы HTTP

| Метод | Обычный смысл | Safe | Idempotent |
|---|---|---:|---:|
| `GET` | получить представление | да | да |
| `POST` | создать/запустить обработку | нет | обычно нет |
| `PUT` | целиком заменить по известному URI | нет | да |
| `PATCH` | частично изменить | нет | зависит от операции |
| `DELETE` | удалить | нет | да |

Safe означает «не запрашивает изменение состояния». Idempotent — повтор того же запроса имеет тот же задуманный эффект, хотя статус второго ответа может отличаться. Идемпотентность не означает отсутствие логов или сетевого трафика.

## 12. Статусы и заголовки — часть контракта

- `200 OK` — успешный ответ с представлением;
- `201 Created` + `Location` — ресурс создан;
- `202 Accepted` — работа принята, но не завершена;
- `204 No Content` — успех без тела;
- `400 Bad Request` — запрос неверной формы;
- `401`/`403` — нет приемлемой identity/нет права;
- `404 Not Found` — ресурс не найден или скрыт;
- `409 Conflict` — конфликт с текущим состоянием;
- `412 Precondition Failed` — не выполнено условие `If-Match`;
- `422 Unprocessable Content` — синтаксис понятен, но данные семантически неприемлемы;
- `500` — непредвиденная ошибка сервера;
- `503` — сервис временно не готов.

Не следует отвечать `200` с телом `{ success:false }` на всякую ошибку: клиент, proxy и мониторинг теряют стандартный смысл HTTP.

## 13. DTO и domain model

DTO — внешняя форма данных. Domain entity содержит внутренние правила. Если выдавать entity напрямую, внутренняя перестройка становится breaking change, а чувствительные поля могут случайно попасть наружу.

Отдельные request/response DTO позволяют задавать обязательность, имена и версию независимо. Валидация DTO проверяет форму; domain проверяет бизнес-инварианты. Нельзя полагаться только на клиентскую валидацию.

## 14. Problem Details

Единая форма ошибки уменьшает количество специальных случаев у клиента. Problem Details содержит `type`, `title`, `status`, `detail`, `instance` и extensions.

```json
{
  "type": "https://example.test/problems/order-state",
  "title": "Order cannot be cancelled",
  "status": 409,
  "instance": "/orders/42/cancellation",
  "traceId": "00-abcd..."
}
```

Stack trace, SQL и внутренние имена классов клиенту не передают. `traceId` помогает сопоставить безопасный ответ с подробным серверным логом.

## 15. Повторы и Idempotency-Key

Клиент не всегда знает, обработал ли сервер запрос до обрыва соединения. Автоматически повторить `GET` обычно безопасно, а повтор `POST /payments` может создать второй платёж.

Клиент передаёт уникальный `Idempotency-Key`, сервер атомарно сохраняет ключ вместе с результатом и при повторе возвращает тот же логический результат. Нужно определить область ключа, TTL, поведение при другом body и конкурентных запросах. Просто проверить ключ, а затем отдельно записать заказ — гонка.

## 16. Optimistic concurrency: ETag

Сервер возвращает версию представления:

```http
ETag: "order-v7"
```

Клиент изменяет ресурс с `If-Match: "order-v7"`. Если текущая версия уже `v8`, сервер отклоняет запрос (`412`), не затирая чужие изменения. ETag также позволяет условный `GET` с `If-None-Match` и ответ `304 Not Modified`.

## 17. Коллекции: фильтры и пагинация

`GET /orders` без границ однажды станет слишком большим. Контракт задаёт limit, сортировку, фильтры и continuation token/cursor. Offset прост, но при вставках может пропускать или повторять элементы; cursor устойчивее при заранее определённом порядке.

Нужны максимальный page size и стабильный tie-breaker, например `(createdAt, id)`. Нельзя позволять произвольную сортировку по любому полю без оценки нагрузки и утечки данных.

## 18. OpenAPI

OpenAPI машиночитаемо описывает paths, operations, parameters, schemas, responses и security schemes. Из документа строят UI и clients, но генерация не заменяет проектирование: описание должно включать реальные статусы, обязательность, ограничения и примеры ошибок.

Contract/integration tests сравнивают поведение с обещанием. Сгенерированный клиент не исправит неоднозначную семантику API.

## 19. Совместимость и версии

Добавление необязательного поля обычно совместимо, если readers терпимы к неизвестным полям. Переименование, удаление, изменение типа или смысла — breaking change.

Сначала предпочитают additive evolution. Если разрыв неизбежен, выбирают стратегию (URI, header или media type), период параллельной поддержки, уведомление и дату прекращения. Версионирование не отменяет миграционный план.

## 20. Миграция SOAP

Безопасная последовательность:

1. зафиксировать WSDL/XSD и фактическое поведение;
2. перечислить клиентов, bindings, безопасность и SLA;
3. покрыть текущий контракт characterization-тестами;
4. отделить бизнес-логику от transport;
5. поставить adapter/anti-corruption layer;
6. переносить клиентов постепенно и наблюдать ошибки;
7. отключать старую точку только после подтверждения.

Механическая замена SOAP на JSON не исправляет плохую модель. Иногда разумно сохранить старый facade, а внутри вызвать новый application layer.

## 21. Типичные ошибки

- считать сетевой вызов обычным методом;
- выбирать технологию по моде, не инвентаризировав клиентов;
- использовать `GET` для изменения состояния;
- раскрывать domain entities и stack traces;
- возвращать один `200` для всех исходов;
- повторять неидемпотентный `POST` без ключа;
- забывать pagination и concurrency;
- объявлять OpenAPI документацией, не проверяя его;
- одномоментно отключать legacy endpoint.

## Итог

ASMX и WCF важны для понимания и сопровождения существующих SOAP-интеграций; ASP.NET Core — основной инструмент современных HTTP API. Качество сервиса определяется не форматом JSON, а ясным контрактом, корректной семантикой HTTP, безопасными повторами, совместимостью и поведением при отказах.
