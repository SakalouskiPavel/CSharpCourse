# Урок 29. WCF, ASMX, Web API и REST

> Сервисные технологии отражают разные эпохи и стили. Важно уметь поддержать legacy SOAP-контракт, но новый HTTP API проектировать вокруг ресурсов, стандартной семантики HTTP, совместимости и наблюдаемости.

## Результаты обучения

- проектировать границу и её контракт;
- называть эксплуатационные и security-компромиссы;
- проверять негативные сценарии и совместимость;
- документировать решение и его ограничения.

## Карта темы

### 1. ASMX

Классические .NET Framework XML Web Services публиковали SOAP/WSDL через атрибуты. Это legacy-технология для сопровождения, а не рекомендуемая основа нового .NET сервиса.

### 2. WCF

Объединял contracts, bindings и behaviors для SOAP/разных transports. Полный WCF server относится к .NET Framework; современный .NET имеет client packages, а CoreWCF помогает миграционным сценариям.

### 3. SOAP

Envelope, WSDL и XSD дают формальный контракт, расширения WS-* — безопасность/транзакции/надёжность. Цена — сложность и строгая toolchain-совместимость.

### 4. ASP.NET Core Web API

Endpoints/minimal APIs/controllers принимают HTTP, model binding/validation и middleware образуют pipeline. DI lifetime, cancellation и ProblemDetails являются частью контракта.

### 5. REST constraints

Client-server, stateless, cacheable, uniform interface, layered system и optional code-on-demand описывают архитектурный стиль. JSON поверх POST ещё не делает API RESTful.

### 6. Ресурсы и HTTP

URI называет ресурс, methods задают семантику: GET safe/idempotent, PUT/DELETE idempotent, POST обычно нет, PATCH частичен. Status codes и headers передают стандартный смысл.

### 7. Контракты ошибок

Problem Details даёт type, title, status, detail и instance с extensions. Не раскрывайте stack trace; correlation id помогает найти серверную диагностику.

### 8. Версионирование

Сначала проектируют additive evolution и tolerant readers. Breaking change требует версии/нового media type/endpoint и периода совместимости; версия в URL — один из вариантов.

### 9. Idempotency и concurrency

Idempotency-Key защищает повтор POST, ETag/If-Match реализует optimistic concurrency. Retry допустим только с пониманием семантики.

### 10. Документация и тесты

OpenAPI — машиночитаемый контракт, но generated document нужно проверять. Consumer/contract tests дополняют integration tests и помогают миграции.

## Минимальный пример

```csharp
app.MapGet("/orders/{id:guid}", async (Guid id, IOrderQueries queries, CancellationToken ct) =>
    await queries.FindAsync(id, ct) is { } order
        ? Results.Ok(order)
        : Results.NotFound());
```

Минимальный пример показывает форму API, но production-границе также нужны валидация, наблюдаемость, timeout/cancellation, защита данных и согласованная модель ошибок.

## Типичные ошибки

- выбирать технологию до формулировки качественных требований;
- скрывать важные зависимости и состояния за «универсальной» абстракцией;
- проверять только happy path;
- считать локальную успешность доказательством эксплуатационной надёжности;
- хранить секреты или персональные данные в коде и логах.

## Итог

Архитектурное решение оценивается через сценарии изменения и отказа. Фиксируйте контекст, последствия и критерий пересмотра, чтобы решение можно было проверить, а не защищать авторитетом.
