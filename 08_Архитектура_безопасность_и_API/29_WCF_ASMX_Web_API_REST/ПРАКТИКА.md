# Практика — урок 29

## Цель

Опубликовать application layer заказов как предсказуемый HTTP API и описать миграцию условного SOAP-контракта.

## Часть 1. HTTP API

Реализуйте:

- `POST /orders` — создание с `Idempotency-Key`;
- `GET /orders/{id}` — чтение с `ETag`;
- `GET /orders` — cursor/continuation pagination и ограниченный `pageSize`;
- `POST /orders/{id}/cancellation` — отмена с `If-Match`;
- `/health` — техническая проверка.

Требования:

1. Request/response DTO не совпадают с domain entities.
2. Создание возвращает `201` и `Location`.
3. Повтор того же idempotency key и body не создаёт второй заказ; тот же ключ с другим body отклоняется.
4. Неверная версия `If-Match` возвращает `412`.
5. Нарушение состояния заказа возвращает `409` в Problem Details.
6. Ошибка формы данных возвращает согласованный validation Problem Details.
7. Необработанная ошибка не раскрывает stack trace, но содержит `traceId`.
8. Все I/O принимают `CancellationToken`.
9. OpenAPI описывает статусы, DTO, pagination, `Idempotency-Key`, `ETag` и security scheme.

## Часть 2. Тесты контракта

Проверьте минимум:

- успешное создание и корректный `Location`;
- повтор создания с ключом;
- конкурентные одинаковые запросы с одним ключом;
- конфликт ключа и другого body;
- чтение существующего и отсутствующего заказа;
- условный `GET` с `If-None-Match`;
- отмену с текущей и устаревшей версией;
- максимальный page size и переход по cursor;
- форму хотя бы трёх Problem Details;
- отсутствие внутренних имён/stack trace в ответе `500`.

## Часть 3. Миграционный документ

Дан legacy SOAP endpoint `CancelOrder(orderId, reason)` с WSDL и тремя неизвестными клиентами. Создайте `SOAP_MIGRATION.md`:

1. сравните ASMX, full WCF, WCF client и CoreWCF;
2. перечислите вопросы об endpoint, binding, security, WS-* и клиентах;
3. предложите anti-corruption adapter к новому `CancelOrder` use case;
4. покажите mapping SOAP Fault ↔ HTTP Problem Details;
5. опишите characterization/contract tests;
6. составьте этапы параллельной эксплуатации, наблюдения и отключения.

## Критерии готовности

- API использует смысл HTTP, а не только JSON;
- повтор и конкуренция имеют определённое поведение;
- OpenAPI соответствует фактическим ответам;
- тесты охватывают негативные сценарии;
- миграция сохраняет старый контракт на переходный период;
- README объясняет, почему выбран ресурсный или command endpoint.

## Дополнительное задание

Сгенерируйте клиент из OpenAPI и напишите consumer test. Затем внесите одно совместимое и одно несовместимое изменение схемы и зафиксируйте, как каждое повлияло на клиент.
