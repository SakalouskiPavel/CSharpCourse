# Урок 19. Структурные шаблоны: как соединять объекты

> Структурные паттерны не столько создают новые действия, сколько меняют форму сотрудничества: переводят несовместимый контракт, оборачивают объект, упрощают подсистему, строят дерево или разделяют две оси изменений.

## 1. Adapter

Приложению нужен контракт:

```csharp
public interface IFileStorage
{
    Stream OpenRead(string path);
}
```

Legacy API возвращает bytes и использует другой тип ошибки:

```csharp
public sealed class LegacyStorageClient
{
    public byte[] Download(string key) { /* ... */ }
}
```

Adapter переводит:

```csharp
public sealed class LegacyStorageAdapter : IFileStorage
{
    private readonly LegacyStorageClient _client;

    public LegacyStorageAdapter(LegacyStorageClient client) => _client = client;

    public Stream OpenRead(string path)
    {
        try
        {
            return new MemoryStream(_client.Download(path), writable: false);
        }
        catch (LegacyNotFoundException ex)
        {
            throw new FileNotFoundException("Объект не найден", path, ex);
        }
    }
}
```

Adapter переводит не только signatures, но и semantics: units, errors, cancellation, ownership, null, sync/async behavior. Если перевод теряет важную гарантию, контракт должен это показать.

Object adapter использует composition. Class adapter через inheritance возможен в языках/ситуациях, но C# single inheritance и связь с legacy class обычно делают composition предпочтительнее.

## 2. Facade

Subsystem требует много шагов:

```csharp
public sealed class CheckoutFacade
{
    public CheckoutResult Checkout(CheckoutRequest request)
    {
        // inventory, pricing, payment, order coordination
    }
}
```

Facade предоставляет простой use-case oriented entry point. Она не обязана скрывать все advanced subsystem APIs и не меняет contracts компонентов.

Хорошая facade:

- координирует связанный сценарий;
- не хранит всю бизнес-логику мира;
- зависит от узких services;
- возвращает понятный результат;
- не становится God Object.

Adapter меняет интерфейс одной зависимости; Facade упрощает работу с группой зависимостей. Один класс может выполнять обе роли, но намерение нужно назвать.

## 3. Decorator

Decorator реализует тот же контракт, содержит inner object и добавляет ответственность:

```csharp
public sealed class LoggingStorage : IFileStorage
{
    private readonly IFileStorage _inner;
    private readonly ILogger _logger;

    public Stream OpenRead(string path)
    {
        _logger.LogInformation("Reading {Path}", path);
        return _inner.OpenRead(path);
    }
}
```

Client использует `IFileStorage` и не знает, обёрнут ли object.

Типичные decorators: logging, metrics, authorization, caching, retry, validation. Они compose-ятся:

```text
Authorization → Cache → Retry → RemoteStorage
```

Порядок меняет поведение. Cache снаружи authorization может выдать данные не тому user; retry снаружи metrics считает одну logical operation, внутри — attempts. Это архитектурное решение.

Decorator должен сохранять обещания inner contract: cancellation, exceptions, ownership, thread safety. Если намеренно меняет (кэширует stale result), это документируется.

## 4. Proxy

Proxy занимает место real subject и контролирует доступ к нему.

Виды:

- virtual proxy — lazy creation/loading;
- remote proxy — представляет объект другого процесса;
- protection proxy — проверяет access;
- caching proxy — контролирует получение;
- smart reference — считает/управляет access.

Форма похожа на Decorator. Различие намерения:

- Decorator добавляет responsibility к существующему object;
- Proxy представляет/контролирует доступ к subject, который может быть remote/lazy/expensive.

Remote proxy не может полностью скрыть network: latency, timeout, partial failure, serialization и cancellation должны оставаться в contract. «Выглядит как local call» может быть опасной иллюзией.

## 5. Bridge

Две независимые оси:

- вид отчёта: summary/detailed;
- renderer: text/HTML/PDF.

Наследование создаёт комбинации: `TextSummaryReport`, `HtmlSummaryReport`, `TextDetailedReport`...

Bridge разделяет abstraction и implementation:

```csharp
public interface IReportRenderer
{
    string Render(ReportModel model);
}

public abstract class Report
{
    protected IReportRenderer Renderer { get; }

    protected Report(IReportRenderer renderer) => Renderer = renderer;
    public abstract string Build();
}
```

Каждая иерархия меняется отдельно и combine-ится composition. Bridge нужен, когда **обе** оси имеют варианты и развиваются независимо. Если renderer один, Strategy/обычная dependency проще.

## 6. Composite

Дерево содержит leaves и groups:

```csharp
public interface IDocumentNode
{
    int CharacterCount { get; }
    void Render(StringBuilder output);
}

public sealed class TextNode : IDocumentNode { /* leaf */ }

public sealed class SectionNode : IDocumentNode
{
    private readonly List<IDocumentNode> _children = new();
    // CharacterCount и Render делегируют children
}
```

Client одинаково работает с отдельным элементом и composite. Рекурсивная структура естественна для UI tree, folders, expression tree, document.

Варианты API:

- uniform interface с Add/Remove у всех — просто обходить, но leaf получает бессмысленные операции;
- безопасные отдельные container interfaces — честнее, но client различает роли.

Не бросайте `NotSupportedException` повсюду ради красивой UML-симметрии.

Нужно определить ownership, cycles, parent links, traversal order и поведение при глубоком дереве.

## 7. Flyweight

Миллион символов документа не должен хранить одинаковый font/color object миллион раз. Flyweight разделяет **intrinsic state**, одинаковое и неизменяемое, а **extrinsic state** (позиция) передаётся извне.

```csharp
public sealed record TextStyle(string Font, int Size, string Color);

public sealed record Glyph(char Symbol, TextStyle SharedStyle, int X, int Y);
```

Factory/cache выдаёт общий immutable style. Если shared object mutable, изменение одного клиента влияет на всех.

Flyweight полезен при измеренном огромном числе повторов. Цена: разделение state, cache lifetime, key equality и сложность. String interning — родственная runtime-техника, но не основание вручную intern-ить весь пользовательский ввод (память может удерживаться долго).

## 8. Proxy, Decorator и Adapter рядом

Все могут содержать другой object:

| Паттерн | Основное намерение | Контракт |
|---|---|---|
| Adapter | сделать несовместимое совместимым | обычно новый |
| Decorator | добавить ответственность | тот же |
| Proxy | представить/контролировать subject | тот же/эквивалентный |
| Facade | упростить подсистему | новый упрощённый |

Кодовая форма не определяет pattern; определяет проблема.

## 9. Decorator и middleware

ASP.NET middleware образует chain вокруг следующего delegate и структурно напоминает Decorator/Chain of Responsibility. Но framework lifecycle и HTTP semantics важнее имени GoF.

Не пытайтесь делать decorator class для каждого cross-cutting concern, если pipeline/interceptor framework уже решает задачу прозрачнее.

## 10. Cache decorator: скрытая сложность

Кэширование меняет observable behavior:

- stale data;
- key semantics;
- invalidation;
- exception/negative caching;
- mutable object sharing;
- concurrent stampede.

Поэтому cache decorator не «просто dictionary». Его контракт должен определить freshness и consistency. Полная тема — урок 30.

## 11. Retry decorator: границы

Retry повторяет вызов только если:

- failure transient;
- operation idempotent или имеет idempotency key;
- attempt ограничен;
- есть backoff/jitter/cancellation;
- timeout budget не исчерпан.

Decorator не знает предметную idempotency автоматически. Retry вокруг non-idempotent payment может удвоить списание.

## 12. Ownership в wrappers

Если wrapper получает inner IDisposable, кто его Dispose-ит?

Варианты:

- decorator владеет inner и cascade Dispose;
- composition root владеет всем graph;
- wrapper принимает `leaveOpen`;
- контракт отдельно запрещает ownership transfer.

Нельзя по привычке Dispose injected inner. То же относится к stream, HttpClient, DbContext и subscriptions.

## 13. Как распознать Structural Pattern

Спросите:

- несовместимы ли contracts? → Adapter;
- слишком сложен типовой вход в subsystem? → Facade;
- нужно динамически добавить responsibility? → Decorator;
- нужно контролировать/представить доступ? → Proxy;
- две оси создают произведение subclasses? → Bridge;
- данные образуют дерево part-whole? → Composite;
- повторяется огромное immutable state? → Flyweight.

Если проблема не наблюдается, дополнительные wrappers только удлиняют call stack.

## Частые ошибки

- Adapter переводит signature, но теряет error/cancellation semantics;
- Facade становится God Object;
- Decorator меняет контракт молча;
- порядок wrappers не проверен;
- remote Proxy скрывает latency/failure как «локальность»;
- Bridge применяется при одной вариативной оси;
- Composite заставляет leaves поддерживать бессмысленные methods;
- Flyweight разделяет mutable state;
- wrapper неправильно Dispose-ит inner;
- cache/retry добавлены без предметной политики.

## Итог

Структурный паттерн выбирается по виду несовместимости/композиции. Одинаковая оболочка может быть Adapter, Decorator или Proxy — название определяется намерением и изменённым контрактом.
