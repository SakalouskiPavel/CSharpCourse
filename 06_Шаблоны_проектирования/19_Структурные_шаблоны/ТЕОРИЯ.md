# Урок 19. Структурные шаблоны проектирования

> Структурные паттерны меняют форму сотрудничества объектов: адаптируют контракт, добавляют поведение оболочкой, скрывают подсистему или разделяют независимо изменяющиеся части.

## Результаты обучения

- различать Adapter, Decorator, Facade, Proxy, Bridge и Composite
- сохранять семантику оборачиваемого контракта
- строить деревья через Composite
- учитывать порядок и lifetime оболочек

## Карта темы

### 1. Adapter

Преобразует несовместимый API в контракт клиента: переводит типы, единицы, ошибки и lifetime, а не только переименовывает методы.

### 2. Decorator

Реализует тот же контракт, делегирует inner-объекту и добавляет ответственность. Цепочки logging, retry, cache и authorization чувствительны к порядку.

### 3. Facade

Даёт упрощённую точку входа в подсистему. Он координирует типовой сценарий, но не обязан скрывать все возможности и не должен становиться god object.

### 4. Proxy

Замещает реальный объект и контролирует доступ: remote, lazy, protection или cache. Техническая форма похожа на Decorator, намерение отличается.

### 5. Bridge

Разделяет две независимо изменяющиеся оси — абстракцию и реализацию — и предотвращает декартово произведение подклассов.

### 6. Composite

Позволяет одинаково обходить лист и группу в дереве. Если операция бессмысленна для листа, безопаснее разделить контракты, чем бросать NotSupportedException повсюду.

### 7. Flyweight

Разделяет неизменяемое внутреннее состояние множества объектов, передавая внешнее в операции. Выгоду по памяти нужно измерить.

### 8. Обещания wrapper

Оболочка сохраняет cancellation, exceptions, thread-safety и ownership исходного контракта либо явно документирует изменение, например cache staleness.

## Минимальный пример

```csharp
public sealed class LoggingStore(IStore inner, ILogger logger) : IStore
{
    public async Task SaveAsync(Item item, CancellationToken ct)
    {
        logger.LogInformation("Saving {Id}", item.Id);
        await inner.SaveAsync(item, ct);
    }
}
```

Паттерн оценивайте по причине изменения, а не по сходству диаграммы. Начните с простого решения и вводите дополнительный объект только тогда, когда можете назвать выигрыш и цену.

## Типичные ошибки

- Decorator с другим смыслом интерфейса;
- потеря CancellationToken при делегировании;
- Facade, знающий всю систему;
- кэширование mutable-ссылок без политики;

## Итог

Название паттерна не доказывает качество. Корректное решение сохраняет прозрачные зависимости, предметные инварианты и возможность проверить поведение изолированно.
