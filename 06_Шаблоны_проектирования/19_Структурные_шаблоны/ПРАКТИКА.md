# Практика — урок 19

## Цель

Создать единый storage contract поверх legacy API и композицию инфраструктурных wrappers. Каждый структурный паттерн должен иметь отдельное наблюдаемое назначение.

## Adapter

Дан учебный `LegacyBlobClient`:

- принимает integer container id и string key;
- возвращает byte[];
- бросает собственные exceptions;
- не принимает CancellationToken;
- caller владеет полученными bytes.

Адаптируйте к `IFileStorage` с async read result/Stream contract. Явно решите:

- mapping path → container/key;
- not found;
- error translation с InnerException;
- невозможность настоящей cancellation legacy-вызова;
- ownership returned Stream.

Не притворяйтесь, что sync legacy I/O стал неблокирующим только из-за `Task.FromResult`.

## Decorators

Реализуйте независимо:

- structured logging;
- metrics/timing;
- authorization;
- cache с малым TTL для immutable byte snapshots.

Соберите минимум две цепочки и покажите, как порядок меняет logs/metrics/access. Выберите основной безопасный порядок.

## Proxy

Создайте lazy proxy, который создаёт дорогой storage client только при первом обращении. Creation thread-safe, но README отдельно анализирует thread safety самого client.

## Facade

`DocumentStorageFacade` выполняет типовой сценарий: проверить документ, сохранить, затем вернуть descriptor. Она координирует узкий use case и не содержит универсальных методов всех subsystem services.

## Composite

Представьте virtual folders/files как дерево. Leaf и folder поддерживают `TotalSize` и `RenderTree`; только folder изменяет children через отдельный container contract. Запретите cycles или обнаруживайте их.

## Проверки

- legacy not found переводится в contract result;
- original exception сохраняется;
- cancellation limitation документирована;
- unauthorized request не получает cached data;
- cache не отдаёт mutable shared buffer;
- wrapper ownership не закрывает чужой resource;
- lazy subject создаётся один раз;
- tree size корректен и cycle обработан.

## Критерии приёмки

- Adapter переводит семантику, а не только имена;
- каждый Decorator заменяем через тот же contract;
- порядок цепочки обоснован и проверен;
- Proxy явно представляет lazy subject;
- Facade остаётся use-case oriented;
- Composite не даёт leaf бессмысленный Add;
- README классифицирует каждый class по проблеме/последствию;
- solution собирается.

## Дополнительное задание

Разделите report kind и renderer через Bridge либо аргументированно откажитесь, если в задаче нет двух независимых осей. Flyweight добавляйте только после измерения повторяющегося state.
