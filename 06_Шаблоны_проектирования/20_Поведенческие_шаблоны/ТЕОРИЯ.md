# Урок 20. Поведенческие шаблоны: алгоритмы, состояния и взаимодействия

> Поведенческие паттерны распределяют решения между объектами: какой алгоритм выбрать, как перейти между состояниями, как представить запрос, как передать его по цепочке и как уведомить участников. Их нельзя применять по принципу «один if — один класс».

## 1. Strategy

Меняется алгоритм расчёта скидки:

```csharp
public interface IDiscountStrategy
{
    decimal Apply(Order order, decimal subtotal);
}

public sealed class RegularDiscount : IDiscountStrategy
{
    public decimal Apply(Order order, decimal subtotal) => subtotal;
}

public sealed class VipDiscount : IDiscountStrategy
{
    public decimal Apply(Order order, decimal subtotal) => subtotal * 0.9m;
}
```

Context делегирует выбранной Strategy. Выбор выполняет composition root/application policy, а не постоянный `if` внутри каждой операции.

Strategy подходит, если algorithms взаимозаменяемы под одним contract. Для одной короткой stateless операции достаточно `Func<Order,decimal,decimal>`; class нужен для dependencies, state и предметного имени.

Цена: больше объектов/навигации, client должен выбрать strategy, общий contract иногда становится слишком бедным для реально разных algorithms.

## 2. State

Поведение заказа зависит от lifecycle:

```text
Draft → Submitted → Paid → Shipped
                  ↘ Cancelled
```

Большой switch по status повторяется во многих methods. State objects инкапсулируют допустимые действия/переходы.

```csharp
public interface IOrderState
{
    void Pay(OrderContext order);
    void Cancel(OrderContext order);
}
```

Отличие от Strategy:

- Strategy обычно выбирает client и алгоритмы независимы;
- State связан с внутренним lifecycle, переходы меняют текущий state.

State Pattern не обязателен для трёх простых состояний: enum + централизованная transition table часто яснее. Классы оправданы, когда поведение каждого состояния значительно и расширяется.

Недопустимый переход должен иметь явный outcome. Не оставляйте default no-op: caller решит, что операция успешна.

## 3. Command

Command представляет запрос объектом:

```csharp
public sealed record ShipOrderCommand(OrderId OrderId);

public sealed class ShipOrderHandler
{
    public Result Handle(ShipOrderCommand command) { /* ... */ }
}
```

Это позволяет:

- поставить запрос в очередь;
- логировать/serialize;
- назначить handler;
- retry/deduplicate;
- хранить history;
- реализовать undo при наличии обратимой операции.

Command не равен произвольному DTO: он выражает намерение изменить состояние и обычно имеет одного логического handler-а.

Если command повторяется после timeout, нужен idempotency key или доказанная идемпотентность. Объектная форма сама не даёт exactly-once.

## 4. Undo и Command

Классический command может иметь `Execute`/`Undo`. Undo должен сохранять достаточное предыдущее state и учитывать внешние изменения.

Отмена текста локально проста; «отменить отправленное письмо» невозможно как технический rollback — нужна предметная compensation («отправить исправление»). Поэтому Undo не добавляют автоматически всем commands.

## 5. Chain of Responsibility

Запрос проходит handlers:

```text
Authentication → Authorization → Validation → Handler
```

Две распространённые семантики:

1. первый подходящий handler завершает обработку;
2. каждый middleware выполняет часть до/после `next`.

```csharp
public delegate Task RequestDelegate(Request request, CancellationToken ct);

public interface IMiddleware
{
    Task InvokeAsync(Request request, RequestDelegate next, CancellationToken ct);
}
```

Нужно определить порядок, termination, error propagation, cancellation и ownership context. Цепочка, собранная в случайном порядке DI scanning, опасна.

Chain отличается от Decorator намерением: Chain ищет/организует обработку по последовательности; Decorator добавляет responsibility к subject. Middleware сочетает черты обоих.

## 6. Observer

Observer уже встречался как .NET events:

```text
Subject → уведомляет набор Observers
```

Subject не знает concrete observers. Варианты реализации: event, `IObservable<T>`, собственный subscription contract.

Нужно определить:

- push/pull data;
- sync/async;
- error isolation;
- subscription lifetime;
- порядок;
- reentrancy;
- событие публикуется до/после commit.

In-process Observer не гарантирует durable delivery. Domain event внутри transaction ещё не является integration event в broker.

## 7. Mediator

Без Mediator компоненты напрямую знают друг друга:

```text
A ↔ B ↔ C ↔ D
```

Mediator централизует протокол взаимодействия:

```csharp
public interface IMediator
{
    TResult Send<TResult>(IRequest<TResult> request);
}
```

Преимущества: коллеги слабее связаны, routing/cross-cutting pipeline унифицированы. Цена: call graph скрыт за Send, mediator/handlers могут стать глобальной шиной и God Object, compile-time navigation хуже.

Mediator library не создаёт архитектуру автоматически. Модули, transactions и contracts всё равно проектируются.

## 8. Memento

Memento сохраняет snapshot внутреннего state для восстановления, не раскрывая его детали caretaker-у.

```csharp
public sealed class TextEditor
{
    public sealed record Snapshot(string Text, int Cursor);

    public Snapshot Save() => new(_text, _cursor);
    public void Restore(Snapshot snapshot) { /* validate + restore */ }
}
```

- Originator создаёт/восстанавливает;
- Memento хранит snapshot;
- Caretaker хранит history, но не изменяет внутренности.

Цена: memory, deep copy, sensitive data, version compatibility. Для больших states используют diffs/event log, но это иные trade-offs.

## 9. Template Method

Базовый class фиксирует skeleton, subclasses переопределяют steps:

```csharp
public abstract class Importer
{
    public ImportReport Run(Stream source)
    {
        Validate(source);
        var records = Parse(source);
        return Save(records);
    }

    protected virtual void Validate(Stream source) { }
    protected abstract IEnumerable<Record> Parse(Stream source);
    protected abstract ImportReport Save(IEnumerable<Record> records);
}
```

Преимущество: единый порядок. Цена: inheritance coupling, fragile base class, subclasses могут нарушить hidden assumptions.

Strategy/композиция лучше, если steps нужно комбинировать независимо или runtime. Template Method уместен для устойчивой иерархии с действительно общим skeleton.

Не вызывайте overridable step из constructor: derived object ещё не создан полностью.

## 10. Iterator

Iterator предоставляет последовательный доступ, не раскрывая представление. В C# это `IEnumerable<T>`, `IEnumerator<T>`, `foreach`, `yield return`.

```csharp
public IEnumerable<Node> DepthFirst()
{
    yield return this;
    foreach (Node child in _children)
        foreach (Node nested in child.DepthFirst())
            yield return nested;
}
```

Нужно определить order, deferred execution, reaction to mutation, cleanup и возможность повторного enumeration. Async iterator — `IAsyncEnumerable<T>`.

Паттерн уже встроен в язык; собственный IEnumerator вручную нужен редко.

## 11. Visitor

Есть стабильная hierarchy элементов, но регулярно добавляются операции:

```csharp
public interface IDocumentVisitor
{
    void Visit(TextNode node);
    void Visit(ImageNode node);
}

public interface IDocumentNode
{
    void Accept(IDocumentVisitor visitor);
}
```

`node.Accept(visitor)` вызывает подходящий `Visit(this)` — double dispatch: выбор зависит от runtime node type и visitor type.

Visitor упрощает добавление новой операции: новый visitor. Но добавление нового element type требует изменить visitor interface и все visitors.

Подходит для compiler AST/document model с относительно стабильными node kinds. Если element types часто растут, methods/pattern matching/functional representation могут быть проще.

Visitor может нарушить инкапсуляцию, если nodes раскрывают слишком много internal state.

## 12. Interpreter

Interpreter представляет grammar правилами/classes и вычисляет expression tree:

```text
expression := number | expression + expression
```

Простые filters/rules/domain mini-language можно моделировать nodes `And`, `Or`, `Equals`. У каждого `Interpret(context)`.

Плюсы: grammar видна в object model, легко комбинировать маленькие правила. Минусы: много classes, плохая производительность/диагностика для сложной grammar.

Для настоящего языка обычно используют lexer/parser generator, AST и отдельные phases. Regex не заменяет nested grammar.

## 13. Strategy, State, Command — как различать

| Паттерн | Что меняется | Кто обычно выбирает |
|---|---|---|
| Strategy | алгоритм | client/configuration |
| State | поведение по lifecycle | context/transition |
| Command | выполняемый запрос | sender создаёт, dispatcher/handler выполняет |

Один class технически может выглядеть одинаково, но intent определяет lifecycle и contracts.

## 14. Observer и Mediator

Observer: publisher не знает subscribers, один факт расходится многим.

Mediator: участники отправляют request централизованному coordinator/router; обычно есть ожидаемый handler/result.

Event bus, mediator и broker нельзя считать синонимами. У них различаются process boundary, durability и delivery guarantees.

## 15. Behavioral pattern или функция

Не создавайте hierarchy, если достаточно:

- dictionary key → function;
- switch expression над закрытым набором;
- Func/Action strategy;
- обычного прямого вызова;
- короткого loop;
- framework pipeline.

Classes оправданы state/dependencies/lifecycle/public extension point. Более простое решение легче читать и менять, пока variability не появилась.

## 16. Ошибки и транзакции

Поведенческая композиция должна определить failure:

- Strategy exception меняет ли state?
- State transition атомарен ли?
- Command повторяем ли?
- Chain продолжает после failure?
- Observer errors влияют на publisher?
- Memento восстановление может провалиться?
- Mediator transaction охватывает какой handler?

Паттерн не отвечает автоматически. Эти решения важнее диаграммы классов.

## Частые ошибки

- class Strategy создаётся для одной однострочной lambda;
- State скрывает transitions/no-op invalid actions;
- Command retry не имеет idempotency;
- Chain order не определён;
- Observer subscriber leak/exception не учтён;
- Mediator скрывает God Object и cross-module coupling;
- Memento shallow-copy-ит mutable state;
- Template Method создаёт fragile base class;
- Visitor применяется к постоянно растущей hierarchy;
- Interpreter строится для сложного языка вместо parser tooling;
- pattern name заменяет анализ errors/transactions.

## Итог

Выбирайте поведенческий паттерн по вопросу: меняется алгоритм, lifecycle, запрос, последовательность handlers, подписчики, координация, snapshot, skeleton, traversal, операция над hierarchy или grammar? Если ответ проще выражается функцией/switch/direct call — паттерн пока не нужен.
