# Урок 20. Поведенческие шаблоны проектирования

> Поведенческие паттерны распределяют алгоритмы, переходы и коммуникацию. Их задача — сделать изменяемое поведение явным, не превращая каждый условный оператор в отдельный класс.

## Результаты обучения

- выбирать Strategy, State, Command, Observer и Chain of Responsibility
- различать сходные по форме паттерны
- контролировать ошибки и переходы
- тестировать композицию поведения

## Карта темы

### 1. Strategy

Инкапсулирует взаимозаменяемый алгоритм. Короткой стратегии может хватить `Func`; отдельный класс полезен для состояния, зависимостей и предметного имени.

### 2. State

Переносит поведение, зависящее от этапа жизненного цикла. В отличие от Strategy, состояния и допустимые переходы образуют модель процесса.

### 3. Command

Представляет запрос объектом, который можно поставить в очередь, логировать, повторять или отменять. Результат и idempotency должны быть явными.

### 4. Chain of Responsibility

Передаёт запрос по цепочке до обработчика либо формирует pipeline. Порядок, остановка и поведение при ошибке являются контрактом.

### 5. Observer

Publisher уведомляет subscribers, не зная конкретных типов. In-process observer не гарантирует долговечную доставку и требует управления подпиской.

### 6. Mediator

Централизует связи коллег. Он уменьшает прямые зависимости, но способен скрыть call graph и стать god object.

### 7. Template Method

Базовый класс фиксирует скелет алгоритма, наследники переопределяют шаги. Strategy через композицию часто слабее связывает варианты.

### 8. Iterator, Visitor, Memento

Iterator скрывает обход; Visitor добавляет операции к стабильной иерархии; Memento хранит снимок для history/undo без раскрытия внутренностей.

## Минимальный пример

```csharp
public interface IDiscountStrategy
{
    decimal Apply(Order order, decimal subtotal);
}

public sealed class Checkout(IDiscountStrategy discount)
{
    public decimal Total(Order order) => discount.Apply(order, order.Subtotal);
}
```

Паттерн оценивайте по причине изменения, а не по сходству диаграммы. Начните с простого решения и вводите дополнительный объект только тогда, когда можете назвать выигрыш и цену.

## Типичные ошибки

- класс-стратегия для каждой однострочной функции;
- скрытые недопустимые переходы State;
- retry неидемпотентного Command;
- in-memory Observer вместо брокера;

## Итог

Название паттерна не доказывает качество. Корректное решение сохраняет прозрачные зависимости, предметные инварианты и возможность проверить поведение изолированно.
