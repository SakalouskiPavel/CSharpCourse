# Практика — урок 20

## Цель

Реализовать lifecycle заказа с явными переходами, командами и validation pipeline. Поведенческие паттерны должны сотрудничать, не скрывая transaction/error semantics.

## State model

Состояния: Draft, Submitted, Paid, Shipped, Cancelled.

Составьте transition table до кода:

| Current | Action | Next | Условие/ошибка |
|---|---|---|---|

Реализуйте enum + централизованную таблицу либо State classes. Выбор обоснуйте размером поведения. Любой invalid transition возвращает явный Result, не no-op.

## Commands

Создайте commands SubmitOrder, PayOrder, ShipOrder, CancelOrder с command id/idempotency key. Handler:

- загружает order;
- проверяет ожидаемую version;
- применяет transition;
- сохраняет атомарно;
- возвращает typed result.

Повтор того же Pay command не удваивает payment side effect в учебной модели.

## Chain

До handler выполняются упорядоченные steps:

1. authentication context exists;
2. authorization;
3. command shape validation;
4. idempotency lookup;
5. handler;
6. result logging/metrics.

Порядок собирается явно. Cancellation и exception проходят по contract. Validation failure не вызывает следующие business steps.

## Strategy

Cancellation policy заказа зависит от customer/category/time. Реализуйте две стратегии либо Func-варианты и объясните выбор формы. State transition использует policy, но не выбирает concrete strategy через `if customerType`.

## Observer

После успешного commit опубликуйте in-process `OrderStateChanged`. Два subscribers: audit и notification collector.

Определите, что происходит, если notification subscriber падает после commit. Не пытайтесь откатить уже сохранённое состояние обычным exception. Зафиксируйте ограничение in-process delivery и будущую потребность outbox.

## Memento experiment

Для Draft order реализуйте snapshot/restore только для локального редактирования до Submit. Snapshot делает независимую копию mutable lines. Не разрешайте Memento отменять внешний payment/shipment.

## Проверки

- все разрешённые transitions;
- все запрещённые transitions;
- повтор command id;
- неверная expected version;
- остановка chain на каждом step;
- strategy differences;
- subscriber failure после commit;
- deep snapshot draft lines.

## Критерии приёмки

- transition table и код согласованы;
- invalid action не меняет state;
- commands имеют явный result/idempotency;
- chain order видим и проверен;
- strategy заменяема;
- observer публикуется после commit с документированной failure policy;
- memento не выдаётся за distributed rollback;
- README объясняет, почему Template Method, Visitor, Interpreter и Mediator не применены либо где были бы уместны;
- solution собирается.

## Дополнительное задание

Представьте transition table как маленькое expression tree и реализуйте Visitor для текстовой документации, только если hierarchy правил стабильна. Сравните с обычным LINQ/switch решением и оставьте более читаемое.
