# Урок 18. Порождающие шаблоны проектирования

> Порождающие паттерны отделяют выбор и сборку объекта от кода, который им пользуется. Они нужны, когда создание имеет варианты, этапы, жизненный цикл или инварианты; обычный конструктор предпочтительнее, пока его достаточно.

## Результаты обучения

- различать Factory Method, Abstract Factory, Builder, Prototype и Singleton
- выбирать паттерн по оси изменения
- сохранять зависимости явными
- тестировать создание без глобального состояния

## Карта темы

### 1. Factory Method

Определяет заменяемую точку создания продукта. Подходит, когда общий алгоритм стабилен, а конкретный создаваемый тип выбирает специализация.

### 2. Abstract Factory

Создаёт согласованное семейство объектов без знания concrete classes. Добавлять новое семейство легко, новый вид продукта — дороже, потому что меняется контракт всех фабрик.

### 3. Builder

Пошагово собирает сложный объект. Он полезен при вариантах представления и множестве optional-частей, но `Build` обязан проверить обязательные данные и не выпускать полусобранный результат.

### 4. Prototype

Создаёт объект копированием шаблона. Необходимо явно определить shallow/deep copy, особенно для изменяемых вложенных коллекций и ресурсов.

### 5. Singleton

Даёт один экземпляр и глобальный доступ, скрывая зависимость и осложняя изоляцию тестов. Singleton lifetime в DI-контейнере управляемее и не требует статической точки доступа.

### 6. Simple factory и named factory

Статический метод `Create` или отдельная функция часто достаточны. Named factories выражают разные допустимые способы появления объекта: создание нового и восстановление из хранилища.

### 7. DI и composition root

Container собирает граф в точке входа. Бизнес-код не должен запрашивать зависимости через service locator: необходимые роли видны в конструкторе.

### 8. Выбор паттерна

Сначала назовите, что меняется: тип продукта, семейство, этапы сборки или шаблон. Без реальной вариативности дополнительный слой только прячет `new`.

## Минимальный пример

```csharp
public interface IStorageFactory
{
    IReader CreateReader();
    IWriter CreateWriter();
}

public sealed class ImportJob(IStorageFactory storage)
{
    public Task RunAsync() => CopyAsync(storage.CreateReader(), storage.CreateWriter());
}
```

Паттерн оценивайте по причине изменения, а не по сходству диаграммы. Начните с простого решения и вводите дополнительный объект только тогда, когда можете назвать выигрыш и цену.

## Типичные ошибки

- фабрика для каждого простого конструктора;
- service locator в доменной логике;
- Singleton как контейнер глобального изменяемого состояния;
- невалидный промежуточный результат Builder;

## Итог

Название паттерна не доказывает качество. Корректное решение сохраняет прозрачные зависимости, предметные инварианты и возможность проверить поведение изолированно.
