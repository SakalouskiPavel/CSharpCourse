# Урок 18. Порождающие шаблоны: управление созданием объектов

> Конструктор — лучший исходный способ создать объект. Порождающий паттерн нужен, когда создание само стало задачей: требуется выбрать тип, собрать сложную конфигурацию, создать согласованное семейство, копировать шаблон или управлять временем жизни.

## 1. Что такое шаблон проектирования

Design pattern — повторяемая форма решения типовой проблемы в определённом контексте. Это не готовая библиотека и не обязательный набор классов.

Паттерн описывают через:

- **контекст** — где возникла задача;
- **проблему/силы** — что конфликтует;
- **решение** — распределение ответственностей;
- **последствия** — что стало легче и что усложнилось.

Название помогает обсуждать дизайн. Само наличие `Factory` в имени класса не доказывает корректность.

## 2. Когда обычного конструктора достаточно

```csharp
var address = new Address(city, street);
```

Оставьте `new`, если:

- тип конкретный и известен client-у;
- создание состоит из понятных аргументов;
- constructor защищает invariants;
- нет нескольких семейств/lifecycle policy;
- подмена типа не является требованием.

Скрывать каждое `new` за `IAddressFactory` создаёт навигацию и mock-объекты без пользы.

## 3. Named/static factory method

Constructor не имеет предметного имени. Несколько способов появления объекта можно назвать:

```csharp
public sealed class Order
{
    private Order(OrderId id, OrderStatus status) { /* ... */ }

    public static Order CreateDraft(OrderId id) =>
        new(id, OrderStatus.Draft);

    public static Order Restore(OrderId id, OrderStatus status) =>
        new(id, status);
}
```

Преимущества:

- имя выражает намерение;
- можно вернуть subtype/cached instance/Result;
- можно отличить создание нового от восстановления;
- constructor остаётся закрытым.

Это часто называют factory method в широком смысле, но классический GoF **Factory Method** использует polymorphic creation point, который subclasses/implementations меняют.

## 4. Simple Factory

```csharp
public static class ExporterFactory
{
    public static IExporter Create(ExportFormat format) => format switch
    {
        ExportFormat.Json => new JsonExporter(),
        ExportFormat.Csv => new CsvExporter(),
        _ => throw new ArgumentOutOfRangeException(nameof(format))
    };
}
```

Централизует выбор concrete type. Это полезная идиома, хотя не отдельный GoF-паттерн.

Цена: factory знает все варианты; новый формат требует её изменения. Если набор закрыт и мал, это честнее Reflection/DI container. Если providers подключаются независимо, нужен registration/plugin mechanism.

## 5. Factory Method (GoF)

Общий процесс использует создаваемый product, а специализация выбирает product:

```csharp
public abstract class ReportJob
{
    protected abstract IReportWriter CreateWriter();

    public void Run(Report report)
    {
        IReportWriter writer = CreateWriter();
        writer.Write(report);
    }
}

public sealed class PdfReportJob : ReportJob
{
    protected override IReportWriter CreateWriter() => new PdfReportWriter();
}
```

Скелет `Run` стабилен, product меняется через override. В современном C# тот же variation point часто проще передать delegate/interface factory через composition:

```csharp
public sealed class ReportJob(Func<IReportWriter> createWriter)
{
    public void Run(Report report) => createWriter().Write(report);
}
```

Наследование оправдано только при настоящей иерархии процесса; иначе composition слабее связывает.

## 6. Abstract Factory

Проблема: нужны согласованные семейства связанных products.

```csharp
public interface IUiFactory
{
    IButton CreateButton();
    IDialog CreateDialog();
}

public sealed class DesktopUiFactory : IUiFactory
{
    public IButton CreateButton() => new DesktopButton();
    public IDialog CreateDialog() => new DesktopDialog();
}

public sealed class WebUiFactory : IUiFactory
{
    public IButton CreateButton() => new WebButton();
    public IDialog CreateDialog() => new WebDialog();
}
```

Client получает factory и не смешивает DesktopButton с WebDialog.

Компромисс:

- добавить новое **семейство** легко: новая factory;
- добавить новый **вид product** трудно: метод добавляется во все factories.

Используйте, когда согласованность семейства — реальный invariant. Для одного независимого product обычная factory проще.

## 7. Builder: поэтапная сборка

Constructor с десятью parameters трудно читать:

```csharp
var report = new Report(title, author, true, false, 20, null, ...);
```

Builder накапливает части и выдаёт готовый object:

```csharp
public sealed class ReportBuilder
{
    private string? _title;
    private readonly List<ReportSection> _sections = new();

    public ReportBuilder WithTitle(string title)
    {
        _title = title;
        return this;
    }

    public ReportBuilder AddSection(string heading, string text)
    {
        _sections.Add(new ReportSection(heading, text));
        return this;
    }

    public Report Build()
    {
        if (string.IsNullOrWhiteSpace(_title))
            throw new InvalidOperationException("Заголовок обязателен");

        return new Report(_title, _sections.ToArray());
    }
}
```

Fluent return `this` позволяет цепочку. `Build` проверяет итоговый invariant и передаёт snapshot, чтобы дальнейшее изменение builder-а не изменяло готовый report.

Builder нужен при этапах/вариантах/повторной сборке. Для трёх optional properties object initializer/record `with` проще.

## 8. Step Builder и типобезопасность

Обычный builder допускает вызвать Build слишком рано и обнаружить ошибку в runtime. Step Builder разными interfaces разрешает только допустимый следующий шаг. Он переносит часть проверки в compile time, но создаёт много типов.

Применяйте для критичного порядка с небольшим числом шагов. Если комбинаций много, state explosion сделает API тяжелее обычного validated Build.

## 9. Director

Классический Builder отделяет builder от **Director**, который знает последовательность:

```csharp
public static Report BuildMonthly(ReportBuilder builder) => builder
    .WithTitle("Месячный отчёт")
    .AddSection("Итоги", "...")
    .Build();
```

В C# director часто обычный method/service. Отдельный class нужен, если recipes меняются/тестируются/переиспользуются как самостоятельная ответственность.

## 10. Prototype

Prototype создаёт новый объект копированием настроенного образца.

```csharp
public sealed record ReportTemplate(
    string Title,
    IReadOnlyList<string> Sections);

ReportTemplate copy = template with { Title = "Новый отчёт" };
```

Главный вопрос — глубина копирования. `with` у record выполняет shallow copy: тот же mutable nested List может разделяться.

- **shallow copy** копирует верхний object и references;
- **deep copy** создаёт независимые mutable части;
- immutable nested values безопасно разделять.

`MemberwiseClone` protected и shallow. Serialization ради deep clone обычно медленна, теряет поведение/типы и скрывает contract; явный copy constructor/factory яснее.

Prototype полезен при runtime-configured templates или дорогой подготовке, если copy semantics определена. Для простого DTO `with`/constructor достаточно.

## 11. Singleton

Singleton GoF гарантирует один instance и глобальную точку доступа:

```csharp
public sealed class AppRegistry
{
    private static readonly Lazy<AppRegistry> _instance =
        new(() => new AppRegistry());

    public static AppRegistry Instance => _instance.Value;
    private AppRegistry() { }
}
```

Проблемы:

- dependency скрыта: метод обращается к глобальному состоянию;
- tests влияют друг на друга;
- lifetime привязан ко всему process;
- mutable state требует synchronization;
- configuration/cleanup усложняются.

DI **singleton lifetime** означает один service instance внутри container scope, но consumers всё равно получают dependency явно. Это обычно предпочтительнее Singleton pattern.

Static immutable utility/config value не обязательно Singleton. Иногда process-wide object обоснован (например, registry инфраструктуры), но глобальная доступность не должна быть автоматической.

## 12. Thread-safe singleton

Static initialization CLR thread-safe, `Lazy<T>` также координирует создание. Double-checked locking легко написать неверно и редко нужен в C#.

Thread-safe **создание** не делает thread-safe mutable state внутри singleton. Это отдельная задача.

## 13. Object Pool

Pool часто относят к порождающим idioms: он выдаёт переиспользуемый дорогой object и принимает обратно.

Подходит, если:

- allocation/initialization измеренно дороги;
- объектов одновременно ограниченное число;
- state можно надёжно сбросить;
- ownership Rent/Return строго определён.

Риски: утечка не возвращённого объекта, использование после Return, данные прошлого клиента, contention. Для обычных lightweight services GC/DI проще.

## 14. DI container и factories

Container умеет создавать graph, выбирать lifetime и factories. Он не отменяет дизайн:

- registrations находятся в composition root;
- business code не вызывает `serviceProvider.GetService` как Service Locator;
- runtime value передаётся factory method/delegate;
- container не должен выбирать domain decision, скрытый в строковом key.

Если обычный constructor injection решает задачу, отдельная factory не нужна.

## 15. Сравнение

| Проблема | Подход |
|---|---|
| Понятное простое создание | constructor |
| Несколько именованных путей | named factory |
| Выбор одного product | simple factory / injected factory |
| Вариативный creation step в иерархии | Factory Method |
| Согласованное семейство products | Abstract Factory |
| Много этапов/вариантов сборки | Builder |
| Копия runtime template | Prototype |
| Один управляемый lifetime | чаще DI singleton; редко Singleton |
| Дорогое измеренное переиспользование | Pool |

## Частые ошибки

- factory создаётся для каждого `new`;
- Factory Method путается с любым static Create;
- Abstract Factory используется для одного product;
- Builder выпускает невалидный object или отдаёт mutable internal list;
- Prototype не определяет shallow/deep semantics;
- Singleton хранит mutable global business state;
- thread-safe creation принимается за thread-safe object;
- Service Locator прячет dependencies;
- pool добавляется без profiling и clear ownership.

## Итог

Порождающий паттерн отвечает не «как убрать `new`», а «какая часть создания меняется и кому принадлежит это решение». Если ось изменения нельзя назвать, constructor почти наверняка лучше.
