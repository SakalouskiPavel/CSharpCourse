# Практика — урок 18

## Цель

Создать генератор документов с двумя согласованными семействами компонентов и безопасной поэтапной сборкой. Паттерны применяются только по явно названным требованиям.

## Семейства

Два target family:

- Plain Text: renderer sections + text exporter;
- HTML: HTML renderer + HTML exporter.

Создайте Abstract Factory, выдающую согласованные `ISectionRenderer` и `IDocumentExporter`. Client не делает `if (format)` и не знает concrete types.

Проверьте, что невозможно случайно смешать HTML renderer с text exporter через обычный composition root. Если интерфейсы технически позволяют смесь, factory должна гарантировать семейство и README описывает эту границу.

## Builder

`DocumentBuilder` собирает:

- обязательный непустой title;
- минимум одну section;
- optional author;
- metadata без повторяющихся keys;
- immutable/read-only finished document.

После `Build` изменение builder-а не меняет уже созданный document. Повторный Build имеет явно документированную семантику: разрешён независимый snapshot либо запрещён.

## Named factories

У `Document` реализуйте осмысленные пути `CreateDraft` и `Restore`. Они должны различаться invariants/initial state, а не быть двумя именами одного constructor без причины.

## Factory Method comparison

Создайте маленький альтернативный вариант export job:

- через protected Factory Method;
- через injected `Func<IDocumentExporter>`.

Оставьте основным один и объясните, почему inheritance/composition лучше в контексте.

## Prototype experiment

Скопируйте `DocumentTemplate` через record `with`, содержащий nested mutable collection. Покажите shallow sharing, затем реализуйте безопасную copy factory/snapshot. Не используйте JSON serialization для клонирования.

## Singleton decision

Глобальный singleton запрещён в основной реализации. Зарегистрируйте stateless exporter как DI singleton lifetime либо создайте явно и передайте. Объясните, почему потребитель остаётся тестируемым.

## Критерии приёмки

- Abstract Factory сохраняет family consistency;
- Builder не выпускает невалидный/изменяемый результат;
- named factories имеют разный предметный смысл;
- Factory Method и composition сравниваются на коде;
- copy semantics продемонстрирована;
- нет Service Locator/global mutable Singleton;
- README содержит таблицу «проблема → паттерн → цена → альтернатива»;
- solution собирается, оба формата формируются.

## Дополнительное задание

Добавьте третье семейство и оцените стоимость. Затем попробуйте добавить новый вид product во все factories — это должно наглядно показать компромисс Abstract Factory.
