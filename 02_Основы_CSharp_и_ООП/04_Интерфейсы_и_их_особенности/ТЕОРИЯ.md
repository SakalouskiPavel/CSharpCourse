# Урок 04. Интерфейсы: договор между клиентом и реализацией

> Класс сообщает, как устроен конкретный объект. Интерфейс сообщает, какую роль объект способен выполнять. Благодаря этому клиент зависит от нужного поведения, а реализацию можно заменить без переписывания клиента.

## Что нужно знать

Нужны понятия класса, объекта, метода, конструктора, модификатора доступа, наследования, композиции и зависимости из урока 03.

## Результаты обучения

Вы сможете объявить и реализовать интерфейс, передать реализацию через конструктор, разделить большой контракт на роли, выбрать между интерфейсом и абстрактным классом, объяснить явную реализацию, default members, variance и test fake.

## 1. Проблема жёсткой зависимости

```csharp
public sealed class RegistrationService
{
    private readonly EmailSender _sender = new();

    public void Register(string email)
    {
        // регистрация
        _sender.Send(email, "Добро пожаловать");
    }
}
```

Сервис сам создаёт `EmailSender`. Чтобы отправить SMS, отключить настоящую отправку при проверке или поменять библиотеку, придётся менять `RegistrationService`.

Сервису на самом деле нужен не конкретный email-класс, а способность отправить уведомление.

## 2. Объявление интерфейса

```csharp
public interface INotificationSender
{
    void Send(string recipient, string message);
}
```

Интерфейс — тип-контракт. Он говорит: любой подходящий объект предоставляет метод `Send` с указанными параметрами. Имя с `I` — соглашение .NET, а не требование компилятора.

Реализации:

```csharp
public sealed class EmailSender : INotificationSender
{
    public void Send(string recipient, string message)
    {
        Console.WriteLine($"Email для {recipient}: {message}");
    }
}

public sealed class SmsSender : INotificationSender
{
    public void Send(string recipient, string message)
    {
        Console.WriteLine($"SMS для {recipient}: {message}");
    }
}
```

После двоеточия класс обещает выполнить контракт. Если сигнатура отсутствует, проект не соберётся.

## 3. Клиент интерфейса

```csharp
public sealed class RegistrationService
{
    private readonly INotificationSender _sender;

    public RegistrationService(INotificationSender sender)
    {
        _sender = sender ?? throw new ArgumentNullException(nameof(sender));
    }

    public void Register(string recipient)
    {
        _sender.Send(recipient, "Добро пожаловать");
    }
}
```

`RegistrationService` — клиент интерфейса. Он знает только операцию, которая ему нужна. Конкретный объект выбирает composition root:

```csharp
INotificationSender sender = new EmailSender();
var service = new RegistrationService(sender);
service.Register("student@example.com");
```

Замена `new EmailSender()` на `new SmsSender()` не меняет сервис. Это не означает, что реализации полностью одинаковы: адрес получателя, длина сообщения и ошибки каналов могут различаться. Контракт должен честно выражать общую семантику.

## 4. Интерфейс не создаёт объект

Нельзя написать `new INotificationSender()`: интерфейс не содержит достаточной конкретной реализации. Переменная интерфейсного типа хранит ссылку на объект реализующего класса.

```csharp
INotificationSender sender = new EmailSender();
```

Через переменную видны только члены контракта. Это полезное ограничение: клиент не начинает зависеть от случайных дополнительных методов `EmailSender`.

## 5. Один класс — несколько ролей

Класс может реализовать несколько интерфейсов:

```csharp
public interface IHealthCheck
{
    bool IsAvailable();
}

public sealed class EmailSender : INotificationSender, IHealthCheck
{
    public void Send(string recipient, string message) { /* ... */ }
    public bool IsAvailable() => true;
}
```

Коду регистрации нужен `INotificationSender`, коду мониторинга — `IHealthCheck`. Ни один клиент не получает лишнего обязательства.

## 6. Interface Segregation Principle

Плохой общий контракт:

```csharp
public interface INotificationChannel
{
    void Send(string recipient, string message);
    bool SupportsAttachments();
    void AttachFile(string path);
    void Schedule(DateTimeOffset time);
}
```

SMS-реализация вынуждена бросать `NotSupportedException` для вложений. Лучше разделить роли:

```csharp
public interface INotificationSender
{
    void Send(string recipient, string message);
}

public interface IAttachmentSender
{
    void SendWithAttachment(string recipient, string message, string path);
}
```

**Interface Segregation Principle (ISP)**: клиент не должен зависеть от операций, которыми не пользуется. «Маленький» означает не минимальное число строк, а одну связанную роль.

## 7. Интерфейс или абстрактный класс

| Интерфейс | Абстрактный класс |
|---|---|
| описывает роль/способность | является базой семейства |
| класс может реализовать несколько | класс наследуется только от одного класса |
| не хранит обычное instance state | может хранить поля и состояние |
| удобен как граница зависимости | удобен для общего защищённого поведения |

Выбирайте интерфейс, когда разные несвязанные классы выполняют одну роль или клиенту нужна заменяемая зависимость. Выбирайте abstract class, когда существует настоящая иерархия с общей реализацией и состоянием.

Не создавайте интерфейс автоматически для каждого класса. Если вариативности нет, контракт не нужен клиенту и тип является внутренней деталью, прямой класс проще.

## 8. Явная реализация

Иногда два интерфейса имеют одинаковое имя с разным смыслом или технический член не нужен в основном API:

```csharp
public interface ITextExporter
{
    string Export();
}

public interface IAuditExporter
{
    string Export();
}

public sealed class Report : ITextExporter, IAuditExporter
{
    string ITextExporter.Export() => "Текст отчёта";
    string IAuditExporter.Export() => "Аудит отчёта";
}
```

Вызов возможен только через выбранную роль:

```csharp
var report = new Report();
string text = ((ITextExporter)report).Export();
```

У явного члена нет `public`: доступ задаёт интерфейс. Не используйте этот механизм, чтобы произвольно прятать неудобный дизайн.

## 9. Реализация по умолчанию

Современный C# позволяет интерфейсу дать default implementation:

```csharp
public interface ILogger
{
    void Log(string message);

    void LogError(string message)
    {
        Log($"ERROR: {message}");
    }
}
```

Это помогает осторожно развивать опубликованный контракт, не ломая все реализации немедленно. Но интерфейс не превращается в обычный базовый класс: он не хранит поля экземпляра, правила разрешения нескольких реализаций сложны, а скрытая логика затрудняет чтение. Для учебного и внутреннего кода сначала рассмотрите extension method или отдельный сервис.

## 10. Статические абстрактные члены

Иногда generic-алгоритму нужна операция самого типа, например сложение чисел. Интерфейс может объявить static abstract member:

```csharp
public interface IParsableValue<TSelf>
    where TSelf : IParsableValue<TSelf>
{
    static abstract bool TryParse(string text, out TSelf value);
}
```

Эта возможность лежит в основе generic math. Она продвинутая: обычную instance-зависимость через неё моделировать не нужно. Главное отличие — член вызывается через параметр типа, а не через объект.

## 11. Ковариантность и контравариантность

Эти слова описывают безопасную совместимость generic-интерфейсов.

Если `Cat` является `Animal`, последовательность кошек безопасно читать как последовательность животных:

```csharp
IEnumerable<Cat> cats = GetCats();
IEnumerable<Animal> animals = cats;
```

`IEnumerable<out T>` **ковариантен**: он выдаёт `T`, но не принимает произвольный `T` для добавления.

Обработчик, умеющий принять любое животное, может принять и кошку:

```csharp
public interface IHandler<in T>
{
    void Handle(T value);
}
```

`in T` обозначает **контравариантность** входа. Mutable `IList<T>` инвариантен: если бы список кошек можно было считать списком животных, в него добавили бы собаку.

Variance работает для безопасных преобразований ссылочных типов и только в разрешённых позициях. На первом проходе достаточно понимать причину: нельзя разрешить преобразование, которое позволит положить значение неправильного типа.

## 12. Test fake

Для проверки сервиса настоящая отправка письма не нужна. Простая подмена сохраняет полученные данные:

```csharp
public sealed class FakeNotificationSender : INotificationSender
{
    public string? LastRecipient { get; private set; }
    public string? LastMessage { get; private set; }

    public void Send(string recipient, string message)
    {
        LastRecipient = recipient;
        LastMessage = message;
    }
}
```

```csharp
var fake = new FakeNotificationSender();
var service = new RegistrationService(fake);

service.Register("student@example.com");
Console.WriteLine(fake.LastRecipient);
```

Fake — простая рабочая реализация для проверки. Mock обычно настраивает ожидаемые взаимодействия и проверяет вызовы; stub возвращает заранее заданные данные. Термины иногда используются свободно, поэтому важнее описать поведение подмены.

Интерфейс создают не «ради mock», а потому что клиенту действительно нужна роль. Fake является дополнительным преимуществом такой границы.

## Частые ошибки

- интерфейс полностью копирует публичный API единственного класса без потребности клиента;
- большой интерфейс заставляет реализации бросать `NotSupportedException`;
- контракт содержит методы, нужные только одной инфраструктурной реализации;
- service locator заменяет явные constructor dependencies;
- default implementation используется как склад общей логики;
- variance применяется наугад без анализа чтения и записи;
- тест проверяет каждую внутреннюю операцию и ломается при безопасном рефакторинге.

## Итоговая карта

```text
Клиенту нужна роль
  ↓
Интерфейс описывает минимальный контракт роли
  ↓
Несколько классов могут реализовать контракт
  ↓
Composition root выбирает реализацию
  ↓
Клиент получает её через конструктор
```

Вы готовы к следующему уроку, если можете выделить интерфейс по потребности клиента, реализовать две версии, заменить их в точке сборки и объяснить, почему большой универсальный интерфейс хуже нескольких связанных ролей.
