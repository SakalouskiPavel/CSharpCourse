# Урок 04. Интерфейсы и их особенности

> Интерфейс описывает роль объекта для клиента. Он уменьшает связанность только тогда, когда выделен по потребности потребителя, а не механически создан для каждого класса.

## Результаты обучения

- проектировать маленькие интерфейсы по ролям
- использовать явную реализацию и default members
- понимать variance generic-интерфейсов
- различать контракт, абстрактный класс и реализацию

## Карта темы

### 1. Контракт интерфейса

Интерфейс задаёт доступные операции, свойства, события и индексаторы без привязки к состоянию реализации. Имя обычно описывает способность или роль.

### 2. Interface Segregation

Клиент не должен зависеть от методов, которые не использует. Несколько узких контрактов позволяют типу участвовать в разных сценариях без «жирного» общего API.

### 3. Интерфейс и абстрактный класс

Интерфейс допускает множество ролей и не хранит instance-состояние; базовый класс может разделять реализацию и состояние, но навязывает одну линию наследования.

### 4. Явная реализация

Член `IContract.Method` виден только через ссылку интерфейса. Это помогает разрешить конфликт имён или скрыть техническую операцию из основного API.

### 5. Default interface members

Интерфейс может содержать реализацию по умолчанию, что помогает развивать публичные контракты. Это не универсальная замена базовым классам: состояние и правила версионирования остаются сложными.

### 6. Статические абстрактные члены

Современный C# позволяет интерфейсам задавать статические операции, что используется generic math. Вызов возможен через параметр типа с соответствующим constraint.

### 7. Ковариантность и контравариантность

`out T` разрешает безопасно возвращать более конкретные типы, `in T` — принимать менее конкретные. Variance работает только для ссылочных типов и только в разрешённых позициях.

### 8. Тестовые двойники

Интерфейс образует шов для подмены внешней зависимости. Не каждую подмену нужно мокать: простой fake часто яснее и устойчивее.

## Минимальный пример

```csharp
public interface IClock
{
    DateTimeOffset UtcNow { get; }
}

public sealed class ExpirationService(IClock clock)
{
    public bool IsExpired(DateTimeOffset deadline) => deadline <= clock.UtcNow;
}
```

Разберите пример построчно: какие гарантии выражены типами, где проверяются предусловия и что увидит вызывающий код. Затем измените один инвариант и проследите, какая часть публичного контракта должна поменяться.

## Типичные ошибки

- создавать интерфейс с единственной реализацией «на всякий случай»;
- копировать весь публичный API класса в интерфейс;
- добавлять методы, нужные только одной реализации;
- считать mock обязательным для каждого интерфейса;

## Итог

Тема считается усвоенной, когда вы можете не только применить механизм, но и объяснить его цену, область корректности и более простой альтернативный вариант. Проверяйте решения сборкой, тестами и небольшими экспериментами, а не только чтением.
