# Практика — урок 04

## Цель

Расширить учебную библиотеку уведомлениями так, чтобы сценарий выдачи не зависел от конкретного канала и не заставлял все каналы поддерживать лишние возможности.

## Этап 1. Роль отправителя

Создайте:

```csharp
public interface INotificationSender
{
    void Send(string recipient, string message);
}
```

Реализуйте `EmailNotificationSender` и `ConsoleNotificationSender`. Обе реализации проверяют обязательные аргументы, но по-разному доставляют результат.

## Этап 2. Клиент

Создайте `LoanNotificationService`, который получает `INotificationSender` через конструктор. Он умеет сформировать и отправить уведомление о выдаче и возврате экземпляра.

Сервис не должен:

- создавать sender через `new` внутри метода;
- проверять concrete type через `is EmailNotificationSender`;
- обращаться к service locator;
- знать технические настройки email.

В console composition root создайте выбранную реализацию и передайте её сервису.

## Этап 3. Разделение ролей

Только email поддерживает вложение. Создайте отдельный `IAttachmentSender`. Не добавляйте `SendAttachment` в общий `INotificationSender` и не заставляйте консольную реализацию бросать `NotSupportedException`.

Покажите класс, реализующий два интерфейса, и два клиента, каждый из которых зависит только от своей роли.

## Этап 4. Fake и проверка

Создайте `FakeNotificationSender`, сохраняющий последнее сообщение и количество вызовов. С его помощью без настоящей отправки проверьте:

1. правильного получателя;
2. наличие названия книги в сообщении;
3. ровно один вызов при успешном сценарии;
4. отсутствие вызова при неуспешной выдаче.

Можно использовать обычное console-приложение с явными сравнениями. Если test framework уже настроен преподавателем, допустимы автоматические тесты, но практика не требует необъяснённого инструмента.

## Этап 5. Явная реализация

Добавьте отдельные интерфейсы `IUserMessageFormatter` и `IAuditMessageFormatter` с одинаковым методом `Format`. Реализуйте оба явно в одном классе. Покажите вызов через каждую интерфейсную ссылку и объясните, почему простой `formatter.Format()` недоступен.

## Критерии приёмки

- клиент зависит от интерфейса нужной роли;
- реализация выбирается только в composition root;
- минимум две взаимозаменяемые реализации проходят одинаковый базовый сценарий;
- возможность вложения отделена от простой отправки;
- fake доказывает четыре заявленных наблюдения;
- явная реализация продемонстрирована осмысленно;
- README содержит схему зависимостей и сравнение interface/abstract class для этой задачи;
- solution собирается, а сценарии повторяются по инструкции.

## Дополнительное задание

Создайте `IHandler<in T>` и продемонстрируйте, почему обработчик общего `Notification` можно использовать там, где требуется обработчик `EmailNotification`. Отдельно письменно объясните, почему mutable-список не может быть ковариантным. Не включайте этот эксперимент в основную архитектуру уведомлений.
