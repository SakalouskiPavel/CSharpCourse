# Практика — Урок 03. Классы как воплощение принципов ООП

## Задача

Создать доменную модель библиотеки: книги, экземпляры, читатели и выдачи. Модель должна сама защищать правила выдачи, а приложение — координировать сценарии.

## Обязательные требования

- запретить выдачу уже занятого экземпляра и превышение лимита читателя;
- сделать обязательное состояние корректным сразу после конструктора;
- использовать композицию минимум в одном месте;
- добавить полиморфное правило срока выдачи для разных типов читателей;
- покрыть инварианты и переходы состояния тестами;

## Порядок работы

1. Зафиксируйте сценарии и спорные решения до написания основной реализации.
2. Создайте production- и test-проекты, добавьте их в solution.
3. Реализуйте самый маленький сквозной сценарий и закоммитьте его.
4. Добавьте проверки ошибок и граничных условий.
5. Выполните `dotnet format --verify-no-changes`, `dotnet build` и `dotnet test` либо объясните недоступность команды.
6. В README проекта опишите запуск и принятые решения.

## Критерии готовности

- все обязательные сценарии воспроизводимы;
- решение демонстрирует именно тему урока, а не обходит её;
- тесты проверяют поведение, включая минимум два негативных или граничных случая;
- публичный API и сообщения об ошибках понятны без чтения реализации;
- в репозитории нет результатов сборки и секретов.

## Дополнительный уровень

- сохранить доменную модель независимой от консоли и файлов;
- добавить журнал доменных событий без внешнего фреймворка;

Дополнительная часть выполняется после устойчивой базовой. Не усложняйте архитектуру только ради количества паттернов.
