# Практика — урок 03

## Цель

Построить небольшую объектную модель библиотечного экземпляра. Главная проверка — не количество классов, а невозможность нарушить правила через публичный API.

## Предметная ситуация

Библиотека хранит экземпляры книг. Экземпляр можно выдать читателю и вернуть. Один экземпляр нельзя одновременно выдать двум читателям.

## Этап 1. `BookCopy`

Создайте `sealed class BookCopy`:

- неизменяемые свойства `InventoryNumber`, `Title`, `Author`;
- читаемые снаружи `IsBorrowed` и `BorrowedBy` без публичных setters;
- конструктор отклоняет пустые обязательные строки;
- `TryBorrow(string readerName)` выдаёт свободный экземпляр и возвращает `true`; занятый — не изменяет и возвращает `false`;
- `Return()` возвращает экземпляр; попытка вернуть свободный экземпляр имеет явно выбранное и документированное поведение.

После любой операции должны выполняться инварианты:

1. идентификатор, название и автор не пусты;
2. если `IsBorrowed == false`, `BorrowedBy` отсутствует;
3. если `IsBorrowed == true`, имя читателя присутствует;
4. повторная выдача не заменяет первого читателя.

## Этап 2. `Reader`

Создайте класс читателя с именем и лимитом одновременных выдач. Не храните публично изменяемый счётчик. Методы читателя и экземпляра должны сотрудничать так, чтобы:

- лимит нельзя было превысить;
- неудачная выдача не меняла ни один объект;
- успешный возврат уменьшал число выдач;
- отрицательный лимит был невозможен.

Сначала письменно определите, какой объект отвечает за координацию. Допустим отдельный `LibraryService`; не помещайте случайно половину одного инварианта в консоль.

## Этап 3. Композиция

Создайте `LoanPolicy`, которая определяет максимальный срок выдачи в днях. Передайте её координирующему сервису через конструктор. Сервис не должен создавать policy внутри метода.

Интерфейс для policy пока необязателен — он появится в уроке 04. Достаточно композиции с конкретным классом.

## Этап 4. Наблюдаемые сценарии

Создайте отдельное console application для демонстрации:

1. успешная выдача;
2. повторная выдача того же экземпляра;
3. достижение лимита читателя;
4. успешный возврат;
5. повторная выдача после возврата;
6. попытка создать объект с пустыми данными.

Console project может печатать результат, но предметные правила находятся в классах библиотеки.

## Проверка без test framework

До отдельного урока о тестировании используйте таблицу:

| Начальное состояние | Операция | Ожидаемый результат | Состояние после |
|---|---|---|---|
| Экземпляр свободен | Выдать Анне | Успех | Выдан Анне |
| Выдан Анне | Выдать Борису | Отказ | По-прежнему у Анны |
| Лимит достигнут | Выдать ещё один | Отказ | Состояние не меняется |
| Выдан Анне | Вернуть | Успех | Свободен |

Добавьте фактические результаты и подтвердите их запуском.

## Ограничения

- никаких public fields;
- никаких public setters, позволяющих обойти правила;
- объект корректен сразу после конструктора;
- не использовать наследование только ради повторного кода;
- не создавать глобальное mutable static state;
- зависимости конструктора проверяются на `null`, если допускается ссылочный тип.

## Критерии приёмки

- solution собирается;
- все шесть сценариев воспроизводятся;
- инварианты нельзя нарушить обычным публичным вызовом;
- классы имеют объяснимые ответственности;
- композиция используется минимум один раз;
- README содержит диаграмму или список объектов, их ответственности и инвариантов;
- ученик объясняет различия class, object, property, constructor и method.

## Дополнительное задание

Добавьте `record Address` для адреса читателя и покажите равенство двух адресов с одинаковыми данными. Затем добавьте в record изменяемый список и объясните, почему value equality не делает содержимое глубоко неизменяемым. Эксперимент не включайте в основную модель, если он ухудшает её.
