# Урок 03. Классы и объектно-ориентированное программирование

> В прошлом уроке данные лежали в отдельных переменных, а действия — в отдельных методах. Класс позволяет описать предмет задачи целиком: какие данные у него есть, какие действия разрешены и какие правила никогда нельзя нарушать.

## Что нужно знать

Перед уроком убедитесь, что понимаете тип, переменную, метод, параметр, условие и возвращаемое значение. Все эти понятия введены в уроке 02.

## Результаты обучения

Вы сможете:

- различать класс, объект и ссылку на объект;
- объявлять поля, свойства, методы и конструкторы;
- управлять доступом через `public`, `private`, `protected` и `internal`;
- защищать состояние объекта и его инварианты;
- объяснять абстракцию, инкапсуляцию, наследование и полиморфизм;
- выбирать композицию вместо необоснованного наследования;
- различать class, struct и record;
- передавать обязательные зависимости через конструктор.

## 1. От разрозненных переменных к объекту

Представим банковский счёт:

```csharp
string accountNumber = "BY01";
decimal balance = 100m;
```

Отдельные переменные не защищают правило: баланс нельзя уменьшить ниже нуля. Любая часть программы может случайно записать `balance = -500m`.

Класс объединяет состояние и допустимое поведение:

```csharp
public sealed class BankAccount
{
    public string Number { get; }
    public decimal Balance { get; private set; }

    public BankAccount(string number, decimal initialBalance)
    {
        if (string.IsNullOrWhiteSpace(number))
            throw new ArgumentException("Номер счёта обязателен", nameof(number));
        if (initialBalance < 0)
            throw new ArgumentOutOfRangeException(nameof(initialBalance));

        Number = number;
        Balance = initialBalance;
    }

    public void Deposit(decimal amount)
    {
        if (amount <= 0)
            throw new ArgumentOutOfRangeException(nameof(amount));

        Balance += amount;
    }

    public bool TryWithdraw(decimal amount)
    {
        if (amount <= 0 || amount > Balance)
            return false;

        Balance -= amount;
        return true;
    }
}
```

**Класс** — описание нового типа. **Объект** — конкретный экземпляр этого класса во время выполнения.

```csharp
var account = new BankAccount("BY01", 100m);
account.Deposit(50m);
bool withdrawn = account.TryWithdraw(30m);
Console.WriteLine(account.Balance); // 120
```

`new` создаёт объект и вызывает конструктор. Переменная `account` хранит ссылку, по которой объект доступен.

## 2. Состояние и поведение

**Состояние** — данные объекта в конкретный момент. У счёта это номер и баланс. **Поведение** — разрешённые действия: пополнение и снятие.

Членами класса могут быть:

- **поле** — внутренняя переменная объекта;
- **свойство** — контролируемый способ читать или изменять значение;
- **метод** — действие;
- **конструктор** — код начального создания;
- событие, индексатор и вложенный тип — они появятся позже.

Простое auto-property:

```csharp
public string Number { get; }
public decimal Balance { get; private set; }
```

`get` разрешает чтение. `private set` разрешает изменение только коду самого класса. Пользователь может увидеть баланс, но не выполнить `account.Balance = -1000m`.

Поле нужно, когда свойству требуется дополнительная логика:

```csharp
private string _ownerName;

public string OwnerName
{
    get => _ownerName;
    set
    {
        if (string.IsNullOrWhiteSpace(value))
            throw new ArgumentException("Имя владельца обязательно");
        _ownerName = value.Trim();
    }
}
```

`value` внутри setter — присваиваемое значение. Поля обычно делают `private`, чтобы внешний код не обходил проверку свойства.

## 3. Конструктор и корректное начало жизни

Конструктор называется как класс и не имеет возвращаемого типа. Его задача — не просто присвоить поля, а создать корректный объект.

```csharp
var invalid = new BankAccount("", -10m); // конструктор выбросит исключение
```

Если обязательное значение отсутствует, лучше отказаться от создания, чем выпустить объект, который все последующие методы вынуждены проверять.

У класса может быть несколько конструкторов, но перегрузки должны приводить к одному набору правил:

```csharp
public BankAccount(string number)
    : this(number, 0m)
{
}
```

`: this(...)` вызывает другой конструктор этого же класса и не дублирует валидацию.

## 4. Инвариант

**Инвариант** — правило, которое остаётся истинным после создания и любой публичной операции. Для счёта:

- номер не пуст;
- начальный баланс не отрицателен;
- пополнение положительно;
- снятие не делает баланс отрицательным.

Инвариант отличается от случайной проверки интерфейса. Если консоль проверила сумму, другой клиент всё равно может вызвать класс напрямую. Поэтому предметное правило защищает сам объект.

## 5. Модификаторы доступа

Модификатор задаёт, откуда виден член:

| Модификатор | Упрощённый смысл |
|---|---|
| `public` | доступен любому клиенту типа |
| `private` | доступен только внутри этого класса |
| `protected` | доступен классу и наследникам |
| `internal` | доступен внутри текущей сборки/проекта |
| `protected internal` | объединённые правила `protected` или `internal` |
| `private protected` | наследник только в той же сборке |

Не делайте всё `public` ради удобства. Публичный член становится обещанием клиентам и позже меняется сложнее. Начинайте с минимально необходимого доступа.

## 6. Инкапсуляция и абстракция

**Инкапсуляция** — объект сам отвечает за своё состояние и скрывает решения, которые клиенту не нужны. Это не только `private`: класс с двадцатью публичными сеттерами технически скрывает поля, но не защищает правила.

**Абстракция** — представление только существенных свойств задачи. Клиент счёта вызывает `TryWithdraw(30m)`, а не получает инструкции «прочитай баланс, сравни, вычисли новый, запиши».

Аналогия: банкомат предлагает операции «снять» и «показать баланс», не раскрывая таблицы базы данных. Аналогия заканчивается там, где реальная банковская система имеет сеть, безопасность и транзакции — учебный класс этого пока не моделирует.

## 7. Ссылочная семантика класса

Класс — ссылочный тип. Две переменные могут указывать на один объект:

```csharp
var first = new BankAccount("BY01", 100m);
var second = first;

second.Deposit(50m);
Console.WriteLine(first.Balance); // 150
```

Копируется ссылка, а не сам счёт. Изменение через `second` видно через `first`.

`null` означает отсутствие ссылки:

```csharp
BankAccount? account = null;
```

Обращение `account.Balance` без объекта вызовет `NullReferenceException`. Nullable-аннотация `?` предупреждает компилятор, что отсутствие допустимо и его нужно проверить.

## 8. `static` и члены экземпляра

Член экземпляра относится к конкретному объекту: у каждого счёта свой баланс. `static` относится к самому типу:

```csharp
public static int CreatedCount { get; private set; }
```

Обращение выполняется как `BankAccount.CreatedCount`. Статическое состояние общее для всего процесса, поэтому изменяемые static-поля требуют осторожности: они скрывают зависимость, мешают изоляции тестов и создают проблемы конкурентного доступа.

## 9. Наследование

Наследование описывает отношение «является разновидностью»:

```csharp
public abstract class Shape
{
    public abstract double CalculateArea();
}

public sealed class Rectangle(double width, double height) : Shape
{
    public override double CalculateArea() => width * height;
}

public sealed class Circle(double radius) : Shape
{
    public override double CalculateArea() => Math.PI * radius * radius;
}
```

- `abstract class` нельзя создать напрямую;
- `abstract`-метод задаёт обязательство без реализации;
- `override` реализует или заменяет виртуальное поведение;
- `sealed` запрещает дальнейшее наследование.

Код может работать с общим типом:

```csharp
double PrintArea(Shape shape)
{
    double area = shape.CalculateArea();
    Console.WriteLine(area);
    return area;
}
```

Фактический метод выбирается по реальному объекту. Это **динамический полиморфизм**.

Наследование корректно, если любой объект потомка можно использовать вместо базового без нарушения обещаний. Квадрат как наследник изменяемого прямоугольника часто ломает ожидания независимой установки ширины и высоты — известный пример неправильной иерархии.

## 10. Композиция

**Композиция** означает: объект содержит другой объект и поручает ему часть работы.

```csharp
public sealed class StatementPrinter
{
    public string Print(BankAccount account)
    {
        return $"Счёт {account.Number}: {account.Balance}";
    }
}
```

Счёт не обязан наследоваться от принтера. Отношение здесь «использует», а не «является». Композиция обычно удобнее, если поведение нужно заменять независимо или если типы не образуют естественную иерархию.

Правило «предпочитайте композицию» не запрещает наследование. Оно просит сначала проверить, действительно ли подтип сохраняет контракт и нужна ли общая иерархия.

## 11. Полиморфизм, перегрузка и переопределение

Три похожих слова:

- **перегрузка (overload)** — методы имеют одно имя, но разные параметры; выбор делает компилятор;
- **переопределение (override)** — наследник заменяет виртуальное поведение; выбор зависит от объекта во время выполнения;
- **полиморфизм** — клиент работает через общий контракт, а разные объекты ведут себя по-своему.

## 12. Class, struct и record

### `class`

Ссылочная семантика, identity и совместно наблюдаемое состояние. Подходит для сущностей и сервисов: счёт остаётся тем же счётом при изменении баланса.

### `struct`

Значимый тип: присваивание обычно копирует значение. Подходит для небольшого самостоятельного значения без отдельной identity, например координаты. Изменяемая крупная структура создаёт неожиданные копии, поэтому структуры обычно делают маленькими и неизменяемыми.

### `record`

Record автоматически даёт равенство по значениям и удобную печать/копирование через `with`. Подходит для value-like данных и сообщений:

```csharp
public sealed record Address(string City, string Street);
```

Два адреса с одинаковыми компонентами считаются равными. Обычные классы по умолчанию сравниваются по ссылке. Record не делает вложенный изменяемый список неизменяемым автоматически.

## 13. Зависимости и их передача

Если сервису нужны часы, хранилище или отправитель писем, это его **зависимости**. Не следует незаметно создавать их внутри каждого метода.

```csharp
public sealed class AccountService
{
    private readonly IClock _clock;

    public AccountService(IClock clock)
    {
        _clock = clock ?? throw new ArgumentNullException(nameof(clock));
    }
}
```

Constructor injection делает обязательную зависимость видимой и не допускает создание сервиса без неё. Интерфейсы подробно изучаются в следующем уроке; пока важно направление: объект получает нужного сотрудника извне, а не ищет его глобально.

## Частые ошибки

- класс служит мешком публичных свойств, а правила разбросаны снаружи;
- объект можно создать некорректным и «донастроить потом»;
- setter принимает значение, нарушающее инвариант;
- наследование выбирается только ради повторного использования нескольких строк;
- virtual-метод вызывается из конструктора до завершения создания наследника;
- mutable static state используется как скрытое глобальное хранилище;
- крупная изменяемая структура копируется в неожиданных местах;
- класс самостоятельно создаёт файловую систему, часы и сеть, скрывая зависимости.

## Итоговая карта

```text
Класс описывает тип
  ├─ состояние: поля и свойства
  ├─ поведение: методы
  └─ корректное начало: конструктор

Объект — экземпляр класса
  └─ публичные операции сохраняют инварианты

Связи объектов
  ├─ «является» → наследование при подстановке
  └─ «использует/содержит» → композиция
```
