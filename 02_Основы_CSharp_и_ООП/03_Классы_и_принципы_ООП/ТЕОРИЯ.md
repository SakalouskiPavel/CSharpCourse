# Урок 03. Классы как воплощение принципов ООП

> Класс полезен не как контейнер полей, а как граница ответственности и носитель инвариантов. В центре урока — состояние, поведение, инкапсуляция, композиция, наследование и полиморфизм.

## Результаты обучения

- создавать типы с корректным состоянием
- разделять абстракцию, инкапсуляцию, наследование и полиморфизм
- выбирать композицию или наследование осознанно
- понимать различия class, struct и record

## Карта темы

### 1. Объект и инвариант

Объект объединяет состояние и операции, которые сохраняют его допустимость. Конструктор должен создать корректное состояние; публичные сеттеры опасны, если позволяют обойти правила.

### 2. Инкапсуляция

Модификаторы доступа — инструмент, но смысл инкапсуляции в сокрытии решений, способных измениться. Публичный контракт должен быть меньше реализации.

### 3. Абстракция

Абстракция оставляет свойства, значимые для задачи. Хороший тип говорит на языке предметной области и не заставляет вызывающий код координировать его внутренние поля.

### 4. Наследование

Наследование выражает отношение подстановки, а не просто повторное использование кода. Базовый класс определяет контракт для наследников; `virtual`, `abstract`, `override` и `sealed` управляют полиморфизмом.

### 5. Композиция

Композиция делегирует часть поведения отдельному объекту и обычно слабее связывает изменения. Предпочтение композиции не абсолютное, но для сменяемых политик она безопаснее глубокой иерархии.

### 6. Полиморфизм

Код работает через общий контракт, а фактическое поведение выбирается по runtime-типу. Перегрузка — статический полиморфизм, переопределение — динамический.

### 7. Классы, структуры и records

Класс имеет ссылочную семантику, структура — значимую и должна быть небольшой/неизменяемой. Records дают value-based equality и удобны для данных, но не отменяют проектирование инвариантов.

### 8. Жизненный цикл и зависимости

Конструктор явно получает обязательные зависимости. Скрытое создание инфраструктуры внутри доменного объекта осложняет тестирование и связывает уровни.

## Минимальный пример

```csharp
public sealed class BankAccount
{
    public decimal Balance { get; private set; }

    public void Deposit(decimal amount)
    {
        if (amount <= 0) throw new ArgumentOutOfRangeException(nameof(amount));
        Balance += amount;
    }

    public void Withdraw(decimal amount)
    {
        if (amount <= 0 || amount > Balance)
            throw new InvalidOperationException("Invalid withdrawal");
        Balance -= amount;
    }
}
```

Разберите пример построчно: какие гарантии выражены типами, где проверяются предусловия и что увидит вызывающий код. Затем измените один инвариант и проследите, какая часть публичного контракта должна поменяться.

## Типичные ошибки

- анемичная модель с публичными сеттерами и правилами снаружи;
- наследование ради повторного использования пары строк;
- вызов виртуальных методов из конструктора;
- крупные изменяемые структуры;

## Итог

Тема считается усвоенной, когда вы можете не только применить механизм, но и объяснить его цену, область корректности и более простой альтернативный вариант. Проверяйте решения сборкой, тестами и небольшими экспериментами, а не только чтением.
