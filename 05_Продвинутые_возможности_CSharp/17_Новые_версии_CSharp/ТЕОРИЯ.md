# Урок 17. Современный C# и безопасное обновление версии

> Новая версия языка полезна не количеством сокращений, а тем, какие ошибки она делает невозможными и насколько ясно выражает намерение. Обновление начинается с понимания SDK, target framework и language version — это три связанные, но разные вещи.

## Актуальная точка курса

На дату обновления главы (июль 2026 года) последняя стабильная версия — **C# 14**, выпущенная в ноябре 2025 года и связанная по умолчанию с .NET 10. Preview-возможности следующей версии могут существовать, но курс не использует их как обязательную основу.

Перед практикой сверяйтесь с:

- [историей версий C#](https://learn.microsoft.com/en-us/dotnet/csharp/whats-new/csharp-version-history);
- [настройкой версии языка](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/configure-language-version);
- [нововведениями C# 14](https://learn.microsoft.com/en-us/dotnet/csharp/whats-new/csharp-14);
- [политикой поддержки .NET](https://dotnet.microsoft.com/en-us/platform/support/policy/dotnet-core).

## 1. SDK, compiler, TFM и runtime

### SDK

.NET SDK содержит compiler Roslyn, `dotnet` CLI, build targets и reference packs. `dotnet --list-sdks` показывает установленные SDK.

### TFM

Target Framework Moniker в `.csproj`:

```xml
<TargetFramework>net10.0</TargetFramework>
```

TFM задаёт набор API и runtime contract, под который компилируется проект. Код может собираться SDK 10, но targeting более старый TFM при наличии reference pack.

### Language version

C# version определяет доступный синтаксис и language rules. По умолчанию SDK выбирает версию, согласованную с TFM. Microsoft рекомендует это соответствие: более новый синтаксис иногда зависит от runtime/library members.

### Runtime

Runtime выполняет приложение. Framework-dependent app требует совместимый runtime на машине; self-contained publication включает выбранный runtime.

Новая language feature не всегда требует новую runtime instruction, но может требовать library types/attributes. Поэтому «compiler принял» не единственная проверка deployment.

## 2. `LangVersion`

Явная настройка:

```xml
<PropertyGroup>
  <LangVersion>14.0</LangVersion>
</PropertyGroup>
```

Обычно оставляйте default, соответствующий TFM. Не задавайте `latest` в воспроизводимом проекте: на разных машинах он означает возможности установленного compiler-а и способен менять сборку без изменения repository.

`preview` допустим для эксперимента, но не production baseline без решения команды. Preview syntax/semantics могут измениться; tooling и libraries не обязаны быть готовы.

Проверить выбранную версию можно временной директивой:

```csharp
#error version
```

Compiler выдаст diagnostic с собственной и language version. После проверки директиву удаляют.

## 3. `global.json`

```json
{
  "sdk": {
    "version": "10.0.100",
    "rollForward": "latestPatch"
  }
}
```

`global.json` управляет выбором SDK CLI, а не target runtime напрямую. Политика roll-forward определяет допустимую замену отсутствующей версии.

Слишком жёсткая фиксация без процесса обновления блокирует security patches; слишком свободная делает сборки разными. Команда выбирает policy и обновляет SDK осознанно.

## 4. Nullable reference types (C# 8+)

```csharp
string requiredName = "Анна";
string? optionalMiddleName = null;
```

Flow analysis отслеживает возможный null. Оператор `!` подавляет warning, но не выполняет runtime check:

```csharp
string definitely = maybe!; // null всё ещё возможен во время выполнения
```

При миграции старого проекта:

1. включайте nullable по module/layer;
2. исправляйте public contracts;
3. добавляйте guards/annotations;
4. не заменяйте каждый warning на `!`;
5. проверяйте deserialization/Reflection/ORM boundaries.

Nullable annotations становятся частью API для compiler consumers и влияют на качество downstream analysis.

## 5. Records, `init` и `with` (C# 9+)

```csharp
public sealed record Address(string City, string Street);

Address first = new("Минск", "Независимости");
Address moved = first with { Street = "Победителей" };
```

Record class имеет value-based equality и удобное non-destructive copying. `with` создаёт shallow copy: вложенный mutable List может оставаться общим.

`init` позволяет assignment только при construction/object initializer/with:

```csharp
public string Name { get; init; } = "";
```

Это не заменяет constructor/factory, если несколько properties должны вместе образовывать invariant.

`record struct` даёт value-type record semantics. Выбор class/struct по-прежнему определяется identity, копированием, размером и lifetime.

## 6. Top-level statements и упрощение структуры

```csharp
Console.WriteLine("Hello");
```

Compiler создаёт entry point. В проекте может быть только один файл с top-level statements, и они должны предшествовать type declarations в этом compilation unit.

Top-level удобен для маленьких console apps, но большой application всё равно нуждается в composition root и методах. Скрытая generated Program type учитывается некоторыми test/hosting scenarios.

## 7. File-scoped namespace и global usings (C# 10+)

```csharp
namespace Course.Catalog;

public sealed class Book
{
}
```

File-scoped namespace уменьшает indentation и допускается один на файл.

```csharp
global using System.Text.Json;
```

Global using доступен всем файлам compilation. Он уместен для действительно общих namespaces. Если добавить десятки domain/infrastructure namespaces, происхождение types становится неясным и растёт риск конфликтов.

ImplicitUsings SDK генерирует часть global usings в зависимости от project SDK.

## 8. Pattern matching

Patterns проверяют форму и одновременно извлекают данные:

```csharp
string Describe(object? value) => value switch
{
    null => "значение отсутствует",
    string { Length: 0 } => "пустая строка",
    string text => $"текст длиной {text.Length}",
    int number and >= 0 and <= 100 => "процент",
    _ => "другое значение"
};
```

Современные patterns:

- type/property/positional;
- relational `<`, `>=`;
- logical `and`, `or`, `not`;
- list patterns `[first, .. rest]`;
- recursive combinations.

Switch expression читается хорошо для ограниченного набора вариантов. Сложные guards и side effects лучше вынести в методы. Exhaustiveness warning помогает, но open class hierarchy не становится автоматически закрытым algebraic data type.

## 9. `required` members (C# 11+)

```csharp
public sealed class CreateBookRequest
{
    public required string Title { get; init; }
    public required string Author { get; init; }
}
```

Compiler требует assignment в object initializer/constructor contract. `required`:

- не проверяет непустую строку;
- не защищает Reflection/deserializer автоматически во всех режимах;
- не выражает связь двух properties;
- может усложнить compatibility при добавлении в public type.

Для сильного invariant constructor/factory всё ещё лучше.

## 10. Raw string literals и UTF-8 literals (C# 11+)

```csharp
string json = """
{
  "course": "C#",
  "active": true
}
""";
```

Raw string уменьшает escaping. Число delimiter quotes выбирают больше максимальной серии кавычек внутри. Indentation результата определяется closing delimiter.

```csharp
ReadOnlySpan<byte> header = "Content-Type: application/json\r\n"u8;
```

`u8` создаёт compile-time UTF-8 data view и полезен для protocol/performance code. Это не null-terminated native string и не `byte[]`; учитывайте target API/lifetime.

## 11. Primary constructors (C# 12+)

```csharp
public sealed class DistanceCalculator(double scale)
{
    public double Convert(double value) => value * scale;
}
```

Parameters доступны во всём type body. Compiler создаёт storage, только если значение нужно сохранить/capture; имя parameter не становится public property автоматически.

Для class primary constructor не гарантирует invariant и не запрещает дополнительные constructors. Если validation/ownership сложны, обычный explicit constructor может читаться лучше.

## 12. Collection expressions (C# 12+)

```csharp
int[] first = [1, 2, 3];
List<int> second = [1, 2, 3];
int[] combined = [0, .. first, 4];
```

Expression target-typed: один синтаксис создаёт подходящее collection-like представление. `..` spreads элементы.

Не путайте с lazy concatenation: результат/стратегия зависит от target. Большие spreads могут копировать данные. Явный constructor остаётся полезен, если capacity/comparer/ownership важны.

## 13. C# 13: `params` collections и low-level улучшения

`params` больше не ограничен массивом и может использовать поддерживаемые collection types, включая Span/interface forms:

```csharp
static void PrintAll(params ReadOnlySpan<string> values)
{
    foreach (string value in values)
        Console.WriteLine(value);
}
```

C# 13 также расширил работу `ref struct`/`ref` в async/iterator contexts с lifetime restrictions, добавил новый `System.Threading.Lock`-ориентированный lock behavior и другие возможности.

Не переписывайте API на новый params type без проверки overload resolution, allocations, binary/source compatibility и потребностей callers.

## 14. C# 14: extension members

До C# 14 был classic extension method:

```csharp
public static bool HasText(this string? value) =>
    !string.IsNullOrWhiteSpace(value);
```

C# 14 добавляет extension blocks, позволяющие группировать methods/properties и объявлять static extension members:

```csharp
public static class StringExtensions
{
    extension(string value)
    {
        public bool HasText => !string.IsNullOrWhiteSpace(value);
    }
}
```

Это всё ещё статически разрешаемые внешние members: type string не меняется, private state недоступно, instance members имеют приоритет, конфликты namespaces возможны.

Используйте новую форму, если property/member group действительно улучшает API. Не превращайте extension block в свалку бизнес-логики.

## 15. C# 14: `field`-backed properties

```csharp
public string Name
{
    get;
    set => field = string.IsNullOrWhiteSpace(value)
        ? throw new ArgumentException("Имя обязательно", nameof(value))
        : value.Trim();
}
```

`field` обращается к скрытому backing field auto-property, избавляя от ручного `_name`. Это contextual keyword в accessor contexts.

Если в старом коде есть field/property с именем `field`, обновление может создать неоднозначность; используйте `this.field` для настоящего member и проверяйте warnings. Сложная property logic всё равно может требовать явное поле/методы.

## 16. Другие возможности C# 14

- null-conditional assignment позволяет присваивание через `?.`/`?[]` только если receiver не null;
- `nameof(List<>)` поддерживает unbound generic types;
- дополнительные implicit conversions для Span/ReadOnlySpan;
- modifiers на simple lambda parameters;
- partial constructors/events для generated code scenarios;
- user-defined compound assignment operators.

Каждую возможность изучайте по конкретному use case и официальной reference page. «Поддерживается C# 14» не означает, что её нужно применить в каждом проекте.

## 17. Language, library и runtime dependency

Некоторые features — в основном syntax lowering и работают на старом runtime; другие требуют attributes/types/methods стандартной библиотеки.

Например, `required` использует metadata attributes, generic math — interfaces библиотеки, UTF-8 literal target — Span APIs. Принудительно задать новый LangVersion при старом TFM может потребовать packages/polyfills и не является поддерживаемым стандартным сочетанием согласно документации Microsoft.

Поэтому обновление рассматривает:

- SDK/compiler;
- TFM/runtime availability;
- NuGet dependencies;
- analyzers/source generators;
- build agents/IDE;
- deployment environment.

## 18. Стратегия миграции

1. Зафиксировать текущую сборку/tests.
2. Обновить SDK/build agents и проверить release notes.
3. Обновить TFM отдельно от массового синтаксического рефакторинга.
4. Устранить warnings/obsolete APIs.
5. Включать nullable по границам.
6. Применять новые features маленькими commits.
7. Не смешивать functional change и cosmetic modernization.
8. Проверить publish/deployment, а не только IDE build.
9. Записать rejected features и team conventions.

## 19. Как оценивать новую конструкцию

Задайте вопросы:

- Какую ошибку/шум она уменьшает?
- Понятна ли команде и поддерживается ли tooling?
- Меняет ли public API/source/binary compatibility?
- Есть ли hidden allocation/copy/lifetime?
- Улучшает ли contract или только сокращает строки?
- Можно ли внедрить локально и откатить?

Новый синтаксис не является целью рефакторинга. Иногда старый explicit code лучше обучает и яснее показывает ownership.

## Частые ошибки

- SDK, TFM, language и runtime считаются одной версией;
- LangVersion `latest` делает build зависимым от машины;
- preview включён транзитивно во все проекты;
- nullable warnings подавлены `!`;
- record считается глубоко immutable;
- required считается runtime validation;
- primary constructor parameter ошибочно принимается за property;
- collection expression скрывает важный comparer/capacity;
- C# 14 extensions используются для unrelated logic;
- проект переписывается целиком ради синтаксиса без behavioral tests.

## Итог

```text
SDK выбирает compiler/build tools
TFM выбирает доступный platform contract
LangVersion выбирает syntax/rules
Runtime выполняет опубликованное приложение

Новая feature полезна, если усиливает контракт
и уменьшает реальный класс ошибок.
```

Вы готовы к следующему блоку, если можете объяснить четыре версии, закрепить воспроизводимую сборку и оценить nullable/record/pattern/extension feature по пользе и цене.
