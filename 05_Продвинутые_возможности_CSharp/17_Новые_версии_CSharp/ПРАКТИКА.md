# Практика — урок 17

## Цель

Модернизировать небольшой legacy-модуль каталога, сохраняя поведение. Каждое изменение должно усиливать контракт или устранять конкретный шум, а не просто использовать новый синтаксис.

## Базовая линия

Создайте/возьмите модуль со следующими намеренными особенностями:

- nullable выключен;
- DTO — mutable class с public setters;
- длинный if/else по типу/состоянию;
- namespace block и повторяющиеся usings;
- JSON/SQL sample со сложным escaping;
- ручной backing field с простой validation;
- helper extension method старой формы;
- массивы создаются обычными initializers.

Сначала зафиксируйте поведение characterization scenarios. Модернизация без baseline не принимается.

## Версии

Для текущего выполнения курса используйте поддерживаемый .NET 10 SDK/TFM и стабильный C# 14 либо обновите значения по официальной support policy на дату прохождения. Зафиксируйте SDK policy через global.json с обоснованным rollForward.

Не используйте `latest`/`preview` в основном варианте.

## Обязательные изменения

1. Включить nullable и исправить warnings без необоснованных `!`.
2. Преобразовать value-like DTO в record либо письменно оставить class из-за identity/mutability.
3. Использовать exhaustive/readable pattern matching вместо подходящей условной цепочки.
4. Применить raw string к JSON/SQL sample.
5. Использовать required/init только там, где они не заменяют domain validation.
6. Перевести один namespace в file-scoped и определить небольшой набор global usings.
7. Применить collection expression с анализом target/copy.
8. Рассмотреть primary constructor; принять или отклонить письменно.
9. Переписать одну простую property на C# 14 `field`, проверив конфликт имени.
10. Сравнить classic extension method и C# 14 extension block; выбрать основной вариант.

## Matrix решений

Для каждой feature заполните:

| Feature | Проблема до | Изменение | Риск/цена | Принято? |
|---|---|---|---|---|

Отклонённая feature с хорошим обоснованием считается корректным результатом.

## Проверки

- behavior scenarios до/после совпадают;
- nullable boundary проверена null inputs/Reflection/JSON;
- record nested mutable data не выдаётся за deep immutable;
- required не заменяет пустую-string validation;
- pattern matching обрабатывает unknown case;
- build выполняется на clean environment выбранным SDK;
- publish/run проверены, не только editor analysis.

## Compatibility experiment

В отдельной ветке/commit временно:

- задайте LangVersion ниже нужной feature и сохраните diagnostic;
- выполните `#error version` и удалите directive;
- попробуйте `latest`, объясните зависимость от installed SDK и верните stable setting.

## Ограничения

- нет массового formatting/refactoring вместе с behavioral change;
- нет preview в main solution;
- нет подавления nullable warnings без justification;
- новые extensions не скрывают I/O/side effects;
- global usings не импортируют случайно все layers;
- feature применяется только после описания пользы.

## Критерии приёмки

- SDK/TFM/LangVersion/runtime объяснены отдельно;
- clean build воспроизводим;
- все обязательные features рассмотрены;
- behavioral baseline сохранён;
- migration commits малы и тематичны;
- README содержит decision matrix и rejected choices;
- ссылки ведут на official versioned documentation.

## Дополнительное задание

Создайте отдельный playground с доступной preview C# следующей версии, исследуйте одну feature по official proposal и не подключайте playground к production projects. Запишите, что может измениться до release и какой fallback существует.
