# Практика — урок 13

## Цель

Создать монитор датчиков, в котором правила фильтрации передаются делегатами, а изменения публикуются событиями с управляемой подпиской.

## Модель

`TemperatureSensor` получает измерения и хранит последнее значение. При реальном изменении публикует `TemperatureChanged` с old/new value и timestamp.

Событие вызывается после успешного изменения состояния. Отрицать измерение нельзя через subscriber: validation выполняется до события.

## Делегаты

Создайте:

- именованный `MeasurementFilter(decimal value) -> bool`;
- `Func<decimal, string>` formatter;
- минимум три filter-а, включая lambda с захваченным threshold;
- метод композиции двух predicates через AND.

Покажите, что изменение захваченного threshold меняет результат closure. Затем создайте вариант со снимком значения и объясните отличие.

## Событие

Используйте `EventHandler<TemperatureChangedEventArgs>`. Подписчики:

1. console display;
2. in-memory history;
3. alert subscriber для порога.

Проверьте порядок вызова, но не делайте его предметной гарантией без документации.

## Управление lifetime

Сделайте `AlertSubscription : IDisposable`, который подписывается при создании и отписывается в Dispose. Повторный Dispose безопасен.

Докажите через счётчик/WeakReference-эксперимент либо схему ссылок, почему отсутствие отписки удерживает subscriber. Не вызывайте `GC.Collect` в production-логике; только контролируемый учебный эксперимент допустим.

## Ошибки handlers

Создайте subscriber, выбрасывающий исключение. Реализуйте два явно разделённых режима publisher-а:

- fail-fast стандартным Invoke;
- isolated notification, вызывающий invocation list и собирающий ошибки.

README объясняет, какой режим выбран основным и почему. Ошибки не проглатываются.

## Ограничения

- внешний код не может вызвать/заменить event;
- publisher не держит lock во время callbacks;
- async void handlers отсутствуют;
- disposable subscription всегда освобождается;
- delegate names и parameter names выражают смысл;
- событие не заменяет durable messaging.

## Критерии приёмки

- фильтры и formatter действительно передаются как values;
- closure experiment воспроизводим;
- три subscribers получают корректные EventArgs;
- отписка прекращает уведомления;
- оба режима ошибок проверены;
- invocation list не используется для скрытого возврата результатов;
- README сравнивает delegate, interface и event.

## Дополнительное задание

Создайте отдельный `AsyncEventHandler<T>` и последовательный publisher, который await-ит всех handlers, поддерживает cancellation и собирает ошибки. Не смешивайте его с обычным EventHandler.
