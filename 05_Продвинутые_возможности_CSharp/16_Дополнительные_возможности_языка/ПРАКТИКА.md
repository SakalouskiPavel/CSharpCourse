# Практика — урок 16

## Цель

Реализовать безопасный parser бинарного заголовка без промежуточных массивов. Сначала — на `ReadOnlySpan<byte>` и `BinaryPrimitives`; unsafe-вариант только как изолированный дополнительный эксперимент.

## Формат

Определите versioned header фиксированной части:

| Поле | Размер | Порядок |
|---|---:|---|
| Magic `CSH1` | 4 bytes | как записано |
| Version | 1 byte | — |
| Flags | 1 byte | bit flags |
| PayloadLength | 4 bytes | big-endian |
| Timestamp | 8 bytes | big-endian Unix seconds |

Зафиксируйте максимальный payload length и допустимые versions/flags.

## Safe parser

```csharp
bool TryParseHeader(
    ReadOnlySpan<byte> source,
    out MessageHeader header,
    out HeaderError error)
```

Требования:

- проверить минимальную длину до каждого slice/read;
- проверить magic без создания string, где разумно;
- явно использовать big-endian operations;
- проверить version, unknown flags, payload bounds и timestamp;
- обычный неверный input возвращает error, а не читает за границей;
- `MessageHeader` не хранит Span и может жить после метода.

## Writer и round trip

Реализуйте `TryWriteHeader(MessageHeader, Span<byte>, out int written)`. Он не пишет частичный header при слишком маленьком destination либо чётко документирует другое атомарное поведение.

Проверьте parse(write(value)) для нескольких значений.

## Stackalloc

В console demo используйте фиксированный `stackalloc byte[HeaderSize]`. Размер не приходит от пользователя. После заполнения запишите buffer в файл/stream до выхода из scope.

## Checked arithmetic

Вычисление общей длины `header + payload` выполняйте checked либо через безопасное сравнение до сложения. Добавьте сценарий переполнения/слишком большого payload.

## Conditional compilation

Добавьте только диагностический вывод под собственным symbol `BINARY_TRACE`. Обе конфигурации обязаны собираться. В README объясните, почему parser behavior не должен различаться.

## Обязательные сценарии

- корректный header;
- source короче каждого критического offset;
- неверный magic;
- неизвестная version;
- unknown flags;
- payload больше лимита;
- big-endian значение проверено известной byte sequence;
- destination writer слишком мал;
- checked overflow/граница.

## Ограничения

- no unsafe в основной реализации;
- no BitConverter без явного анализа endianness;
- no `ToArray`/string allocation в parsing path без причины;
- Span не сохраняется в class/async operation;
- пользовательский размер не передаётся в stackalloc;
- warnings не подавляются глобально.

## Критерии приёмки

- parser не читает вне source;
- endianness подтверждён tests/examples;
- ошибки различимы;
- writer/reader round trip работает;
- обе symbol configurations собираются;
- README описывает lifetime и safe API boundary;
- простой baseline и allocation assumptions объяснены.

## Дополнительное задание

Создайте минимальный unsafe parser в одном internal method, вызываемый безопасным wrapper-ом после полной проверки длины. Сравните benchmark с Span-вариантом. Если измеримого выигрыша нет, оставьте Span основной реализацией.
