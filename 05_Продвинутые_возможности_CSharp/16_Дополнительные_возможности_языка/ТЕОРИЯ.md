# Урок 16. Дополнительные и низкоуровневые возможности C#

> Большую часть прикладного кода можно писать без указателей и условной компиляции. Эти инструменты нужны на границах платформ, в interop, системных библиотеках и измеренных горячих участках. Чем ниже уровень, тем больше гарантий становится ответственностью разработчика.

## 1. Директивы препроцессора

Строки, начинающиеся с `#`, обрабатываются compiler-ом:

```csharp
#if DEBUG
Console.WriteLine("Диагностический режим");
#endif
```

C# preprocessor не выполняет текстовые macro substitutions как C/C++. Он включает/исключает области и управляет compiler information.

Основные директивы:

- `#if`, `#elif`, `#else`, `#endif` — conditional compilation;
- `#define`, `#undef` — symbols внутри файла до обычного кода;
- `#error`, `#warning` — сообщение сборки;
- `#nullable` — nullable context;
- `#pragma warning` — точечное управление diagnostics;
- `#line` — source mapping для generated code;
- `#region` — folding hints редактору.

## 2. Compilation symbols

Symbols задаются project configuration:

```xml
<PropertyGroup>
  <DefineConstants>FEATURE_IMPORT;PLATFORM_X</DefineConstants>
</PropertyGroup>
```

`DEBUG` часто определяется Debug configuration; `TRACE`/другие зависят от проекта.

Conditional compilation подходит, если код действительно нельзя собрать/поставить на платформе или нужен compile-time variant. Для включения бизнес-функции в уже развёрнутом приложении лучше configuration/feature flag: `#if` требует новой сборки и создаёт несколько фактически разных программ.

Чем больше комбинаций symbols, тем больше вариантов нужно собирать и тестировать. Непроверенная ветвь «гниёт» незаметно.

## 3. `#nullable`

```csharp
#nullable enable

string required = "text";
string? optional = null;
```

Nullable reference types — compile-time static analysis. Они не меняют runtime representation string и не запрещают получить null через Reflection, старую library или `null!`.

Режимы:

- `enable` — annotations и warnings;
- `disable` — старое oblivious поведение;
- `warnings` / `annotations` — управление частями анализа;
- `restore` — вернуться к project setting.

Предпочитайте включение в project. Локальная directive полезна при постепенной миграции/generated code, но множество переключений затрудняет контракт.

## 4. Подавление warnings

```csharp
#pragma warning disable CS0618
LegacyApi.Call();
#pragma warning restore CS0618
```

Подавление должно быть минимальным и сопровождаться причиной. Нельзя отключать category на весь проект, чтобы скрыть непонятые warnings.

Альтернативы: исправить код, обновить dependency, изменить configuration, использовать suppression attribute с justification. Warning — не всегда defect, но каждое подавление создаёт долг.

## 5. `checked` и переполнение

Integer type имеет ограниченный диапазон:

```csharp
int maximum = int.MaxValue;
int wrapped = unchecked(maximum + 1); // int.MinValue
```

`checked` требует exception при integral overflow:

```csharp
int result = checked(maximum + 1); // OverflowException
```

Контекст можно задать project setting и локально переопределить. Explicit numeric conversion также может потерять диапазон:

```csharp
long large = 3_000_000_000;
int value = checked((int)large);
```

Floating-point overflow даёт Infinity, а `decimal` имеет собственные checked semantics и обычно бросает OverflowException.

Для денег/счётчиков переполнение нельзя молча принимать. Для криптографии/хешей wraparound может быть намеренным и документируется через `unchecked`.

## 6. `ref`-семантика

Урок 02 ввёл `ref`, `out`, `in`. Дополнительные формы позволяют ссылаться на место хранения:

```csharp
ref int Find(int[] values, int index)
{
    return ref values[index];
}

int[] numbers = { 10, 20 };
ref int selected = ref Find(numbers, 1);
selected = 99;
Console.WriteLine(numbers[1]); // 99
```

`ref return` не копирует value, а предоставляет alias элемента. Compiler следит, чтобы reference не пережила storage. Нельзя вернуть `ref` на local variable, исчезающий после метода.

`ref readonly` позволяет ссылку без изменения, `in` — readonly reference parameter. Выигрыш заметен главным образом для крупных structs в hot path; для маленького int сложность не оправдана.

## 7. `ref struct`

`Span<T>` — `ref struct`. Такой тип ограничен stack/lifetime safety:

- нельзя boxing в object/interface;
- нельзя field обычного class;
- нельзя пересечь `await`/`yield` при несовместимом lifetime;
- нельзя захватить обычной lambda так, чтобы он ушёл в heap.

Ограничения compiler-а предотвращают ссылку на уже недействительную память. Новые версии C# постепенно смягчают отдельные случаи, но фундаментальный lifetime остаётся.

## 8. `Span<T>` и `Memory<T>`

```csharp
ReadOnlySpan<byte> header = data.AsSpan(0, 8);
uint length = BinaryPrimitives.ReadUInt32LittleEndian(header[0..4]);
```

Span — view непрерывной памяти без владения: array, stackalloc, unmanaged memory. Slice обычно не копирует данные.

`ReadOnlySpan<T>` запрещает изменение через view. `Memory<T>`/`ReadOnlyMemory<T>` можно хранить в heap и передавать через async boundary; `.Span` получают для синхронной обработки.

View не продлевает разумный логический lifetime pooled buffer: после Return массив использовать нельзя, даже если ссылка технически осталась.

## 9. `stackalloc`

```csharp
Span<byte> buffer = stackalloc byte[32];
buffer.Clear();
```

Память выделяется в stack frame и исчезает при выходе. Это быстро и не создаёт managed heap allocation, но stack мал.

Опасно:

```csharp
// Размер пришёл от пользователя — возможен stack overflow
Span<byte> data = stackalloc byte[userSize];
```

Используйте маленький фиксированный/ограниченный размер; большой buffer — array/ArrayPool. Нулевое значение памяти и её очистка зависят от формы/контекста, поэтому инициализируйте данные, прежде чем читать или передавать наружу.

## 10. `unsafe` и pointers

```csharp
unsafe
{
    int value = 42;
    int* pointer = &value;
    Console.WriteLine(*pointer);
}
```

Pointer хранит адрес. `&` берёт адрес, `*` разыменовывает, `->` обращается к member pointed struct. Unsafe code требует project option:

```xml
<AllowUnsafeBlocks>true</AllowUnsafeBlocks>
```

`unsafe` не означает автоматически вредный код; оно означает, что compiler/runtime не могут гарантировать обычную type/memory safety. Developer отвечает за bounds, lifetime, alignment, null pointer и ownership.

## 11. `fixed` и pinning

GC может перемещать managed objects во время compaction. Pointer на array/string стал бы неверным. `fixed` временно закрепляет объект:

```csharp
unsafe
{
    byte[] bytes = { 1, 2, 3 };
    fixed (byte* pointer = bytes)
    {
        UseNative(pointer, bytes.Length);
    }
}
```

Pointer нельзя использовать после block. Долгое/массовое pinning мешает compaction и фрагментирует heap. Иногда pinned object heap/Memory.Pin или copying buffer подходит лучше; выбор измеряют.

## 12. Pointer arithmetic и bounds

`pointer + 1` сдвигается на размер pointed type. Runtime не проверяет выход за buffer. Ошибка может повредить память процесса, проявиться позже и не иметь обычного exception.

Span сохраняет bounds checks и предпочтителен, если решает задачу. Unsafe optimization применяют в маленьком изолированном методе с safe wrapper и exhaustive tests.

## 13. Interop и P/Invoke

P/Invoke вызывает функцию native library:

```csharp
internal static partial class NativeMethods
{
    [LibraryImport("mylibrary", EntryPoint = "add")]
    internal static partial int Add(int left, int right);
}
```

Source-generated `LibraryImport` предпочтителен во многих современных сценариях; `DllImport` остаётся распространённым и необходимым для поддерживаемых случаев.

Нужно согласовать ABI:

- library/entry-point name;
- calling convention;
- integer sizes;
- struct layout/alignment;
- character encoding;
- bool representation;
- кто выделяет и освобождает memory;
- error reporting и thread safety.

Одинаковое имя C# type не гарантирует одинаковый native layout.

## 14. Struct layout

```csharp
[StructLayout(LayoutKind.Sequential)]
public struct NativePoint
{
    public int X;
    public int Y;
}
```

`Sequential` сохраняет порядок fields с учётом packing/alignment. `Explicit` позволяет `FieldOffset`, но повышает риск. Не применяйте layout attributes к обычной domain model без interop/binary requirement.

Blittable types можно передавать без сложного преобразования; strings, booleans, arrays часто требуют marshalling rules.

## 15. Function pointers

```csharp
unsafe delegate* unmanaged<int, int, int> operation;
```

Function pointer — низкоуровневая ссылка на функцию без delegate object/target. Он нужен в interop/performance libraries и требует unsafe/calling convention. Для обычных callbacks используйте delegate: он безопаснее, удобнее и удерживает target lifetime.

## 16. Endianness и binary data

Multi-byte number имеет порядок bytes: little-endian или big-endian. Не читайте network/file header через случайный pointer cast: format должен явно задавать endianness.

```csharp
uint value = BinaryPrimitives.ReadUInt32BigEndian(source);
```

Дополнительно проверяйте длину до slice/read. Binary parsing — работа с недоверенным input, даже если файл локален.

## 17. Partial types и generated code

```csharp
public partial class ApiClient
{
    partial void OnRequestCreated();
}
```

`partial` позволяет разделить declaration типа/метода между файлами, особенно human-written и generated code. Все части собираются в один type. Partial не является способом скрыть бесконечно большой класс; для ручного кода композиция обычно лучше.

Source generators используют partial hooks/types, не переписывая исходный файл. Generated code не редактируют вручную: изменение исчезнет при следующей генерации.

## 18. Выбор уровня

Порядок предпочтения:

1. обычный безопасный API;
2. Span/Memory и BinaryPrimitives;
3. pool/pinning при измеренной необходимости;
4. unsafe/pointers в изолированной границе;
5. native interop с документированным ABI.

Переход ниже оправдывается отсутствующим API, interop или профилированием, а не желанием сделать код «профессиональнее».

## Частые ошибки

- `#if` используется как runtime feature flag;
- варианты conditional compilation не собираются в CI;
- warning подавлен глобально без причины;
- `unchecked` скрывает денежное переполнение;
- ref return раскрывает изменяемое внутреннее состояние;
- stackalloc size поступает напрямую от пользователя;
- Span сохраняется за пределами lifetime;
- pointer используется после fixed;
- native ownership/encoding не определены;
- unsafe применяется до профилирования;
- generated partial file редактируется вручную.

## Итог

```text
Safe managed code
  ↓ при необходимости
Span / Memory / BinaryPrimitives
  ↓ измеренный interop/hot path
pinning / unsafe / pointers
  ↓
явный lifetime, bounds, ABI и ownership
```

Вы готовы дальше, если можете выбрать безопасный уровень, ограничить stackalloc, объяснить pinning и спроектировать safe wrapper вокруг low-level реализации.
