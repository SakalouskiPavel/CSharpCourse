# Урок 15. Garbage Collector, ресурсы и Disposable Pattern

> GC освобождает managed memory, когда объект больше недостижим. Но он не обещает вовремя закрыть файл, сокет или native handle. Память и внешний ресурс — разные задачи с разными механизмами.

## 1. Managed memory

.NET runtime управляет памятью объектов. Программист создаёт:

```csharp
var book = new Book("B-001", "C# с нуля");
```

Runtime выделяет память, а Garbage Collector позднее находит недостижимые objects и освобождает их. Обычно не нужно вручную вычислять адрес и освобождать память как в языках без managed runtime.

Managed не означает «бесплатно»: allocations, traversal, compaction и удержание больших graphs имеют цену.

## 2. Stack и managed heap без мифов

Каждый thread имеет stack frames вызовов: parameters, local state и return information. Objects class обычно размещаются в managed heap, а local variable может хранить ссылку на них.

Value type не означает «всегда stack». Он может быть:

- local частью stack frame;
- field внутри heap object;
- элементом array;
- boxed в отдельном object;
- оптимизирован JIT register-ами.

Правильная модель — value semantics/reference semantics и lifetime, а не упрощённое «struct на стеке, class в куче».

## 3. Достижимость и GC roots

GC начинает с roots:

- references в активных thread stacks/registers;
- static fields;
- GC handles runtime/interoperability;
- некоторые runtime structures.

Он проходит по ссылкам. Всё достижимое считается живым; недостижимое можно освободить.

```text
root → Service → Cache → Book
```

Пока root удерживает Service, а он Cache, Book жив. Цикл сам по себе не утечка:

```text
A ↔ B, но пути от root нет → оба собираются
```

**Managed memory leak** — объекты логически больше не нужны, но остаются достижимыми из-за static collection, event subscription, timer, cache или незавершённой task.

## 4. Поколения

Большинство объектов живёт недолго. Generational GC использует это наблюдение:

- Generation 0 — новые короткоживущие objects;
- Generation 1 — промежуточный buffer;
- Generation 2 — долго живущие objects.

Объект, переживший collection, может быть promoted. Gen 0 собирается чаще и обычно дешевле; Gen 2 затрагивает большой долгоживущий graph и дороже.

Поколение — runtime implementation strategy, а не business lifecycle. Не пытайтесь вручную «поместить объект в Gen 1».

## 5. Mark, sweep, compact

Упрощённо collection:

1. останавливает/координирует managed threads на нужных фазах;
2. отмечает достижимые objects;
3. считает остальные мусором;
4. освобождает;
5. при необходимости уплотняет память и обновляет references.

.NET GC имеет workstation/server, background/concurrent и другие режимы. Детали зависят от runtime/configuration. Прикладной вывод: уменьшайте ненужные allocations/удержание только после измерения и избегайте предположений о точном моменте GC.

## 6. Large Object Heap

Большие allocations (традиционно около 85 000 bytes и выше; детали зависят от object/layout/runtime) попадают в Large Object Heap. Их collection обычно связана с Gen 2; перемещение/compaction имеет особые trade-offs.

Большие временные arrays создают pressure и fragmentation. Решения: streaming, reuse/pooling, chunks — после профилирования. Не используйте порог как вечную спецификацию platform behavior.

## 7. Почему `GC.Collect()` почти всегда не решение

GC сам выбирает момент по allocation budget и memory pressure. Принудительный full collection может:

- остановить приложение;
- собрать мало памяти, если graph достижим;
- преждевременно promote objects;
- ухудшить throughput.

`GC.Collect` допустим в редких измерительных/host scenarios с доказательством. Он не исправляет удерживающую static reference/event.

## 8. Managed memory и unmanaged resource

`FileStream` — managed object, но владеет OS file handle. Если перестать ссылаться на stream, GC когда-нибудь освободит managed object, однако файл может оставаться открытым неопределённое время.

Ресурсы с ограниченным/внешним lifetime:

- file/socket handles;
- database connections/readers;
- locks/semaphores;
- native memory;
- graphics handles;
- transaction/subscription.

Для них нужен deterministic cleanup — освобождение в известной точке.

## 9. `IDisposable` и `using`

```csharp
using FileStream stream = File.OpenRead(path);
// использовать stream
```

Compiler создаёт try/finally и вызывает `Dispose`, даже если внутри exception/return.

Область using declaration продолжается до конца текущего scope. Классический block делает границу явной:

```csharp
using (FileStream stream = File.OpenRead(path))
{
    // ресурс нужен только здесь
}
```

После Dispose объект может оставаться в памяти, но ресурс освобождён. GC и Dispose решают разные задачи.

## 10. Владение

Кто создал resource, обычно владеет и освобождает его, если контракт не передал ownership.

```csharp
public sealed class ReportWriter
{
    private readonly Stream _stream;

    public ReportWriter(Stream stream)
    {
        _stream = stream; // владеет ли writer? контракт обязан сказать
    }
}
```

Если injected stream создаёт caller и планирует использовать дальше, writer не должен неожиданно Dispose. Варианты:

- writer не владеет stream;
- constructor принимает `leaveOpen`;
- writer сам создаёт resource через factory и владеет;
- API документирует transfer ownership.

Double Dispose стандартных ресурсов часто безопасен, но рассчитывать на это без контракта нельзя.

## 11. Реализация `IDisposable` в sealed class

```csharp
public sealed class ReportWriter : IDisposable
{
    private readonly Stream _stream;
    private readonly bool _ownsStream;
    private bool _disposed;

    public ReportWriter(Stream stream, bool ownsStream)
    {
        _stream = stream ?? throw new ArgumentNullException(nameof(stream));
        _ownsStream = ownsStream;
    }

    public void Write(string text)
    {
        ObjectDisposedException.ThrowIf(_disposed, this);
        // записать text в stream
    }

    public void Dispose()
    {
        if (_disposed) return;

        if (_ownsStream)
            _stream.Dispose();

        _disposed = true;
    }
}
```

Dispose идемпотентен. Методы после cleanup либо явно не поддерживаются и бросают `ObjectDisposedException`, либо имеют документированное безопасное поведение.

Если class только владеет другими IDisposable managed objects и sealed, finalizer/сложный canonical inheritance pattern не нужен.

## 12. Наследуемый Dispose Pattern

Для non-sealed base class, предназначенного для наследования, pattern сложнее:

```csharp
public class ResourceOwner : IDisposable
{
    private bool _disposed;

    protected virtual void Dispose(bool disposing)
    {
        if (_disposed) return;

        if (disposing)
        {
            // dispose managed owned resources
        }

        // free direct unmanaged resources, если есть
        _disposed = true;
    }

    public void Dispose()
    {
        Dispose(true);
        GC.SuppressFinalize(this);
    }
}
```

Не добавляйте этот шаблон механически. Sealed classes проще, а наследование владельцев ресурсов требует строгого порядка base/derived cleanup.

## 13. Finalizer

```csharp
~NativeResourceOwner()
{
    Dispose(false);
}
```

Finalizer вызывается GC thread-ом в неопределённый момент для object, зарегистрированного на finalization. Такой object обычно переживает дополнительный цикл: сначала попадает в finalization queue, затем после выполнения становится собираемым.

Finalizer:

- не гарантирован при аварийном завершении;
- не должен обращаться к уже финализированным managed objects;
- не должен бросать исключения;
- нужен только при прямом владении unmanaged resource как страховка.

После успешного Dispose `GC.SuppressFinalize` убирает ненужную finalization work.

## 14. `SafeHandle`

Вместо собственного finalizer для OS handle предпочтителен `SafeHandle`/его subclasses. Он инкапсулирует надёжное освобождение handle и взаимодействие с runtime.

Managed wrapper владеет SafeHandle и Dispose-ит его. Это уменьшает сложность и риск утечки при async exceptions/interop.

Если ваш class владеет только SafeHandle, собственный finalizer обычно не нужен: finalization обеспечивает SafeHandle.

## 15. `IAsyncDisposable`

Cleanup иногда требует I/O, например async flush/transaction close:

```csharp
await using AsyncResource resource = await OpenAsync(cancellationToken);
```

```csharp
public async ValueTask DisposeAsync()
{
    await _stream.DisposeAsync().ConfigureAwait(false);
}
```

`await using` гарантирует ожидание `DisposeAsync`. Тип может реализовать IDisposable и IAsyncDisposable, но должен документировать, какой путь предпочтителен и эквивалентны ли гарантии.

Будьте осторожны с cancellation cleanup: после отменённой операции ресурс всё равно нужно освободить; часто cleanup выполняют с отдельным ограниченным token/без исходной отмены.

## 16. Event subscriptions как ресурс

Подписка имеет lifetime и может быть представлена IDisposable:

```csharp
using IDisposable subscription = publisher.Subscribe(handler);
```

Dispose выполняет unsubscribe. Это тот же принцип ownership, хотя OS handle нет. Timers, callbacks, registrations и linked cancellation tokens также часто IDisposable.

## 17. Memory leak diagnostics

Симптомы: managed heap растёт после нагрузки, Gen 2 collections не возвращают память, число объектов определённого типа постоянно растёт.

Путь диагностики:

1. воспроизвести steady workload;
2. снять memory dump/GC trace;
3. сравнить object counts/retained size;
4. найти path to root;
5. исправить ownership/retention;
6. повторить измерение.

Частые roots: static dictionaries, events, timers, caches без eviction, pending Tasks, thread locals.

Операционная система может не сразу показать уменьшение process working set после GC: runtime сохраняет segments для будущих allocations. Это не само по себе доказательство leak.

## 18. WeakReference

`WeakReference<T>` не удерживает объект живым. Подходит для специальных caches/наблюдения, где отсутствие допустимо.

Weak reference может исчезнуть в любой момент collection; после `TryGetTarget` target нужно удерживать strong local reference на время использования. Это не замена unsubscribe/eviction и усложняет reasoning.

## 19. Pools

`ArrayPool<T>` и object pools переиспользуют buffers/objects, уменьшая allocations в hot path.

```csharp
byte[] buffer = ArrayPool<byte>.Shared.Rent(minimumLength);
try
{
    // использовать только нужный диапазон
}
finally
{
    ArrayPool<byte>.Shared.Return(buffer, clearArray: true);
}
```

Rented array может быть больше запрошенного и содержать старые данные. Нельзя использовать после Return. Sensitive data очищают. Pool вводит сложный ownership, поэтому применяется после измерения.

## 20. Allocation reduction без преждевременности

Полезный порядок:

1. корректный простой код;
2. метрики/trace/benchmark;
3. найти dominant allocation/retention;
4. уменьшить lifetime, материализацию или копирование;
5. применить Span/pool только при доказанной пользе;
6. повторно измерить и проверить безопасность.

Нулевые allocations не являются целью любого business code. Читаемость и корректность часто важнее микровыигрыша.

## Частые ошибки

- GC считается механизмом своевременного закрытия файлов;
- вызывается GC.Collect вместо удаления root;
- любой value type считается stack allocation;
- injected dependency Dispose-ится без ownership;
- finalizer добавляется классу без direct unmanaged resource;
- Dispose не идемпотентен;
- method работает после Dispose с неясным состоянием;
- event/timer registration не освобождается;
- WeakReference маскирует неправильный lifetime;
- pooled buffer используется после Return или не очищает секреты;
- process memory metric принимается за точное число live objects.

## Итог

```text
Managed object memory → GC по достижимости
External/limited resource → Dispose в известной точке
Direct native handle → SafeHandle (обычно)
Async cleanup → IAsyncDisposable / await using
```

Вы готовы дальше, если можете нарисовать path to root, объяснить почему event удерживает subscriber, определить ownership ресурса и корректно реализовать sealed IDisposable wrapper.
