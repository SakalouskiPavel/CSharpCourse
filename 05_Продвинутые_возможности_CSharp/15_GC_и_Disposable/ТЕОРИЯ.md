# Урок 15. Сборщик мусора, финализация и Disposable Pattern

> GC управляет памятью managed-объектов, но не временем освобождения файлов, сокетов и native handles. Deterministic cleanup выражается IDisposable/await using, а финализатор остаётся страховочной сеткой для непосредственного владения unmanaged-ресурсом.

## Результаты обучения

- объяснять generations и roots GC
- корректно реализовывать/использовать IDisposable
- понимать finalization и SafeHandle
- диагностировать удержание объектов и давление на GC

## Карта темы

### 1. Достижимость

GC начинает с roots: стеков, static fields, handles и runtime structures. Недостижимый managed-объект освобождается независимо от циклов ссылок; «утечка» означает нежелательную достижимость.

### 2. Поколения

Gen 0 оптимизирует короткоживущие объекты, выжившие продвигаются. Gen 2 и Large Object Heap собираются реже; стоимость зависит от allocation rate и живого графа.

### 3. GC не про ресурсы

Файл может оставаться открыт до неопределённого момента, даже если объект скоро станет мусором. `using` компилируется в try/finally и вызывает Dispose.

### 4. IDisposable

В sealed-классе Dispose освобождает owned resources и становится idempotent. Не следует dispose-ить внедрённую зависимость, если владение ею не передано явно.

### 5. Финализатор

Finalizable objects переживают дополнительный цикл и увеличивают стоимость. Финализатор нужен только владельцу unmanaged handle; `GC.SuppressFinalize` убирает страховку после успешного Dispose.

### 6. SafeHandle

SafeHandle инкапсулирует native handle и надёжную финализацию, поэтому собственный finalizer часто не нужен. Managed wrapper владеет SafeHandle и dispose-ит его.

### 7. IAsyncDisposable

Async cleanup нужен, когда освобождение требует I/O. `await using` ожидает `DisposeAsync`; смешанный контракт требует ясной политики.

### 8. Пулы и давление памяти

ArrayPool/ObjectPool уменьшают аллокации в горячих местах, но вводят владение и очистку данных. Сначала измеряйте allocation profile.

### 9. Удержание

Static collections, event subscriptions, timers и caches часто удерживают граф. WeakReference — специализированный инструмент, не универсальный fix.

## Минимальный пример

```csharp
public sealed class ReportWriter(Stream stream, bool ownsStream) : IDisposable
{
    private bool _disposed;
    public void Dispose()
    {
        if (_disposed) return;
        if (ownsStream) stream.Dispose();
        _disposed = true;
    }
}
```

Пример — отправная точка, а не готовая архитектура. Проверьте предусловия, lifetime объектов, обработку ошибок и поведение на границах. Для инфраструктурных тем отдельно фиксируйте, какие гарантии даёт библиотека, а какие должен обеспечить ваш код.

## Типичные ошибки

- вызывать GC.Collect для «оптимизации»;
- надеяться на finalizer для своевременного закрытия файла;
- dispose-ить dependency без соглашения о владении;
- возвращать pooled buffer без очистки чувствительных данных;
