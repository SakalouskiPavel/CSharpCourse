# Практика — урок 15

## Цель

Создать пакетный writer отчётов с явным владением потоками, синхронным/асинхронным cleanup и проверкой всех путей завершения.

## `ReportWriter`

Sealed class получает Stream и `leaveOpen` либо `ownsStream`. Он:

- проверяет constructor arguments;
- пишет несколько строк отчёта;
- запрещает Write после Dispose;
- flush-ит согласно документированному контракту;
- Dispose идемпотентен;
- не имеет finalizer, потому что владеет managed Stream.

Покажите два сценария:

1. caller сохраняет ownership и использует stream после writer;
2. ownership передан writer, stream закрыт вместе с ним.

## Async-вариант

Создайте `AsyncReportWriter : IAsyncDisposable` или единый тип с обоими контрактами. `DisposeAsync` ожидает async flush/dispose. Опишите различие гарантий sync/async пути.

## Fake resource

Реализуйте учебный `TrackingStream`/wrapper, который фиксирует:

- Dispose/DisposeAsync вызван;
- сколько раз;
- были ли записи после cleanup;
- выполнялся ли cleanup после исключения.

Проверьте:

- обычный успех;
- exception в середине записи;
- ранний return;
- повторный Dispose;
- leaveOpen true/false;
- cancellation async operation с последующим cleanup.

## Subscription resource

Из урока 13 оформите event subscription как IDisposable. С помощью схемы или controlled WeakReference experiment покажите path:

```text
static/long-lived publisher → delegate → subscriber
```

После Dispose subscriber перестаёт получать события и publisher больше не удерживает его.

## Pool experiment

Отдельно возьмите byte[] из ArrayPool в try/finally:

- не предполагаете точную длину;
- используете только requested range;
- возвращаете ровно один раз;
- очищаете sensitive content;
- не читаете после Return.

Объясните, почему pool не включён в основной writer без профилирования.

## Ограничения

- нет GC.Collect в production path;
- нет finalizer без direct unmanaged handle;
- ownership виден из API/README;
- cleanup выполняется при exception;
- Dispose не скрывает исходную ошибку;
- после Dispose поведение определено;
- WeakReference не заменяет unsubscribe.

## Критерии приёмки

- sync/async cleanup проверен;
- owned и borrowed resources различаются;
- double dispose безопасен;
- event subscription освобождается;
- pool experiment соблюдает ownership;
- README содержит root graph и объяснение GC vs Dispose;
- solution собирается, все сценарии воспроизводимы.

## Дополнительное задание

Оберните учебный native handle через подходящий SafeHandle subclass и managed wrapper. Собственный finalizer wrapper-а не добавляйте, если finalization уже обеспечивает SafeHandle. Опишите только на поддерживаемой платформе и не используйте реальный опасный ресурс без необходимости.
