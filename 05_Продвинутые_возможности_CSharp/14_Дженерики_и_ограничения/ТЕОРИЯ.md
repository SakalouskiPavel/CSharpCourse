# Урок 14. Generics: типы и алгоритмы с параметрами

> Обычный метод получает значения-параметры. Generic-код получает ещё и тип-параметр. Так один алгоритм работает с разными типами, сохраняя compile-time проверки и не превращая всё в `object`.

## 1. Проблема повторения и `object`

```csharp
static int First(int[] values) => values[0];
static string First(string[] values) => values[0];
```

Логика одинакова. Вариант с `object` теряет информацию:

```csharp
static object First(object[] values) => values[0];
```

Клиент должен cast-ить результат, можно случайно смешать типы, а value types подвергаются boxing.

Generic method:

```csharp
static T First<T>(IReadOnlyList<T> values)
{
    if (values.Count == 0)
        throw new ArgumentException("Последовательность пуста", nameof(values));

    return values[0];
}
```

`T` — type parameter. При вызове он заменяется конкретным типом:

```csharp
int firstNumber = First<int>(new[] { 10, 20 });
string firstName = First(new[] { "Анна", "Борис" });
```

Во втором вызове compiler вывел `T=string` из аргумента.

## 2. Generic type

```csharp
public sealed class Box<T>
{
    public T Value { get; }

    public Box(T value)
    {
        Value = value;
    }
}
```

`Box<int>` и `Box<string>` — разные закрытые constructed types. Нельзя присвоить один другому. Open type `Box<>` используется в Reflection, но variable/instance требует заданный argument.

Type parameter может использоваться в fields, properties, parameters, return values и nested types.

## 3. Несколько параметров типа

```csharp
public sealed record Pair<TKey, TValue>(TKey Key, TValue Value);
```

Имена начинают с `T`: `T`, `TKey`, `TValue`, `TResult`. Они должны сообщать роль. Слишком много parameters часто означает, что abstraction объединяет несвязанные ответственности.

## 4. Type inference

```csharp
static TResult Convert<TSource, TResult>(
    TSource source,
    Func<TSource, TResult> converter) => converter(source);

string text = Convert(42, number => number.ToString());
```

Compiler выводит types из arguments и lambda target. Только ожидаемый return type обычно недостаточен:

```csharp
// Create<T>() без аргументов часто требует Create<Book>()
```

При неоднозначности укажите arguments явно или улучшите API.

## 5. Почему над неизвестным `T` нельзя делать всё

```csharp
static T Add<T>(T left, T right)
{
    // return left + right; // compiler не знает operator + для любого T
}
```

Без constraints compiler гарантирует только операции `object` и присваивание. Constraints описывают обязательные способности.

## 6. Основные constraints

```csharp
where T : class
where T : struct
where T : notnull
where T : unmanaged
where T : BaseClass
where T : IComparable<T>
where T : new()
```

- `class` — reference type (nullable variant зависит от annotation context);
- `struct` — non-nullable value type;
- `notnull` — не допускает nullable type argument на уровне анализа;
- `unmanaged` — value type без managed references, пригоден для низкоуровневых операций;
- base/interface — `T` поддерживает их members;
- `new()` — публичный parameterless constructor.

```csharp
static T Clamp<T>(T value, T minimum, T maximum)
    where T : IComparable<T>
{
    if (minimum.CompareTo(maximum) > 0)
        throw new ArgumentException("Неверный диапазон");

    if (value.CompareTo(minimum) < 0) return minimum;
    if (value.CompareTo(maximum) > 0) return maximum;
    return value;
}
```

Constraint даёт доступ к `CompareTo` и документирует requirement.

## 7. Комбинация и порядок constraints

Можно объединять:

```csharp
where T : BaseEntity, IAuditable, new()
```

Class/value constraint идёт первым, base class — до interfaces, `new()` обычно последним. Некоторые комбинации противоречат друг другу. Compiler проверяет declaration и каждый use.

Не добавляйте `new()` только чтобы generic service сам создавал dependencies: factory/delegate/DI обычно лучше выражает construction with arguments.

## 8. Generic interface и реализации

```csharp
public interface IRepository<TEntity, in TId>
{
    TEntity? Find(TId id);
    void Add(TEntity entity);
}
```

Generic interface задаёт семейство контрактов. Однако универсальный repository не всегда полезен: разные aggregates требуют разных queries, transactions и operations. Generics устраняют техническое повторение, но не делают бизнес-семантику одинаковой.

Класс может реализовать несколько закрытых вариантов одного интерфейса, если contracts не конфликтуют.

## 9. `default(T)` и nullable

`default` зависит от типа:

- reference type → null;
- numeric value type → 0;
- bool → false;
- struct → все fields default.

```csharp
T? FindOrDefault<T>(IEnumerable<T> source, Func<T, bool> predicate)
{
    return source.FirstOrDefault(predicate);
}
```

Знак `T?` имеет разный смысл в зависимости от constraints/nullable context. Для API, где отсутствие должно быть отличимо от допустимого default (например, `0`), Result/Option-like type может быть яснее.

`notnull` помогает compiler и dictionary key constraints, но не является runtime guard от Reflection/deserialization/disabled analysis.

## 10. Boxing и производительность

```csharp
int number = 42;
object boxed = number;      // boxing
int unboxed = (int)boxed;   // unboxing
```

Boxing упаковывает копию value type в managed object. Generic `List<int>` хранит ints без boxing, в отличие от старой non-generic collection of object.

Не любой generic-код автоматически быстрее. JIT часто разделяет machine code для reference type instantiations и специализирует value types, но детали runtime могут меняться. Измеряйте hot path.

## 11. Reified generics

.NET runtime знает type arguments:

```csharp
Console.WriteLine(typeof(List<int>));
Console.WriteLine(list.GetType().GenericTypeArguments[0]);
```

Это отличается от type erasure моделей некоторых платформ. Можно проверять `is List<int>` и читать generic metadata. Но Reflection не должен заменять normal generic dispatch.

## 12. Invariance

```csharp
List<Cat> cats = new();
// List<Animal> animals = cats; // запрещено
```

Иначе через `animals.Add(new Dog())` собака попала бы в список кошек. Mutable generic types, одновременно принимающие и возвращающие `T`, обычно инвариантны.

## 13. Covariance и contravariance

```csharp
public interface IProducer<out T>
{
    T Produce();
}

public interface IConsumer<in T>
{
    void Consume(T value);
}
```

- `out T` используется только в output-safe positions: producer cats можно читать как producer animals;
- `in T` — input-safe positions: consumer animals умеет принять cats.

Variance действует для interface/delegate type parameters и reference conversions. `IEnumerable<Cat>` → `IEnumerable<Animal>` безопасно, но value types не получают такую boxing-конверсию через variance.

## 14. Generic methods и overload resolution

Generic overload конкурирует с non-generic. Compiler выбирает best applicable candidate по conversions и inference. Слишком похожие overloads создают неоднозначность, особенно с null/default/lambdas.

Не пытайтесь «победить compiler» случайными casts: API с разными смыслами лучше дать разные names.

## 15. Static abstract members и generic math

Современный C# позволяет interface требовать static operation:

```csharp
static T Sum<T>(IEnumerable<T> values)
    where T : INumber<T>
{
    T result = T.Zero;
    foreach (T value in values)
        result += value;
    return result;
}
```

`INumber<T>` из generic math сообщает наличие Zero, arithmetic operators и других numeric capabilities. Dispatch проверяется compile time без `dynamic`/Reflection.

Это продвинутый инструмент для математических libraries. Обычный domain `Money` не обязан реализовывать полный numeric contract: валюта и округление имеют собственную семантику.

## 16. Когда generics не нужны

Не используйте type parameter, если:

- метод работает ровно с одним domain type;
- разные types требуют разного поведения, скрытого `if typeof(T)`;
- interface/base class уже выражает нужную polymorphic role;
- generic repository стирает важные queries;
- abstraction используется один раз и усложняет чтение.

Если generic method постоянно проверяет runtime type, параметрический полиморфизм не достигнут: алгоритм не одинаков для всех T.

## Частые ошибки

- object/casts используются вместо generics;
- T не имеет constraint, но код ожидает member/operator;
- `new()` скрывает сложное создание dependency;
- `default(T)` принимается за однозначное «не найдено»;
- `notnull` считается runtime validation;
- ожидается covariance у mutable List;
- generic API имеет слишком много несвязанных parameters;
- внутри выполняются `typeof(T)` ветки для каждого известного типа;
- универсальный repository скрывает domain semantics;
- generic math применяется ради демонстрации.

## Итог

```text
Generic parameter T
  ↓
один алгоритм для семейства типов
  ↓ constraints
compiler знает доступные способности
  ↓
type-safe use без object casts/boxing
```

Вы готовы дальше, если можете написать generic method/class, выбрать constraints, объяснить invariance и распознать abstraction, где типы на самом деле ведут себя по-разному.
