# Урок 14. Дженерики, реализация и ограничения

> Generics позволяют выразить семейство алгоритмов без потери типобезопасности и без упаковки значимых типов. Ограничения сообщают компилятору, какие операции допустимы над неизвестным T.

## Результаты обучения

- проектировать generic types и methods
- применять constraints и variance
- понимать runtime reification и статические члены интерфейсов
- избегать чрезмерной generic-абстракции

## Карта темы

### 1. Параметрический полиморфизм

Generic-код задаёт алгоритм через T, а компилятор проверяет применения. В отличие от `object`, не нужны приведения и boxing для value types.

### 2. Вывод типов

Compiler выводит type arguments из аргументов метода, но обычно не из return type. Иногда явный `<T>` нужен для снятия неоднозначности.

### 3. Constraints

`class`, `struct`, `notnull`, `unmanaged`, `new()`, base type и interface constraints дают операции и документируют требования. Порядок и совместимость ограничений задаются языком.

### 4. Variance

`out` и `in` доступны интерфейсам/делегатам и только для reference-type conversions. Mutable `IList<T>` инвариантен, потому что и принимает, и возвращает T.

### 5. Реализация CLR

Generics reified: runtime знает type arguments. Код для reference types часто разделяется, value types получают специализированные версии; это отличается от Java erasure.

### 6. default и nullability

`default(T)` может быть null или нулевым значением. Nullable annotations и `notnull` помогают статическому анализу, но не являются полной runtime-валидацией.

### 7. Generic math

Static abstract interface members позволяют писать числовые алгоритмы с `INumber<T>`. Это compile-time dispatch без dynamic/reflection.

### 8. Generic repository и абстракции

Универсальность полезна при действительно одинаковой семантике. `IRepository<T>` нередко скрывает различающиеся запросы и транзакции; предметный контракт бывает честнее.

## Минимальный пример

```csharp
public static T Clamp<T>(T value, T min, T max) where T : IComparable<T>
{
    if (min.CompareTo(max) > 0) throw new ArgumentException("Invalid range");
    if (value.CompareTo(min) < 0) return min;
    return value.CompareTo(max) > 0 ? max : value;
}
```

Пример — отправная точка, а не готовая архитектура. Проверьте предусловия, lifetime объектов, обработку ошибок и поведение на границах. Для инфраструктурных тем отдельно фиксируйте, какие гарантии даёт библиотека, а какие должен обеспечить ваш код.

## Типичные ошибки

- заменять generics на object и casts;
- ставить `new()` только ради service locator-подобного создания;
- ожидать covariance у mutable коллекции;
- создавать универсальный тип с десятком несвязанных параметров;
