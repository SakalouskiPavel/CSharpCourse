# Практика — урок 34

## Цель

Создать Worker для outbox урока 31/33 и pipeline, который собирает, проверяет и публикует один прослеживаемый артефакт.

## Часть 1. Worker Service

1. Создайте проект `OrderOutbox.Worker` через Generic Host.
2. `BackgroundService` пакетами читает outbox и публикует сообщения.
3. Каждая итерация создаёт DI scope; scoped repository/DbContext не удерживается между итерациями.
4. Поддержите cancellation и graceful shutdown: новая работа не берётся, текущая завершается безопасно.
5. Ошибки отдельного сообщения классифицируются; неожиданный fatal failure завершает процесс ненулевым кодом или явно выбранной host policy.
6. Конфигурация (batch size, interval, broker endpoint) приходит извне и валидируется при старте.
7. Добавьте structured logs и метрики last success, processed/failed, duration и oldest outbox age.
8. Реализуйте management health endpoint или heartbeat; различите liveness/readiness.

## Часть 2. Console utility

Создайте `orders-admin replay-dlq`:

- аргументы `--from`, `--limit`, `--dry-run`;
- результат в stdout, диагностика в stderr;
- `0` при успехе и документированные ненулевые exit codes;
- неинтерактивный режим для CI;
- cancellation по Ctrl+C;
- audit запуска replay без вывода payload/secrets.

## Часть 3. CI pipeline

Pipeline на pull request и main выполняет:

1. checkout с минимальными permissions;
2. установку зафиксированного .NET SDK;
3. restore с корректным dependency cache;
4. format/analyzers;
5. build с warnings policy;
6. unit и integration tests;
7. публикацию test results/coverage как диагностики;
8. dependency/security scan;
9. `dotnet publish` worker и utility;
10. упаковку immutable artifact с commit SHA/version metadata.

Release job должен использовать **этот же** artifact, а не пересобирать код. Секретов в repository и логах нет; для cloud deployment опишите OIDC/workload identity.

## Часть 4. Deployment design

Создайте `DELIVERY.md`:

- различия CI, Delivery и Deployment в вашем процессе;
- среды и external configuration;
- strategy rollout и rollback;
- expand/contract для одной migration;
- smoke tests и release metrics;
- canary/abort criteria;
- feature flag lifecycle;
- provenance/SBOM и permissions;
- incident procedure для утечки секрета.

## Проверки

- pipeline падает при compile/test error;
- локальные команды совпадают с CI;
- повторный запуск даёт эквивалентный artifact;
- worker корректно останавливается во время обработки;
- broker outage отражается readiness/metrics и не теряет outbox;
- две replicas не создают повторный бизнес-эффект;
- utility возвращает разные exit codes для usage и runtime error;
- artifact можно связать с commit.

## Критерии готовности

- pipeline читаем и воспроизводим;
- build once/promote соблюдается;
- permissions и secrets ограничены;
- worker имеет корректный lifetime и observability;
- failure/rollback сценарии проверены;
- документация позволяет новому разработчику запустить всё без скрытых ручных шагов.
