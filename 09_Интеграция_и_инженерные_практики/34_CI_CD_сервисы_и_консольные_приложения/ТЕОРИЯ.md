# Урок 34. CI/CD, консольные приложения и фоновые сервисы

> Pipeline превращает конкретный commit в проверенный и прослеживаемый артефакт, а затем управляемо доставляет этот же артефакт в среду. Автоматизация полезна лишь тогда, когда отказ понятен и воспроизводим.

## Результаты обучения

Вы сможете:

- различать Continuous Integration, Delivery и Deployment;
- строить pipeline из быстрых проверок, тестов, публикации и развёртывания;
- обеспечивать воспроизводимость, traceability и безопасное хранение секретов;
- выбирать deployment/rollback strategy и quality gates;
- создавать .NET console и Worker Service с Generic Host;
- корректно обрабатывать configuration, cancellation, shutdown и exit codes;
- проектировать health checks и observability фонового процесса;
- защищать software supply chain и артефакты сборки.

## 1. От ручной инструкции к исполняемому процессу

Ручное «соберите на машине Васи, скопируйте DLL и поправьте JSON» не воспроизводимо: неизвестны версии инструментов, точный исходный код и сделанные руками изменения. Pipeline хранится как код и на каждый commit выполняет одинаковые шаги в чистой среде.

Главные свойства: repeatability, быстрый feedback, audit trail и возможность доказать, какой commit породил работающий артефакт.

## 2. Continuous Integration

CI — частое объединение небольших изменений в общую ветвь с автоматической проверкой. Это не просто «у нас запускается build server».

Практики CI:

- короткоживущие ветви и небольшие commits/PR;
- автоматические build, tests и static checks;
- быстрое исправление красной основной ветви;
- отсутствие долгой ручной фазы интеграции;
- одинаковые команды локально и в pipeline.

Если ветвь живёт месяц, интеграция всё равно отложена, даже при ежедневной сборке самой ветви.

## 3. Continuous Delivery и Continuous Deployment

**Continuous Delivery** поддерживает основную ветвь в состоянии, пригодном для выпуска; production deployment может требовать осознанного подтверждения. **Continuous Deployment** автоматически выпускает каждое прошедшее изменение.

Непрерывность не означает отсутствие контроля. Approval, change window или compliance gate могут быть частью Delivery. Выбор зависит от риска, возможности быстрого отката и зрелости наблюдаемости.

## 4. Типичный pipeline

```text
checkout → restore → format/lint → build → unit tests
         → integration/security checks → publish/package
         → attest/sign → deploy test → smoke tests
         → promote production → verify/observe
```

Дешёвые быстрые проверки идут раньше дорогих. Независимые jobs запускают параллельно. Однако слишком сложный DAG затрудняет диагностику — pipeline тоже является программой и требует поддержки.

## 5. Restore, build, test, publish

- `dotnet restore` разрешает зависимости и создаёт assets;
- `dotnet build --no-restore` компилирует;
- `dotnet test --no-build` запускает тесты на уже собранном коде;
- `dotnet publish` создаёт deployable output.

Важно не пересобирать «почти тот же» код между тестом и выпуском. Build once, promote same artifact уменьшает расхождение: hash артефакта остаётся тем же для test и production.

## 6. Быстрые и надёжные проверки

Unit tests дают ранний feedback. Integration tests проверяют БД, broker и HTTP-контракты. End-to-end покрывают несколько критичных потоков, но медленны и хрупки.

Flaky test, который «перезапускают до зелёного», разрушает доверие. Его локализуют, назначают владельца и исправляют; quarantine может быть только временной и видимой. Quality gate должен блокировать по осмысленному риску, а не ради красивого процента coverage.

## 7. Воспроизводимость и lock files

Версии SDK фиксируют через `global.json`, зависимости — точными правилами/lock file там, где это принято. Runner image и инструменты версионируют. Случайный `latest` способен поменять сборку без изменения Git.

Кэш зависимостей ускоряет restore, но ключ должен зависеть от OS, SDK и dependency files. Кэш не является артефактом релиза и не должен скрывать отсутствие объявленной зависимости.

## 8. Артефакт, версия и provenance

Артефакт — immutable package/container/published directory, полученный из commit. Версия должна связываться с tag/commit SHA и содержать метаданные сборки.

Provenance/attestation криптографически связывает артефакт с repository, workflow и commit. Подпись/attestation помогает проверить происхождение, но не доказывает отсутствие уязвимости или правильность бизнес-логики.

## 9. Секреты в CI/CD

Нельзя хранить production password в YAML, repository, artifact или build log. Используют secret store и минимальные права. Предпочтительны короткоживущие credentials через workload identity/OIDC вместо бессрочного cloud key.

Pull request из недоверенного fork не должен получать deployment secrets. Нельзя бездумно печатать environment или включать shell trace. Секрет после утечки отзывают и ротируют, а не только маскируют лог.

## 10. Permissions и dependency risks

Pipeline исполняет код с доступом к репозиторию и иногда инфраструктуре. Actions/tasks pin-ят на доверенную версию/commit, permissions задают минимально, production environment защищают rules.

Проверяют уязвимости и лицензии зависимостей, создают SBOM, обновляют base images. Security scanner выдаёт сигнал для triage, а не магическое доказательство безопасности. Исключение vulnerability должно иметь владельца и срок.

## 11. Среды и конфигурация

Один артефакт продвигается между средами, а конфигурация поступает извне: environment variables, mounted config, secret store. Пароль production не встраивают в образ.

Environment parity уменьшает сюрпризы, но production всегда отличается размером и реальными интеграциями. Infrastructure as Code и migrations versioning делают различия видимыми.

## 12. Database migrations

Приложение и схема не обновляются атомарно. Безопасен expand/contract:

1. добавить совместимую новую колонку/таблицу;
2. выпустить код, умеющий старое и новое;
3. перенести/backfill данные;
4. переключить чтение;
5. после периода удалить старое.

Destructive migration перед rollout старого кода ломает rollback. Миграции делают идемпотентными/контролируемыми, блокировки и длительность проверяют на объёме, близком к production.

## 13. Deployment strategies

- rolling: экземпляры обновляются постепенно; нужна совместимость версий;
- blue/green: две среды, трафик переключается; быстрее rollback, дороже ресурсы;
- canary: малый процент получает новую версию, решение принимают по метрикам;
- recreate: старая версия останавливается, затем новая; просто, но с downtime.

Стратегия не спасёт, если новая версия необратимо изменила данные. Rollback кода и roll-forward/fix-forward данных проектируют отдельно.

## 14. Feature flags

Flag отделяет deployment от release: код уже развёрнут, но функция включается аудитории постепенно. Полезны kill switch и canary.

Flag — временная ветвь в production. У него должны быть владелец, default, аудит изменений, срок удаления и тесты обеих сторон. Секретные возможности нельзя защищать только flag: authorization остаётся обязательной.

## 15. Проверка после deployment

Успешная команда deploy не означает работающий продукт. Выполняют smoke tests и наблюдают technical/business metrics: error rate, latency, saturation, queue lag, успешные заказы. Сравнивают с baseline и заранее задают abort/rollback criteria.

Автоматический rollback при одном шумном сигнале может создать цикл. Решение использует несколько показателей, окно и понятный runbook.

## 16. Console application

Console app подходит командам, миграциям, одноразовым jobs и утилитам. Её контракт — arguments/stdin, stdout для результата, stderr для диагностики и exit code.

Обычно `0` означает успех, ненулевое значение — ошибку. Скрипт CI не должен разбирать русскую фразу «вроде получилось», если доступен машинный exit code/JSON output. Неинтерактивный режим не должен зависать в ожидании `ReadLine`.

## 17. Generic Host

Generic Host объединяет DI, configuration, logging и lifetime:

```csharp
var builder = Host.CreateApplicationBuilder(args);
builder.Services.AddHostedService<OutboxRelayWorker>();

using var host = builder.Build();
await host.RunAsync();
```

Console tool тоже может использовать Host, если ей нужны эти инфраструктурные возможности. Для простой функции на двадцать строк он может быть лишним.

## 18. `IHostedService` и `BackgroundService`

`IHostedService` задаёт `StartAsync` и `StopAsync`. `BackgroundService` упрощает долгий цикл через `ExecuteAsync(stoppingToken)`.

```csharp
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    while (!stoppingToken.IsCancellationRequested)
    {
        await relay.PublishPendingAsync(stoppingToken);
        await Task.Delay(TimeSpan.FromSeconds(1), stoppingToken);
    }
}
```

Нельзя запускать fire-and-forget работу и возвращать из `ExecuteAsync`, теряя lifetime/ошибки. Исключение верхнего уровня должно иметь определённую политику: завершить процесс для рестарта supervisor или обработать ожидаемый отказ внутри итерации.

## 19. Graceful shutdown

При SIGTERM/service stop Host отменяет token и даёт ограниченное время. Worker перестаёт брать новую работу, завершает/безопасно прерывает текущую, делает commit/ack в правильном порядке и освобождает ресурсы.

Если shutdown timeout истёк, процесс может быть убит. Поэтому операция должна быть идемпотентной, а незавершённое сообщение — повторно доставляемым. Игнорировать cancellation в бесконечном delay нельзя.

## 20. Scoped dependencies в singleton worker

Hosted service регистрируется singleton. Нельзя напрямую удерживать scoped `DbContext` всю жизнь. Для итерации создают scope:

```csharp
await using var scope = scopeFactory.CreateAsyncScope();
var relay = scope.ServiceProvider.GetRequiredService<IOutboxRelay>();
await relay.RunBatchAsync(stoppingToken);
```

Это допустимое использование service provider на composition/lifetime boundary, а не Service Locator в domain logic.

## 21. Scheduling и overlap

`PeriodicTimer`/delay дают интервал после или между запусками, но не полноценный distributed scheduler. Нужно решить: допускается ли overlap, что будет при долгой итерации, пропущенном времени, переходе часов и запуске нескольких replicas.

Для singleton job в кластере нужны leader election/distributed lock или внешний scheduler. Lock имеет lease/timeout и не отменяет идемпотентность.

## 22. Windows Service, systemd и containers

Worker — приложение; Windows Service/systemd/container orchestrator — способы supervision. Они запускают процесс, передают stop, рестартуют и собирают состояние.

Для Windows используется интеграция hosting Windows Services; под Linux — systemd integration или container. Бизнес-код не должен зависеть от конкретного supervisor. Container не требует бесконечного самодельного keep-alive: главный процесс worker и есть lifetime контейнера.

## 23. Health и observability worker

Liveness отвечает, не завис ли процесс; readiness — способен ли он принимать работу. Worker без HTTP может публиковать heartbeat/metrics или предоставить небольшой management endpoint.

Следят за last successful iteration, processing latency, failure count, queue/outbox age, throughput и stuck items. Лог «worker работает» каждую секунду создаёт шум; structured события нужны на начало/итог batch и аномалии.

## 24. Типичные ошибки

- различать Delivery/Deployment только по названию инструмента;
- собирать новый артефакт для каждой среды;
- использовать `latest` и неприкреплённые actions;
- передавать долгоживущий cloud secret;
- считать coverage/security scan доказательством качества;
- выкатывать несовместимую migration;
- не проверять продукт после deploy;
- ловить все исключения worker и молча продолжать;
- игнорировать cancellation и exit code;
- держать scoped DbContext в singleton;
- запускать неидемпотентный job в нескольких replicas.

## Итог

CI быстро интегрирует изменения, Delivery сохраняет готовность к выпуску, Deployment автоматизирует сам выпуск. Надёжный pipeline производит один прослеживаемый артефакт, защищает цепочку поставки и проверяет результат после развёртывания. Console и Worker приложения следуют тем же требованиям: явный контракт, конфигурация извне, корректный lifetime, наблюдаемость и безопасный повтор.
