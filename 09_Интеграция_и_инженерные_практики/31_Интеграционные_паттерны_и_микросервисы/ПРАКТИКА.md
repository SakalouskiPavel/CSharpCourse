# Практика — урок 31

## Цель

Расширить сервис заказов асинхронным взаимодействием со складом и уведомлениями. Реальный broker пока не обязателен: важнее корректная модель сообщений и отказов.

## Обязательная часть

1. Опишите bounded contexts Orders, Inventory и Notifications, владельцев данных и границы доверия.
2. Реализуйте локальный transactional outbox рядом с `PlaceOrder`.
3. Создайте immutable integration event `OrderPlacedV1` с message, correlation, causation ids, UTC-временем и минимальным payload.
4. Outbox relay публикует событие через интерфейс `IMessagePublisher`; тестовая реализация может быть in-memory.
5. Inventory consumer обрабатывает сообщение идемпотентно через inbox и создаёт резерв.
6. Notifications consumer независимо реагирует на то же событие.
7. Transient error получает ограниченный retry с backoff; permanent error направляется в учебную DLQ с причиной.
8. Ack моделируется только после commit consumer.
9. Добавьте метрики pending outbox, retry count, DLQ count и processing age.

## Saga

Опишите процесс:

`PlaceOrder → ReserveStock → ChargePayment → ConfirmOrder`.

Для каждого шага укажите владельца, command/event, timeout, повтор и компенсацию. Реализуйте небольшой orchestrator state machine хотя бы для успешного пути, отказа оплаты и повторной доставки сообщения.

## Тесты

Проверьте:

- заказ и outbox entry появляются атомарно;
- ошибка relay не теряет entry;
- повтор relay не создаёт второй резерв;
- два consumers получают независимые копии события;
- consumer crash до ack приводит к повтору без повторного эффекта;
- permanent error оказывается в DLQ;
- saga компенсирует резерв после отказа оплаты;
- повтор compensation безопасен;
- correlation id проходит через весь поток.

## Архитектурное решение

Создайте ADR: оставить контексты модулями монолита или выделить сервисы. Сравните deployment independence, транзакции, нагрузку, размер команды, observability, on-call и стоимость миграции. Решение должно следовать из требований, а не из предпочтения технологии.

## Критерии готовности

- контракты не раскрывают внутренние entities;
- outbox/inbox участвуют в локальных транзакциях;
- retries ограничены и различают типы ошибок;
- промежуточные состояния видимы пользователю и мониторингу;
- тесты воспроизводят дубли и частичные отказы;
- README объясняет границу каждой гарантии.
