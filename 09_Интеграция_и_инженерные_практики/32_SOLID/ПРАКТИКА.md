# Практика — урок 32

## Цель

Рефакторить намеренно неудобный checkout под защитой тестов. Не требуется «применить пять паттернов»: каждое изменение должно обслуживать названный сценарий.

## Исходная проблема

Создайте `LegacyCheckoutService`, который в одном методе:

- валидирует DTO;
- рассчитывает скидку через `switch`;
- создаёт заказ;
- пишет напрямую в хранилище;
- вызывает конкретный payment SDK;
- формирует email;
- читает текущее время;
- ловит все исключения и возвращает `false`.

Сначала напишите characterization tests существующего поведения.

## Сценарии изменения

1. Добавить новую discount policy без изменения checkout orchestration.
2. Подключить второй payment provider с тем же поведенческим контрактом.
3. Отчёт должен только читать заказы.
4. Тест должен управлять временем без ожидания реальных часов.
5. Email временно переносится в outbox, не меняя domain rules.

## Обязательная часть

- SRP: отделите use case orchestration, domain policy, persistence и adapters.
- OCP: создайте узкую, содержательную точку расширения скидок.
- LSP: сформулируйте contract payment gateway и общий набор contract tests для двух реализаций.
- ISP: разделите read/write capabilities по потребностям клиентов.
- DIP: domain/application не ссылаются на SDK, EF/SQL или SMTP.
- Соберите object graph в composition root через constructor injection.
- Не внедряйте `IServiceProvider` в business code.

## Проверки контракта

Для payment gateways одинаково проверьте:

- успешное списание;
- отказ банка как ожидаемый результат;
- timeout/cancellation;
- повтор с одинаковым idempotency key;
- отсутствие неожиданного исключения для документированных исходов.

## Отчёт о решениях

Создайте `SOLID_REVIEW.md`. Для каждого изменения укажите:

- исходную боль;
- применённый принцип;
- какой сценарий стал локальнее;
- добавленную сложность;
- почему более простого решения недостаточно.

Отдельно назовите хотя бы одну потенциальную абстракцию, которую вы сознательно **не** ввели.

## Критерии готовности

- characterization tests проходят до и после рефакторинга;
- новая discount policy добавляется без правки orchestration;
- обе payment реализации проходят один contract suite;
- read-only consumer не видит write methods;
- направления project references ведут к business policy;
- типы названы по бизнес-намерению, а не `Manager/Helper`;
- решение не содержит пустых интерфейсов «на будущее».

## Дополнительное задание

Создайте намеренно нарушающую LSP payment implementation. Покажите красный contract test, затем исправьте модель контракта или реализацию и объясните выбор.
