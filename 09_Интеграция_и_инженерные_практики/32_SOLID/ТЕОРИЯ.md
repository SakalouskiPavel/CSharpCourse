# Урок 32. Насколько твёрдые принципы SOLID

> SOLID — эвристики управления изменениями, а не чек-лист количества интерфейсов. Принцип полезен, когда связывает наблюдаемую боль с конкретной границей и более дешёвым будущим изменением.

## Результаты обучения

- выбирать подход по требованиям и ограничениям;
- проектировать отказоустойчивый рабочий/интеграционный поток;
- делать гарантии, ownership и метрики явными;
- проверять решение на сбоях и изменениях.

## Карта темы

### 1. SRP

Модуль должен иметь одну связанную причину изменения — обычно одного актора/политику, а не одну функцию. Смешение расчёта, persistence и UI заставляет менять код по несвязанным причинам.

### 2. OCP

Поведение должно расширяться через устойчивую точку вариации без переписывания проверенного ядра. Нельзя быть закрытым ко всем изменениям; abstraction выбирают по вероятной оси.

### 3. LSP

Подтип должен сохранять предусловия, постусловия, инварианты и ожидаемое поведение базового контракта. Throw NotSupported в обычной операции часто показывает ложную иерархию.

### 4. ISP

Клиенты зависят от узких ролей, которые используют. Размер интерфейса определяется cohesion клиента, а не формальным числом методов.

### 5. DIP

Высокоуровневая политика и детали зависят от абстракции, принадлежащей потребности политики. Dependency injection — техника сборки, но сама не гарантирует правильного направления.

### 6. Связь принципов

ISP помогает LSP, SRP выявляет границы OCP, DIP стабилизирует policy. Одно изменение может улучшить несколько принципов, поэтому не нужно по классу на букву.

### 7. Стоимость abstraction

Интерфейс, indirection и generic framework имеют cognitive cost. Извлекайте variation point из подтверждённых сценариев, а не из воображаемого будущего.

### 8. Тестируемость

Хорошие seams позволяют проверить политику без инфраструктуры. Но интерфейс только ради mock framework может закрепить implementation details и хрупкие interaction tests.

### 9. Метрики и smell

Large class, shotgun surgery, unstable dependencies и частые совместные изменения — сигналы, не приговор. Решение подтверждает история изменений и простота use case.

## Минимальный пример

```csharp
public sealed class InvoiceService(ITaxPolicy taxPolicy, IInvoiceStore store)
{
    public async Task<Invoice> CreateAsync(Order order, CancellationToken ct)
    {
        var invoice = Invoice.Create(order, taxPolicy.Calculate(order));
        await store.SaveAsync(invoice, ct);
        return invoice;
    }
}
```

Пример фиксирует только форму. В инженерной системе дополнительно опишите владельца, гарантии доставки/завершения, лимиты, совместимость, безопасность и сигнал, по которому решение считается здоровым.

## Типичные ошибки

- выбирать инструмент по популярности вместо требований;
- обещать exactly-once или нулевой downtime без определения границ;
- оставлять retry, rollback и ownership «инфраструктуре»;
- измерять активность вместо ценности и времени потока;
- усложнять систему до появления подтверждённой причины.

## Итог

Зрелое решение включает не только happy path, но и миграцию, частичный отказ, повтор, наблюдаемость и способ безопасно изменить контракт. Компромиссы следует записывать до того, как они станут неожиданными свойствами production.
