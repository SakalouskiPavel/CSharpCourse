# Урок 32. SOLID без догм

> SOLID — пять эвристик для управления изменениями в объектном коде. Они не требуют делать интерфейс для каждого класса и не заменяют понимание предметной области.

## Результаты обучения

Вы сможете:

- находить причины изменения класса и его клиентов;
- применять SRP, OCP, LSP, ISP и DIP на конкретных сценариях;
- проверять подстановку через контракт, а не наследование синтаксиса;
- отличать dependency inversion от dependency injection;
- распознавать чрезмерные абстракции и выбирать простой дизайн;
- безопасно рефакторить код под тестами.

## 1. Зачем нужны принципы

Работающий сегодня код может быть дорогим завтра: изменение расчёта скидки ломает email, добавление доставки требует править один гигантский `switch`, тест зависит от сети. SOLID помогает локализовать будущие изменения.

Принцип — не закон компилятора. У каждого выделения есть цена: больше типов, навигации и концепций. Сначала называют реальную ось изменения, затем вводят ровно столько структуры, сколько её обслуживает.

## 2. Cohesion, coupling и причины изменения

Высокая cohesion означает, что части модуля служат одной понятной цели. Низкая coupling — модуль мало знает о деталях других. Эти характеристики важнее числа файлов.

Класс `OrderService`, который валидирует заказ, считает налог, пишет SQL, формирует HTML и отправляет SMTP, соединяет разные причины изменения: бизнес-правила, БД, представление и инфраструктуру.

## 3. SRP: Single Responsibility Principle

SRP формулируют как «модуль должен иметь одну причину изменения» или быть ответственным перед одним актором/группой требований. Это не «один метод на класс».

```csharp
public sealed class PlaceOrderHandler
{
    public async Task<Result<OrderId>> Handle(Command command, CancellationToken ct)
    {
        var order = Order.Place(command.CustomerId, command.Lines);
        await repository.AddAsync(order, ct);
        return order.Id;
    }
}
```

Handler координирует один use case. Domain object защищает правила, repository отвечает за persistence, presenter/API — за HTTP. Разделение следует границам решений, а не стремлению получить крошечные классы.

## 4. Признаки нарушения SRP

- в описании класса много несвязанных «и»;
- разные команды постоянно правят один файл по разным причинам;
- тест use case требует SMTP, БД и HTTP одновременно;
- изменение формата отчёта задевает расчёт;
- класс знает слишком много инфраструктурных деталей.

Большой класс не обязательно нарушает SRP, а маленький — не обязательно соблюдает. Важна связанность ответственности.

## 5. OCP: Open/Closed Principle

Модуль открыт для расширения и закрыт для модификации в отношении **предвиденной оси вариации**. Новую стратегию оплаты желательно добавить реализацией, не меняя стабильный checkout.

```csharp
public interface IDiscountPolicy
{
    Money Calculate(Order order);
}

public sealed class LoyalCustomerDiscount : IDiscountPolicy { /* ... */ }
```

OCP не требует никогда не менять код. Исправление ошибки и новая бизнес-модель закономерно меняют модуль. Абстракция оправдана, если варианты действительно независимы и имеют устойчивый общий контракт.

## 6. Полиморфизм вместо растущего condition

Большой `switch` по типу доставки, повторяющийся в расчёте, UI и persistence, показывает разбросанную вариацию. Strategy или handler registry может собрать её в реализации.

Но один маленький исчерпывающий `switch` по закрытому набору вариантов иногда яснее десяти классов. Pattern matching и discriminated-like hierarchy полезны, когда набор закрыт; полиморфизм — когда расширяется независимо.

## 7. Extension points должны быть узкими

Интерфейс `IPlugin.Execute(object context)` формально расширяем, но его контракт ничего не говорит. Хорошая точка расширения определяет вход, результат, ошибки, lifetime и invariants.

Не следует проектировать все возможные будущие плагины заранее. Rule of three и история изменений помогают отличить устойчивую вариацию от фантазии.

## 8. LSP: Liskov Substitution Principle

Если код ожидает базовый тип `T`, объект подтипа `S` должен использоваться без нарушения корректности клиента. Важен поведенческий контракт, не только совпадение сигнатур.

Подтип не должен:

- усиливать предусловия;
- ослаблять постусловия;
- нарушать invariants;
- вводить неожиданные исключения;
- менять обещанную семантику side effects.

Если `IOrderRepository.FindAsync` обещает `null` при отсутствии, реализация не должна в этом случае бросать `NotFoundException` без изменения контракта.

## 9. Предусловия и постусловия

Предусловие — что требуется до вызова; постусловие — что гарантируется после. Если базовый shipping calculator принимает любой неотрицательный вес, подтип, работающий только от 10 кг, усиливает предусловие и ломает клиента.

Подтип может принимать **больше** допустимых входов и обещать **не меньше** результата. Контракт включает документацию, nullable-аннотации, ошибки, ordering, mutability и performance, если клиент на них обоснованно рассчитывает.

## 10. Классические ловушки наследования

Квадрат как изменяемый прямоугольник ломает ожидание независимого изменения ширины и высоты. `ReadOnlyCollection` как подтип изменяемой коллекции с `NotSupportedException` на `Add` также нарушает ожидаемую способность.

Решение часто не в хитром override, а в другой модели: immutable `Shape`, отдельные read/write interfaces, composition вместо inheritance.

## 11. Как проверять LSP

Один набор **contract tests** запускают для всех реализаций интерфейса. Тесты фиксируют наблюдаемое клиентом поведение: отсутствие, дубли, cancellation, ошибки, порядок и side effects.

Моки тоже обязаны уважать контракт. Fake repository, разрешающий невозможные состояния, создаёт зелёные, но ложные тесты.

## 12. ISP: Interface Segregation Principle

Клиент не должен зависеть от методов, которыми не пользуется. Большой `IOrderService` с Create/Read/Cancel/Export/Admin вынуждает read-only consumer знать лишнее и затрудняет реализацию.

Role interfaces описывают потребность клиента:

```csharp
public interface IOrderReader
{
    Task<OrderDto?> FindAsync(OrderId id, CancellationToken ct);
}

public interface IOrderWriter
{
    Task SaveAsync(Order order, CancellationToken ct);
}
```

Интерфейс принадлежит абстракции со стороны потребителя, а не обязан зеркалить все public methods реализации.

## 13. ISP и безопасность изменений

Узкий интерфейс уменьшает поверхность mock, разрешений и случайного использования. Компонент отчёта с `IOrderReader` физически не может отменить заказ через эту зависимость.

Однако интерфейс из одного метода не всегда лучше: методы, меняющиеся вместе и представляющие одну роль, логично оставить рядом. Нельзя дробить до потери смысла.

## 14. DIP: Dependency Inversion Principle

Высокоуровневая бизнес-политика не должна зависеть от низкоуровневой детали; обе зависят от абстракции. Абстракция не должна формироваться вокруг устройства конкретной БД или SMTP.

`PlaceOrderHandler` зависит от `IOrderRepository` и `IUnitOfWork`, определённых нуждами application/domain. Infrastructure реализует их через EF Core. Направление исходных зависимостей идёт к политике, даже если runtime-вызов идёт в БД.

## 15. DIP, DI и IoC — не одно и то же

- DIP — принцип направления зависимостей;
- Dependency Injection — способ передать объекту зависимости извне;
- IoC container — инструмент построения object graph.

Можно использовать constructor injection и всё равно нарушить DIP: handler принимает конкретный `SqlOrderRepository`. Можно соблюдать DIP без контейнера, вручную создав graph в composition root.

## 16. Composition Root и constructor injection

Composition root — одно место на внешней границе, где выбирают реализации и lifetimes.

```csharp
services.AddScoped<IOrderRepository, EfOrderRepository>();
services.AddScoped<PlaceOrderHandler>();
```

Constructor injection делает обязательные зависимости видимыми и позволяет создать валидный объект сразу. Service Locator (`serviceProvider.GetService` внутри business code) скрывает требования и переносит ошибку в runtime.

## 17. Stable Abstractions

Хорошая абстракция выражает стабильную бизнес-потребность, например `IPaymentGateway.Charge`, а не копирует API одного поставщика. Adapter переводит её в SDK провайдера.

Интерфейс над каждым классом «для тестов» часто даёт только дублирование. Абстрагируют volatile boundary: время, сеть, файловую систему, БД, внешний API или реальную взаимозаменяемую policy.

## 18. SOLID вместе на одном примере

В checkout:

- SRP отделяет orchestration, pricing, persistence и notification;
- OCP позволяет добавить discount policy;
- LSP требует одинакового контракта payment gateways;
- ISP даёт use case только нужные capabilities;
- DIP направляет зависимости от adapters к business policy.

Принципы поддерживают друг друга, но могут конфликтовать с простотой. Для одной неизменной реализации прямой код может быть честнее framework из фабрик и стратегий.

## 19. Code smells — сигналы, не доказательства

Long method, shotgun surgery, feature envy, divergent change, giant interface и switch могут указать на проблему. Сначала выясняют причину изменения и тестируют текущее поведение. Автоматическое «лечение» smell способно ухудшить cohesion.

Duplication знаний опаснее совпадения строк. Два похожих расчёта с разными владельцами могут развиваться независимо; преждевременное объединение создаст ложную абстракцию.

## 20. Безопасный рефакторинг

1. Зафиксировать observable behavior тестами.
2. Назвать конкретную боль/сценарий изменения.
3. Делать маленькое преобразование.
4. Запускать тесты и анализировать diff.
5. Только затем добавлять новое поведение.

Рефакторинг меняет структуру без намеренного изменения поведения. Если одновременно переписать архитектуру и требования, источник ошибки трудно найти.

## 21. Когда остановиться

Признаки overengineering: интерфейс с одной реализацией без volatile boundary, фабрика фабрик, абстрактные имена `Manager/Processor`, навигация через десятки файлов для простой операции, конфигурация варианта, который никогда не меняется.

Цель — не максимальное количество extension points, а минимальная стоимость вероятных изменений при понятном коде сегодня.

## 22. Типичные ошибки

- трактовать SRP как «один метод»;
- создавать strategy для каждого `if`;
- считать наследование достаточным для LSP;
- бросать `NotSupportedException` вместо честного интерфейса;
- размещать интерфейс по форме реализации, а не нужде клиента;
- путать DIP с IoC container;
- внедрять service provider в domain code;
- мокировать всё, включая простые value objects;
- применять SOLID без сценария изменения.

## Итог

SOLID помогает делать изменения локальными и контракты честными. SRP группирует одну ответственность, OCP защищает стабильную политику от ожидаемой вариации, LSP сохраняет обещания, ISP уменьшает ненужные зависимости, DIP направляет код к бизнес-правилам. Простота и доказанные требования остаются главным ограничителем.
