# Урок 09. Атрибуты

> Атрибуты добавляют декларативные метаданные к коду. Они ничего не делают сами: значение появляется только вместе с потребителем — компилятором, runtime, фреймворком или вашей инфраструктурой.

## Результаты обучения

- объявлять и применять custom attributes
- задавать AttributeUsage
- читать метаданные без лишнего создания объектов
- проектировать декларативный API без скрытой магии

## Карта темы

### 1. Модель атрибутов

Класс наследуется от `Attribute`; позиционные аргументы идут в конструктор, именованные — в settable свойства/поля. Значения ограничены константно представимыми типами.

### 2. Цели применения

Атрибут можно поставить на assembly, type, member, parameter, return value и другие цели. Префикс цели (`assembly:`, `return:`) устраняет неоднозначность.

### 3. AttributeUsage

`ValidOn` ограничивает цели, `AllowMultiple` разрешает несколько экземпляров, `Inherited` управляет видимостью на производных типах. Настройки являются частью контракта.

### 4. Чтение

`GetCustomAttribute` создаёт экземпляр атрибута. `CustomAttributeData` читает метаданные без выполнения конструктора и лучше для анализаторов или недоверенных сборок.

### 5. Наследование

Наследование атрибутов зависит и от `AttributeUsage`, и от API чтения. Для переопределённых методов поведение требует явной проверки, а не предположения.

### 6. Стандартные атрибуты

`Obsolete`, nullable-аннотации, conditional и caller info влияют на компилятор; serialization/validation/test attributes читают конкретные библиотеки.

### 7. Дизайн потребителя

Сначала определяется алгоритм и поведение при конфликте/отсутствии метаданных, затем форма атрибута. Ошибки конфигурации следует выявлять при старте, а не в середине запроса.

### 8. Альтернативы

Интерфейс лучше, когда объект действительно обладает поведением; конфигурация — когда данные меняются без перекомпиляции; source generator — когда нужна compile-time проверка.

## Минимальный пример

```csharp
[AttributeUsage(AttributeTargets.Class, AllowMultiple = false, Inherited = false)]
public sealed class CommandAttribute(string name) : Attribute
{
    public string Name { get; } = name;
    public string? Description { get; init; }
}

[Command("import", Description = "Imports a file")]
public sealed class ImportCommand { }
```

Разберите пример построчно: какие гарантии выражены типами, где проверяются предусловия и что увидит вызывающий код. Затем измените один инвариант и проследите, какая часть публичного контракта должна поменяться.

## Типичные ошибки

- ожидать, что атрибут автоматически изменит поведение;
- помещать изменяемую бизнес-конфигурацию в metadata;
- выполнять тяжёлую работу в конструкторе атрибута;
- молча принимать конфликтующие атрибуты;

## Итог

Тема считается усвоенной, когда вы можете не только применить механизм, но и объяснить его цену, область корректности и более простой альтернативный вариант. Проверяйте решения сборкой, тестами и небольшими экспериментами, а не только чтением.
