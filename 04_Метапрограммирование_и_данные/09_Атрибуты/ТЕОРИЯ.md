# Урок 09. Атрибуты: декларативные метаданные

> Атрибут позволяет прикрепить к типу, методу или свойству дополнительное описание. Он не выполняет действие сам: поведение появляется только у кода, который этот атрибут читает и интерпретирует.

## 1. Знакомый пример

```csharp
[Obsolete("Используйте SendAsync")]
public void Send()
{
}
```

`[Obsolete]` — атрибут. Компилятор знает его смысл и предупреждает при вызове `Send`. Если создать `[CourseNote]`, компилятор не начнёт показывать заметку автоматически: нужен собственный consumer.

Атрибут — metadata, consumer — поведение. Это главная граница урока.

## 2. Объявление собственного атрибута

```csharp
[AttributeUsage(AttributeTargets.Class, Inherited = false)]
public sealed class CommandAttribute : Attribute
{
    public string Name { get; }
    public string? Description { get; init; }

    public CommandAttribute(string name)
    {
        if (string.IsNullOrWhiteSpace(name))
            throw new ArgumentException("Имя команды обязательно", nameof(name));

        Name = name;
    }
}
```

Класс наследуется от `System.Attribute`, обычно имеет суффикс `Attribute`, хотя при применении суффикс можно опустить:

```csharp
[Command("import", Description = "Импортирует файл")]
public sealed class ImportCommand : ICommandHandler
{
}
```

`"import"` — positional argument конструктора. `Description = ...` — named argument свойства.

## 3. Допустимые значения

Аргументы атрибута должны быть представлены в metadata на этапе компиляции. Допустимы простые константные типы, `string`, `Type`, enum, некоторые массивы этих типов и `null`.

Нельзя передать произвольный объект, результат сетевого вызова или `DateTime.Now`:

```csharp
// [Command(GetNameFromDatabase())] // не компилируется
```

Атрибут не является runtime-конфигурацией. Часто меняющиеся адреса, таймауты и секреты должны приходить из configuration, а не перекомпилироваться в metadata.

## 4. `AttributeUsage`

```csharp
[AttributeUsage(
    AttributeTargets.Class | AttributeTargets.Method,
    AllowMultiple = true,
    Inherited = false)]
```

- `ValidOn`/первый аргумент задаёт допустимые targets;
- `AllowMultiple` разрешает несколько атрибутов одного типа на одной цели;
- `Inherited` определяет, виден ли атрибут базового типа на производном при соответствующем способе чтения.

Ограничивайте targets по реальному смыслу. Атрибут команды на поле создаёт вопрос, который consumer всё равно не умеет осмысленно решить.

## 5. Цели применения

Атрибуты ставятся на assembly, module, type, constructor, method, property, field, event, parameter, return value и generic parameter.

Префикс уточняет цель:

```csharp
[assembly: AssemblyMetadata("Course", "CSharp")]

[return: NotNullIfNotNull(nameof(input))]
public string? Normalize(string? input) => input?.Trim();
```

Для auto-property атрибут иногда направляют на сгенерированное поле:

```csharp
[field: NonSerialized]
public string Cache { get; set; } = "";
```

Префикс нужен, когда одна синтаксическая конструкция создаёт несколько возможных metadata targets.

## 6. Чтение через Reflection

```csharp
CommandAttribute? attribute = type.GetCustomAttribute<CommandAttribute>();

if (attribute is not null)
    Console.WriteLine(attribute.Name);
```

`GetCustomAttribute<T>` создаёт экземпляр атрибута. Другие операции:

- `IsDefined` — проверить наличие;
- `GetCustomAttributes<T>` — получить все;
- параметр `inherit` — учитывать наследование, если конкретный API и AttributeUsage его поддерживают.

Consumer обязан определить поведение:

- атрибут отсутствует;
- атрибутов несколько;
- имена конфликтуют;
- атрибут стоит на неподходящей сигнатуре;
- constructor атрибута выбросил исключение.

Лучше валидировать metadata один раз при старте, а не обнаруживать конфликт на первом пользовательском запросе.

## 7. `CustomAttributeData`

```csharp
foreach (CustomAttributeData data in type.CustomAttributes)
{
    Console.WriteLine(data.AttributeType.Name);
    foreach (CustomAttributeTypedArgument argument in data.ConstructorArguments)
        Console.WriteLine(argument.Value);
}
```

`CustomAttributeData` читает описание constructor/named arguments без создания экземпляра атрибута. Это полезно анализаторам и при изучении внешней assembly, где выполнять attribute constructor нежелательно.

Конструктор атрибута всё равно должен быть простым: сохранять значения, а не читать файл, сеть или service locator. Неожиданное выполнение при Reflection сделает систему хрупкой.

## 8. Наследование атрибутов

```csharp
[AttributeUsage(AttributeTargets.Class, Inherited = true)]
public sealed class AuditedAttribute : Attribute
{
}

[Audited]
public class BaseService { }

public class OrderService : BaseService { }
```

При подходящем API `OrderService` может считаться audited. Но правила отличаются для types, overridden members, parameters и properties. Нельзя делать вывод только по `Inherited = true`; проверяйте документацию конкретного метода чтения и тестируйте иерархию.

## 9. Стандартные атрибуты

### Компилятор и анализ

- `Obsolete` — предупреждение/ошибка для устаревшего API;
- `Conditional` — включает вызов при compilation symbol;
- caller info (`CallerMemberName`, `CallerFilePath`, `CallerLineNumber`) подставляет сведения о месте вызова;
- nullable analysis attributes уточняют потоковый анализ null.

### Runtime и библиотеки

- `Flags` меняет ожидаемое использование enum;
- `Serializable` имеет смысл для конкретных старых механизмов, но не делает тип автоматически JSON-контрактом;
- `JsonPropertyName`, `JsonIgnore`, `JsonConstructor` читает System.Text.Json;
- validation/test/framework attributes читают соответствующие библиотеки.

Одинаковая форма `[Something]` не означает одинакового consumer.

## 10. Декларативный подход

Императивный код говорит «выполни шаги». Декларативные metadata говорят «этот элемент обладает характеристикой», а инфраструктура решает как её обработать.

Преимущества:

- описание находится рядом с целью;
- можно сканировать единообразно;
- уменьшается повтор registration code.

Цена:

- поведение скрыто от прямого чтения метода;
- ошибки переходят в runtime;
- сложнее найти consumer;
- чрезмерная «магия» затрудняет debugging;
- trimming/AOT требуют явности.

Хороший framework документирует, кто читает атрибут, когда выполняется валидация и что происходит при конфликте.

## 11. Атрибут, интерфейс или configuration

Выберите **интерфейс**, когда объект реально выполняет роль и клиент вызывает поведение.

Выберите **атрибут**, когда элементу нужна компактная compile-time metadata, которую читает инфраструктура: route name, serialization name, marker.

Выберите **configuration**, когда значение меняется по environment или deployment: URL, лимит, feature flag, secret.

Выберите **source generator**, когда metadata может породить типизированный код и ошибки желательно увидеть при сборке.

Пример: `[Retry(3)]` выглядит удобно, но retry зависит от вида операции, idempotency и окружения. Configuration/policy object может быть честнее одного числа в metadata.

## 12. Проектирование custom attribute

Порядок:

1. Опишите consumer и его алгоритм.
2. Определите target и нужен ли multiple/inheritance.
3. Разделите обязательные positional и optional named values.
4. Ограничьте значения простыми metadata.
5. Определите отсутствие, конфликт и дубликаты.
6. Валидируйте при старте.
7. Кэшируйте обработанный descriptor.
8. Документируйте поведение, а не только синтаксис.

## Частые ошибки

- ожидать, что атрибут сам запускает код;
- помещать environment configuration и secrets в metadata;
- не ограничивать AttributeTargets;
- разрешать multiple, не определив порядок/конфликт;
- выполнять I/O в constructor атрибута;
- сканировать атрибуты на каждом запросе;
- предполагать наследование без проверки API;
- использовать marker attribute там, где interface яснее;
- скрывать сложную бизнес-логику за набором annotations.

## Итог

```text
Атрибут записывает metadata
  ↓
Reflection/компилятор/framework читает metadata
  ↓
Consumer проверяет правила
  ↓
Только consumer создаёт поведение
```

Вы готовы дальше, если можете создать attribute с корректным AttributeUsage, применить positional/named arguments, прочитать его двумя способами и объяснить, почему атрибут без consumer ничего не меняет.
