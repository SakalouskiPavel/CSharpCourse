# Практика — урок 09

## Цель

Заменить имя/описание команд урока 08 декларативной metadata и построить registry, который валидирует атрибуты один раз при запуске.

## Атрибут команды

Создайте `CommandAttribute`:

- применяется только к class;
- не наследуется;
- разрешён один экземпляр;
- обязательное имя передаётся constructor argument;
- optional description — named property;
- constructor проверяет пустое имя.

Примените его минимум к трём `ICommandHandler`.

## Атрибут роли

Создайте `RequiresRoleAttribute`:

- применяется к class и method;
- допускает несколько экземпляров;
- хранит одну непустую роль.

Определите и запишите семантику нескольких ролей: требуется любая или все. Registry не должен угадывать.

## Consumer

При старте consumer:

1. находит handler types по интерфейсу;
2. требует `CommandAttribute`;
3. проверяет уникальность имени ordinal-ignore-case;
4. проверяет допустимую сигнатуру handler;
5. собирает immutable/read-only descriptor;
6. агрегирует все ошибки конфигурации;
7. после успешной сборки не читает attributes на каждый Execute.

Descriptor отделите от экземпляра attribute. Он содержит только данные, нужные runtime registry.

## Автоматическая справка

Сформируйте таблицу команд: имя, описание, роли. Сортировка выполняется явно. Команда без description получает документированное значение, например «Описание отсутствует».

## `CustomAttributeData`

Отдельный diagnostic mode выводит constructor/named arguments через CustomAttributeData без создания attribute object. Сравните результат с `GetCustomAttribute<T>`.

## Ошибочные случаи

Создайте учебные типы:

- handler без CommandAttribute;
- два handler с одинаковым именем;
- пустая роль;
- наследник типа с CommandAttribute при `Inherited=false`;
- несколько RequiresRole attributes.

Для каждого запишите ожидаемое поведение до запуска.

## Ограничения

- атрибуты не содержат service calls и mutable runtime state;
- authorization не считается реализованной только наличием marker-а: consumer обязан проверить роль перед вызовом;
- metadata сканируется один раз;
- конфликт не перезаписывается молча в dictionary;
- environment values и secrets не помещаются в атрибуты.

## Критерии приёмки

- AttributeUsage соответствует смыслу обоих атрибутов;
- positional/named arguments используются осознанно;
- все ошибочные metadata выявляются до выполнения команд;
- справка формируется из descriptors;
- роли реально проверяются consumer-ом;
- CustomAttributeData продемонстрирован отдельно;
- README отвечает: кто, когда и как читает каждый атрибут.

## Дополнительное задание

Добавьте attribute к return value или parameter и прочитайте его через Reflection. Затем предложите, как source generator мог бы перенести проверку дубликатов в compile time.
