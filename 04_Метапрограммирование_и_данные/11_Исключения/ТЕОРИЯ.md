# Урок 11. Исключения и нюансы работы с ними

> Исключение сообщает, что операция не может выполнить обещанный контракт. Надёжный код различает ожидаемый альтернативный исход и исключительный сбой, сохраняет исходную диагностику и освобождает ресурсы.

## Результаты обучения

- выбирать между результатом и исключением
- строить точные try/catch/finally границы
- сохранять stack trace и causal chain
- проектировать собственные исключения

## Карта темы

### 1. Модель исключений

При `throw` CLR разворачивает стек до подходящего handler. `finally` выполняется при выходе из блока и служит cleanup, но `using` обычно выражает владение яснее.

### 2. Исключение или результат

Невалидный аргумент API и сломанная инфраструктура обычно исключительны; «пользователь не найден» может быть нормальным результатом. Исключения не стоит использовать как управление частым ветвлением.

### 3. Стандартные типы

`ArgumentNullException`, `ArgumentOutOfRangeException`, `InvalidOperationException`, `NotSupportedException` передают стандартный смысл. Точный тип помогает вызывающему коду и диагностике.

### 4. Catch и filters

Ловите только то, что можете обработать, на границе с нужным контекстом. Exception filters `when` выбирают обработчик без изменения исходного stack trace.

### 5. Повторный throw

`throw;` сохраняет стек, `throw ex;` начинает его заново. Оборачивать исключение стоит, когда новый тип добавляет смысл уровня, сохраняя исходное в InnerException.

### 6. Пользовательские исключения

Custom exception полезен для устойчивой категории ошибки, которую клиент должен различать. Он получает понятное имя, стандартные constructors по необходимости и неизменяемый контекст без секретов.

### 7. Async-исключения

Исключение async-метода хранится в Task и возникает при `await`. Несколько параллельных ошибок требуют понимания WhenAll; `async void` почти невозможно корректно ожидать и обрабатывать.

### 8. Логирование

Логировать следует один раз на границе ответственности. Сообщение содержит контекст, structured fields и correlation id, но не токены, пароли или персональные данные.

### 9. Надёжность

Не ловите фатальные состояния без ясной стратегии. Cleanup не должен скрывать исходную ошибку; cancellation обычно распространяется, а не превращается в «неизвестную ошибку».

## Минимальный пример

```csharp
try
{
    await repository.SaveAsync(order, cancellationToken);
}
catch (DbException ex) when (IsUniqueViolation(ex))
{
    throw new DuplicateOrderException(order.Id, ex);
}
```

Пример — отправная точка, а не готовая архитектура. Проверьте предусловия, lifetime объектов, обработку ошибок и поведение на границах. Для инфраструктурных тем отдельно фиксируйте, какие гарантии даёт библиотека, а какие должен обеспечить ваш код.

## Типичные ошибки

- пустой `catch (Exception)`;
- `throw ex` и потеря исходного стека;
- логирование и повторный throw на каждом слое;
- возвращать `null` для любого сбоя;
- включать секреты в exception message;

## Итог
