# Урок 11. Исключения: ошибки, диагностика и восстановление

> Исключение сообщает, что выполняемая операция не смогла выполнить обещанный контракт. Задача разработчика — не «поймать всё», а отличить ожидаемый альтернативный результат от настоящего сбоя, сохранить причину и восстановиться только там, где это возможно.

## 1. Три вида проблем

Из урока 01:

- **ошибка компиляции** — нарушены правила языка; программа не собрана;
- **ошибка выполнения** — во время корректно собранной операции возникла невозможная ситуация;
- **логическая ошибка** — программа завершилась, но дала неверный результат.

Исключения относятся к выполнению. Они не исправляют логическую ошибку и не заменяют проверку требований.

```csharp
int number = int.Parse("не число");
```

Код компилируется, но `Parse` выбросит `FormatException`.

## 2. Что происходит при `throw`

```csharp
decimal Divide(decimal left, decimal right)
{
    if (right == 0)
        throw new DivideByZeroException("Делитель равен нулю");

    return left / right;
}
```

`throw` немедленно прекращает обычный путь метода. CLR ищет подходящий `catch`:

1. в текущем методе;
2. в вызвавшем методе;
3. выше по call stack;
4. до верхней границы процесса.

Этот подъём называется **stack unwinding**. По пути выполняются `finally`-блоки. Если обработчика нет, приложение/операция завершается, а host выводит необработанное исключение.

## 3. Объект исключения и stack trace

Все обычные исключения наследуются от `System.Exception`. Полезные свойства:

- `Message` — описание;
- `StackTrace` — цепочка вызовов и места кода;
- `InnerException` — исходная причина при оборачивании;
- `Data` — дополнительный словарь, использовать сдержанно.

Stack trace отвечает «каким путём выполнение пришло к ошибке». Поэтому нельзя заменять диагностику только сообщением `Что-то пошло не так`.

Не включайте в message пароли, токены, полные персональные данные и содержимое секретных файлов.

## 4. `try`, `catch`, `finally`

```csharp
try
{
    string json = File.ReadAllText(path);
    return JsonSerializer.Deserialize<BookDto>(json);
}
catch (FileNotFoundException ex)
{
    Console.Error.WriteLine($"Файл не найден: {ex.FileName}");
    return null;
}
catch (JsonException ex)
{
    Console.Error.WriteLine($"Повреждён JSON: {ex.Message}");
    return null;
}
finally
{
    Console.WriteLine("Попытка чтения завершена");
}
```

- `try` ограничивает потенциально падающую операцию;
- первый совместимый `catch` обрабатывает исключение;
- `finally` выполняется при обычном выходе и при исключении (кроме аварийного завершения процесса и редких фатальных ситуаций).

Ставьте более конкретные catch выше общего. `catch (Exception)` первым сделает остальные недостижимыми.

Try-блок должен быть достаточно узким, чтобы было понятно, какая операция обрабатывается. Огромный try вокруг всего приложения смешивает несвязанные сбои.

## 5. Ожидаемый результат или исключение

Пользователь ввёл нечисловой текст — обычный сценарий:

```csharp
if (!int.TryParse(input, out int value))
    Console.WriteLine("Введите целое число");
```

Файл конфигурации, обязательный для запуска, недоступен — операция не может выполнить контракт, исключение уместно.

Полезный вопрос: может ли вызывающий код разумно ожидать этот исход как обычную ветку?

- поиск не нашёл необязательный товар → `null`, `Try...`, Result;
- аргумент метода нарушает documented precondition → argument exception;
- база данных недоступна → exception;
- пользователь отменил операцию → cancellation, отдельный семантический исход;
- дубликат при регистрации → зависит от контракта: Result/Conflict часто яснее exception.

Исключения дороги при создании stack trace и плохо читаются как частое управление циклом.

## 6. Стандартные типы

- `ArgumentNullException` — обязательный аргумент null;
- `ArgumentException` — аргумент имеет неверную форму;
- `ArgumentOutOfRangeException` — значение вне диапазона;
- `InvalidOperationException` — текущее состояние объекта не допускает операцию;
- `NotSupportedException` — операция принципиально не поддерживается;
- `KeyNotFoundException`, `FormatException`, `IOException`, `TimeoutException` — более конкретные категории.

```csharp
ArgumentNullException.ThrowIfNull(order);

if (quantity <= 0)
    throw new ArgumentOutOfRangeException(
        nameof(quantity),
        quantity,
        "Количество должно быть положительным");
```

Точный стандартный тип лучше собственного `CourseException` без устойчивого смысла.

## 7. Catch только там, где можно действовать

Обработать исключение означает выполнить осмысленное действие:

- повторить безопасную transient operation по политике;
- переключиться на fallback;
- преобразовать в понятный результат границы;
- добавить контекст и пробросить;
- гарантировать cleanup;
- завершить приложение с корректным exit code.

Пустой catch повреждает наблюдаемость:

```csharp
try { Save(); }
catch { } // ошибка потеряна, вызывающий думает об успехе
```

Если слой не знает, что делать, он обычно не ловит исключение.

## 8. Повторный проброс

```csharp
catch (IOException)
{
    CleanupTemporaryFile();
    throw;
}
```

`throw;` сохраняет исходный stack trace. `throw ex;` начинает trace с текущего места и скрывает полезный путь — так делать не следует.

Если слой добавляет новый смысл, оборачивайте с inner exception:

```csharp
catch (IOException ex)
{
    throw new CatalogStorageException(
        $"Не удалось загрузить каталог '{catalogId}'",
        ex);
}
```

Обёртка оправдана, когда новый тип — стабильная категория уровня приложения, а сообщение добавляет безопасный контекст.

## 9. Exception filters

```csharp
catch (HttpRequestException ex) when (ex.StatusCode == HttpStatusCode.NotFound)
{
    return null;
}
```

`when` выбирает handler по условию. Filter вычисляется до входа в catch и не портит исходный stack trace. Он удобен для категории/кода, но не должен выполнять side effects или сложную бизнес-логику.

## 10. Собственное исключение

```csharp
public sealed class CatalogStorageException : Exception
{
    public string CatalogId { get; }

    public CatalogStorageException(
        string catalogId,
        string message,
        Exception innerException)
        : base(message, innerException)
    {
        CatalogId = catalogId;
    }
}
```

Custom exception нужен, если вызывающий должен различать устойчивую предметную/прикладную категорию. Контекст лучше хранить отдельным property, а не заставлять parser разбирать Message.

Не создавайте новый тип для каждой строки ошибки. Не храните mutable state и секреты. Старое требование `[Serializable]`/serialization constructors относится главным образом к legacy cross-AppDomain/.NET Framework scenarios и не должно копироваться механически в современный код.

## 11. Cleanup, `finally` и `using`

Ресурс нужно освободить при успехе и исключении:

```csharp
using FileStream stream = File.OpenRead(path);
// работа с потоком
```

`using` компилируется в `try/finally`, вызывающий `Dispose`. Это яснее ручного finally. Для async cleanup есть `await using` и `IAsyncDisposable`.

Cleanup сам может завершиться ошибкой. Нельзя бездумно скрыть исходное исключение вторичной ошибкой; библиотеки ресурсов обычно проектируют Dispose устойчивым и идемпотентным.

## 12. Исключения в async-коде

```csharp
Task<string> task = LoadAsync(path, cancellationToken);
string text = await task;
```

Исключение async operation сохраняется в `Task` и повторно выбрасывается при `await`. Если Task не await-ится/не наблюдается, ошибка теряется для нормального control flow.

```csharp
try
{
    await LoadAsync(path, cancellationToken);
}
catch (IOException ex)
{
    // обработка
}
```

`async void` допускается главным образом для event handlers: вызывающий не получает Task и не может нормально ждать/ловить ошибку.

## 13. Несколько async-ошибок

```csharp
Task first = SaveFirstAsync();
Task second = SaveSecondAsync();
await Task.WhenAll(first, second);
```

Обе операции могут завершиться ошибкой. Await сообщает failure, а все детали доступны в состояниях tasks/Exception агрегированной задачи. Не прекращайте наблюдение остальных tasks после первой ошибки.

Политика частичного успеха должна быть явной: отменить оставшиеся, дождаться всех, собрать ошибки или применить компенсацию.

## 14. Cancellation — не обычная ошибка

`CancellationToken` сообщает cooperative cancellation. Поддерживающий метод обычно бросает `OperationCanceledException`/`TaskCanceledException`, и Task получает состояние Canceled.

```csharp
catch (OperationCanceledException) when (cancellationToken.IsCancellationRequested)
{
    throw; // сохраняем семантику отмены
}
```

Не превращайте отмену в «успех с пустым результатом» и не логируйте ожидаемую пользовательскую отмену как системную аварию. Timeout иногда проявляется тем же типом, поэтому API должен позволять различить source tokens/контекст.

## 15. Логирование

Исключение обычно логируется один раз на границе, которая завершает операцию или превращает его во внешний ответ. Если каждый слой логирует и пробрасывает, одна ошибка создаёт пять одинаковых записей.

Полезный structured log содержит:

- тип операции;
- безопасные identifiers;
- correlation id;
- exception object целиком;
- outcome/retry decision.

Не делайте `logger.LogError(ex.Message)`: теряется stack trace. Передавайте exception отдельным аргументом logging API.

## 16. Границы приложения

На верхней границе console app:

```csharp
try
{
    return await RunAsync(args);
}
catch (ConfigurationException ex)
{
    Console.Error.WriteLine(ex.Message);
    return 2;
}
catch (Exception ex)
{
    logger.LogCritical(ex, "Unhandled application error");
    return 1;
}
```

Граница гарантирует диагностику и exit code, но не притворяется, что неизвестный сбой исправлен. В Web API аналогичная граница превращает известные ошибки в status/problem details и не раскрывает stack trace клиенту.

## 17. Надёжность

Не всякую операцию безопасно повторять. Retry записи заказа может создать дубликат без idempotency. Не ловите `OutOfMemoryException`, `StackOverflowException` и повреждение процесса как обычные recoverable cases.

После исключения объект/транзакция могут иметь неопределённое состояние. Контракт должен указывать, можно ли повторить использование или ресурс нужно закрыть/создать заново.

## Частые ошибки

- исключение используется вместо `TryParse`;
- `catch (Exception)` возвращает успех/null;
- пустой catch скрывает дефект;
- `throw ex` теряет stack trace;
- каждое исключение оборачивается на каждом слое;
- одна ошибка логируется многократно;
- message содержит секрет;
- Task запускается без await/наблюдения;
- cancellation превращается в обычную ошибку;
- retry выполняется без проверки transient/idempotent;
- try охватывает слишком много несвязанных операций.

## Итог

```text
Ожидаемая ветка → Try/Result/null по контракту
Невыполнимый контракт → exception
  ↓
catch только там, где есть действие
  ↓
сохранить stack trace и cause
  ↓
граница логирует/преобразует исход один раз
```

Вы готовы дальше, если можете выбрать между Try/result и exception, написать узкий catch, сохранить cause при оборачивании и объяснить поведение ошибки в async Task.
