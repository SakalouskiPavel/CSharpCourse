# Практика — урок 11

## Цель

Создать импорт JSON Lines: каждая строка файла содержит отдельный `BookDto`. Ошибки отдельных записей являются ожидаемым результатом, а сбой файла/сериализатора — отдельной категорией.

## Контракты результата

Создайте:

- `ImportItemError`: номер строки, безопасный код и понятное сообщение;
- `ImportReport`: успешные книги, ошибки записей, количество строк;
- `CatalogImportException`: ошибка всей операции с path/id и InnerException.

Не используйте exception для каждой невалидной строки после её разбора. Собирайте item errors и продолжайте, если формат задания это допускает.

## Алгоритм

1. Открыть файл через async stream/reader с `using`.
2. Читать строки последовательно с CancellationToken.
3. Пустую строку обработать по зафиксированному правилу.
4. Deserialize DTO в узком try/catch `JsonException`.
5. Validate/map в domain; ожидаемую validation error добавить в report.
6. Неожиданную I/O error обернуть один раз в CatalogImportException.
7. Вернуть report при успешном завершении чтения.

## Категории

Различите:

- файл не найден;
- доступ запрещён;
- I/O failure в середине;
- повреждённый JSON одной строки;
- корректный JSON с невалидной книгой;
- cancellation;
- неожиданную programming error.

README объясняет, какие категории входят в report, какие завершают импорт, какие логируются на границе.

## Обязательные эксперименты

1. Покажите, что `throw;` сохраняет исходное место.
2. В отдельной учебной ветке замените на `throw ex;`, сравните stack trace и верните корректный вариант.
3. Оберните IOException с InnerException и добавьте catalog context.
4. Используйте exception filter для одного точного, обоснованного случая.
5. Отмените import после нескольких строк и сохраните статус Canceled, не «успешный частичный отчёт» без явного контракта.

## Верхняя граница

Console app:

- печатает summary успешного report;
- для известной operation error выдаёт понятное сообщение и ненулевой exit code;
- неожиданную ошибку логирует/печатает с диагностикой разработчику, но не выдаёт секреты;
- не логирует одну ошибку повторно на каждом слое.

## Ограничения

- нет пустых catch;
- нет `catch (Exception) { return null; }`;
- нет `throw ex` в финальном коде;
- message не содержит содержимое всей пользовательской строки/секреты;
- все streams/readers освобождаются;
- Task импорта всегда await-ится;
- cancellation не преобразуется в generic failure.

## Критерии приёмки

- mixed file даёт частичный report согласно правилам;
- infrastructure failure отличается от item error;
- InnerException и stack trace сохранены;
- cleanup происходит при успехе, ошибке и отмене;
- exit codes документированы;
- ученик объясняет каждый catch и что конкретно он делает;
- solution собирается, сценарии воспроизводимы.

## Дополнительное задание

Запустите два независимых импорта через Task.WhenAll, заставьте оба завершиться ошибками и соберите полную диагностику, не теряя ни одну task. Опишите политику частичного успеха.
