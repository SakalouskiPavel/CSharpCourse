# Урок 08. Reflection: как программа исследует собственные типы

> Обычно компилятор заранее знает тип объекта и проверяет вызовы. Reflection нужен, когда тип становится известен только во время работы: для сериализации, тестовых фреймворков, dependency injection, плагинов и анализаторов.

## Что нужно знать

Нужны классы, интерфейсы, свойства, методы, generics на уровне коллекций и LINQ. Reflection намеренно изучается после них: он описывает уже знакомые конструкции как данные.

## 1. Код и метаданные

При сборке .NET-проекта создаётся **assembly** — обычно файл `.dll` или `.exe`. В нём находятся:

- IL — промежуточные инструкции программы;
- **метаданные** — описание типов, методов, параметров, свойств, наследования и ссылок;
- manifest — имя, версия и состав сборки;
- иногда embedded resources.

Метаданные похожи на каталог библиотеки: они сообщают, какие книги есть и как они устроены, не читая каждую страницу. CLR использует их для загрузки типов, а приложение может читать их через Reflection API.

## 2. `Type` — описание типа

```csharp
Type bookType = typeof(Book);
Book book = new("C# с нуля", "Анна Автор");
Type runtimeType = book.GetType();

Console.WriteLine(bookType.Name);      // Book
Console.WriteLine(bookType.FullName);  // пространство имён + Book
```

- `typeof(Book)` получает тип, известный в коде;
- `book.GetType()` получает фактический runtime-тип объекта;
- `Type.GetType(string)` пытается разрешить тип по строковому имени.

`GetType()` нельзя вызвать у `null`. Строковое разрешение хрупко: нужны правильные namespace/assembly names, а переименование не проверяется компилятором.

`Type` сам является объектом, описывающим другой тип. Это первый пример метапрограммирования: программа работает со структурой программы как с данными.

## 3. Основные сведения `Type`

```csharp
Console.WriteLine(bookType.IsClass);
Console.WriteLine(bookType.IsAbstract);
Console.WriteLine(bookType.BaseType?.Name);

foreach (Type contract in bookType.GetInterfaces())
    Console.WriteLine(contract.Name);
```

Полезны `Namespace`, `Assembly`, `IsValueType`, `IsEnum`, `IsGenericType`. Методы `IsAssignableFrom`/`IsAssignableTo` отвечают на вопрос совместимости:

```csharp
bool isHandler = typeof(ICommandHandler).IsAssignableFrom(candidateType);
```

Это означает: объект `candidateType` можно присвоить переменной `ICommandHandler`.

## 4. `MemberInfo` и виды членов

Базовое описание члена — `MemberInfo`. Специализации:

- `MethodInfo` — метод;
- `PropertyInfo` — свойство;
- `FieldInfo` — поле;
- `ConstructorInfo` — конструктор;
- `EventInfo` — событие;
- `ParameterInfo` — параметр метода/конструктора.

```csharp
foreach (PropertyInfo property in bookType.GetProperties())
{
    Console.WriteLine($"{property.PropertyType.Name} {property.Name}");
}
```

`GetProperties()` по умолчанию возвращает публичные свойства. `GetProperty("Title")` возвращает описание либо `null`, поэтому отсутствие нужно обрабатывать.

## 5. `BindingFlags`

Флаги уточняют поиск:

```csharp
BindingFlags flags =
    BindingFlags.Instance |
    BindingFlags.Public |
    BindingFlags.NonPublic |
    BindingFlags.DeclaredOnly;

MethodInfo[] methods = bookType.GetMethods(flags);
```

- `Instance` / `Static` — члены объекта или типа;
- `Public` / `NonPublic` — доступность;
- `DeclaredOnly` — только объявленные здесь, без унаследованных;
- `IgnoreCase` — поиск имени без регистра, применять осторожно.

Если указать flags неполно, можно неожиданно получить пустой результат. Для большинства прикладных задач начинайте с публичного контракта; чтение private members связывает инфраструктуру с реализацией.

## 6. Чтение значения свойства

```csharp
PropertyInfo? titleProperty = bookType.GetProperty("Title");

if (titleProperty is not null && titleProperty.CanRead)
{
    object? value = titleProperty.GetValue(book);
    Console.WriteLine(value);
}
```

Reflection возвращает `object?`, потому что конкретный тип свойства выясняется во время выполнения. Для value type возникает boxing. Перед `SetValue` проверяют `CanWrite`, тип присваиваемого значения и предметные правила.

Reflection может обойти обычный удобный путь API, но не должен использоваться, чтобы систематически ломать инкапсуляцию.

## 7. Поиск и вызов метода

```csharp
MethodInfo? method = bookType.GetMethod(
    "Rename",
    new[] { typeof(string) });

method?.Invoke(book, new object?[] { "Новое название" });
```

`Invoke` принимает объект-владельца (`null` для static) и массив аргументов. Ошибки, которые обычный вызов обнаружил бы при компиляции, теперь происходят во время выполнения:

- метод не найден;
- выбрана неверная перегрузка;
- число или тип аргументов не совпадает;
- нет доступа;
- сам метод выбросил исключение.

Исключение вызванного метода обычно оказывается внутри `TargetInvocationException.InnerException`. Нельзя терять исходную причину в диагностике.

## 8. Создание объекта

```csharp
object? instance = Activator.CreateInstance(
    typeof(Book),
    "C# с нуля",
    "Анна Автор");
```

`Activator` ищет подходящий конструктор. Результат имеет тип `object?`, его проверяют:

```csharp
if (instance is Book createdBook)
    Console.WriteLine(createdBook.Title);
```

Для plugin host обычно сначала проверяют:

- тип concrete: не abstract/interface;
- реализует разрешённый контракт;
- имеет допустимый конструктор;
- не является open generic type;
- assembly входит в доверенный набор.

## 9. Generic-типы

`List<>` — открытое generic-определение: параметр элемента ещё не задан. `List<Book>` — закрытый тип, который можно создать.

```csharp
Type openList = typeof(List<>);
Type closedList = openList.MakeGenericType(typeof(Book));
object? list = Activator.CreateInstance(closedList);
```

`MakeGenericType` проверяет constraints. Аналогично `MethodInfo.MakeGenericMethod` закрывает generic method. Не создавайте тип из непроверенного пользовательского имени.

## 10. Сборки

```csharp
Assembly current = typeof(Program).Assembly;

foreach (Type type in current.GetTypes())
    Console.WriteLine(type.FullName);
```

`Assembly.GetTypes()` способен выбросить `ReflectionTypeLoadException`, если часть зависимостей не загрузилась. В production-диагностике полезны `LoaderExceptions`.

Сканировать все загруженные сборки на каждый запрос дорого и непредсказуемо. Обычно список сборок ограничивают, анализируют при старте и кэшируют descriptors.

## 11. Пример: поиск обработчиков

```csharp
Type contract = typeof(ICommandHandler);

List<Type> handlerTypes = typeof(Program).Assembly
    .GetTypes()
    .Where(type =>
        type.IsClass &&
        !type.IsAbstract &&
        contract.IsAssignableFrom(type))
    .ToList();
```

Затем каждый тип валидируется и создаётся. Важно разделить этапы:

1. discovery — найти кандидатов;
2. validation — проверить контракт и конфликты;
3. activation — создать объекты;
4. execution — вызвать через обычный интерфейс.

После создания лучше работать через `ICommandHandler`, а не продолжать `Invoke`: типизированный контракт безопаснее и быстрее.

## 12. Производительность

Поиск metadata, boxing, создание массива аргументов и `Invoke` дороже прямого вызова. Это не означает, что Reflection всегда медленный и запрещён.

Практические правила:

- сканируйте один раз, а не в горячем цикле;
- кэшируйте `Type`, `PropertyInfo` и готовые descriptors;
- после discovery переходите к интерфейсу;
- для горячего доступа можно один раз построить delegate/expression;
- измеряйте, прежде чем усложнять.

## 13. Безопасность

Опасный дизайн:

```csharp
Type type = Type.GetType(userInput)!;
object instance = Activator.CreateInstance(type)!;
```

Пользователь фактически выбирает исполняемый тип. Нужен whitelist разрешённых контрактов/сборок и проверка конфигурации. Reflection не является sandbox: загруженный плагин выполняется с правами процесса.

Для изоляции недоверенного кода нужен отдельный процесс/container и ограниченные права, а не только interface.

## 14. `AssemblyLoadContext`

Современный .NET использует `AssemblyLoadContext` для управления загрузкой и зависимостями плагинов. Collectible context можно выгрузить, только когда не осталось живых ссылок на его assembly, types и objects.

Это продвинутая тема: default context достаточен большинству приложений. Несколько контекстов могут загрузить типы с одинаковым именем, которые runtime считает разными из-за различной identity сборки/контекста.

## 15. Trimming и Native AOT

Linker при публикации может удалить код, который не виден через обычные статические ссылки. Если член используется только по строковому имени Reflection, инструмент не всегда понимает, что его нужно сохранить.

Варианты:

- сделать зависимость статически видимой;
- использовать `DynamicallyAccessedMembers` и related annotations;
- descriptor configuration;
- source generator, создающий код на этапе сборки;
- отказаться от trimming/AOT для несовместимого сценария.

Аннотация — обещание linker-у и требует точного понимания, а не случайного подавления warning.

## 16. Когда Reflection не нужен

Не используйте Reflection, если:

- обычный interface решает заменяемость;
- generic constraint выражает требование при компиляции;
- словарь функций явно задаёт маршрутизацию;
- конфигурация может выбрать заранее зарегистрированную реализацию;
- source generator может перенести проверку в compile time.

Reflection оправдан, когда структура действительно обнаруживается в runtime и цена динамичности принята осознанно.

## Частые ошибки

- строковые имена заменяют типизированный контракт;
- private fields читаются как основной API;
- metadata сканируется при каждом запросе;
- `GetMethod` игнорирует перегрузки;
- `TargetInvocationException` скрывает inner exception;
- создаётся произвольный тип из пользовательской строки;
- plugin считается изолированным только из-за interface;
- trimming warnings подавляются без анализа;
- Reflection применяется там, где достаточно словаря или DI registration.

## Итог

```text
Assembly содержит metadata
  ↓
Type описывает тип
  ↓
MemberInfo описывает член
  ↓
Discovery → Validation → Activation
  ↓
Дальше лучше работать через обычный интерфейс
```

Вы готовы дальше, если можете получить Type тремя способами, перечислить публичные свойства, безопасно найти реализации интерфейса и объяснить, какие проверки теряются при переходе от обычного вызова к Reflection.
