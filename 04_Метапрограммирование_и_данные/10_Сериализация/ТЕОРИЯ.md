# Урок 10. Сериализация: данные между памятью, файлом и сетью

> Объект живёт в памяти конкретного процесса. Чтобы сохранить его в файл или передать другой программе, данные превращают в согласованный формат. Этот формат становится контрактом и может жить дольше текущих классов.

## 1. Сериализация и десериализация

**Сериализация** преобразует данные объекта в представление для хранения/передачи. **Десериализация** читает представление и создаёт данные в программе.

```text
OrderDto в памяти
   ↓ serialize
JSON-текст / UTF-8 bytes
   ↓ deserialize
OrderDto в другом процессе или позже
```

Сериализация не сохраняет «весь объект» буквально: методы, открытый файл, thread, database connection и текущий stack не превращаются в полезный JSON. Сохраняются выбранные данные согласно правилам serializer-а.

## 2. Форматы

- **JSON** — текстовый, распространён в Web API, удобен человеку, имеет объекты, массивы, строки, числа, boolean и null.
- **XML** — текстовый, поддерживает namespaces, attributes и schema; часто встречается в enterprise/legacy contracts.
- **CSV** — строки и колонки; прост для таблиц, но требует точных правил delimiter, quoting, encoding и schema.
- **Бинарные форматы** (Protocol Buffers, MessagePack и другие) компактнее/быстрее в конкретных сценариях, но требуют schema/tooling.

Выбор определяется потребителями, совместимостью, размером, производительностью и эксплуатацией. Бинарный формат не становится безопасным автоматически.

В курсе используется `System.Text.Json` — стандартная библиотека современного .NET.

## 3. Первый JSON

```csharp
public sealed record BookDto(
    string InventoryNumber,
    string Title,
    decimal Price);

var book = new BookDto("B-001", "C# с нуля", 25.50m);

string json = JsonSerializer.Serialize(book);
Console.WriteLine(json);
```

Пример результата:

```json
{"InventoryNumber":"B-001","Title":"C# с нуля","Price":25.50}
```

Обратное преобразование:

```csharp
BookDto? restored = JsonSerializer.Deserialize<BookDto>(json);

if (restored is null)
    throw new JsonException("JSON не содержит книгу");
```

Метод возвращает nullable: JSON `null` допустим. Неверный JSON обычно приводит к `JsonException`.

## 4. JSON-синтаксис и C#-типы

```json
{
  "name": "Анна",
  "age": 25,
  "active": true,
  "tags": ["csharp", "dotnet"],
  "address": null
}
```

JSON string не знает `DateTimeOffset`, `Guid`, enum или `decimal` как отдельные типы. Serializer применяет соглашение преобразования. Другая платформа должна знать формат даты, enum и числа. «Корректный JSON» ещё не означает «совместимый контракт».

## 5. Единые `JsonSerializerOptions`

```csharp
var options = new JsonSerializerOptions
{
    PropertyNamingPolicy = JsonNamingPolicy.CamelCase,
    WriteIndented = true,
    PropertyNameCaseInsensitive = false,
    MaxDepth = 32
};
```

Options определяют naming, converters, number handling, ignore rules, reference handling и другое. Создавайте согласованный набор на границе приложения и переиспользуйте его. Если разные места случайно используют разные options, один и тот же тип получает несколько контрактов.

Options после первого использования следует считать неизменяемыми; в ASP.NET Core настройка обычно централизована.

## 6. Атрибуты JSON

```csharp
public sealed class BookDto
{
    [JsonPropertyName("inventoryNumber")]
    public required string InventoryNumber { get; init; }

    public required string Title { get; init; }

    [JsonIgnore]
    public string? InternalNote { get; init; }
}
```

- `JsonPropertyName` фиксирует внешнее имя;
- `JsonIgnore` исключает член;
- `JsonInclude` может включать поддерживаемый непубличный accessor/member;
- `JsonConstructor` выбирает constructor при неоднозначности;
- polymorphism attributes задают разрешённые derived types.

Атрибут связывает CLR-тип с конкретным transport contract. Для нескольких различных API-контрактов лучше отдельные DTO, а не множество конфликтующих annotations на domain model.

## 7. DTO и domain model

**DTO (Data Transfer Object)** — тип, форма которого предназначена для границы передачи. Domain object защищает поведение и инварианты.

Почему не всегда сериализовать domain object напрямую:

- private/internal state может случайно раскрыться;
- рефакторинг домена ломает внешний JSON;
- serializer способен создать состояние в обход domain factory;
- циклы навигации раздувают payload;
- API нужны другие названия, optional fields и версии;
- секретное поле может попасть наружу.

```csharp
BookDto ToDto(Book book) =>
    new(book.InventoryNumber, book.Title, book.Price);

Book ToDomain(BookDto dto) =>
    Book.Create(dto.InventoryNumber, dto.Title, dto.Price);
```

Mapping — дополнительный код, зато граница и место валидации становятся явными.

## 8. Валидация после parsing

Serializer проверяет синтаксис и совместимость типов, но не знает бизнес-правила.

```json
{"inventoryNumber":"","title":"","price":-500}
```

JSON корректен. DTO может быть создан. Domain factory должна отклонить пустой номер и отрицательную цену.

Разделяйте:

1. JSON syntax error;
2. data shape/type error;
3. validation/domain error;
4. storage/I/O error.

Они требуют разных сообщений и реакции.

## 9. Конструкторы и неизменяемые типы

System.Text.Json сопоставляет JSON properties параметрам подходящего constructor-а/свойствам. Имена constructor parameters должны соответствовать CLR property names по правилам serializer-а, даже если JSON name изменён атрибутом.

```csharp
public sealed class UserDto
{
    public string Name { get; }

    [JsonConstructor]
    public UserDto(string name)
    {
        Name = name;
    }
}
```

`required` и nullable annotations улучшают compile-time контракт, но входной JSON всё равно недоверен. Поддержка required members serializer-ом зависит от целевой версии .NET/настроек; предметную валидацию не удаляют.

## 10. Отсутствующее поле, `null` и default

Это разные состояния:

- property отсутствует в JSON;
- property присутствует со значением `null`;
- property содержит пустую строку/ноль;
- property содержит валидное значение.

Для reference property отсутствие часто приводит к default (`null`), для `int` — к `0`. Если `0` имеет смысл, невозможно понять, поле отсутствовало или явно задано, без nullable/собственного converter/model.

При проектировании версии заранее решите обязательность, default и поведение старого reader-а.

## 11. Converter

Converter управляет записью/чтением конкретного типа:

```csharp
public sealed class CourseCodeConverter : JsonConverter<CourseCode>
{
    public override CourseCode Read(
        ref Utf8JsonReader reader,
        Type typeToConvert,
        JsonSerializerOptions options)
    {
        string? text = reader.GetString();
        return CourseCode.Parse(text);
    }

    public override void Write(
        Utf8JsonWriter writer,
        CourseCode value,
        JsonSerializerOptions options)
    {
        writer.WriteStringValue(value.Value);
    }
}
```

Converter регистрируется в options или атрибутом. Он должен точно определить accepted tokens, null, формат и exception. Не превращайте converter в service locator или место database calls.

## 12. Циклы и ссылки

Объектный граф может иметь цикл: `Department → Employees → Department`. JSON естественно представляет дерево; наивный обход цикла не закончится.

Варианты:

- создать DTO без обратной ссылки;
- передавать identifiers вместо вложенного объекта;
- игнорировать цикл (`ReferenceHandler.IgnoreCycles`) с потерей части данных;
- сохранять reference metadata (`ReferenceHandler.Preserve`), меняя JSON-контракт.

Для публичного API отдельный DTO/identifier обычно понятнее технических `$id`/`$ref`.

## 13. Полиморфизм

Если property имеет базовый тип, serializer должен знать разрешённые derived types и discriminator:

```csharp
[JsonPolymorphic(TypeDiscriminatorPropertyName = "kind")]
[JsonDerivedType(typeof(EmailNotificationDto), "email")]
[JsonDerivedType(typeof(SmsNotificationDto), "sms")]
public abstract record NotificationDto;
```

JSON содержит `kind`, по которому выбирается тип. Разрешённые варианты перечисляются явно. Никогда не создавайте произвольный CLR type по имени из payload: это security risk и жёсткая связь с assembly names.

Нужно определить поведение неизвестного discriminator и совместимость старого consumer с новым вариантом.

## 14. Версионирование

Контракт изменяется независимо от кода. Обычно безопаснее:

- добавлять optional property с понятным default;
- терпеть неизвестные properties;
- не менять смысл существующего поля;
- не переиспользовать старое имя для другого значения;
- сохранять reader tests для старых payloads.

Потенциально breaking:

- переименование/удаление;
- смена типа числа на объект;
- optional → required;
- изменение enum representation;
- смена timezone/date format;
- новый derived type для consumer с exhaustive logic.

Миграция может прочитать старую versioned DTO и преобразовать её в текущую domain model.

## 15. Файлы и кодировка

```csharp
await using FileStream stream = File.Create(path);
await JsonSerializer.SerializeAsync(stream, dto, options, cancellationToken);
```

При чтении используйте async API и `CancellationToken`, если операция I/O. Не пишите напрямую поверх единственной корректной копии: временный файл + flush + atomic replace/rename снижает риск половинчатого файла.

JSON обычно передаётся в UTF-8. Не превращайте большой UTF-8 payload сначала в огромную string без необходимости.

## 16. Потоковая обработка

`DeserializeAsyncEnumerable<T>` позволяет читать большой верхнеуровневый массив по элементам. `Utf8JsonReader`/`Utf8JsonWriter` дают низкоуровневый forward-only доступ.

Потоковая обработка уменьшает peak memory, но усложняет validation, partial failure и повторное чтение. Начинайте с обычного serializer-а, если данные ограничены и помещаются в память.

## 17. Безопасность

Любой внешний payload недоверен:

- ограничивайте request/file size;
- задавайте разумный `MaxDepth`;
- используйте timeout/cancellation на I/O-границе;
- явно разрешайте polymorphic types;
- проверяйте значения после deserialize;
- не включайте secrets через публичные properties;
- не возвращайте raw exception/stack trace клиенту;
- учитывайте чрезмерные строки, массивы и числа.

Десериализация не должна автоматически запускать бизнес-действия. Сначала данные создаются и проверяются, затем application service выполняет use case.

## 18. Reflection, source generation, trimming

По умолчанию System.Text.Json может строить metadata с Reflection. Для Native AOT/trimming и быстрого старта можно использовать source-generated `JsonSerializerContext`:

```csharp
[JsonSerializable(typeof(BookDto))]
internal partial class CourseJsonContext : JsonSerializerContext
{
}
```

```csharp
string json = JsonSerializer.Serialize(
    dto,
    CourseJsonContext.Default.BookDto);
```

Source generation делает набор типов явным и переносит часть работы в build. У режимов есть ограничения; проверяйте документацию целевой версии .NET. Не оптимизируйте без требования, но учитывайте публикационный режим заранее.

## Частые ошибки

- внешний контракт совпадает с domain model случайно;
- `Deserialize` считается полной валидацией;
- options создаются по-разному в каждом месте;
- отсутствие, null и default смешиваются;
- cycles «исправляются» игнорированием без анализа потери данных;
- произвольный CLR type выбирается из payload;
- breaking change выпускается без compatibility tests;
- огромный файл читается целиком в string;
- serializer раскрывает secret/internal property;
- Reflection-based contract ломается после trimming/AOT.

## Итог

```text
External bytes/text
  ↓ syntax + shape
DTO
  ↓ validation + mapping
Domain model

Обратно: domain → DTO → versioned contract
```

Вы готовы дальше, если можете спроектировать DTO, настроить единые options, различить ошибки синтаксиса/валидации, обработать старую версию и объяснить, почему десериализованный объект ещё нельзя считать доверенным.
