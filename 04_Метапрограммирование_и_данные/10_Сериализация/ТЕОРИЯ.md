# Урок 10. Как устроена сериализация

> Сериализация превращает граф объектов в формат хранения или передачи, но формат — это публичный контракт, а не снимок внутренней памяти. Основные риски связаны с совместимостью схемы, типами, ссылками, безопасностью и границами доверия.

## Результаты обучения

- проектировать DTO и версионируемый контракт
- настраивать System.Text.Json
- объяснять reference handling и polymorphism
- безопасно обрабатывать недоверенные данные

## Карта темы

### 1. Контракт и формат

JSON, XML и бинарные форматы различаются читаемостью, схемой и стоимостью. Внешний контракт следует отделять от доменной модели, чтобы её рефакторинг не ломал потребителей.

### 2. System.Text.Json

`JsonSerializer` использует публичные свойства/конструкторы и настраивается options. Naming policy, converters, ignore rules и number handling должны быть едины на границе приложения.

### 3. DTO

DTO описывает транспортные данные без поведения. Явное отображение DTO↔domain — дополнительный код, зато даёт место для валидации и эволюции.

### 4. Конструкторы и неизменяемость

Неизменяемые типы десериализуются через подходящий конструктор; `[JsonConstructor]` снимает неоднозначность. Успешное создание CLR-объекта ещё не означает корректность предметных данных.

### 5. Полиморфизм

Разрешённые производные типы и discriminator задаются явно. Автоматическая загрузка произвольного типа из payload опасна и связывает контракт с CLR-именами.

### 6. Ссылки и циклы

Обычный JSON представляет дерево; циклический граф требует игнорирования циклов, сохранения ссылок или отдельной модели идентификаторов. Каждый вариант меняет смысл контракта.

### 7. Версионирование

Добавление optional-поля обычно совместимо, переименование/смена типа — нет. Reader должен терпеть неизвестные поля, а обязательность и defaults проектируются заранее.

### 8. Потоковая обработка

Async stream APIs и `Utf8JsonReader/Writer` уменьшают пиковую память. Для больших массивов не следует сначала читать весь файл в строку.

### 9. Безопасность

Ограничивайте размер, глубину и время обработки, валидируйте данные после parsing, не доверяйте type metadata. Секреты нельзя сериализовать по умолчанию.

## Минимальный пример

```csharp
var options = new JsonSerializerOptions
{
    PropertyNamingPolicy = JsonNamingPolicy.CamelCase,
    WriteIndented = true,
    MaxDepth = 32
};

var json = JsonSerializer.Serialize(orderDto, options);
var restored = JsonSerializer.Deserialize<OrderDto>(json, options)
    ?? throw new JsonException("Empty payload");
```

Пример — отправная точка, а не готовая архитектура. Проверьте предусловия, lifetime объектов, обработку ошибок и поведение на границах. Для инфраструктурных тем отдельно фиксируйте, какие гарантии даёт библиотека, а какие должен обеспечить ваш код.

## Типичные ошибки

- сериализовать доменный граф прямо во внешний API;
- считать десериализованный объект валидным;
- включать произвольные CLR-типы из входных данных;
- читать большой payload целиком без лимитов;

## Итог

Умение использовать механизм включает понимание его runtime-модели, цены и альтернатив. Подтверждайте выводы небольшими тестами, логами, generated output или профилированием — в зависимости от темы.
