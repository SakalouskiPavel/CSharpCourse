# Практика — урок 12

## Цель

Перенести каталог книг в реляционную SQLite database через EF Core, проверить schema constraints, parameterization, transaction и optimistic concurrency. SQLite выбран для воспроизводимого локального старта; архитектурные решения не должны зависеть от случайных особенностей provider-а.

## Проекты и зависимости

Создайте persistence project и отдельное console application. Подключите актуальные совместимые packages EF Core SQLite и design tools согласно официальной документации/целевому SDK. Версии package зафиксируйте в project/lock configuration, а не в тексте курса.

## Schema

Минимальные таблицы:

- `Books`: key, unique inventory number, required title, price с ограничением, author FK, concurrency token;
- `Authors`: key, required normalized/display name;
- `Tags` и join table `BookTags` с составным unique/primary key.

Создайте migration. Просмотрите SQL/schema и объясните каждый key, constraint и index.

## EF queries

Реализуйте:

1. поиск книги по inventory number;
2. read-only projection `BookSummary` с AsNoTracking;
3. фильтрацию по цене в database;
4. отчёт author + book count;
5. поиск по tag;
6. ограниченную page выдачи со стабильным order.

Для каждого сохраните/покажите generated SQL и убедитесь, что фильтр не выполняется после ToList.

## ADO.NET parameter experiment

Один read query выполните напрямую через provider `DbConnection/DbCommand` с parameter. Передайте строку, содержащую кавычку и SQL-подобный текст, и покажите, что она остаётся value, а не частью команды.

Identifiers из input не вставляйте. Для выбора sort column используйте enum/whitelist, который сопоставляется заранее написанным выражениям.

## Transaction

Use case «переместить книгу от одного автора к новому и записать audit row» должен быть атомарным. Если всё помещается в один SaveChanges, используйте его default transaction и объясните достаточность. Затем создайте учебный вариант с двумя SaveChanges в explicit transaction и вызовите ошибку между ними — после rollback частичного результата быть не должно.

Не выполняйте внешних сетевых действий внутри transaction.

## Optimistic concurrency

Откройте два отдельных DbContext:

1. оба читают одну Book;
2. первый меняет и сохраняет;
3. второй меняет старую версию и сохраняет;
4. приложение получает concurrency conflict;
5. выведите current, attempted и database values;
6. примените выбранную политику: сообщить/перечитать либо явно merge.

Автоматический blind retry запрещён.

## Integration tests/проверки

Используйте настоящую SQLite database с изолированным временным файлом либо корректно настроенным in-memory connection lifetime. Не подменяйте relational semantics коллекцией `List<T>`.

Проверьте:

- unique inventory constraint;
- invalid foreign key/required value;
- transaction rollback;
- concurrency conflict;
- query projection;
- parameterized input;
- migration создаёт schema с чистого состояния.

## NoSQL decision note

Не внедряя вторую DB, представьте document-модель `BookCatalog` и ответьте:

- что является aggregate/document;
- какие запросы дешевле/сложнее;
- где дублируется author;
- как обновить имя автора;
- какая transaction boundary;
- какой partition key возможен;
- почему выбран relational вариант для текущих требований.

## Ограничения

- никакой SQL-конкатенации input;
- DbContext короткоживущий и не разделяется параллельно;
- read queries не загружают ненужные columns;
- constraints существуют в schema;
- transaction короткая;
- secrets/connection credentials не коммитятся;
- migration и database-файлы обрабатываются согласно правилам репозитория;
- ошибки не скрываются catch-all.

## Критерии приёмки

- database воспроизводится migrations с чистого состояния;
- все шесть queries выполняют ожидаемый SQL;
- injection experiment безопасен благодаря parameter;
- rollback и concurrency conflict доказаны;
- indexes связаны с конкретными запросами;
- README описывает setup, cleanup, schema, transaction и NoSQL trade-off;
- решение согласуется с актуальной официальной документацией EF Core по transactions, concurrency и efficient querying.

## Дополнительное задание

Сравните offset и keyset pagination на растущем наборе, изучите execution plans до/после index и измерьте. Либо реализуйте streaming большого read query через `AsAsyncEnumerable`, документируя lifetime DbContext/connection.
