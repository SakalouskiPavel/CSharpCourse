# Урок 12. Базы данных в .NET: SQL, EF Core и NoSQL

> Память процесса исчезает после завершения. База данных хранит состояние долговременно, обеспечивает поиск, ограничения и совместную работу нескольких клиентов. Код приложения и база вместе защищают данные, но делают это на разных границах.

## 1. Что такое база данных и СУБД

**База данных** — организованные долговременные данные. **Система управления базами данных (СУБД, DBMS)** — программа, которая хранит их, выполняет запросы, управляет конкурентным доступом, журналом, правами и восстановлением.

Приложение обычно не редактирует файл базы как текст. Оно подключается к DBMS и отправляет команды через driver/provider.

```text
C# application → provider/driver → DBMS → data files
```

SQLite работает как embedded library с файлом; PostgreSQL, SQL Server и другие обычно работают отдельным server process. Это влияет на deployment и concurrency, но основные relational понятия общие.

## 2. Реляционная модель

Данные представлены отношениями, практически — таблицами.

`Books`:

| Id | InventoryNumber | Title | AuthorId | Price |
|---:|---|---|---:|---:|
| 1 | B-001 | C# с нуля | 10 | 25.50 |

- **row** — одна запись;
- **column** — именованное свойство с database type;
- **schema** — таблицы, колонки, ограничения, связи, индексы;
- **primary key** — уникально идентифицирует row;
- **foreign key** — ссылается на primary/unique key другой таблицы;
- `NULL` — отсутствие значения, не пустая строка и не ноль.

Entity в domain и row в таблице похожи, но не обязаны совпадать один к одному.

## 3. Ограничения целостности

База должна защищать правила независимо от конкретного приложения:

- `PRIMARY KEY` — уникальность identity;
- `NOT NULL` — значение обязательно;
- `UNIQUE` — уникальность candidate key;
- `FOREIGN KEY` — ссылка существует;
- `CHECK` — условие значения, если поддерживается/применяется;
- database type/length — допустимое представление.

```sql
CREATE TABLE Books (
    Id INTEGER PRIMARY KEY,
    InventoryNumber TEXT NOT NULL UNIQUE,
    Title TEXT NOT NULL,
    Price NUMERIC NOT NULL CHECK (Price >= 0)
);
```

Проверка в C# даёт быстрый понятный ответ, constraint защищает от другого приложения, race condition и ручной команды. Это не дублирование без смысла, а defense in depth.

## 4. Нормализация простыми словами

Если имя автора повторяется в каждой книге, его исправление требует менять много rows и легко создаёт разные написания. Отдельная таблица Authors уменьшает повторение:

```text
Authors(Id, Name)
Books(Id, Title, AuthorId)
```

Нормализация разделяет факты и уменьшает update/insert/delete anomalies. Но чрезмерное дробление усложняет запросы. Модель выбирают по смыслу данных, целостности и чтению, а денормализацию добавляют осознанно ради измеренного сценария.

## 5. SQL: язык запросов

SQL декларативен: запрос описывает нужный результат, optimizer выбирает план.

```sql
SELECT Id, Title, Price
FROM Books
WHERE Price >= 20
ORDER BY Title;
```

Основные команды:

- `SELECT` — чтение;
- `INSERT` — добавление;
- `UPDATE` — изменение;
- `DELETE` — удаление;
- `CREATE/ALTER/DROP` — schema (DDL).

Всегда называйте нужные columns вместо `SELECT *` в стабильном контракте: меньше данных, яснее mapping и меньше зависимость от schema.

## 6. Join

```sql
SELECT b.Title, a.Name AS AuthorName
FROM Books AS b
INNER JOIN Authors AS a ON a.Id = b.AuthorId;
```

`INNER JOIN` оставляет rows с совпавшей парой. `LEFT JOIN` сохраняет все rows слева, а отсутствующие правые columns становятся NULL.

Join объединяет rows в результате, но не «вкладывает объект навсегда». Несколько one-to-many joins могут многократно повторить данные и создать cartesian explosion.

## 7. Параметризованные запросы

Опасно:

```csharp
command.CommandText =
    $"SELECT Id, Title FROM Books WHERE Title = '{userInput}'";
```

Ввод становится частью SQL-кода. Злоумышленник может изменить структуру команды — SQL injection.

Правильно:

```csharp
await using DbCommand command = connection.CreateCommand();
command.CommandText = """
    SELECT Id, Title, Price
    FROM Books
    WHERE Price >= @minimumPrice
    """;

DbParameter parameter = command.CreateParameter();
parameter.ParameterName = "@minimumPrice";
parameter.Value = minimumPrice;
command.Parameters.Add(parameter);
```

SQL-text и value передаются отдельно. Parameters также помогают типизации и повторному использованию query plans.

Parameter обычно не заменяет имя table/column/order direction. Динамические identifiers выбирают из whitelist заранее известных вариантов, а не вставляют произвольный input.

## 8. ADO.NET

ADO.NET — низкоуровневый стандартный набор abstractions/providers:

- `DbConnection` — соединение;
- `DbCommand` — SQL и parameters;
- `DbDataReader` — последовательное чтение rows;
- `DbTransaction` — transaction;
- provider packages дают конкретную реализацию для SQLite/PostgreSQL/SQL Server.

```csharp
await using DbConnection connection = CreateConnection(connectionString);
await connection.OpenAsync(cancellationToken);

await using DbCommand command = connection.CreateCommand();
command.CommandText = "SELECT Id, Title FROM Books ORDER BY Id";

await using DbDataReader reader =
    await command.ExecuteReaderAsync(cancellationToken);

while (await reader.ReadAsync(cancellationToken))
{
    long id = reader.GetInt64(0);
    string title = reader.GetString(1);
}
```

Reader читает текущую row; ordinal columns нужно согласовать с SELECT. Для nullable column сначала проверяют `IsDBNull`.

## 9. Connection lifetime и pooling

Соединение открывают поздно и закрывают рано через `await using`. Это не означает физическое сетевое подключение каждый раз: provider обычно использует connection pool и возвращает закрытый logical connection в пул.

Не держите connection/reader открытыми во время ожидания пользователя или внешнего HTTP-вызова. Pool ограничен; долгие владельцы создают starvation.

Connection string содержит адрес и credentials. Реальные secrets не коммитят и не логируют.

## 10. ORM и EF Core

**ORM (Object-Relational Mapper)** отображает .NET objects на relational schema, строит SQL и materializes results.

```csharp
public sealed class CatalogDbContext : DbContext
{
    public DbSet<BookEntity> Books => Set<BookEntity>();

    public CatalogDbContext(DbContextOptions<CatalogDbContext> options)
        : base(options)
    {
    }
}
```

`DbContext` — короткоживущая unit of work:

- хранит настройки/provider;
- переводит LINQ expression tree в SQL;
- отслеживает загруженные entities;
- сохраняет изменения через `SaveChanges`;
- координирует transaction одной операции.

DbContext не потокобезопасен. Не делайте его static/singleton и не запускайте несколько параллельных операций на одном instance.

## 11. Mapping

Convention mapping можно уточнять Fluent API:

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<BookEntity>(book =>
    {
        book.HasKey(x => x.Id);
        book.Property(x => x.InventoryNumber)
            .IsRequired()
            .HasMaxLength(50);
        book.HasIndex(x => x.InventoryNumber).IsUnique();
        book.Property(x => x.Price).HasPrecision(18, 2);
    });
}
```

Persistence entity может отличаться от domain model. Не позволяйте ORM convenience разрушить private constructors/invariants; используйте mapping, backing fields или отдельную persistence model по сложности проекта.

## 12. Запросы EF Core

```csharp
List<BookSummary> books = await context.Books
    .AsNoTracking()
    .Where(book => book.Price >= minimumPrice)
    .OrderBy(book => book.Title)
    .Select(book => new BookSummary(
        book.Id,
        book.Title,
        book.Price))
    .ToListAsync(cancellationToken);
```

До terminal operation это `IQueryable`: provider переводит expression tree в SQL. Важные правила:

- фильтруйте в БД;
- проецируйте только нужные columns;
- используйте `AsNoTracking` для read-only query, если не нужна identity tracking;
- ограничивайте result (`Take`, pagination);
- смотрите generated SQL и query plan;
- не вызывайте `ToList` до фильтра, иначе данные попадут в память.

`AsNoTracking` не «всегда быстрее»: tracking нужен для updates и identity resolution. Выбор следует use case.

## 13. Change tracking и SaveChanges

```csharp
BookEntity book = await context.Books
    .SingleAsync(x => x.Id == id, cancellationToken);

book.Title = newTitle;
await context.SaveChangesAsync(cancellationToken);
```

Context хранит original/current values и формирует UPDATE. Для одного вызова SaveChanges relational provider обычно использует transaction: либо все изменения сохранены, либо ни одно. Это подтверждено актуальной документацией EF Core; ручная transaction нужна не всегда.

Не смешивайте долгоживущий context с множеством независимых user operations: tracking растёт, данные устаревают, lifetime зависимостей запутывается.

## 14. Связи и N+1

Загрузка authors, затем отдельный запрос books для каждого автора создаёт 1 + N round trips.

Варианты:

- projection с join/subquery;
- eager loading `Include` при необходимости entity graph;
- explicit loading осознанно;
- отказаться от lazy loading, если она скрывает запросы.

`Include` не универсален: несколько collections могут вызвать cartesian explosion. Split queries уменьшают дублирование, но добавляют round trips/consistency trade-offs. Проверяйте SQL и объём, а не количество методов LINQ.

## 15. Транзакции и ACID

Transaction объединяет операции в одно логическое изменение.

- **Atomicity** — всё или ничего;
- **Consistency** — constraints/invariants сохраняются;
- **Isolation** — конкурентные transactions имеют определённую видимость;
- **Durability** — commit переживает ожидаемый сбой согласно гарантиям DBMS.

```csharp
await using var transaction =
    await context.Database.BeginTransactionAsync(cancellationToken);

try
{
    // несколько SaveChanges/commands одной database
    await context.SaveChangesAsync(cancellationToken);
    await transaction.CommitAsync(cancellationToken);
}
catch
{
    await transaction.RollbackAsync(CancellationToken.None);
    throw;
}
```

Ручная transaction нужна, когда одна бизнес-операция содержит несколько отдельных saves/commands, которые обязаны быть атомарны. Держите её короткой. Не выполняйте внутри сетевой email/HTTP: внешняя система не откатится вместе с SQL.

## 16. Уровни изоляции

Конкурентные transactions могут вызвать:

- dirty read — чтение незафиксированного;
- non-repeatable read — повторное чтение row изменилось;
- phantom — повторный range query получил другие rows;
- lost update — одно изменение затёрло другое.

Isolation level задаёт компромисс consistency/concurrency и зависит от DBMS. Более сильная изоляция не бесплатна: locks/version storage, blocking, deadlocks и retries.

## 17. Оптимистическая конкурентность

Два пользователя прочитали одну книгу и изменили разные/одни поля. **Concurrency token** сохраняется при чтении и участвует в WHERE обновления.

Условно:

```sql
UPDATE Books
SET Title = @title
WHERE Id = @id AND Version = @originalVersion;
```

Если row уже изменён, affected rows = 0. EF Core выбрасывает `DbUpdateConcurrencyException`.

Дальше нужно предметное решение:

- сообщить пользователю и перечитать;
- повторить всю операцию на свежих данных, если безопасно;
- показать current/proposed/database values и объединить;
- отказаться от изменения.

Автоматический retry без пересчёта бизнес-решения может затереть данные.

## 18. Индексы

Индекс — дополнительная структура для ускорения определённых поисков/сортировок. Цена:

- место на диске/в памяти;
- обновление при INSERT/UPDATE/DELETE;
- maintenance/statistics;
- неправильный индекс может не использоваться.

Composite index `(AuthorId, Price)` обычно помогает запросам по AuthorId и по AuthorId+Price, но не обязательно по Price alone. Порядок columns важен.

Индекс выбирают по реальным query patterns и проверяют execution plan. «Индексировать все поля» ухудшает записи и не решает плохой запрос.

## 19. Миграции

Migration — versioned изменение schema. EF Core migration описывает переход Up/Down, но production deployment требует больше:

- review generated SQL;
- backup/restore plan;
- объём и locks;
- backward-compatible expand/contract при нескольких версиях приложения;
- отдельное выполнение с правами schema owner;
- проверка на копии данных.

Удаление/переименование column не должно выпускаться одновременно с кодом, если старые instances ещё работают.

## 20. NoSQL — семейство моделей

NoSQL не означает «без запросов» или «быстрее SQL». Основные модели:

- key-value — доступ по ключу;
- document — агрегат хранится JSON-подобным документом;
- wide-column — большие разреженные наборы/partitioned access;
- graph — вершины/рёбра и traversal;
- time-series/search — специализированные модели.

Document store удобен, когда aggregate читается/пишется целиком и допускается денормализация. Но joins, uniqueness и multi-document transactions могут быть ограниченнее/дороже. Дублированные данные требуют стратегии обновления.

Выбирайте по:

- access patterns;
- consistency requirements;
- transaction boundary;
- partition key и scale;
- query/index capabilities;
- operations, backup, cost и team experience.

Polyglot persistence допустима, но каждый новый storage увеличивает эксплуатационную цену.

## 21. Безопасность и эксплуатация

- минимальные DB privileges;
- secrets вне repository/logs;
- encrypted transport при сетевом DB;
- параметризованные values;
- command/query timeout;
- cancellation;
- connection pool metrics;
- slow query logs и plans;
- migrations/backup;
- обязательная проверка restore.

Backup без регулярного restore test — надежда, а не стратегия восстановления.

## Частые ошибки

- SQL строится конкатенацией input;
- DbContext становится singleton/static;
- query загружает всю entity/таблицу до фильтра;
- lazy loading создаёт N+1;
- transaction держится во время HTTP/email;
- constraints существуют только в C#;
- concurrency exception перезаписывается автоматическим retry;
- индекс добавляется без query plan;
- NoSQL выбирается «для масштабирования» без access patterns;
- migration запускается без совместимости и backup/restore plan.

## Итог

```text
Use case
  ↓ query/transaction boundary
C# (ADO.NET или EF Core)
  ↓ parameterized SQL
DBMS
  ├─ schema + constraints
  ├─ indexes + plans
  ├─ transactions + isolation
  └─ durable data + recovery
```

Вы готовы дальше, если можете спроектировать таблицу с keys/constraints, выполнить parameterized query, объяснить lifetime DbContext, transaction boundary и обработать optimistic concurrency conflict.
