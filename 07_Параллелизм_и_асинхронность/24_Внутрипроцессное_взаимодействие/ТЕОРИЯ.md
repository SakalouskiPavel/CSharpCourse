# Урок 24. Внутрипроцессное взаимодействие: вызовы, очереди и Channels

> Объекты одного процесса могут вызвать друг друга напрямую или передать сообщение через буфер. Очередь нужна не для «слабой связанности вообще», а когда требуется разнести участников во времени, ограничить нагрузку, организовать workers или явно задать backpressure.

## 1. Прямой вызов — нормальная отправная точка

```csharp
OrderResult result = orderService.Place(command);
```

Преимущества:

- типизированные параметры/результат;
- exception идёт по call stack;
- порядок виден;
- нет промежуточного buffer;
- caller знает момент завершения.

Temporal coupling — caller и callee активны одновременно — не всегда проблема. Не заменяйте понятный вызов message bus только ради слова «асинхронный».

## 2. Callback и event

Delegate передаёт продолжение, event уведомляет subscribers. Обычный event синхронен: handlers выполняются в call stack publisher-а.

Event хорош для локального факта с небольшим числом подписчиков, если определены lifetime/errors. Он не обеспечивает buffer, backpressure, retry и durability.

Async callbacks должны возвращать Task; `async void` не даёт publisher-у дождаться ошибок.

## 3. Сообщение

Message — immutable описание намерения/факта/запроса:

- **Command** просит выполнить действие, обычно один handler;
- **Event** сообщает свершившийся факт, handlers может быть много;
- **Query** просит данные и не должна менять observable state;
- **Job/Work item** описывает единицу фоновой обработки.

Имена events обычно в прошедшем времени (`OrderPaid`). Называть command событием опасно: неизвестно, обязательно ли кто-то должен его выполнить.

Message содержит data, correlation/idempotency metadata при необходимости, но не open Stream/DbContext/delegate, если должна пережить границу/очередь.

## 4. `ConcurrentQueue<T>`

```csharp
var queue = new ConcurrentQueue<Job>();
queue.Enqueue(job);

if (queue.TryDequeue(out Job? next))
    Process(next);
```

Thread-safe enqueue/dequeue, но нет:

- async ожидания новых данных;
- capacity/backpressure;
- автоматического completion;
- worker lifecycle;
- гарантии атомарности нескольких вызовов.

Polling с `while`/Delay тратит latency/CPU и усложняет shutdown. Для producer/consumer обычно Channel.

## 5. `BlockingCollection<T>`

Оборачивает concurrent collection, умеет bounded capacity, blocking Add/Take и CompleteAdding. Подходит sync producer/consumer и legacy thread-based code.

Она блокирует threads. Для async pipeline `Channel<T>` естественнее. Не смешивайте BlockingCollection с async methods через `.Result`.

## 6. `Channel<T>`

Channel разделён:

- `ChannelWriter<T>` — producers;
- `ChannelReader<T>` — consumers.

```csharp
Channel<Job> channel = Channel.CreateBounded<Job>(
    new BoundedChannelOptions(100)
    {
        FullMode = BoundedChannelFullMode.Wait,
        SingleWriter = false,
        SingleReader = false
    });
```

`WriteAsync` ждёт место, `ReadAllAsync` выдаёт до completion:

```csharp
await channel.Writer.WriteAsync(job, ct);

await foreach (Job item in channel.Reader.ReadAllAsync(ct))
    await HandleAsync(item, ct);
```

SingleWriter/SingleReader — optimization promises. Указывайте true только если архитектура действительно гарантирует.

## 7. Bounded и unbounded

Unbounded channel принимает данные быстрее processing и может расти до exhaustion memory. Он оправдан, если поток строго ограничен другим invariant-ом и это доказано.

Bounded channel имеет capacity и full policy:

- `Wait` — producer ждёт, создавая backpressure;
- `DropNewest` — удалить newest buffered item;
- `DropOldest` — удалить oldest;
- `DropWrite` — отбросить записываемый item.

Выбор предметный. Финансовую команду нельзя молча drop; telemetry sample иногда можно. Drop требует metrics/notification, иначе потеря невидима.

## 8. Backpressure

Backpressure переносит сигнал перегрузки upstream. Если consumers обрабатывают 100/s, а producers создают 1000/s, бесконечный buffer не решает разницу — только откладывает отказ.

Варианты:

- ждать свободную capacity;
- отклонить запрос;
- drop/coalesce допустимые данные;
- масштабировать consumers;
- уменьшить producer rate;
- persisted queue/broker.

Capacity связывают с memory per item, допустимой latency и burst, затем измеряют depth/oldest age.

## 9. Completion

Producer сообщает отсутствие новых items:

```csharp
channel.Writer.TryComplete();
```

При failure:

```csharp
channel.Writer.TryComplete(exception);
```

Readers дообрабатывают buffered items, затем enumeration завершается или сообщает error. Completion — не cancellation:

- completion: новых данных не будет, drain возможно;
- cancellation: caller больше не хочет ждать/операция должна остановиться по policy.

Writer ownership должен быть ясен. Если producers много, coordinator завершает channel после всех, а не первый producer случайно.

## 10. Worker lifecycle

Host:

1. создаёт channel;
2. запускает известное число worker Tasks;
3. публикует writer/узкий producer API;
4. при shutdown прекращает вход;
5. завершает writer;
6. ждёт workers;
7. наблюдает все errors.

Fire-and-forget worker без сохранённого Task теряет exception и не даёт graceful shutdown.

## 11. Несколько consumers и порядок

Channel сохраняет порядок чтения items, но несколько workers завершают работу в другом порядке.

Если нужен order per customer/order key:

- partition по key в несколько channels;
- один consumer на partition;
- sequence/version check;
- reorder buffer (цена memory/latency).

Global order ограничивает parallelism. Определяйте минимально необходимый scope порядка.

## 12. Ошибки item и pipeline

Handler failure options:

- завершить весь channel failure;
- записать Result/ошибку и продолжить;
- bounded retry;
- перенести в local dead-letter collection;
- отменить workers.

In-memory retry теряется при crash и может блокировать очередь poison item. Для обязательной durable processing нужен broker/storage (урок 33), а не усложнённый Channel.

## 13. Fan-out и fan-in

Fan-out: один поток работ распределяется нескольким workers. Fan-in: результаты нескольких producers собираются в один channel/aggregator.

При fan-in нужен correlation id, completion всех producers и policy partial failure. Multiple writers безопасны, если channel настроен соответственно; business deduplication всё равно отдельно.

## 14. Mediator и in-process bus

Mediator routes request к handler без прямой reference. Полезен для унификации pipeline, но:

- может скрыть call graph;
- не даёт buffer/durability;
- Send часто выполняется сразу в том же процессе;
- transaction/lifetime остаются responsibility application.

Notification mediator похож на Observer. Не называйте его broker: после process crash messages исчезнут.

## 15. TPL Dataflow и alternatives

Dataflow blocks дают Buffer/Transform/Action, linking, bounded capacity и completion для сложных graphs. Это отдельный package/API. Channels — более низкоуровневая async queue.

Выбор:

- прямой вызов — sync request/response;
- event — локальное уведомление;
- Channel — custom producer/consumer;
- Dataflow — graph transformations;
- broker — durable/distributed delivery.

## 16. Ownership передаваемых данных

Message должен быть immutable или ownership transferred. Если producer после Write меняет `byte[]`, consumer увидит race/corruption.

Варианты:

- immutable values;
- defensive copy;
- owner object (`IMemoryOwner<T>`) с явным Dispose consumer-ом;
- pool buffer возвращает последний owner, не producer сразу после Write.

Channel передаёт reference, не клонирует object.

## 17. Observability

Измеряйте:

- writes/reads;
- current/estimated depth;
- wait duration producer;
- queue age;
- processing duration;
- active workers;
- failures/retries/drops;
- shutdown/drain time.

Один throughput без age скрывает растущий backlog.

## 18. Deterministic tests

Tests используют TaskCompletionSource/gates, а не Delay. Проверяют:

- второй Write ждёт при полной capacity;
- drop policy наблюдаема;
- completion drains;
- failure propagation;
- cancellation;
- max active workers;
- early consumer exit cleanup.

Каждый test имеет timeout и awaits all workers.

## Частые ошибки

- очередь заменяет простой direct call;
- ConcurrentQueue polling крутится в цикле;
- unbounded buffer используется при неизвестной нагрузке;
- drop невидим;
- writer никогда не completed;
- первый из многих producers завершает channel;
- ожидается order completion при нескольких workers;
- mutable buffer меняется после enqueue;
- worker fire-and-forget;
- in-memory bus принимается за durable broker.

## Итог

```text
Direct call — совместное время и прямой результат
Event — уведомление
Channel — async buffer + backpressure + completion
Broker — durable/distributed guarantees
```

Вы готовы дальше, если можете выбрать capacity/full policy, назначить владельца completion, дождаться workers и определить порядок/ошибки/ownership messages.
