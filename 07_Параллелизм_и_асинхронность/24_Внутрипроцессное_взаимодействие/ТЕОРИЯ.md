# Урок 24. Внутрипроцессное взаимодействие

> Компоненты одного процесса могут вызываться напрямую, публиковать события или передавать сообщения через очередь. Выбор задаёт временную связанность, backpressure, порядок, ошибки и lifecycle.

## Результаты обучения

- объяснять runtime-модель темы;
- выбирать механизм по требуемой гарантии;
- сохранять ошибки, отмену и завершение;
- проверять корректность и производительность экспериментом.

## Карта темы

### 1. Прямой вызов

Самый простой типобезопасный путь: результат и исключение возвращаются по стеку. Временная связанность часто является честным и полезным свойством.

### 2. Callbacks и events

Позволяют publisher уведомить подписчиков, но обычный вызов синхронен. Медленный или ошибочный subscriber влияет на publisher, если политика не изолирует его.

### 3. ConcurrentQueue

Потокобезопасна, но сама не даёт ожидания данных и лимита. BlockingCollection добавляет bounded blocking model для синхронных producer/consumer.

### 4. Channel

Channel<T> поддерживает async чтение/запись, bounded capacity и full mode. Completion writer-а сообщает, что новых элементов не будет.

### 5. Backpressure

При заполнении producer ждёт, отбрасывает новый/старый элемент или завершает ошибкой. Политика определяется ценностью сообщения; unbounded buffer переносит перегрузку в память.

### 6. Порядок

Канал сохраняет порядок записи, однако несколько workers могут завершить обработку иначе. Для порядка внутри ключа поток partition-ят по ключу.

### 7. Command, event, query

Command просит действие, event сообщает факт, query читает без изменения. In-process message не становится автоматически durable.

### 8. Mediator

Уменьшает прямые ссылки, но скрывает call graph и не заменяет границы модулей. Он не даёт гарантий брокера.

### 9. Lifecycle

Владелец запускает workers, передаёт cancellation, завершает writers и ожидает readers. Ошибки фоновых задач нельзя оставлять ненаблюдаемыми.

## Минимальный пример

```csharp
var channel = Channel.CreateBounded<Job>(100);
await channel.Writer.WriteAsync(job, ct);
channel.Writer.Complete();
await foreach (var item in channel.Reader.ReadAllAsync(ct))
    await HandleAsync(item, ct);
```

Для конкурентного кода нарисуйте участников, разделяемые данные, точки ожидания и владельца завершения. Потокобезопасность отдельного вызова ещё не делает атомарной последовательность вызовов.

## Типичные ошибки

- путать Task и Thread;
- блокировать асинхронную работу через Wait или Result;
- запускать бесконтрольное число операций;
- терять исключение, cancellation или сигнал завершения;
- проверять гонки только единичным успешным запуском.

## Итог

Сначала докажите корректность простейшей версии, затем измеряйте и добавляйте параллелизм. Планировщик не обязан выбирать удобный interleaving, поэтому решение должно быть корректным при любом допустимом порядке.
