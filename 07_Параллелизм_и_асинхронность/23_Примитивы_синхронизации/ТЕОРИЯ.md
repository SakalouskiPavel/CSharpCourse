# Урок 23. Примитивы синхронизации потоков

> Синхронизация защищает сформулированный инвариант, а не отдельную строку кода. Примитив выбирают по задаче: атомарное изменение, взаимное исключение, сигнал, лимит или межпроцессная координация.

## Результаты обучения

- объяснять runtime-модель темы;
- выбирать механизм по требуемой гарантии;
- сохранять ошибки, отмену и завершение;
- проверять корректность и производительность экспериментом.

## Карта темы

### 1. lock и Monitor

lock даёт mutual exclusion и memory barriers. Объект блокировки должен быть private и стабильным; внутри critical section не вызывают долгий I/O или чужой callback.

### 2. Interlocked

Выполняет атомарные операции над простыми значениями и CompareExchange. Для согласованности нескольких полей обычный lock часто яснее.

### 3. Volatile

Управляет видимостью и ordering отдельного чтения/записи, но не делает составную операцию атомарной. Это специализированный инструмент.

### 4. SemaphoreSlim

Ограничивает число одновременных владельцев и поддерживает WaitAsync. Каждый успешный Wait требует Release в finally.

### 5. ReaderWriterLockSlim

Разрешает несколько readers или одного writer. Он оправдан только при измеренном read-heavy contention и не допускает await внутри владения.

### 6. Сигналы

ManualResetEvent остаётся установленным до Reset, AutoResetEvent освобождает одного waiter. CountdownEvent ждёт N завершений, Barrier координирует фазы.

### 7. Межпроцессные primitives

Named Mutex/Semaphore работают через ОС и дороже. Нужны имена, ACL и стратегия на случай аварии владельца.

### 8. Deadlock

Цикл ожидания устраняют общим порядком захвата, малыми секциями и отсутствием sync-over-async. Timeout помогает диагностике, но не исправляет инвариант.

### 9. Async lock

Обычный lock нельзя удерживать через await. SemaphoreSlim(1,1) даёт асинхронное ожидание, но его ownership/reentrancy не равны Monitor.

## Минимальный пример

```csharp
private readonly object _gate = new();
public void Credit(decimal amount)
{
    lock (_gate)
    {
        _balance += amount;
    }
}
```

Для конкурентного кода нарисуйте участников, разделяемые данные, точки ожидания и владельца завершения. Потокобезопасность отдельного вызова ещё не делает атомарной последовательность вызовов.

## Типичные ошибки

- путать Task и Thread;
- блокировать асинхронную работу через Wait или Result;
- запускать бесконтрольное число операций;
- терять исключение, cancellation или сигнал завершения;
- проверять гонки только единичным успешным запуском.

## Итог

Сначала докажите корректность простейшей версии, затем измеряйте и добавляйте параллелизм. Планировщик не обязан выбирать удобный interleaving, поэтому решение должно быть корректным при любом допустимом порядке.
