# Урок 23. Примитивы синхронизации: защищаем инварианты и передаём сигналы

> Синхронизация нужна не «вокруг многопоточного кода вообще», а для конкретной гарантии: один владелец, атомарное изменение, ограничение количества, ожидание события или согласованный порядок. Неверно выбранный primitive может убрать одну гонку и создать deadlock.

## 1. Сначала сформулируйте инвариант

```csharp
public sealed class Account
{
    public decimal Balance { get; private set; }
}
```

Для перевода важен не отдельный field, а правило:

- у отправителя достаточно денег;
- debit и credit происходят как одна операция;
- общая сумма сохраняется;
- никто не наблюдает половину перехода, если contract этого не допускает.

Lock «на Balance» недостаточен, если два accounts/ledger должны меняться согласованно. Synchronization boundary следует invariant boundary.

## 2. `lock` и `Monitor`

```csharp
private readonly object _gate = new();
private decimal _balance;

public void Deposit(decimal amount)
{
    lock (_gate)
    {
        _balance += amount;
    }
}
```

`lock` обеспечивает mutual exclusion: только один thread держит тот же lock object. Compiler гарантирует release через try/finally даже при exception.

В основе классического object lock — `Monitor.Enter/Exit`; современная версия C#/.NET также оптимизирует `lock` с `System.Threading.Lock`. Используйте statement, а не ручные Enter/Exit без необходимости.

Важно: все accesses к защищаемому state используют **тот же** primitive. Два разных `_gate` не синхронизируют друг друга.

## 3. Выбор lock object

Lock object должен быть:

- private;
- stable (не заменяется);
- предназначен только для этого synchronization contract.

Не используйте:

```csharp
lock (this) { }
lock ("shared") { }
lock (typeof(Account)) { }
```

Внешний код/интернированные строки/static type object могут взять тот же lock и создать неожиданную блокировку/deadlock.

## 4. Critical section

Под lock помещают минимальный, но **полный** набор действий invariant-а:

```csharp
lock (_gate)
{
    if (_balance < amount)
        return false;

    _balance -= amount;
    return true;
}
```

Если check вне lock, другой thread изменит state до write (time-of-check/time-of-use race).

Не выполняйте под lock:

- network/file/database I/O;
- `await` (обычный lock запрещает);
- callback/event чужого кода;
- долгую CPU-work, не требующую protected state;
- blocking wait на Task.

Скопируйте нужное state под lock, отпустите, затем делайте внешнюю работу; но убедитесь, что snapshot semantics допустима.

## 5. Reentrancy и Monitor

Monitor reentrant: тот же thread может повторно взять тот же object; внутренний count требует соответствующих releases. Reentrancy предотвращает self-deadlock простого вложенного вызова, но может скрыть сложный call graph.

`Monitor.TryEnter` позволяет timeout, однако timeout — диагностика/политика, не доказательство корректности. `Monitor.Wait/Pulse/PulseAll` поддерживают condition waiting:

- Wait атомарно отпускает monitor и ждёт;
- после сигнала снова захватывает monitor;
- condition проверяют в `while`, потому что состояние мог изменить другой thread;
- Pulse не хранится как долговечное сообщение.

Для producer/consumer чаще Channel/BlockingCollection яснее ручного Wait/Pulse.

## 6. `Interlocked`

```csharp
long next = Interlocked.Increment(ref _counter);
```

Interlocked выполняет atomic read-modify-write для поддерживаемых простых locations:

- Increment/Decrement/Add;
- Exchange;
- CompareExchange;
- atomic Read для нужных типов/platform semantics.

`CompareExchange(ref location, newValue, expected)` записывает, только если текущее значение равно expected, и возвращает старое. На нём строят lock-free algorithms.

Для счётчика/однослотового state Interlocked удобен. Для нескольких fields и сложного invariant lock обычно понятнее и безопаснее.

## 7. `Volatile`

```csharp
Volatile.Write(ref _stopping, true);
bool stopping = Volatile.Read(ref _stopping);
```

Volatile задаёт visibility/ordering одного read/write. Он не делает sequence атомарной:

```csharp
volatile int _count;
_count++; // всё ещё read + add + write
```

`volatile` keyword поддерживает ограниченный набор field types. Используйте для простых publication flags только при понимании memory model; CancellationToken/Task/synchronization primitive обычно лучше выражает протокол.

## 8. Memory barriers и happens-before

Synchronization создаёт порядок, при котором writes до release видимы thread-у после соответствующего acquire. Это называют happens-before relation.

Lock решает и mutual exclusion, и visibility. Interlocked имеет ordering guarantees. Без них «один thread точно уже присвоил» не означает, что другой читает ожидаемое значение.

Не добавляйте random barrier/volatile для «надёжности»: сначала определите protocol.

## 9. `SemaphoreSlim`

Semaphore имеет count permits:

```csharp
private readonly SemaphoreSlim _gate = new(4, 4);

await _gate.WaitAsync(cancellationToken);
try
{
    await UseLimitedResourceAsync(cancellationToken);
}
finally
{
    _gate.Release();
}
```

До четырёх операций входят одновременно. `SemaphoreSlim(1,1)` часто используется как async-compatible mutual exclusion.

Отличия от lock:

- no thread affinity: release может выполнить другой continuation;
- async WaitAsync;
- count может быть >1;
- не reentrant;
- fairness не следует предполагать без документации.

Каждый успешный Wait имеет один Release. Release без permit может превысить max/count или скрыть protocol bug.

## 10. Async lock

Обычный lock нельзя удерживать через await. `SemaphoreSlim(1,1)`:

```csharp
await _mutex.WaitAsync(ct);
try
{
    await CriticalAsync(ct);
}
finally
{
    _mutex.Release();
}
```

Но держать async mutex во время произвольного I/O всё равно опасно для latency. Иногда правильнее сериализованный worker/Channel, optimistic concurrency или убрать shared state.

SemaphoreSlim не знает owner и допускает ошибочный Release из другого path. Wrapper `AsyncLock` может возвращать IDisposable releaser, но должен корректно работать с cancellation/exceptions.

## 11. `ReaderWriterLockSlim`

Разрешает many readers или one writer, имеет upgradeable read mode.

Полезен, если:

- reads намного чаще/дольше writes;
- read operations действительно безопасны параллельно;
- измерено contention обычного lock;
- code sync и не await-ит внутри.

Цена: сложность, overhead, риск starvation/upgrade deadlock. Для короткого dictionary access обычный lock или immutable snapshot/ConcurrentDictionary часто лучше.

## 12. `Mutex`

Mutex даёт mutual exclusion и может быть named OS object для межпроцессной синхронизации.

Он тяжелее Monitor, имеет thread affinity и `AbandonedMutexException`, если owner thread завершился без release. Abandoned означает: lock получен, но protected state может быть повреждено — нельзя просто игнорировать.

Named mutex names/permissions/platform scope требуют осторожности. Для одного процесса используйте lock, если нет особого требования.

## 13. `Semaphore` и OS handles

`Semaphore` на WaitHandle может быть named/interprocess на поддерживаемой OS. `SemaphoreSlim` lightweight и только in-process.

Не выбирайте OS primitive «потому что надёжнее»: kernel transition дороже, disposal обязателен, platform semantics отличаются.

## 14. События синхронизации

Это не C# events из урока 13.

`ManualResetEventSlim` — gate:

- unsignaled waiters ждут;
- `Set` открывает всем и остаётся signaled;
- `Reset` закрывает.

`AutoResetEvent` — turnstile:

- `Set` освобождает одного waiter;
- автоматически возвращается unsignaled.

Signals могут coalesce; это не очередь сообщений/count. Для async code TaskCompletionSource часто лучше, потому что WaitHandle blocks thread.

## 15. `CountdownEvent` и `Barrier`

`CountdownEvent(N)` ждёт N вызовов Signal — удобно для группы sync workers. `Task.WhenAll` естественнее для Tasks.

`Barrier` координирует phases: все participants доходят до точки, затем переходят к следующей фазе. Participant failure/cancellation требуют обработки, иначе остальные зависнут.

Не применяйте Barrier к независимым operations: forced synchronization уменьшает parallelism.

## 16. `SpinWait` и `SpinLock`

Spin не блокирует thread, а активно проверяет короткое ожидание. Может быть быстрее при микроскопической critical section и низком contention, но сжигает CPU и плохо работает при oversubscription/single core.

Runtime primitives уже используют adaptive spinning. Прикладной SpinLock нужен редко и только после profiling. Его struct-copy semantics особенно опасна: копия — другой lock.

## 17. Concurrent collections

`ConcurrentDictionary`, `ConcurrentQueue`, `ConcurrentStack`, `ConcurrentBag`, `BlockingCollection` дают thread-safe operations.

Используйте составные atomic APIs:

```csharp
int value = dictionary.GetOrAdd(key, CreateValue);
```

Но value factory может быть вызвана больше одного раза при race, даже если сохранится одно значение; side effects внутри опасны. Точные гарантии метода читайте в документации.

Enumeration часто является moment-in-time/weakly consistent view, не global transaction. Несколько вызовов не становятся атомарными.

## 18. Deadlock

Классические условия: mutual exclusion, hold-and-wait, no preemption, circular wait.

```text
Thread A: lock Account 1 → ждёт Account 2
Thread B: lock Account 2 → ждёт Account 1
```

Главная защита — общий порядок acquisition:

```csharp
Account first = left.Id.CompareTo(right.Id) < 0 ? left : right;
Account second = ReferenceEquals(first, left) ? right : left;

lock (first.Gate)
lock (second.Gate)
{
    // transfer invariant
}
```

Одинаковый account обрабатывают отдельно, чтобы не ошибиться в order/equality. Gate не должен быть public; transfer operation может находиться в coordinator с доступом к internal locking API.

## 19. Livelock и starvation

**Livelock**: threads активны, но постоянно уступают/retry и не продвигаются. Jitter/backoff/random ordering помогают конкретному protocol.

**Starvation**: один participant долго не получает resource из-за unfair contention/priority/thread-pool exhaustion.

Deadlock timeout не исправляет livelock/starvation. Нужны metrics: wait duration, queue length, contention, progress.

## 20. Lock contention и granularity

Один global lock прост и корректен, но сериализует всё. Многие fine-grained locks повышают concurrency и deadlock complexity.

Выбор:

- сначала простой lock по invariant;
- измерить contention;
- сократить work под lock;
- partition state;
- immutable snapshot/message passing;
- только затем сложные primitives/lock-free.

Lock-free не означает wait-free и не автоматически быстрее. Correct lock-free algorithms требуют глубокого memory model/ABA/reclamation analysis.

## 21. Как выбрать primitive

| Требование | Стартовый кандидат |
|---|---|
| сложный sync invariant | `lock` |
| простой counter/reference swap | `Interlocked` |
| async mutual exclusion/limit | `SemaphoreSlim` |
| many readers, rare writers, измеренное contention | `ReaderWriterLockSlim` |
| sync gate/signal | reset event |
| несколько процессов | named Mutex/Semaphore/Event по platform |
| producer/consumer | Channel/BlockingCollection |
| Task completion signal | TaskCompletionSource |

## Частые ошибки

- разные locks защищают один resource;
- check вне lock, write внутри;
- lock(this/string/type);
- I/O/callback/await под lock;
- volatile используется для increment;
- SemaphoreSlim Release забывается/дублируется;
- ReaderWriterLock выбирается без measurement;
- WaitHandle блокирует async flow;
- GetOrAdd factory имеет side effect;
- два account locks берутся в разном порядке;
- timeout маскирует deadlock;
- lock-free выбирается ради названия.

## Итог

```text
Сначала invariant/protocol
  ↓
минимальный primitive с нужной гарантией
  ↓
единый owner/order/cleanup
  ↓
stress + metrics, а не sleeps
```

Вы готовы дальше, если можете защитить многополевой invariant одним protocol, отличить mutex/limit/signal и объяснить deadlock через цикл ожидания.
