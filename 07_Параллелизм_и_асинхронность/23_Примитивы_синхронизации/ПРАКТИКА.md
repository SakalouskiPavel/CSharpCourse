# Практика — урок 23

## Цель

Реализовать потокобезопасные переводы между счетами, сохраняя общую сумму, запрещая отрицательный баланс и исключая deadlock.

## Invariants

До кода запишите:

- Balance каждого account неотрицателен;
- перевод либо полностью произошёл, либо нет;
- сумма всех balances не меняется переводами;
- transfer account→тот же account имеет определённый outcome;
- currency должна совпадать;
- history соответствует успешным transfers по выбранной модели.

## Небезопасный baseline

Создайте намеренно небезопасную версию check/debit/credit и stress test. Покажите lost updates/negative balance либо объясните, почему race nondeterministic и как увеличить шанс без sleeps как synchronization.

## Lock implementation

Каждый Account имеет private/internal stable gate. Transfer coordinator берёт два locks в общем порядке по устойчивому unique id. Same-account случай обрабатывается отдельно.

Под locks выполняется только проверка и изменение in-memory state. Logging/events/I/O происходят после release с immutable result data.

## Deadlock experiment

В отдельной учебной реализации возьмите locks в caller order и синхронизируйте старт двух противоположных transfers через Barrier/TaskCompletionSource. Test имеет timeout и обнаруживает зависание. Финальная реализация использует ordering и проходит тот же сценарий.

Не оставляйте намеренно deadlocked threads в основном test process без изоляции/контроля.

## Interlocked experiment

Счётчик успешных transfers ведите через Interlocked. Объясните, почему balances нельзя согласованно обновить двумя отдельными Interlocked operations без более сложного protocol.

## Async-вариант

Создайте отдельный implementation через `SemaphoreSlim(1,1)` на account с тем же global ordering:

- WaitAsync с CancellationToken;
- release только для успешно acquired permits;
- finally в обратном порядке;
- cancellation между acquisitions не теряет первый permit;
- никакого callback/I/O до releases.

Сравните semantics с thread-affine Monitor.

## Stress checks

- много random transfers с фиксированным seed;
- общий balance до/после;
- ни одного отрицательного;
- timeout;
- противоположные transfers;
- insufficient funds;
- cancellation async acquisition;
- exception path releases locks/permits.

## Ограничения

- no lock(this/public object/string/type);
- no Thread.Sleep как proof correctness;
- no I/O/event under lock;
- no mixed lock order;
- no volatile increment;
- no catch, скрывающий worker failure;
- test awaits/joins all workers.

## Критерии приёмки

- invariants выдерживают повторный stress;
- deadlock variant обнаруживается контролируемо, fixed variant завершается;
- lock order объяснён;
- Interlocked используется только для независимого counter;
- async permits освобождаются на всех paths;
- metrics содержат wait time/contention без изменения correctness;
- README сравнивает lock, Interlocked и SemaphoreSlim.

## Дополнительное задание

Реализуйте immutable ledger/message-passing вариант без locks на Account и сравните простоту invariant-а. ReaderWriterLock/lock-free добавляйте только после измеренного bottleneck.
