# Урок 21. Конкурентность и параллелизм: процесс, поток и Task

> Слова «одновременно», «асинхронно» и «параллельно» описывают разные свойства. Сначала построим модель выполнения, затем увидим, почему общее изменяемое состояние делает результат зависимым от порядка действий.

## 1. Последовательное выполнение

В простейшей модели инструкции идут одна за другой:

```text
прочитать файл → вычислить hash → сохранить результат
```

Если задач десять, последовательный код обрабатывает следующую после предыдущей. Он прост: порядок виден, состояние меняется в одном месте, exception идёт по одному stack.

Параллелизм не является улучшением сам по себе. Он добавляет scheduling, synchronization и nondeterministic ordering. Сначала нужен корректный последовательный baseline.

## 2. Конкурентность и параллелизм

**Concurrency** — несколько работ находятся в процессе выполнения и могут продвигаться поочерёдно. Один человек готовит несколько блюд, переключаясь во время ожидания.

**Parallelism** — несколько работ физически выполняются одновременно, например на разных CPU cores.

Одна core может конкурентно выполнять много threads через переключение, но не исполнять их инструкции одновременно. Несколько cores дают реальный parallel execution.

Async I/O может быть concurrent без отдельного thread на каждое ожидание. Parallel CPU calculation обычно использует несколько worker threads.

## 3. Процесс

**Process** — запущенный экземпляр программы с virtual address space и ресурсами OS:

- memory mappings;
- handles/files/sockets;
- security identity;
- threads;
- loaded modules;
- environment/configuration.

Процессы изолированы: обычная reference одного процесса не указывает на object другого. Обмен требует IPC: pipe, socket, shared memory, file и serialization.

Изоляция ограничивает последствия crash и даёт security boundary лучше thread-а, но процесс дороже создавать/координировать.

## 4. Поток

**Thread** — последовательность выполняемых инструкций внутри процесса. У thread есть:

- call stack;
- instruction position/register state;
- scheduling priority/state;
- thread-local data.

Threads процесса разделяют managed heap и большинство process resources. Поэтому они быстро обмениваются objects, но могут одновременно изменять одно state.

```csharp
Console.WriteLine(Environment.CurrentManagedThreadId);
```

Managed thread id полезен для диагностики, но business logic не должна зависеть от случайного номера.

## 5. Планировщик и переключение контекста

OS scheduler распределяет runnable threads по cores. Thread может быть Running, Ready, Waiting/Blocked и т. п. Точный порядок не обещан.

Context switch сохраняет state одного thread и восстанавливает другого. Это имеет цену: scheduler work, cache disruption, memory traffic. Больше threads, чем полезной работы/cores, не означает больше throughput.

Программа не должна быть корректна только при «обычно наблюдаемом» порядке. Scheduler вправе остановить thread между почти любыми инструкциями.

## 6. Thread Pool

Создание OS thread и stack дорого. .NET thread pool переиспользует worker threads для коротких работ и continuations.

```csharp
Task task = Task.Run(() => Compute());
```

`Task.Run` обычно планирует delegate в pool. Pool динамически регулирует threads; это не фиксированное соответствие «Task = один thread».

Долгая блокировка множества pool threads приводит к **thread-pool starvation**: новая работа/continuations ждут свободных workers, latency растёт. Поэтому I/O следует ожидать async API, а не блокировать thread.

Dedicated Thread нужен редко: thread-affine API, специальный long-running loop или OS integration. Его lifetime/shutdown всё равно проектируются явно.

## 7. Task

`Task` представляет будущее завершение операции:

- `Waiting/Running` в упрощённой модели;
- `RanToCompletion`;
- `Faulted` с exception;
- `Canceled`.

`Task<T>` дополнительно имеет результат `T`.

Task — не thread. Он может:

- выполняться на pool thread;
- завершиться синхронно;
- представлять I/O, пока ни один managed thread не занят ожиданием;
- состоять из нескольких продолжений на разных threads;
- быть promise, завершённым `TaskCompletionSource` внешним событием.

## 8. AppDomain: исторический контекст

В .NET Framework AppDomain позволял загружать/выгружать assemblies и давал частичную изоляцию внутри процесса. В современном .NET дополнительные AppDomains для этой модели не поддерживаются.

Для plugin loading/unloading используется `AssemblyLoadContext`; он не является security sandbox. Для настоящей изоляции недоверенного кода нужен отдельный process/container с ограниченными правами.

## 9. CPU-bound и I/O-bound

**CPU-bound**: большую часть времени processor вычисляет — image transform, compression, hash, simulation.

**I/O-bound**: операция в основном ждёт внешний ресурс — disk, network, database.

Для CPU-bound parallelism может использовать cores. Для I/O-bound async API освобождает thread на время ожидания; добавление Task.Run вокруг уже async HTTP/file call обычно только добавляет scheduling.

Смешанная операция разделяется: async read → ограниченный CPU transform → async write.

## 10. Race condition

```csharp
int count = 0;

Parallel.For(0, 100_000, _ =>
{
    count++;
});
```

Ожидается 100000, но `count++` — не одно неделимое действие:

1. прочитать count;
2. прибавить 1;
3. записать.

Два threads могут прочитать одинаковое старое значение и один update потеряется. **Race condition** — результат зависит от timing/interleaving.

Однократный правильный запуск ничего не доказывает: опасный порядок мог не случиться.

## 11. Atomicity

Atomic operation наблюдается как неделимое действие относительно других участников. `Interlocked.Increment(ref count)` атомарно увеличивает счётчик.

Но несколько atomic fields не дают atomic business invariant:

```text
debit account A + credit account B
```

Даже атомарные отдельные изменения могут оставить промежуточное/некорректное состояние. Защищают весь invariant или исключают shared mutable state.

## 12. Visibility и ordering

Processors/JIT могут cache/reorder memory operations в рамках memory model. Без synchronization thread не обязан наблюдать записи другого в ожидаемом порядке.

`lock`, `Interlocked`, `Volatile`, task completion и concurrent collections создают определённые happens-before/visibility guarantees. Нельзя исправить visibility случайным `Thread.Sleep`.

`volatile` не превращает `count++` в atomic operation. Подробности — урок 23.

## 13. Shared mutable state

Самая простая конкурентная программа минимизирует совместно изменяемые данные.

Вместо:

```csharp
Parallel.ForEach(items, item => sharedList.Add(Process(item)));
```

используйте независимые slots/partition-local result, thread-safe collection с осознанной семантикой или pure transform:

```csharp
Result[] results = new Result[items.Length];
Parallel.For(0, items.Length, index =>
{
    results[index] = Process(items[index]);
});
```

Каждая iteration пишет в отдельный индекс. Нужно подтвердить, что `Process` также не использует hidden shared state.

## 14. Data parallelism

Один algorithm применяется к независимым chunks/elements:

```csharp
Parallel.ForEach(
    files,
    new ParallelOptions
    {
        MaxDegreeOfParallelism = Environment.ProcessorCount,
        CancellationToken = cancellationToken
    },
    file => ComputeHash(file));
```

`Parallel.For/ForEach`, PLINQ распределяют CPU-work. Они блокируют calling thread до завершения и не предназначены для массового async I/O.

Условия выгоды:

- достаточно дорогая независимая работа;
- данных достаточно для amortize overhead;
- минимум synchronization;
- нет требуемого strict order;
- cores доступны и process не конкурирует за них с latency-sensitive work.

## 15. Task parallelism

Несколько разных независимых операций:

```csharp
Task<Report> sales = BuildSalesReportAsync(ct);
Task<Report> stock = BuildStockReportAsync(ct);

await Task.WhenAll(sales, stock);
```

Task parallelism/concurrency описывает независимые работы, а data parallelism — деление набора. `WhenAll` сам не создаёт threads; операции уже начались при вызове async methods.

## 16. PLINQ

```csharp
var hashes = files
    .AsParallel()
    .WithDegreeOfParallelism(Environment.ProcessorCount)
    .Select(ComputeHash)
    .ToArray();
```

PLINQ может менять order. `AsOrdered` сохраняет order ценой координации. Exceptions обычно агрегируются. Side effects внутри query делают reasoning трудным.

Не добавляйте `AsParallel` к короткому LINQ «для ускорения». Измерьте data size/work cost.

## 17. Ограничение параллелизма

Degree of parallelism — число одновременно выполняемых работ. Для CPU начальная оценка связана с cores; для I/O — с limits внешнего service, connections, rate limit и memory.

Unbounded fan-out из миллиона Tasks не создаёт миллион полезных операций: растут memory, queue, timeouts и pressure на downstream.

Инструменты: `ParallelOptions`, `Parallel.ForEachAsync`, `SemaphoreSlim`, Channel/worker pool. Выбор зависит от CPU/I/O и backpressure.

## 18. Cancellation и exceptions

Parallel/TPL operations принимают `CancellationToken`. Cancellation cooperative: long CPU loop периодически проверяет token.

Несколько workers могут завершиться errors. Нужна политика:

- остановить дальнейшую работу;
- дождаться started operations;
- собрать все failures;
- не выдавать частичный результат как полный;
- cleanup resources.

Не ловите exception внутри каждой iteration и не теряйте его ради «чтобы цикл продолжился», если partial failure не определён.

## 19. Закон Амдала

Если доля программы `S` остаётся последовательной, максимальное ускорение при бесконечных cores ограничено примерно `1/S`.

Если 20% времени нельзя распараллелить, теоретический предел — около 5×, а на практике ниже из-за overhead. Оптимизировать tiny parallel part бессмысленно, если bottleneck — sequential I/O/lock.

## 20. Измерение

Сравнение:

- Release без debugger;
- одинаковые входы/результаты;
- warmup и несколько запусков;
- CPU time, elapsed, allocations, throughput;
- разные data sizes/degrees;
- нагрузка на machine/downstream.

Stopwatch одного запуска даёт подсказку, не benchmark. BenchmarkDotNet помогает CPU microbenchmark, но end-to-end concurrency требует load test/tracing.

## Частые ошибки

- concurrency, parallelism и asynchrony считаются синонимами;
- Task считается отдельным thread;
- Task.Run оборачивает настоящий async I/O;
- count++ считается atomic;
- обычный List изменяется из Parallel.ForEach;
- Thread.Sleep используется как synchronization;
- parallelism unbounded;
- порядок PLINQ предполагается без контракта;
- один успешный запуск считается доказательством отсутствия race;
- parallel variant не сравнивается с baseline.

## Итог

```text
Process содержит threads и memory
Thread исполняет instructions
Task представляет завершение операции
Concurrency = перекрывающийся прогресс
Parallelism = одновременное выполнение
Correctness начинается с ownership/shared state
```

Вы готовы дальше, если можете классифицировать CPU/I/O, объяснить race `count++`, выбрать независимые outputs и измерить последовательный/параллельный варианты.
