# Урок 21. Введение в параллелизм: процесс, поток, домен и Task

> Конкурентность означает одновременный прогресс, параллелизм — одновременное выполнение. Процесс изолирует память, поток исполняет инструкции, Task представляет будущее завершение и не обязан занимать отдельный поток.

## Результаты обучения

- объяснять runtime-модель темы;
- выбирать механизм по требуемой гарантии;
- сохранять ошибки, отмену и завершение;
- проверять корректность и производительность экспериментом.

## Карта темы

### 1. Процесс

Процесс имеет отдельное виртуальное адресное пространство и ресурсы ОС. Изоляция повышает устойчивость, но обмен данными требует IPC и сериализации.

### 2. Поток и пул

Поток имеет stack и scheduler state, разделяя heap процесса. Thread pool переиспользует workers и подходит для коротких работ; блокировка множества workers ведёт к starvation.

### 3. AppDomain сегодня

В .NET Framework AppDomain давал частичную изоляцию и выгрузку. В современном .NET для загрузки есть AssemblyLoadContext, а настоящую границу безопасности даёт отдельный процесс или контейнер.

### 4. Task

Task хранит состояние завершения, результат или ошибку. I/O Task может ждать сигнал ОС без занятого worker; Task.Run обычно планирует CPU-работу в pool.

### 5. Race condition

Результат зависит от interleaving. `count++` — чтение, вычисление и запись; корректность требует синхронизации либо исключения общего mutable state.

### 6. Memory visibility

Без happens-before гарантий поток может не наблюдать запись ожидаемым образом. lock, Interlocked и другие primitives определяют атомарность и порядок памяти.

### 7. Data parallelism

Parallel.For и PLINQ делят независимые элементы. Они полезны для достаточно крупной CPU-bound работы, если side effects и порядок не нужны.

### 8. Task parallelism

Независимые операции координируются через Tasks. Бесконтрольный fan-out истощает pool, соединения или память, поэтому concurrency ограничивают.

### 9. Измерение

Закон Амдала ограничивает ускорение последовательной долей; scheduling и synchronization стоят времени. Сравнивайте Release-сборки на репрезентативных данных.

## Минимальный пример

```csharp
var totals = new long[partitions.Count];
Parallel.For(0, partitions.Count, i =>
{
    totals[i] = Compute(partitions[i]);
});
var total = totals.Sum();
```

Для конкурентного кода нарисуйте участников, разделяемые данные, точки ожидания и владельца завершения. Потокобезопасность отдельного вызова ещё не делает атомарной последовательность вызовов.

## Типичные ошибки

- путать Task и Thread;
- блокировать асинхронную работу через Wait или Result;
- запускать бесконтрольное число операций;
- терять исключение, cancellation или сигнал завершения;
- проверять гонки только единичным успешным запуском.

## Итог

Сначала докажите корректность простейшей версии, затем измеряйте и добавляйте параллелизм. Планировщик не обязан выбирать удобный interleaving, поэтому решение должно быть корректным при любом допустимом порядке.
