# Практика — урок 25

## Цель

Создать два отдельных процесса — client и server — с duplex named-pipe протоколом, length-prefixed JSON frames, version handshake и concurrent request correlation.

## Frame

- 4-byte unsigned big-endian payload length;
- UTF-8 JSON payload;
- max frame, например 64 KiB, задаётся protocol constants;
- `ReadExactlyAsync` корректно обрабатывает partial reads/EOF;
- oversized/negative-by-interpretation/invalid length отклоняется до allocation;
- partial/invalid frame закрывает connection.

## Envelope

Request/response имеют version, type, correlationId, requestId, payload/error. Разрешённые message types перечислены явно. CLR type names/BinaryFormatter запрещены.

Handshake согласует version и max frame. Incompatible version завершается protocol error.

## Client

- один reader loop;
- pending dictionary correlation→TaskCompletionSource;
- serialized outgoing writes через Channel/single writer;
- per-request timeout и caller cancellation;
- при disconnect все pending Tasks завершаются;
- late/unknown response логируется по policy, не падает dictionary leak.

## Server

- принимает несколько sequential/concurrent clients через отдельные pipe instances;
- валидирует frame/envelope до handler;
- поддерживает `echo`, `getTime`, `delay` и idempotent `incrementOnce(requestId)`;
- не раскрывает stack trace;
- ограничивает concurrent requests;
- graceful shutdown ждёт active handlers по policy.

## Failure experiments

- fragmented write/read small chunks;
- disconnect после отправки до response;
- oversized frame;
- malformed JSON;
- unknown type/version;
- timeout delay;
- duplicate increment requestId;
- concurrent responses в другом порядке.

## Security note

Настройте доступ endpoint для текущего пользователя там, где API/platform поддерживает, либо явно опишите ограничение учебной среды. Pipe name не считается секретом.

## Критерии приёмки

- client/server запускаются отдельными processes;
- framing не зависит от границ Read;
- allocation ограничена;
- correlation поддерживает concurrency;
- disconnect не оставляет pending Tasks;
- duplicate command идемпотентен;
- stdout/stderr roles разделены;
- README содержит protocol specification и threat/failure model.

## Дополнительное задание

Запустите server как child Process с redirected streams, асинхронно читайте stdout/stderr и корректно интерпретируйте exit code. Либо перенесите тот же application protocol на TCP loopback и сравните только transport-specific части.
