# Урок 25. Межпроцессное взаимодействие

> IPC пересекает границу адресных пространств, поэтому требует протокола, фрейминга, сериализации, identity и обработки частичных отказов. Локальный транспорт всё равно может оборваться.

## Результаты обучения

- объяснять runtime-модель темы;
- выбирать механизм по требуемой гарантии;
- сохранять ошибки, отмену и завершение;
- проверять корректность и производительность экспериментом.

## Карта темы

### 1. Pipes

Anonymous pipes удобны для parent-child и однонаправленны; named pipes позволяют независимое подключение и duplex. Доступ определяется настройками ОС.

### 2. Sockets

TCP — ordered byte stream, а не поток сообщений: один Read не обязан совпадать с одним Write. Unix domain socket локален, но framing всё равно нужен.

### 3. Shared memory

Memory-mapped files быстры, но требуют layout/version, синхронизации и защиты от повреждения. Message passing обычно проще.

### 4. Файлы

Drop-folder наблюдаем, но требует atomic rename, claim, дедупликации и уборки. FileSystemWatcher — подсказка об изменении, не надёжная очередь.

### 5. Дочерний процесс

ProcessStartInfo умеет перенаправлять stdin/stdout/stderr. Оба выходных потока читают асинхронно, иначе заполненный buffer способен заблокировать child.

### 6. Протокол

Определите version handshake, length prefix/delimiter, encoding, max size, correlation id и error envelope. Длину проверяют до выделения памяти.

### 7. Частичный отказ

Peer может исчезнуть после приёма запроса до ответа. Retry требует idempotency/deduplication; exactly-once нельзя получить одной настройкой транспорта.

### 8. Безопасность

Минимальные ACL, проверка peer, отсутствие секретов в command line и безопасный endpoint защищают локальную границу.

### 9. Наблюдаемость

Correlation id связывает логи сторон; метрики показывают connect failures, latency, bytes и protocol errors.

## Минимальный пример

```csharp
using var pipe = new NamedPipeServerStream(
    "course-ipc", PipeDirection.InOut, 1,
    PipeTransmissionMode.Byte, PipeOptions.Asynchronous);
await pipe.WaitForConnectionAsync(ct);
```

Для конкурентного кода нарисуйте участников, разделяемые данные, точки ожидания и владельца завершения. Потокобезопасность отдельного вызова ещё не делает атомарной последовательность вызовов.

## Типичные ошибки

- путать Task и Thread;
- блокировать асинхронную работу через Wait или Result;
- запускать бесконтрольное число операций;
- терять исключение, cancellation или сигнал завершения;
- проверять гонки только единичным успешным запуском.

## Итог

Сначала докажите корректность простейшей версии, затем измеряйте и добавляйте параллелизм. Планировщик не обязан выбирать удобный interleaving, поэтому решение должно быть корректным при любом допустимом порядке.
