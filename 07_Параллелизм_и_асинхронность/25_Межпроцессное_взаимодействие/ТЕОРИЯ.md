# Урок 25. Межпроцессное взаимодействие: транспорт и протокол

> В другом процессе нет ваших object references. IPC передаёт bytes через транспорт, поэтому участники обязаны договориться о framing, encoding, version, identity, limits и поведении при частичном отказе.

## 1. Граница процесса

Каждый process имеет отдельное address space. Передать `Order` как reference нельзя. Нужно:

1. выбрать данные контракта;
2. serialize в bytes;
3. передать;
4. определить границу сообщения;
5. deserialize/validate;
6. связать response с request;
7. обработать disconnect/timeout/duplicate.

Даже localhost IPC может задержаться, оборваться или подключить неправильного клиента. «На одной машине» не отменяет distributed properties, лишь уменьшает latency.

## 2. Выбор транспорта

| Транспорт | Сильная сторона | Цена/ограничение |
|---|---|---|
| stdin/stdout | простой parent-child | один lifecycle, framing/errors |
| anonymous pipe | parent-child, handle inheritance | обычно one-way, локально |
| named pipe | независимые local client/server | platform security/instances |
| TCP socket | сеть и стандарты | ports, auth/TLS, byte stream |
| Unix domain socket | local socket + file permissions | platform/path semantics |
| memory-mapped file | throughput/shared memory | synchronization/layout corruption |
| file/drop folder | простота/наблюдаемость | latency, claiming, duplicates |
| broker | durability/routing | отдельная инфраструктура |

Сначала требования: topology, direction, message size/rate, latency, durability, security, portability, operations.

## 3. Дочерний процесс

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = executable,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false
};

using Process process = Process.Start(startInfo)
    ?? throw new InvalidOperationException("Процесс не запущен");
```

Arguments не должны содержать secrets: command line может быть видна другим tools/users. Используйте ArgumentList для корректного quoting вместо ручной строки.

Читайте stdout и stderr конкурентно async. Если parent ждёт process exit, не читая заполненный pipe buffer, а child ждёт записи — deadlock. Официальная документация отдельно предупреждает о зависимости synchronous redirected reads.

Назначьте timeout/cancellation, kill policy, exit code interpretation и cleanup descendants.

## 4. stdout как протокол

Не смешивайте machine messages и human logs в stdout. Например:

- stdout — JSON Lines responses;
- stderr — diagnostics;
- exit code — итог process-level outcome.

Новая строка как delimiter работает, только если serialization гарантирует escaped embedded newline и limits. Для arbitrary binary нужен length prefix.

## 5. Anonymous pipes

Обычно создаются parent-ом, handle передаётся child. Они удобны для однонаправленного local stream. Для duplex нужны два pipe-а. Независимый process без унаследованного handle обычно не обнаружит anonymous pipe по имени.

Dispose parent copy ненужного client handle после запуска, иначе EOF может не прийти: система всё ещё видит открытого writer-а.

## 6. Named pipes

Server создаёт endpoint по имени, client подключается:

```csharp
await using var server = new NamedPipeServerStream(
    "course-ipc",
    PipeDirection.InOut,
    maxNumberOfServerInstances: 4,
    PipeTransmissionMode.Byte,
    PipeOptions.Asynchronous);

await server.WaitForConnectionAsync(cancellationToken);
```

Каждый server instance обслуживает одно соединение; для нескольких clients server создаёт/принимает новые instances.

Named pipe хорошо интегрируется с Windows ACL/identity; platform behavior .NET различается. Имя не является секретом или authentication. Задавайте permissions/allowed users и не доверяйте подключившемуся process автоматически.

Byte mode рассматривайте как stream. Message mode доступен не везде/имеет platform constraints; application framing всё равно делает protocol portable/явным.

## 7. Sockets

TCP гарантирует упорядоченный поток bytes между endpoints, но не message boundaries:

- один Send может читаться несколькими Receive;
- несколько Send могут прийти одним Read;
- Read возвращает меньше requested bytes;
- 0 означает orderly EOF.

Нужна функция `ReadExactly`/loop, framing и max length. TCP не гарантирует application delivery после disconnect: данные могли дойти, а response потеряться.

Unix domain sockets используют local filesystem-like endpoint/permissions и избегают network port, но protocol/framing/failures остаются.

## 8. Framing

Варианты:

- delimiter (`\n`) — simple text records с escaping;
- fixed-size messages — редко, wastes/limits;
- length prefix — сначала fixed integer length, затем payload;
- self-describing format — parser всё равно должен знать конец;
- connection per message — дорого.

Length-prefixed frame:

```text
4 bytes big-endian length | N payload bytes
```

Receiver:

1. читает ровно 4 bytes;
2. декодирует unsigned length;
3. проверяет 0/min/max **до allocation**;
4. читает ровно N bytes;
5. deserialize/validate.

Нельзя доверять length из peer: `2 GB` может вызвать memory exhaustion.

## 9. Protocol envelope

```json
{
  "version": 1,
  "type": "getBook",
  "correlationId": "...",
  "requestId": "...",
  "payload": { },
  "traceId": "..."
}
```

Response содержит correlation id, success/error code, payload. Error contract не раскрывает stack trace/secrets, но даёт stable machine code и human-safe message.

Type whitelist обязателен. Не используйте BinaryFormatter/произвольные CLR type names.

## 10. Handshake и versioning

До business messages стороны обмениваются:

- protocol version/range;
- capabilities/features;
- max frame size;
- optional authentication identity;
- compression/encoding;
- session/correlation metadata.

Если версии несовместимы, соединение завершается понятной ошибкой до interpretation payload.

Additive fields обычно безопаснее; readers терпят unknown optional fields. Breaking change требует новой version и migration period.

## 11. Request/response correlation

При нескольких concurrent requests responses могут прийти в другом порядке. Client хранит:

```text
correlationId → TaskCompletionSource<Response>
```

Reader loop единолично читает stream, routes response нужному waiter. Несколько methods не должны одновременно читать один stream без protocol coordinator.

При disconnect все pending requests завершаются failure/cancellation; dictionary очищается. Correlation id уникален в session и ограничен по lifetime.

## 12. Partial reads/writes

Stream `ReadAsync` не обещает заполнить buffer. Пишите helper loop. WriteAsync для Stream обычно принимает buffer, но protocol должен считать frame отправленным только после успешного полного write/flush по нужной семантике.

Cancellation во время frame может оставить peer с частичным сообщением. Без resynchronization magic/connection reset safest policy — закрыть connection.

Один writer coordinator предотвращает interleaving bytes двух frames. Lock/Channel around outgoing messages, не параллельные writes в один stream.

## 13. Shared memory

Memory-mapped file позволяет processes видеть общий bytes region. Высокая скорость ценой сложной корректности:

- binary layout/version/alignment;
- ownership slots;
- memory ordering/synchronization;
- process crash в середине write;
- corruption detection/checksum;
- ACL;
- cleanup names/files.

Reference objects/GC pointers туда не записывают. Используйте fixed schema/offsets и interprocess primitives. Для большинства control messages pipe/socket проще.

## 14. File/drop-folder

Producer пишет temp file, fsync по требованиям и atomically rename в ready directory. Consumer claim-ит file через rename/lock, обрабатывает и перемещает archive/error.

Нужны:

- unique/idempotency id;
- duplicate handling;
- partial file prevention;
- cleanup/retention;
- permissions;
- retry/quarantine;
- multi-consumer claim.

FileSystemWatcher — уведомление, не durable queue: events могут coalesce/overflow, startup scan всё равно нужен.

## 15. Partial failures и delivery uncertainty

Client отправил command, server committed, connection оборвалась до response. Client не знает outcome.

Retry без idempotency может выполнить command дважды. Решения:

- request/idempotency id;
- server deduplication/result cache;
- operation naturally idempotent;
- status query;
- reconciliation.

Transport не даёт exactly-once business effect. At-most/at-least-once — результат protocol + application state.

## 16. Timeout, heartbeat и liveness

Connect/read/write/overall operation имеют разные timeouts. Timeout не доказывает, что peer умер или command не выполнен.

Heartbeat обнаруживает отсутствие связи вероятностно и создаёт traffic/state. Он не заменяет request timeout. Slow/paused process может считаться dead.

Определите reconnect/backoff/jitter и общий deadline. Бесконечный immediate reconnect создаёт storm.

## 17. Security

- authenticate/authorize peer при необходимости;
- least privilege ACL endpoint;
- не передавать secret command line/log;
- limit frame/rate/connections;
- validate before allocation/action;
- protect network transport TLS;
- prevent local endpoint squatting/spoofing;
- separate protocol errors from internal exception details.

Local process может быть скомпрометирован; localhost не автоматически trusted.

## 18. Observability

Связывайте обе стороны correlation/trace id. Metrics:

- connections/handshake failures;
- active/pending requests;
- frame sizes/bytes;
- latency/timeouts;
- protocol/version errors;
- disconnect/reconnect;
- duplicates;
- child exit codes/stderr.

Не логируйте raw payload, если там secrets/PII.

## Частые ошибки

- ожидается один Read на один Write;
- length выделяется без max check;
- несколько callers читают один stream;
- concurrent writes перемешивают frames;
- stdout logs смешаны с protocol;
- stderr не читается и child deadlock-ится;
- retry non-idempotent command;
- pipe name считается authentication;
- watcher считается queue;
- shared memory выбирается без crash protocol;
- BinaryFormatter/CLR type metadata используется с недоверенным input.

## Итог

```text
IPC transport переносит bytes
Protocol задаёт frames, versions, identity и errors
Application задаёт idempotency и business outcome
```

Вы готовы дальше, если можете реализовать length-prefix loop, ограничить allocation, завершить pending requests при disconnect и объяснить неопределённый outcome после timeout.
