# Урок 26. Async/await без магии: state machine, context и continuations

> `async/await` — удобный синтаксис над Task, awaiter и compiler-generated state machine. Знание механики нужно не для ручного переписывания state machine, а чтобы понимать synchronous fast path, deadlock, context capture, exceptions и allocations.

## 1. `async` не часть публичной сигнатуры

Для caller-а эти methods имеют одинаковый contract `Task<int>`:

```csharp
public async Task<int> FirstAsync()
{
    await Task.Delay(10);
    return 42;
}

public Task<int> SecondAsync()
{
    return Task.FromResult(42);
}
```

`async` описывает implementation и разрешает await; в metadata публичной сигнатуры caller видит Task return. Метод можно переписать без async, не меняя source-level contract, если semantics сохранена.

## 2. Метод начинает выполняться синхронно

```csharp
public async Task DemoAsync()
{
    Console.WriteLine("A");
    await Task.Delay(100);
    Console.WriteLine("B");
}

Console.WriteLine("1");
Task task = DemoAsync();
Console.WriteLine("2");
await task;
Console.WriteLine("3");
```

Обычно вывод: `1 A 2 B 3`. Код до первого **незавершённого** await выполняется прямо при вызове. Async method не отправляется автоматически в thread pool.

Если awaited operation уже completed, method может пройти дальше синхронно и даже полностью завершиться до возврата caller-у.

## 3. Что генерирует compiler

Упрощённо compiler создаёт:

- state machine struct/class representation;
- integer state;
- method builder (`AsyncTaskMethodBuilder` и варианты);
- fields для parameters/locals, нужных после await;
- awaiter fields;
- `MoveNext`, продолжающий выполнение;
- entry method, запускающий machine и возвращающий Task.

Концептуально:

```text
state -1: начать
  выполнить код до await
  если не готово: сохранить state/locals, зарегистрировать continuation, return
state 0: продолжить после completion
  GetResult (включая exception)
  выполнить остаток
  SetResult / SetException
```

Реальный generated code сложнее и меняется между compiler/runtime versions. Не полагайтесь на имена generated fields.

## 4. Suspension и сохранённые locals

```csharp
public async Task<int> CalculateAsync()
{
    int before = 10;
    int loaded = await LoadAsync();
    return before + loaded;
}
```

`before` нужен после await, поэтому становится полем state machine. Local, жизнь которого заканчивается до await, может не сохраняться.

Большой object, сохранённый local-ом через длинное ожидание, удерживается state machine. Можно сократить scope/обнулить reference, но только после профилирования/понимания читаемости.

## 5. Awaiter pattern

C# может await-ить не только Task. Expression должна предоставить подходящий awaiter:

- `GetAwaiter()`;
- `IsCompleted`;
- `GetResult()`;
- `OnCompleted`/`UnsafeOnCompleted` через completion interfaces.

Логика:

1. получить awaiter;
2. если `IsCompleted`, вызвать GetResult сразу;
3. иначе сохранить state и зарегистрировать continuation;
4. после сигнала вызвать MoveNext;
5. GetResult возвращает value или бросает exception.

Custom awaitable возможен, как custom enumerable, но production API обычно использует Task/ValueTask.

## 6. Continuation

Continuation — работа, которая должна продолжиться после завершения awaitable. Это не обязательно новый thread. Она может:

- выполниться inline в завершившем thread при допустимых условиях;
- попасть в thread pool;
- быть posted в UI SynchronizationContext;
- планироваться custom TaskScheduler.

Поэтому нельзя предполагать, что строки до/после await исполняются на одном thread, кроме context/thread-affinity guarantees конкретного environment.

## 7. `SynchronizationContext`

UI framework обычно имеет один UI thread. Его SynchronizationContext принимает continuation и ставит в UI message queue.

```csharp
string text = await LoadAsync();
label.Text = text; // продолжение вернулось в UI context
```

ASP.NET Core обычно не устанавливает custom SynchronizationContext для requests; continuation может продолжиться на pool thread. Это не отменяет ExecutionContext и thread-safety.

Console app тоже обычно не имеет UI context. Не переносите правила WPF/WinForms механически на server.

## 8. Deadlock sync-over-async

UI thread:

1. вызывает async method;
2. method await-ит incomplete Task и захватывает UI context;
3. caller делает `.Result`, блокируя UI thread;
4. completion пытается post continuation на UI thread;
5. thread ждёт Task, Task ждёт thread.

Решение — `await` до верхней границы, а не `Task.Run` как случайный пластырь.

Даже без context synchronous blocking тратит thread и может вызвать thread-pool starvation.

## 9. `ConfigureAwait(false)`

```csharp
string text = await client.GetStringAsync(uri, ct)
    .ConfigureAwait(false);
```

Для Task означает: continuation не обязана возвращаться в captured SynchronizationContext/non-default scheduler. Это полезно reusable library code, которому context не нужен.

`ConfigureAwait(false)`:

- не запускает operation в background;
- не гарантирует другой thread;
- не подавляет ExecutionContext/AsyncLocal;
- действует на конкретный await;
- не исправляет blocking caller полностью как дизайн.

Application/UI code часто нуждается в context. ASP.NET Core context capture обычно отсутствует, поэтому механическое добавление везде даёт мало пользы и шум.

## 10. `TaskScheduler`

TaskScheduler планирует Tasks. Default использует thread pool. Custom scheduler может ограничивать concurrency или привязывать работу.

Await для Task сначала учитывает custom SynchronizationContext, затем non-default TaskScheduler (детали runtime). `Task.Run` явно планирует на default scheduler.

Не используйте `Task.Factory.StartNew(async () => ...)` без понимания scheduler/unwrapping: легко получить `Task<Task>`. Для обычного offload — Task.Run.

## 11. `ExecutionContext` и `AsyncLocal<T>`

ExecutionContext переносит ambient logical state через async flow:

- AsyncLocal values;
- security/culture-related context по platform;
- tracing Activity/correlation context и др.

```csharp
static readonly AsyncLocal<string?> Correlation = new();
```

Значение течёт через await даже при смене thread. Это отличается от `[ThreadStatic]`/ThreadLocal, привязанных к физическому thread.

Ambient state удобно для tracing, но скрытые dependencies/large objects создают coupling/overhead. `ConfigureAwait(false)` не отключает flow ExecutionContext.

## 12. Synchronous fast path

```csharp
public async Task<Book> GetAsync(string id)
{
    if (_cache.TryGetValue(id, out Book? cached))
        return cached;

    return await LoadAsync(id);
}
```

Cache hit может завершить async method синхронно. Task/runtime имеют optimizations для некоторых completed results, но async method с suspension обычно требует state machine storage/Task.

Не предполагайте always asynchronous. API caller должен быть корректен и при synchronous completion: не держать lock, ожидая callback позже, не создавать reentrancy bug.

## 13. Exceptions до/внутри async method

В `async Task` method exception, возникшая при выполнении body, обычно фиксируется в returned Task, даже если произошла до первого await:

```csharp
Task task = AsyncMethod(); // обычно возвращает faulted Task
await task;                // exception здесь
```

Но non-async wrapper может валидировать синхронно:

```csharp
public Task<string> LoadAsync(string path, CancellationToken ct)
{
    ArgumentException.ThrowIfNullOrWhiteSpace(path);
    return LoadCoreAsync(path, ct);
}

private async Task<string> LoadCoreAsync(...) { ... }
```

Теперь argument exception выбрасывается при вызове до возврата Task. Это изменение observable timing; API должен быть последовательным и tests учитывать.

## 14. Exception при `await`

`GetResult` awaiter-а повторно выбрасывает исходное exception. В отличие от `.Wait()`/`.Result`, await обычно не оборачивает одиночную ошибку в AggregateException.

`try/catch/finally` может охватывать await:

```csharp
try
{
    await WorkAsync();
}
finally
{
    await CleanupAsync();
}
```

Если cleanup тоже падает, он может скрыть исходную ошибку — проектируйте cleanup/reporting осознанно.

## 15. `WhenAll` и exceptions

Несколько tasks могут fault. Await итоговой WhenAll Task выбрасывает exception по установленным правилам, но итоговая Task содержит aggregate information о failures. Если нужна полная диагностика, дождитесь всех и изучите outcomes/преобразуйте каждую operation в Result.

Не ловите только первое и не оставляйте остальные unobserved. Точный порядок exception не должен быть business contract.

## 16. Cancellation state

Async method становится Canceled, когда завершает через подходящую `OperationCanceledException`, связанную с cancellation semantics. Просто вернуть default после token request — success, а не canceled.

```csharp
ct.ThrowIfCancellationRequested();
```

Catch-all не должен превращать cancellation в Faulted/Success. `TaskCanceledException` является OperationCanceledException; обычно ловят базовый semantic type.

## 17. `async void`

`async void` не возвращает Task:

- caller не знает completion;
- exception доставляется SynchronizationContext/unhandled path;
- нельзя compose/WhenAll;
- tests сложнее;
- lifetime/cleanup не наблюдаются.

Допустим event handler:

```csharp
private async void Button_Click(object? sender, EventArgs e)
{
    try
    {
        await HandleClickAsync();
    }
    catch (Exception ex)
    {
        ShowError(ex);
    }
}
```

Основная логика возвращает Task; void adapter только UI boundary.

## 18. `TaskCompletionSource<T>`

Создаёт Task, завершение которого контролирует producer:

```csharp
var source = new TaskCompletionSource<Result>(
    TaskCreationOptions.RunContinuationsAsynchronously);

source.TrySetResult(result);
source.TrySetException(error);
source.TrySetCanceled(token);
```

`RunContinuationsAsynchronously` снижает риск inline continuation под внутренним lock/callback producer-а. Но это не гарантия отдельного thread для всей работы.

Используйте TrySet при возможной гонке нескольких completion sources (response/timeout/disconnect), чтобы loser не бросал InvalidOperationException.

Всегда удаляйте registrations/pending dictionary entries.

## 19. `ValueTask` internals на уровне потребителя

ValueTask может содержать:

- синхронный result;
- Task;
- IValueTaskSource-backed operation.

Поэтому его обычно await-ят ровно один раз. Нельзя бездумно хранить, await-ить повторно, использовать `.Result` или WhenAll напрямую. Если нужна многократная/combinator semantics — `.AsTask()` с возможной allocation.

Custom pooling builder/IValueTaskSource — library-level optimization с token/version/lifetime complexity.

## 20. Allocation model

Возможные costs:

- returned Task;
- boxed state machine при suspension;
- continuations;
- captured ExecutionContext;
- captured locals/closures;
- CancellationToken registrations.

Современный runtime оптимизирует common paths. Не переписывайте ясный Task API ради allocations без trace/benchmark. Часто I/O latency намного больше state machine overhead.

## 21. Custom awaitable: учебная модель

Можно сделать awaitable delay/signal с GetAwaiter. Production implementation обязана корректно решить:

- synchronous completion race;
- continuation exactly once;
- ExecutionContext;
- exception/cancellation;
- thread safety;
- lifetime.

Поэтому учебный custom awaiter демонстрирует protocol, а не заменяет Task/ValueTask.

## 22. Отладка async

Debugger показывает logical async stack, но physical thread stack меняется между continuations. Полезны:

- Tasks window/async call stacks;
- structured logs с trace id;
- Activity/OpenTelemetry;
- dump/trace thread-pool starvation;
- exception first-chance осторожно;
- decompiler generated state machine для обучения.

Не логируйте только thread id как identity operation: одна operation меняет threads, один thread выполняет много operations.

## 23. Тестирование fast/suspended paths

Dependency должна уметь:

- вернуть already completed Task;
- вернуть incomplete TaskCompletionSource;
- fault/cancel до и после suspension;
- завершить continuation inline/asynchronously в допустимых случаях.

Это обнаруживает reentrancy и assumption «await всегда переключает thread».

Tests возвращают Task, не async void, используют timeout и не блокируют `.Result`.

## Частые ошибки

- ожидается, что async method сразу уйдёт на другой thread;
- await считается обязательным suspension;
- lock держится при вызове async method, который может завершиться sync/reenter;
- ConfigureAwait(false) считается Task.Run;
- ExecutionContext путается с SynchronizationContext;
- StartNew async lambda создаёт Task<Task>;
- `.Result` блокирует UI context;
- async void используется вне event boundary;
- TaskCompletionSource continuation запускается под lock;
- ValueTask await-ится несколько раз;
- thread id используется как operation id;
- optimization делается без measurement.

## Итог

```text
async method стартует синхронно
await проверяет IsCompleted
  ├─ готово → fast path
  └─ не готово → сохранить state + continuation
completion → MoveNext → GetResult
context определяет место продолжения, Task — outcome
```

Вы готовы дальше, если можете объяснить state machine на временной диаграмме, воспроизвести UI deadlock, отличить contexts и протестировать synchronous/asynchronous completion paths.
