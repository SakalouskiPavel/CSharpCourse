# Урок 26. Магические слова async/await без магии

> Компилятор превращает async-метод в state machine. Понимание continuation, context capture, исключений и fast path объясняет deadlock, лишние переключения и правила хорошего API.

## Результаты обучения

- объяснять runtime-модель темы;
- выбирать механизм по требуемой гарантии;
- сохранять ошибки, отмену и завершение;
- проверять корректность и производительность экспериментом.

## Карта темы

### 1. State machine

Код до первого незавершённого await идёт синхронно. Compiler создаёт state, builder и MoveNext; значения, нужные после await, становятся полями машины.

### 2. Awaiter pattern

Await использует GetAwaiter, IsCompleted, GetResult и регистрацию continuation. Если операция уже завершилась, suspension и планирование могут не понадобиться.

### 3. Context capture

UI SynchronizationContext возвращает continuation на UI thread. ASP.NET Core обычно не имеет такого context, но TaskScheduler и execution context всё ещё важны.

### 4. ConfigureAwait

ConfigureAwait(false) просит не возвращаться в захваченный context, если он есть. Это не команда перейти в thread pool; библиотечный код применяет её, когда context не нужен.

### 5. Sync-over-async deadlock

UI thread блокируется через Result/Wait, а continuation ждёт этот же thread. Устойчивое решение — async all the way.

### 6. Исключения

Ошибки внутри state machine сохраняются в Task и выбрасываются при await. Повторный await завершённой задачи возвращает тот же результат/ошибку.

### 7. Cancellation

Ожидаемая отмена выражается canceled Task и OperationCanceledException. Её не стоит превращать в успешный результат или неизвестную ошибку.

### 8. Async void

Приемлем для event handler; его нельзя await, а ошибка уходит в context. Рабочую логику выносят в Task-returning method.

### 9. Цена

State machine и Task имеют стоимость при suspension. ValueTask и другие оптимизации допустимы после профилирования, когда API-контракт не становится хрупким.

### 10. Тесты

Async test возвращает Task. TaskCompletionSource и управляемые fakes координируют продолжения детерминированно вместо Thread.Sleep.

## Минимальный пример

```csharp
public async Task<string> LoadAsync(Uri uri, CancellationToken ct)
{
    using var response = await client.GetAsync(uri, ct).ConfigureAwait(false);
    response.EnsureSuccessStatusCode();
    return await response.Content.ReadAsStringAsync(ct).ConfigureAwait(false);
}
```

Для конкурентного кода нарисуйте участников, разделяемые данные, точки ожидания и владельца завершения. Потокобезопасность отдельного вызова ещё не делает атомарной последовательность вызовов.

## Типичные ошибки

- путать Task и Thread;
- блокировать асинхронную работу через Wait или Result;
- запускать бесконтрольное число операций;
- терять исключение, cancellation или сигнал завершения;
- проверять гонки только единичным успешным запуском.

## Итог

Сначала докажите корректность простейшей версии, затем измеряйте и добавляйте параллелизм. Планировщик не обязан выбирать удобный interleaving, поэтому решение должно быть корректным при любом допустимом порядке.
