# Практика — урок 26

## Цель

Создать лабораторию async semantics, где каждый сценарий детерминированно показывает fast path, suspension, context capture, exceptions, cancellation и continuation scheduling.

## Trace recorder

Записывайте sequence number, operation name, thread id и AsyncLocal correlation id. Thread id используется только для наблюдения, не для correctness.

## Сценарии

1. Already completed Task: код после await выполняется до возврата caller-у.
2. Incomplete TaskCompletionSource: method приостанавливается, затем продолжается.
3. TCS с/без RunContinuationsAsynchronously: сравнить допустимое место continuation без хрупкого утверждения о конкретном id.
4. Exception до/после await.
5. Non-async validation wrapper и observable timing exception.
6. Cancellation до start и после suspension.
7. AsyncLocal течёт через await и ConfigureAwait(false).
8. ValueTask synchronous result и Task-backed path, каждый await ровно один раз.

## Custom SynchronizationContext

Создайте минимальный single-thread test context/очередь либо используйте UI test host. Воспроизведите deadlock `.Result` только в изолированном process/test с timeout, чтобы не зависнуть. Затем исправьте async all the way.

Покажите, что ConfigureAwait(false) у library await не требует возвращения в context, но не выдавайте его за универсальное исправление blocking caller.

## State machine inspection

Соберите Debug/Release, откройте generated/decompiled async method и найдите:

- state field;
- builder;
- hoisted local;
- awaiter;
- MoveNext;
- SetResult/SetException.

Не копируйте generated names как стабильный API. Сопоставьте с исходными строками.

## TaskCompletionSource race

Смоделируйте response, timeout и disconnect, которые конкурируют за completion. Используйте TrySetResult/TrySetCanceled/TrySetException; loser выполняет cleanup без второго completion. Pending entry удаляется ровно один раз.

## Async void boundary

Создайте event handler `async void`, который только await-ит Task-returning core и локально обрабатывает boundary error. Покажите, что core можно тестировать обычным async test.

## Ограничения

- no sleeps как coordination;
- no hanging deadlock in main test process;
- no `.Result` кроме намеренного isolated scenario;
- no assertions о конкретном thread id без context guarantee;
- ValueTask не consumed повторно;
- all Tasks observed;
- tests имеют timeout.

## Критерии приёмки

- trace доказывает fast/suspended paths;
- context deadlock воспроизведён безопасно и исправлен;
- exceptions/cancellation имеют правильный Task state;
- AsyncLocal и ConfigureAwait объяснены корректно;
- TCS race не даёт double completion/leak;
- generated state machine сопоставлена с source;
- README содержит временные диаграммы минимум трёх сценариев.

## Дополнительное задание

Создайте минимальный custom awaitable только для демонстрации GetAwaiter/IsCompleted/GetResult/OnCompleted. Перечислите гонки и гарантии, которых не хватает для production, и оставьте Task основным API.
