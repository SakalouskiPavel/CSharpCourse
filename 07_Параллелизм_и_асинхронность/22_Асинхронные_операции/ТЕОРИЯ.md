# Урок 22. Асинхронные операции: не занимать поток ожиданием

> Асинхронность особенно полезна, когда программа ждёт сеть, файл или базу. Она не ускоряет внешний сервер и не означает автоматический параллелизм; она позволяет thread-у заниматься другой работой, пока операция ждёт завершения.

## 1. Блокирующее ожидание

Синхронный read:

```csharp
string text = File.ReadAllText(path);
```

Calling thread не может продолжить до результата. Для console utility это иногда приемлемо. В UI thread интерфейс замерзает; в server request thread блокировка снижает scalability.

Async read:

```csharp
string text = await File.ReadAllTextAsync(path, cancellationToken);
```

Когда операция ещё не завершена, method приостанавливается и возвращает управление caller-у. Thread не обязан стоять рядом с диском. После completion продолжение будет запланировано runtime.

## 2. I/O completion простыми словами

Для поддерживаемого async I/O приложение просит OS начать operation и регистрирует continuation/completion. OS/device/network работают независимо. Когда результат готов, runtime узнаёт и планирует продолжение.

Это не означает «вообще нет threads»: системные реализации различаются, а часть подготовки/обработки выполняется threads. Главное — нет выделенного blocked managed worker на всё время типичного ожидания.

## 3. TAP

Task-based Asynchronous Pattern:

- имя обычно заканчивается `Async`;
- возвращается `Task` или `Task<T>`;
- optional `CancellationToken` обычно последний parameter;
- operation начинает работу при вызове, если документация не говорит о lazy behavior;
- exceptions/cancellation входят в Task;
- progress при необходимости передаётся отдельным contract.

```csharp
Task SaveAsync(Document document, CancellationToken cancellationToken);
Task<Document> LoadAsync(string id, CancellationToken cancellationToken);
```

Не создавайте overload с optional bool `runAsync`; sync/async APIs имеют разные execution contracts.

## 4. `async` и `await` на базовом уровне

```csharp
public async Task<string> LoadUpperAsync(string path, CancellationToken ct)
{
    string text = await File.ReadAllTextAsync(path, ct);
    return text.ToUpperInvariant();
}
```

`async` разрешает `await` и заставляет compiler построить state machine. Метод начинает выполняться синхронно до первого await, который ещё не завершён. Подробная трансформация — урок 26.

`await`:

- не блокирует текущий thread как `.Result`;
- приостанавливает async method;
- при completion получает result;
- повторно выбрасывает exception;
- сохраняет обычную форму try/catch/finally.

Если awaited Task уже completed, продолжение может выполниться сразу без suspension.

## 5. Возвращаемые типы

- `Task` — операция без значения;
- `Task<T>` — операция с результатом;
- `IAsyncEnumerable<T>` — поток результатов во времени;
- `ValueTask<T>` — специализированная оптимизация;
- `async void` — практически только event handlers.

Обычный public async API начинает с Task/Task<T>. Caller может await, compose, cancel/observe errors. `async void` невозможно нормально await-ить; exception не находится в Task caller-а.

## 6. Async all the way

Плохо:

```csharp
string text = LoadAsync().Result;
```

`.Result`, `.Wait()` блокируют thread, могут вызвать deadlock с SynchronizationContext и ухудшают scalability.

Правильно распространять async вверх:

```csharp
string text = await LoadAsync();
```

Иногда legacy boundary не позволяет async. Это отдельное integration decision с рисками, а не повод оборачивать каждый вызов в Task.Run.

## 7. I/O-bound и `Task.Run`

```csharp
// Лишняя оболочка:
await Task.Run(() => File.ReadAllTextAsync(path, ct), ct);
```

Если API уже настоящее async I/O, Task.Run добавляет thread-pool scheduling без пользы.

Task.Run уместен в UI/client boundary для значимого CPU-bound method, чтобы UI thread оставался responsive:

```csharp
Report report = await Task.Run(() => BuildReport(data), ct);
```

Server library обычно предоставляет sync CPU method; caller решает scheduling. Массовый Task.Run в ASP.NET не создаёт дополнительные cores и может истощить pool.

## 8. Последовательное и конкурентное ожидание

Последовательно:

```csharp
User user = await LoadUserAsync(id, ct);
Orders orders = await LoadOrdersAsync(id, ct);
```

Второй вызов начинается после первого. Если независимы:

```csharp
Task<User> userTask = LoadUserAsync(id, ct);
Task<Orders> ordersTask = LoadOrdersAsync(id, ct);

await Task.WhenAll(userTask, ordersTask);

User user = await userTask;
Orders orders = await ordersTask;
```

Вызов async method обычно запускает operation до await. `WhenAll` не «распараллеливает» ленивый код, а создаёт Task, завершающийся после всех переданных tasks.

Конкурентность безопасна только при независимости и допустимой нагрузке downstream.

## 9. `Task.WhenAll`

```csharp
Result[] results = await Task.WhenAll(tasks);
```

Результаты массива сохраняют порядок input tasks, а не completion order. Если tasks faulted, итог faulted; все tasks следует считать запущенными и наблюдать их завершение/ошибки.

Partial success policy должна быть явной. Когда нужно собрать каждую ошибку как data, каждая operation может преобразовать свой outcome в Result вместо исключения.

Не запускайте миллион tasks и затем WhenAll: все objects/operations уже созданы. Нужен bounded concurrency.

## 10. `Task.WhenAny`

```csharp
Task<DownloadResult> completed = await Task.WhenAny(tasks);
DownloadResult result = await completed;
```

WhenAny возвращает first completed Task, даже если она faulted/canceled; её нужно await повторно для outcome.

Применения: обработка по мере завершения, race двух источников, timeout pattern. Проигравшие operations не отменяются автоматически — caller обязан решить их lifetime.

## 11. CancellationToken

Cancellation cooperative:

```csharp
public async Task<string> DownloadAsync(Uri uri, CancellationToken ct)
{
    ct.ThrowIfCancellationRequested();
    return await httpClient.GetStringAsync(uri, ct);
}
```

Token передаётся во все поддерживающие nested operations. CPU loop периодически проверяет. После точки irreversible commit отмена может быть проигнорирована или иметь отдельную compensation policy.

`CancellationToken.None` не следует подставлять внутри library, если caller передал token. Не используйте один global CancellationTokenSource навсегда; source имеет lifecycle и может быть IDisposable.

## 12. Timeout

Timeout ограничивает ожидание caller-а:

```csharp
string result = await operation.WaitAsync(timeout, cancellationToken);
```

Но прекращение ожидания не всегда прекращает underlying operation. Если возможно, передайте связанный token самой операции:

```csharp
using var timeoutSource = CancellationTokenSource.CreateLinkedTokenSource(ct);
timeoutSource.CancelAfter(timeout);
await OperationAsync(timeoutSource.Token);
```

Нужно различать user cancellation и timeout в результате/логах. Retry использует общий time budget, иначе каждая попытка заново умножает latency.

## 13. Ограничение concurrency через SemaphoreSlim

```csharp
var gate = new SemaphoreSlim(8);

async Task<Result> RunLimitedAsync(Item item, CancellationToken ct)
{
    await gate.WaitAsync(ct);
    try
    {
        return await ProcessAsync(item, ct);
    }
    finally
    {
        gate.Release();
    }
}
```

Каждый успешный Wait соответствует одному Release в finally. Создать tasks для миллиона items всё равно затратно; Channel/worker pool/Parallel.ForEachAsync может лучше ограничить и creation, и active work.

Limit выбирается по connection pool, API rate, memory и измерениям, не по случайной константе.

## 14. `Parallel.ForEachAsync`

```csharp
await Parallel.ForEachAsync(
    items,
    new ParallelOptions
    {
        MaxDegreeOfParallelism = 8,
        CancellationToken = ct
    },
    async (item, token) => await ProcessAsync(item, token));
```

Подходит для bounded processing набора. Не гарантирует order completion. Shared output требует thread-safe/partitioned design. Для сложного producer/consumer и backpressure Channel яснее (урок 24).

## 15. Async streams

```csharp
public async IAsyncEnumerable<Book> ReadBooksAsync(
    [EnumeratorCancellation] CancellationToken ct = default)
{
    while (await reader.ReadAsync(ct))
        yield return MapBook(reader);
}
```

Потребление:

```csharp
await foreach (Book book in ReadBooksAsync(ct).WithCancellation(ct))
    Console.WriteLine(book.Title);
```

`IAsyncEnumerable<T>` выдаёт элементы по мере async получения, уменьшая buffering и позволяя consumer naturally задавать pace. Однако producer resources остаются открыты, пока enumeration не завершена/disposed.

Нужно определить error после части элементов: consumer уже получил partial data.

## 16. Progress

```csharp
IProgress<DownloadProgress>? progress
```

`Progress<T>` может публиковать callback в captured context (например UI). Library не должна сообщать progress слишком часто или предполагать context. Progress — наблюдение, не cancellation и не result.

Для server/background pipeline часто лучше metrics/events/channel, а не UI-oriented IProgress.

## 17. `ValueTask`

ValueTask может уменьшить Task allocations, если operation **часто** завершается синхронно (cache/buffer) и hot path подтверждён.

Цена:

- struct больше ссылки;
- не всегда можно await несколько раз/использовать combinators без AsTask;
- сложнее implementation/consumption;
- неправильное использование опасно с custom IValueTaskSource.

Public API по умолчанию возвращает Task. ValueTask — измеренная оптимизация, не «новая лучшая Task».

## 18. Ошибки и cleanup

```csharp
await using Stream stream = await OpenAsync(ct);
try
{
    await ProcessAsync(stream, ct);
}
catch (OperationCanceledException) when (ct.IsCancellationRequested)
{
    throw;
}
```

Try/catch/finally работают через await. Resources живут до завершения async scope. Fire-and-forget method теряет нормального владельца exception/cleanup/shutdown.

Если background work действительно нужна, host/service хранит Task, наблюдает errors и ждёт при shutdown.

## 19. Тестирование

Async test возвращает Task и await-ит system under test. Не используйте `Thread.Sleep`/`Task.Delay(100)` для надежды на порядок.

Инструменты deterministic coordination:

- `TaskCompletionSource<T>` с `RunContinuationsAsynchronously`;
- fake async dependency;
- controlled CancellationTokenSource;
- fake clock/time provider для timeout logic;
- assertions о незавершённости/сигналах.

Test должен иметь общий timeout, чтобы deadlock не зависал бесконечно.

## 20. Официальная модель

Официальная документация .NET рекомендует для I/O-bound использовать async/await без Task.Run, для CPU-bound responsiveness — Task.Run на appropriate boundary, и не блокировать Tasks через Wait/Result. Эти правила являются отправной точкой, а конкретный runtime/provider проверяется измерением.

## Частые ошибки

- async считается отдельным thread;
- Task.Run оборачивает async I/O;
- `.Result`/`.Wait()` блокируют call chain;
- независимые операции случайно await-ятся последовательно;
- создаётся unbounded fan-out;
- CancellationToken теряется на одном layer;
- timeout прекращает ожидание, но orphan operation забыта;
- WhenAny winner не await-ится;
- ValueTask используется по умолчанию;
- async stream resource не disposed;
- fire-and-forget exception не наблюдается;
- tests координируются sleeps.

## Итог

```text
I/O-bound → настоящий async API, await без blocked thread
CPU-bound → sync computation; Task.Run/parallelism на boundary
Много операций → явная independence + bounded concurrency
Каждая операция → result/error/cancellation/lifetime
```

Вы готовы дальше, если можете отличить Task от thread, запустить независимые operations вместе, ограничить concurrency и провести cancellation/timeout до самого нижнего API.
