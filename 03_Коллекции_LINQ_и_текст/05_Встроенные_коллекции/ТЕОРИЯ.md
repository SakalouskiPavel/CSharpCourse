# Урок 05. Коллекции: как хранить группы данных

> До сих пор отдельная переменная хранила одно значение. Реальным программам нужны списки товаров, множество ролей, очередь заявок и поиск пользователя по идентификатору. Коллекции решают эти разные задачи, поэтому универсальной «лучшей коллекции» не существует.

## Что нужно знать

Понадобятся переменные, циклы, методы, классы, равенство и интерфейсы из уроков 02–04. LINQ пока не используется: сначала важно понять сами структуры данных.

## Результаты обучения

Вы сможете:

- работать с массивами и `List<T>`;
- перечислять элементы через `for` и `foreach`;
- выбирать `Dictionary`, `HashSet`, `Queue` или `Stack` по смыслу задачи;
- понимать `Count`, `Capacity`, индекс и ключ;
- оценивать основные операции через O(1), O(n) и O(log n);
- задавать равенство и comparer;
- отличать read-only представление от неизменяемых данных;
- понимать назначение sorted, immutable и concurrent collections.

## 1. Что называется коллекцией

**Коллекция** — объект, хранящий группу элементов. В generic-типе запись `<T>` обозначает тип элемента:

```csharp
List<string> names = new List<string>();
```

Это список строк. В современном C# тип справа можно сократить:

```csharp
List<string> names = new();
```

Компилятор всё равно знает точный тип. Список строк не позволит случайно добавить число:

```csharp
names.Add("Анна");
// names.Add(42); // ошибка компиляции
```

Generic-коллекции дают типобезопасность: ошибка обнаруживается при сборке, а при чтении не требуется приводить каждый `object` к нужному типу.

## 2. Массив

Массив хранит фиксированное число элементов одного типа:

```csharp
string[] weekdays =
{
    "Понедельник",
    "Вторник",
    "Среда"
};
```

Индекс начинается с нуля:

```csharp
Console.WriteLine(weekdays[0]); // Понедельник
weekdays[1] = "ВТ";
Console.WriteLine(weekdays.Length); // 3
```

`weekdays[3]` вызовет `IndexOutOfRangeException`: допустимы индексы 0, 1 и 2. Размер после создания не меняется, но элементы менять можно.

Массив удобен, когда размер известен и фиксирован, нужен быстрый доступ по индексу или API требует массив. Если элементы часто добавляются, удобнее список.

## 3. Перечисление через `for` и `foreach`

`for` даёт индекс:

```csharp
for (int index = 0; index < weekdays.Length; index++)
{
    Console.WriteLine($"{index}: {weekdays[index]}");
}
```

Условие использует `<`, а не `<=`, потому что последний индекс на единицу меньше длины.

`foreach` читает элементы без ручного индекса:

```csharp
foreach (string day in weekdays)
{
    Console.WriteLine(day);
}
```

Выбирайте `foreach`, если важен каждый элемент, а индекс не нужен. Не изменяйте структуру обычной коллекции (`Add`/`Remove`) во время `foreach`: enumerator обнаружит изменение и обычно выбросит исключение.

## 4. `List<T>` — динамический массив

```csharp
var products = new List<string>();
products.Add("Книга");
products.Add("Тетрадь");
products.Insert(1, "Ручка");

Console.WriteLine(products[0]);
Console.WriteLine(products.Count); // 3
```

Полезные операции:

- `Add` — добавить в конец;
- `AddRange` — добавить несколько;
- `Insert` — вставить по индексу;
- `Remove` — удалить первое равное значение;
- `RemoveAt` — удалить по индексу;
- `Contains` — проверить наличие;
- `IndexOf` — найти индекс;
- `Clear` — удалить всё.

`Count` — фактическое число элементов. `Capacity` — размер внутреннего массива, уже выделенного под элементы. Когда места не хватает, список создаёт более крупный внутренний массив и копирует данные.

Поэтому добавление в конец обычно O(1), но иногда требует O(n) копирования. В среднем на длинной серии операций говорят **амортизированное O(1)**.

Вставка в начало или середину требует сдвинуть следующие элементы и имеет O(n). Поиск `Contains` также обычно просматривает элементы до совпадения — O(n).

## 5. Что означают O(1), O(n), O(log n)

Big O описывает, как растёт объём работы при росте входа, а не точное время в миллисекундах.

- **O(1)** — число шагов почти не зависит от размера: взять элемент списка по известному индексу.
- **O(n)** — в худшем случае пройти все `n` элементов: найти имя в обычном списке.
- **O(log n)** — на каждом шаге отбрасывается значительная часть вариантов: поиск в сбалансированном дереве.

O(1) не означает «мгновенно», а O(n) не означает «плохо». Список из десяти элементов часто проще и быстрее сложной структуры. Оценка помогает при больших объёмах и частых операциях.

## 6. `Dictionary<TKey, TValue>` — поиск по ключу

Словарь связывает уникальный ключ со значением:

```csharp
var prices = new Dictionary<string, decimal>();
prices.Add("book", 25.50m);
prices["pen"] = 3.20m;

if (prices.TryGetValue("book", out decimal price))
{
    Console.WriteLine(price);
}
```

Ключи уникальны. Повторный `Add` того же ключа выбросит исключение, а индексатор `prices[key] = value` добавит или заменит значение.

Для чтения неизвестного ключа предпочитайте `TryGetValue`. Индексатор `prices["unknown"]` выбросит `KeyNotFoundException`.

Словарь обычно реализован хеш-таблицей и даёт ожидаемое O(1) добавление/поиск. Он вычисляет hash code ключа, выбирает область хранения, затем проверяет равенство кандидатов.

Правило: если два ключа равны по `Equals`, их `GetHashCode` обязан совпадать. Обратное не требуется: разные ключи могут иметь одинаковый hash — это collision, которую словарь умеет разрешать.

Нельзя логически изменять часть ключа, влияющую на равенство/hash, пока ключ находится в словаре: объект окажется не в той области поиска.

## 7. Сравнение строк и comparer

```csharp
var users = new Dictionary<string, string>(
    StringComparer.OrdinalIgnoreCase);

users["Anna"] = "Анна";
Console.WriteLine(users["ANNA"]); // Анна
```

Comparer задаёт правила равенства и hash code. Для технических ключей, логинов и кодов обычно выбирают `Ordinal` либо `OrdinalIgnoreCase`. Преобразование всех ключей через `ToLower()` создаёт строки, зависит от выбранной культуры и хуже выражает намерение.

## 8. `HashSet<T>` — множество уникальных значений

```csharp
var tags = new HashSet<string>(StringComparer.OrdinalIgnoreCase);
tags.Add("csharp");
bool addedAgain = tags.Add("CSHARP"); // false
```

Set хранит уникальные элементы и обычно быстро проверяет наличие. Он умеет операции множеств:

```csharp
tags.UnionWith(otherTags);       // объединение
tags.IntersectWith(required);    // пересечение
tags.ExceptWith(blocked);        // разность
```

Если главное требование — уникальность и `Contains`, `HashSet<T>` честнее списка с постоянными проверками. Не полагайтесь на его порядок как на предметный контракт.

## 9. Очередь и стек

### `Queue<T>` — first in, first out

```csharp
var requests = new Queue<string>();
requests.Enqueue("Заявка A");
requests.Enqueue("Заявка B");

string first = requests.Dequeue(); // Заявка A
```

Очередь моделирует порядок обслуживания: первым пришёл — первым вышел. `Peek` смотрит следующий элемент без удаления; `TryDequeue` безопасно обрабатывает пустую очередь.

### `Stack<T>` — last in, first out

```csharp
var history = new Stack<string>();
history.Push("Шаг 1");
history.Push("Шаг 2");

string last = history.Pop(); // Шаг 2
```

Стек подходит для undo, обхода вложенности и возврата по последним действиям. `Peek` читает вершину, `TryPop` безопасно пытается удалить.

## 10. `LinkedList<T>`

Связный список состоит из узлов со ссылками на соседей. Если конкретный узел уже известен, вставка/удаление рядом выполняется O(1). Но поиск узла остаётся O(n), доступ `list[500]` отсутствует, а элементы хуже расположены в памяти для процессора.

`LinkedList<T>` нужен реже `List<T>`. Формулировка «в середину часто вставляем» недостаточна: если перед каждой вставкой середину приходится искать, вся операция остаётся O(n).

## 11. Отсортированные коллекции

- `SortedSet<T>` — уникальные элементы в порядке comparer;
- `SortedDictionary<TKey,TValue>` — ключи в порядке, основные операции O(log n);
- `SortedList<TKey,TValue>` — компактное хранение в массивах, быстрое чтение, но дорогие вставки со сдвигом.

Сортировка требует согласованного сравнения. Если comparer говорит, что элементы равны по порядку (`Compare` возвращает 0), sorted set считает их одним элементом, даже если другое определение `Equals` расходится. Такие противоречия нужно исключать.

## 12. Интерфейсы коллекций

Интерфейс в параметре сообщает, какие возможности нужны методу:

- `IEnumerable<T>` — значения можно перечислить;
- `ICollection<T>` — есть количество и операции изменения;
- `IList<T>` — есть индекс и позиционные операции;
- `IReadOnlyCollection<T>` — можно читать количество и перечислять;
- `IReadOnlyList<T>` — дополнительно читать по индексу.

```csharp
public void PrintAll(IEnumerable<string> values)
{
    foreach (string value in values)
        Console.WriteLine(value);
}
```

Методу печати не нужны `Add` и индекс, поэтому широкий `List<string>` в контракте избыточен.

## 13. Read-only не означает immutable

```csharp
var source = new List<string> { "A" };
IReadOnlyList<string> view = source;
source.Add("B");
Console.WriteLine(view.Count); // 2
```

Через `view` нельзя вызвать `Add`, но исходный список продолжает меняться. Read-only ограничивает конкретный путь доступа, а **immutable collection** создаёт новую версию при изменении.

`System.Collections.Immutable` содержит `ImmutableList<T>`, `ImmutableDictionary<TKey,TValue>` и другие структуры. Операция `Add` возвращает новую коллекцию, часто разделяя внутренние данные со старой. Это упрощает передачу снимков, но имеет стоимость и не делает сами изменяемые элементы immutable.

## 14. Конкурентные коллекции

Обычные коллекции не предназначены для одновременной записи из нескольких потоков. `ConcurrentDictionary`, `ConcurrentQueue` и другие дают потокобезопасные отдельные операции.

Важно: две безопасные операции не образуют автоматически одну атомарную:

```csharp
if (!dictionary.ContainsKey(key))
    dictionary[key] = value;
```

Другой поток может изменить словарь между строками. Для связанной операции есть `GetOrAdd`, `AddOrUpdate` или внешняя синхронизация. Подробно конкурентность изучается в уроках 21–23.

## 15. Как выбирать коллекцию

Сначала задайте вопросы:

1. Нужен ли порядок?
2. Нужен ли доступ по числовому индексу?
3. Должны ли значения быть уникальны?
4. Есть ли уникальный ключ для поиска?
5. Какая операция выполняется чаще всего?
6. Меняется ли набор после создания?
7. Есть ли конкурентная запись?

Обычный старт:

| Требование | Кандидат |
|---|---|
| фиксированный размер, индекс | массив |
| изменяемая последовательность | `List<T>` |
| поиск значения по ключу | `Dictionary<TKey,TValue>` |
| уникальность/быстрый Contains | `HashSet<T>` |
| FIFO | `Queue<T>` |
| LIFO | `Stack<T>` |
| постоянно отсортированные ключи | sorted collection |

## Частые ошибки

- индекс на единицу выходит за границу;
- список изменяется внутри `foreach`;
- `Contains` по большому списку вызывается во вложенном цикле;
- неизвестный ключ читается индексатором без проверки;
- изменяемый объект используется как dictionary key;
- код полагается на случайный порядок hash collection;
- `IReadOnlyList` принимается за гарантию неизменяемости;
- concurrent collection считается атомарной для произвольной последовательности действий;
- сложная коллекция выбирается без измеримой необходимости.

## Итоговая карта

```text
Последовательность → array / List
Уникальность      → HashSet
Ключ → значение   → Dictionary
Первый вошёл      → Queue
Последний вошёл   → Stack

Выбор определяют операции и правила данных,
а не привычка использовать List везде.
```

Вы готовы к LINQ, если можете создать и перечислить список, безопасно получить значение словаря, объяснить равенство ключей и подобрать коллекцию для пяти типовых требований.
