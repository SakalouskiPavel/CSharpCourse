# Практика — урок 05

## Цель

Создать каталог учебных книг и подобрать отдельную коллекцию для каждого требования. LINQ использовать нельзя: операции реализуются циклами, чтобы устройство коллекций оставалось видимым.

## Модель

Создайте `Book` с неизменяемыми:

- `InventoryNumber` — уникальный технический ключ;
- `Title`;
- `Author`;
- `Price`;
- набором тегов без дубликатов и без учёта регистра.

Конструктор защищает обязательные значения и не принимает отрицательную цену.

## Индексы каталога

`BookCatalog` должен использовать:

1. `Dictionary<string, Book>` для поиска по инвентарному номеру;
2. `HashSet<string>` для набора всех известных тегов;
3. `List<Book>` только там, где важна последовательность вывода;
4. `Queue<string>` для заявок на добавление книг;
5. `Stack<string>` для истории последних успешных команд.

Для технических строк выберите и в README обоснуйте `StringComparer`.

## Обязательные операции

- добавить книгу и отказать при повторном номере;
- безопасно найти книгу по номеру без исключения для обычного отсутствия;
- удалить книгу с согласованным обновлением индексов;
- вернуть read-only представление каталога, не отдавая изменяемое внутреннее состояние;
- найти книги с указанным тегом обычным циклом;
- обработать следующую заявку FIFO;
- показать последнюю выполненную команду LIFO.

## Проверка согласованности

После каждой изменяющей операции вызывайте внутреннюю проверку либо отдельный диагностический метод:

- число ключей совпадает с числом книг;
- каждый ключ соответствует `InventoryNumber` книги;
- после удаления поиск не находит книгу;
- набор тегов соответствует выбранной политике. Решите заранее: удалять тег, когда не осталось книг, или хранить историю тегов.

## Таблица сценариев

Проверьте минимум:

| Сценарий | Ожидаемый результат |
|---|---|
| Добавление уникальной книги | Успех и доступ по ключу |
| Тот же номер в другом регистре | Результат согласно выбранному comparer |
| Дубликат номера | Отказ без порчи индексов |
| Неизвестный номер | Нормальный результат «не найдено» |
| Два одинаковых тега разного регистра | Один логический тег |
| Удаление | Все индексы согласованы |
| Две заявки | Обрабатываются FIFO |
| Две команды | История читается LIFO |

## Ограничения

- не использовать LINQ;
- не искать ключ постоянным перебором списка;
- не отдавать внутренний `List` или `Dictionary` вызывающему коду;
- не полагаться на порядок `Dictionary`/`HashSet`;
- не изменять ключ книги после добавления;
- не ловить `KeyNotFoundException` как обычный способ поиска.

## Критерии приёмки

- каждая коллекция выбрана по названному требованию;
- основные операции и их сложность описаны в README;
- дубликаты и отсутствие обрабатываются предсказуемо;
- внутреннее состояние нельзя изменить через возвращённую ссылку;
- все сценарии воспроизводятся в console application;
- solution собирается без ошибок;
- ученик объясняет разницу между read-only view и immutable data.

## Дополнительное задание

Реализуйте второй вариант индекса тегов: `Dictionary<string, HashSet<string>>`, где тег связан с номерами книг. Сравните сложность поиска по тегу с полным проходом каталога. Не утверждайте, что вариант быстрее, пока не назовёте размер данных и цену обновления индекса.
