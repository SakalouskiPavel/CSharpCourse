# Урок 07. Строки, Unicode и регулярные выражения

> Текст для человека — слова и символы. Для программы — последовательность числовых кодов, которую нужно сравнивать, хранить и разбирать по точным правилам. Ошибки здесь проявляются в именах, языках, денежных форматах, безопасности и производительности.

## Что нужно знать

Понадобятся типы, методы, условия, коллекции и LINQ. Regex изучается после обычных строковых операций: регулярное выражение полезно не для любой задачи с текстом.

## 1. `string` и `char`

```csharp
string greeting = "Здравствуйте";
char first = greeting[0];
```

`string` — ссылочный тип для последовательности UTF-16 code units. `char` хранит один 16-битный code unit. Для многих распространённых букв один `char` соответствует видимому символу, но это не универсальное правило Unicode.

Строка может быть:

- `null` — ссылка отсутствует;
- `""` — существует, но длина ноль;
- `"   "` — содержит пробельные символы.

Проверка пользовательского текста:

```csharp
if (string.IsNullOrWhiteSpace(input))
    Console.WriteLine("Введите непустой текст");
```

## 2. Неизменяемость строк

Строка **immutable**: после создания её содержимое не меняется.

```csharp
string original = "кот";
string upper = original.ToUpperInvariant();

Console.WriteLine(original); // кот
Console.WriteLine(upper);    // КОТ
```

Метод вернул новую строку. Неизменяемость упрощает совместное использование, хеширование и потокобезопасное чтение, но серия изменений создаёт промежуточные объекты.

```csharp
string text = "";
for (int i = 0; i < 10_000; i++)
    text += i; // много новых строк
```

Для небольшого числа частей обычная интерполяция яснее. Для большого неизвестного числа добавлений используют `StringBuilder` или потоковую запись.

## 3. Экранирование и разные литералы

В обычной строке обратная косая черта начинает escape sequence:

```csharp
string lineBreak = "Первая строка\nВторая строка";
string quote = "Он сказал: \"Привет\"";
string path = "C:\\Temp\\data.txt";
```

Verbatim string с `@` почти не обрабатывает `\`:

```csharp
string path = @"C:\Temp\data.txt";
```

Raw string удобна для многострочного JSON, SQL или regex:

```csharp
string json = """
{
  "name": "Анна",
  "active": true
}
""";
```

Количество внешних кавычек выбирают так, чтобы содержимое не завершило литерал. Это синтаксис C#, а не часть итогового текста.

## 4. Unicode: буква не всегда равна `char`

Unicode назначает **code point** абстрактному символу. UTF-16 кодирует code point одним или двумя `char` (surrogate pair). Кроме того, видимый пользователю знак — **grapheme cluster** — может состоять из нескольких code points, например буква плюс combining accent или emoji с модификатором.

```csharp
string emoji = "😀";
Console.WriteLine(emoji.Length); // 2, а не 1
```

`Length` считает UTF-16 code units. Для перечисления Unicode scalar values используйте `Rune`:

```csharp
foreach (Rune rune in emoji.EnumerateRunes())
    Console.WriteLine(rune.Value);
```

Для пользовательских текстовых элементов служит `StringInfo`. Даже grapheme segmentation зависит от версии Unicode и задачи интерфейса.

Практический вывод: не обрезайте имя «до N символов» через бездумный `Substring(0, N)`, если нельзя разорвать surrogate pair или пользовательский знак.

## 5. Нормализация Unicode

Одинаково выглядящий текст может иметь разную последовательность code points. Буква `é` бывает одним code point или `e` плюс combining accent.

```csharp
string normalized = input.Normalize(NormalizationForm.FormC);
```

Нормализация приводит эквивалентные представления к выбранной форме. Она не выполняет перевод, исправление опечаток или универсальное сравнение «как человек». Решение нормализовать должно применяться одинаково на границе данных.

## 6. Сравнение строк

У сравнения есть цель. Для технических идентификаторов обычно нужны точные ordinal-правила:

```csharp
bool sameCode = string.Equals(
    left,
    right,
    StringComparison.OrdinalIgnoreCase);
```

- `Ordinal` сравнивает кодовые единицы;
- `OrdinalIgnoreCase` добавляет независимое от текущей культуры сравнение регистра;
- `CurrentCulture` следует языковым правилам пользователя;
- `InvariantCulture` использует стабильную, но лингвистическую культуру без привязки к пользователю.

Для логина, protocol token, extension и внутреннего кода чаще выбирают ordinal. Для сортировки отображаемых фамилий — нужную `CultureInfo` и `CompareInfo`.

Преобразование обеих строк через `ToLower()` хуже явного `StringComparison`: создаёт объекты, может использовать неожиданную культуру и скрывает намерение.

## 7. Основные операции

```csharp
string cleaned = input.Trim();
bool starts = cleaned.StartsWith("ID-", StringComparison.Ordinal);
bool contains = cleaned.Contains("error", StringComparison.OrdinalIgnoreCase);
string replaced = cleaned.Replace("-", ":", StringComparison.Ordinal);
string[] parts = cleaned.Split(':', StringSplitOptions.TrimEntries);
string joined = string.Join(", ", parts);
```

Всегда проверяйте точную перегрузку. Например, сравнение без `StringComparison` может иметь не ту семантику, которую предполагает читатель.

`Substring`, диапазоны `text[start..end]` и `IndexOf` работают в индексах UTF-16. Отрицательный/слишком большой индекс вызывает исключение.

## 8. Форматирование и культура

Интерполяция:

```csharp
decimal price = 1234.5m;
Console.WriteLine($"Цена: {price:F2}");
```

Формат `F2` показывает две цифры после разделителя. Фактический разделитель и оформление зависят от текущей культуры.

Данные для человека форматируют в его культуре. Машинные протоколы, persisted technical values и стабильные логи часто требуют `InvariantCulture`:

```csharp
string machineValue = price.ToString(
    "F2",
    CultureInfo.InvariantCulture);
```

Разбор тоже зависит от культуры:

```csharp
bool parsed = decimal.TryParse(
    input,
    NumberStyles.Number,
    culture,
    out decimal value);
```

Не заменяйте запятую на точку вручную: правила группировки, знак и пробелы сложнее.

## 9. `StringBuilder`

```csharp
var builder = new StringBuilder();

foreach (Book book in books)
{
    builder.Append(book.InventoryNumber)
           .Append(": ")
           .AppendLine(book.Title);
}

string report = builder.ToString();
```

Builder хранит изменяемый буфер и уменьшает число промежуточных строк при множественных добавлениях. Для `a + b` или одной интерполяции он избыточен. Это mutable object: не делите один экземпляр между потоками без синхронизации.

## 10. `Span<char>` и `ReadOnlySpan<char>`

Span — лёгкое представление непрерывного участка памяти без копирования:

```csharp
ReadOnlySpan<char> text = "ID:12345";
ReadOnlySpan<char> numberPart = text[3..];
bool ok = int.TryParse(numberPart, out int id);
```

Срез span не создаёт новую строку. `Span<T>` — `ref struct`: он не может храниться в обычном поле класса, использоваться после завершения памяти-источника или переживать `await`. Это инструмент оптимизации parsing в горячих участках. Сначала пишите понятный вариант на string и измеряйте.

## 11. Когда нужны регулярные выражения

**Regular expression (regex)** — компактный язык описания текстового шаблона. Подходит для:

- проверки кода фиксированной формы;
- поиска фрагментов;
- извлечения групп;
- простых замен по шаблону.

Regex не подходит для произвольно вложенной грамматики, полноценного HTML/JSON/C# parsing и задачи, где несколько строковых методов читаются яснее.

Для префикса достаточно:

```csharp
value.StartsWith("ID-", StringComparison.Ordinal)
```

Не превращайте это в regex без причины.

## 12. Первый шаблон

Требование: код состоит из двух заглавных латинских букв, дефиса и четырёх цифр.

```regex
\A[A-Z]{2}-\d{4}\z
```

Разбор:

- `\A` — начало всей строки;
- `[A-Z]` — одна буква от A до Z;
- `{2}` — ровно два повторения;
- `-` — обычный дефис;
- `\d` — цифра по правилам regex;
- `{4}` — ровно четыре;
- `\z` — конец всей строки.

Если нужны только ASCII-цифры, честнее `[0-9]`: `\d` в .NET совпадает и с некоторыми цифрами других письменностей.

В C# обычная строка должна экранировать `\`, поэтому удобна verbatim string:

```csharp
string pattern = @"\A[A-Z]{2}-[0-9]{4}\z";
bool valid = Regex.IsMatch(value, pattern);
```

Есть два слоя правил: синтаксис строки C# и синтаксис regex. `@` уменьшает двойное экранирование.

## 13. Основные элементы regex

### Классы и точка

- `[abc]` — один из символов;
- `[^abc]` — любой, кроме перечисленных;
- `\s` — whitespace;
- `\w` — word character по Unicode-правилам .NET;
- `.` — почти любой символ, обычно кроме перевода строки.

### Квантификаторы

- `?` — ноль или один;
- `*` — ноль или больше;
- `+` — один или больше;
- `{m,n}` — от `m` до `n`.

По умолчанию квантификатор greedy: берёт максимально возможное и при необходимости отдаёт назад. `+?`, `*?` — lazy-варианты, берущие минимально возможное. Lazy не означает автоматически быстро или безопасно.

### Альтернативы и группы

```regex
\A(?<level>INFO|WARN|ERROR):\s*(?<message>.+)\z
```

- `|` — один из вариантов;
- `( ... )` — capturing group;
- `(?: ... )` — non-capturing group;
- `(?<name> ... )` — именованная группа.

```csharp
Match match = Regex.Match(line, pattern, options, timeout);
if (match.Success)
{
    string level = match.Groups["level"].Value;
    string message = match.Groups["message"].Value;
}
```

Сначала проверяется `Success`, затем читаются группы.

## 14. Anchors, boundaries и режимы

`^` и `$` обозначают начало/конец строки, но с `RegexOptions.Multiline` работают для каждой строки многострочного текста. `\A` и `\z` всегда ограничивают весь input.

`\b` — word boundary между word/non-word категориями; это не «граница слова по правилам любого языка». Опции `IgnoreCase`, `Multiline`, `Singleline`, `CultureInvariant`, `IgnorePatternWhitespace` меняют смысл шаблона и должны быть видимы рядом с ним.

## 15. Lookaround

Lookahead и lookbehind проверяют окружение, не добавляя его в совпадение:

- `(?=...)` — положительный lookahead;
- `(?!...)` — отрицательный lookahead;
- `(?<=...)` — положительный lookbehind;
- `(?<!...)` — отрицательный lookbehind.

Например, `\d+(?= ₽)` находит число перед обозначением рубля, но не включает пробел и знак в результат. Lookaround полезен, но несколько вложенных проверок быстро делают шаблон нечитаемым. Часто яснее разобрать этапами.

## 16. Backtracking и ReDoS

Обычный regex engine .NET пробует варианты и возвращается назад — **backtracking**. Неоднозначные вложенные квантификаторы, например `(a+)+$`, на специально подобранной длинной строке могут потребовать огромного числа попыток. Это создаёт Regular Expression Denial of Service (ReDoS).

Для недоверенного текста:

```csharp
bool valid = Regex.IsMatch(
    input,
    pattern,
    RegexOptions.CultureInvariant,
    TimeSpan.FromMilliseconds(200));
```

Дополнительно:

- ограничивайте длину input;
- избегайте неоднозначных вложенных квантификаторов;
- используйте anchors;
- тестируйте длинные почти подходящие строки;
- рассмотрите `RegexOptions.NonBacktracking`, если шаблон поддерживается этим engine;
- не считайте timeout заменой понятному шаблону.

## 17. `GeneratedRegex`

Для постоянного известного шаблона source generator создаёт реализацию при сборке:

```csharp
public static partial class CourseCodeRegex
{
    [GeneratedRegex(
        @"\A[A-Z]{2}-[0-9]{4}\z",
        RegexOptions.CultureInvariant,
        matchTimeoutMilliseconds: 200)]
    public static partial Regex CourseCode();
}
```

Класс и метод `partial`, метод не имеет тела. Вызов:

```csharp
bool valid = CourseCodeRegex.CourseCode().IsMatch(input);
```

Generated regex уменьшает runtime parsing шаблона и даёт compile-time generation, но не исправляет логически неверный или уязвимый pattern.

## 18. Пошаговая стратегия разбора текста

1. Опишите формат примерами правильного и неправильного ввода.
2. Ограничьте длину и проверьте `null`.
3. Попробуйте `StartsWith`, `Split`, `IndexOf`, `TryParse`.
4. Если структура действительно шаблонная, напишите regex с anchors.
5. Используйте именованные группы для нужных частей.
6. Добавьте timeout и тест почти совпадающей длинной строки.
7. После синтаксического разбора проверьте смысл: дата существует, число в диапазоне, код разрешён.

Regex проверяет форму, но не всю предметную корректность. Шаблон даты `\d{2}.\d{2}.\d{4}` пропустит 99.99.0000; затем нужен `DateTime.TryParseExact`.

## Частые ошибки

- `Length` принимается за число видимых символов;
- технические ключи сравниваются через текущую культуру;
- строки приводятся к lower case вместо comparer;
- числа форматируются/разбираются без явной культуры на машинной границе;
- `+=` используется в большом цикле;
- regex применяется к простому префиксу или вложенному языку;
- шаблон не ограничен началом и концом при полной валидации;
- `\d` ошибочно считается только `[0-9]`;
- группы читаются без проверки `Match.Success`;
- недоверенный input обрабатывается без лимита и timeout;
- синтаксическое совпадение принимается за предметную валидность.

## Итоговая карта

```text
Текст
  ├─ сравнить → выбрать ordinal или culture
  ├─ собрать  → interpolation / Join / StringBuilder
  ├─ разобрать простой формат → string methods + TryParse
  ├─ разобрать шаблон → Regex + groups + timeout
  └─ обработать Unicode → Rune / StringInfo / normalization по задаче
```
