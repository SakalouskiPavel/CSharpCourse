# Практика — урок 07

## Цель

Создать безопасный анализатор текстового журнала. Он принимает строки, отделяет синтаксически неверные записи, преобразует поля в типы и строит читаемый отчёт.

## Формат строки

```text
2026-07-06T14:30:15Z | ERROR | ORD-1234 | Не удалось сохранить заказ
```

Поля:

1. время в ISO 8601 UTC;
2. уровень `INFO`, `WARN` или `ERROR` без учёта ASCII-регистра;
3. код операции: три заглавные ASCII-буквы, дефис, четыре ASCII-цифры;
4. Unicode-сообщение, которое может содержать пробелы и emoji.

## Этап 1. Правила до кода

Запишите:

- максимально допустимую длину строки;
- допустимы ли пробелы вокруг разделителя;
- может ли сообщение содержать символ `|`;
- поведение для неизвестного уровня;
- какую культуру использует дата;
- является ли пустое сообщение допустимым.

Выберите либо простой `Split` с ограничением, либо regex с именованными группами. Обоснуйте выбор.

## Этап 2. Результат parsing

Создайте `LogEntry` с типизированными `DateTimeOffset Timestamp`, `LogLevel Level`, `string OperationCode`, `string Message`.

Обычная неверная строка не должна выбрасывать необработанное исключение. Метод возвращает результат Try-формы либо отдельный объект с ошибкой:

```csharp
bool TryParseLogEntry(
    string? line,
    out LogEntry? entry,
    out string? error)
```

После проверки формы:

- время разбирается `TryParseExact`/подходящим ISO API с явной культурой;
- уровень сопоставляется явно;
- operation code проверяется полностью от начала до конца;
- сообщение сохраняет Unicode.

## Этап 3. Regex safety

Если используется regex:

- шаблон ограничен `\A` и `\z`;
- ASCII-цифры записаны `[0-9]`;
- нужные поля имеют именованные группы;
- задан timeout;
- input length проверяется до regex;
- добавлена длинная почти совпадающая строка.

Если основной parser построен без regex, создайте отдельный GeneratedRegex только для operation code и примените те же ограничения.

## Этап 4. Сравнение

Уровни и operation codes являются техническими значениями. Используйте ordinal-сравнение. Не преобразуйте все значения в lower case ради сравнения.

Добавьте сообщения:

- на русском;
- с emoji;
- с composed и decomposed Unicode-представлением одной буквы.

Покажите длину в UTF-16 code units и число `Rune`. Не обрезайте сообщение по `Length` как по числу видимых символов.

## Этап 5. Отчёт

Из корректных записей сформируйте:

- количество по уровням;
- список уникальных operation codes;
- первую и последнюю дату;
- строки ошибок parsing с номером исходной строки;
- полный многострочный отчёт через `StringBuilder`.

Для машинного вывода дат используйте стабильный invariant ISO format. Для поясняющего текста допустима выбранная культура пользователя.

## Обязательные сценарии

| Вход | Ожидаемый исход |
|---|---|
| корректная строка | `LogEntry` |
| `error` в другом регистре | принято согласно правилу уровня |
| неправильная дата | ошибка parsing, не crash |
| `AB-1234` | неверный operation code |
| Unicode-цифры вместо ASCII | отклонены кодом операции |
| пустое сообщение | по зафиксированному правилу |
| слишком длинная строка | отклонена до regex |
| emoji в сообщении | сохранён без повреждения |
| мусор до/после записи | отклонён полной валидацией |

## Ограничения

- не использовать exception как обычный результат неверной строки;
- не применять regex без timeout к пользовательскому вводу;
- не сравнивать технические ключи через текущую культуру;
- не считать `char` пользовательским символом во всех случаях;
- не собирать большой отчёт через `+=` в цикле;
- не считать совпадение regex достаточной проверкой даты.

## Критерии приёмки

- все обязательные сценарии имеют предсказанный и фактический результат;
- parser отделяет форму от смыслового преобразования;
- ошибки содержат номер строки и понятную причину без stack trace;
- Unicode-сообщения не повреждаются;
- отчёт воспроизводим и использует явные правила культуры;
- решение ограничивает input и regex time;
- ученик объясняет каждый элемент шаблона и может предложить вариант без regex.

## Дополнительное задание

Реализуйте второй parser на `ReadOnlySpan<char>` без промежуточного массива от `Split`. Сначала зафиксируйте корректность общими сценариями, затем измерьте аллокации. Если выигрыш несущественен для объёма задачи, оставьте простой вариант основным и объясните решение.
