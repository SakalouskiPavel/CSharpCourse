# Урок 06. LINQ: запросы к последовательностям

> LINQ позволяет описать не перебор по шагам, а преобразование данных: «оставить дорогие книги, взять названия и отсортировать». Чтобы пользоваться LINQ уверенно, нужно понимать источник, оператор, лямбда-выражение и момент фактического выполнения.

## Что нужно знать

Понадобятся методы, условия, классы, интерфейсы и коллекции. Особое значение имеет `IEnumerable<T>`: он обещает, что значения можно получать по одному. Делегаты подробно изучаются в уроке 13; здесь лямбда вводится как способ передать LINQ небольшое правило.

## 1. Что такое LINQ

LINQ — Language Integrated Query, «встроенные в язык запросы». Это набор стандартных операций над источниками данных.

Обычный цикл:

```csharp
var expensiveTitles = new List<string>();

foreach (Book book in books)
{
    if (book.Price >= 50m)
        expensiveTitles.Add(book.Title);
}
```

Тот же смысл через LINQ:

```csharp
List<string> expensiveTitles = books
    .Where(book => book.Price >= 50m)
    .Select(book => book.Title)
    .ToList();
```

Читайте слева направо:

1. взять `books`;
2. оставить книги с ценой не ниже 50;
3. из каждой книги взять название;
4. выполнить запрос и сохранить результат списком.

LINQ не всегда короче цикла и не обязан быть быстрее. Его преимущество — ясное описание цепочки преобразований без ручного управления промежуточным состоянием.

## 2. Лямбда-выражение

В `book => book.Price >= 50m`:

- `book` — параметр для очередного элемента;
- `=>` читается «переходит в»;
- справа — выражение результата.

Это короткая форма функции. Эквивалентный обычный метод:

```csharp
bool IsExpensive(Book book)
{
    return book.Price >= 50m;
}

IEnumerable<Book> result = books.Where(IsExpensive);
```

Для сложного правила именованный метод часто понятнее. Параметр лямбды называйте по смыслу (`book`, не `x`).

## 3. `IEnumerable<T>` и перечисление

`IEnumerable<T>` — источник, у которого можно запросить enumerator и получать следующий элемент. Он не обещает:

- наличие индекса;
- возможность изменения;
- известное заранее количество;
- хранение всех элементов в памяти;
- возможность повторно получить те же данные.

Источником может быть список, массив, вычисляемая последовательность, строки файла или значения, создаваемые по мере запроса.

`foreach` скрывает работу enumerator:

```csharp
foreach (Book book in books)
    Console.WriteLine(book.Title);
```

При каждом новом перечислении источник может выполняться заново.

## 4. Отложенное выполнение

Большинство операторов, возвращающих `IEnumerable<T>`, не перебирает источник сразу:

```csharp
IEnumerable<Book> query = books.Where(book => book.Price >= 50m);
```

Здесь создано описание запроса. Он выполнится при `foreach`, `ToList`, `Count`, `First` или другом потреблении.

```csharp
books.Add(new Book("3", "Новая книга", 100m));

foreach (Book book in query)
    Console.WriteLine(book.Title); // новая книга участвует
```

Это **deferred execution** — отложенное выполнение. Оно позволяет строить цепочку и читать данные потоково, но результат может измениться между перечислениями.

`ToList()` выполняет запрос сейчас и создаёт снимок ссылок/значений:

```csharp
List<Book> snapshot = query.ToList();
```

Снимок списка не клонирует сами изменяемые объекты `Book`.

## 5. `Where`, `Select` и `SelectMany`

`Where` фильтрует и сохраняет форму элемента:

```csharp
IEnumerable<Book> cheap = books.Where(book => book.Price < 20m);
```

`Select` проецирует каждый элемент в новую форму:

```csharp
IEnumerable<string> titles = books.Select(book => book.Title);
```

`SelectMany` разворачивает вложенные последовательности:

```csharp
IEnumerable<string> allTags = books.SelectMany(book => book.Tags);
```

Обычный `Select(book => book.Tags)` вернул бы последовательность наборов тегов. `SelectMany` возвращает один поток отдельных тегов.

Обычно выгодно отфильтровать ненужные элементы до дорогого преобразования, если это не меняет смысл.

## 6. Поиск одного элемента и кардинальность

**Кардинальность** здесь — сколько элементов ожидает операция.

| Оператор | Ожидание |
|---|---|
| `First` | нужен первый, отсутствие — ошибка |
| `FirstOrDefault` | первый либо default при отсутствии |
| `Single` | должен существовать ровно один |
| `SingleOrDefault` | ноль или один; два — ошибка |
| `Last` | последний, отсутствие — ошибка |

```csharp
Book? found = books.FirstOrDefault(book => book.InventoryNumber == number);
```

`First` не проверяет уникальность. `Single` проверяет и потому может пройти источник дальше после первого совпадения. Выбор выражает предметное ожидание, а не предпочтение короткого имени.

Для value type default может выглядеть как допустимое значение (`0`, `false`), поэтому отсутствие иногда лучше моделировать nullable или отдельным результатом.

## 7. Проверки и количество

```csharp
bool anyFree = books.Any(book => book.IsAvailable);
bool allPriced = books.All(book => book.Price > 0m);
int count = books.Count(book => book.Price >= 50m);
```

`Any` прекращает работу на первом совпадении. Проверка `collection.Any()` обычно лучше `collection.Count() > 0` для произвольного `IEnumerable`, потому что не требует полного подсчёта.

`All` для пустой последовательности возвращает `true`: не найдено элемента, нарушающего условие. Это называется vacuous truth и иногда требует отдельной проверки `Any`, если бизнес-смысл требует хотя бы один элемент.

## 8. Сортировка

```csharp
IOrderedEnumerable<Book> sorted = books
    .OrderBy(book => book.Author)
    .ThenBy(book => book.Title);
```

- `OrderBy` задаёт первичный ключ;
- `ThenBy` — следующий ключ при равенстве;
- варианты `Descending` меняют направление.

Повторный `OrderBy` начинает новую сортировку и отменяет роль предыдущего ключа. Для строк можно передать comparer.

Сортировка должна увидеть элементы, прежде чем выдать первый в правильном порядке, и обычно стоит O(n log n) плюс память. Не сортируйте, если нужен только minimum/maximum.

## 9. Агрегация

Агрегат превращает много элементов в одно значение:

```csharp
decimal total = books.Sum(book => book.Price);
decimal average = books.Average(book => book.Price);
decimal min = books.Min(book => book.Price);
decimal max = books.Max(book => book.Price);
```

`Average`, `Min` и `Max` для пустой последовательности non-nullable значений могут выбросить исключение. Определите смысл пустого набора заранее.

`Aggregate` задаёт собственную свёртку:

```csharp
string line = titles.Aggregate(
    "Названия:",
    (current, title) => current + " " + title);
```

Для построения большого текста такая конкатенация неэффективна; лучше `string.Join` или `StringBuilder`. `Aggregate` полезен, когда операция действительно является последовательной свёрткой и готового оператора нет.

## 10. Уникальность и множества

```csharp
IEnumerable<string> uniqueTags = books
    .SelectMany(book => book.Tags)
    .Distinct(StringComparer.OrdinalIgnoreCase);
```

`Distinct`, `Union`, `Intersect` и `Except` используют равенство. `DistinctBy(book => book.Author)` оставляет первый элемент для каждого ключа. Нужно понимать, какой representative сохраняется и важен ли порядок источника.

## 11. Группировка

```csharp
var groups = books.GroupBy(book => book.Author);

foreach (var group in groups)
{
    Console.WriteLine(group.Key);
    foreach (Book book in group)
        Console.WriteLine($"  {book.Title}");
}
```

Каждая `IGrouping<TKey,TElement>` имеет `Key` и сама перечисляется. Группировка не равна словарю: ключ может соответствовать многим элементам, а результат запроса обычно строится при перечислении.

Отчёт:

```csharp
var report = books
    .GroupBy(book => book.Author)
    .Select(group => new
    {
        Author = group.Key,
        BookCount = group.Count(),
        TotalPrice = group.Sum(book => book.Price)
    });
```

`new { ... }` создаёт anonymous type, удобный для локального результата внутри метода. Не используйте его как публичный контракт между слоями.

## 12. `Join` и `GroupJoin`

Пусть отдельно хранятся книги и издатели:

```csharp
var result = books.Join(
    publishers,
    book => book.PublisherId,
    publisher => publisher.Id,
    (book, publisher) => new { book.Title, Publisher = publisher.Name });
```

`Join` сопоставляет равные ключи и похож на inner join SQL: элементы без пары не попадают в результат.

`GroupJoin` связывает внешний элемент с группой внутренних и моделирует one-to-many. Left join в LINQ to Objects обычно строится через `GroupJoin` и `DefaultIfEmpty`. Для начинающего важнее сначала уверенно освоить словарь/прямые связи; сложный join не всегда лучше понятного предметного индекса.

## 13. Method syntax и query syntax

До сих пор использовался method syntax. Query syntax:

```csharp
var titles =
    from book in books
    where book.Price >= 50m
    orderby book.Title
    select book.Title;
```

Компилятор преобразует его в вызовы операторов. Query syntax удобен для нескольких `from`, `join`, `let`, но не имеет отдельного оператора для каждой возможности. В одном проекте выбирайте форму по читаемости, а не смешивайте без причины.

## 14. `yield return` и собственный iterator

```csharp
IEnumerable<int> PositiveNumbers(IEnumerable<int> source)
{
    foreach (int number in source)
    {
        if (number > 0)
            yield return number;
    }
}
```

`yield return` выдаёт очередной элемент и сохраняет место продолжения. Метод выполняется лениво при перечислении. `yield break` завершает последовательность. Нельзя одновременно использовать обычный `return value` для результата такого iterator method.

## 15. Повторное перечисление

```csharp
IEnumerable<Book> query = LoadBooks().Where(book => book.Price > 0);

int count = query.Count();
foreach (Book book in query) { /* ... */ }
```

`LoadBooks()` может выполниться дважды, повторно прочитать файл или получить другие данные. Варианты:

- перечислить один раз;
- материализовать `ToList`, если нужен снимок/повторный доступ;
- изменить API, чтобы источник явно обещал коллекцию;
- не материализовать, если поток большой и нужен один проход.

Материализация — решение о времени выполнения и памяти, а не обязательный финал каждой строки LINQ.

## 16. `IEnumerable<T>` и `IQueryable<T>`

`IEnumerable<T>` выполняет .NET-код над объектами. `IQueryable<T>` строит expression tree — структуру описания выражения, которую provider, например EF Core, пытается перевести в SQL.

```csharp
IQueryable<BookEntity> query = db.Books
    .Where(book => book.Price >= 50m);
```

Запрос к БД обычно выполняется при `ToListAsync`, `FirstAsync` и других terminal operations. Не вся C#-логика переводится. Вызов `AsEnumerable` переводит дальнейшую обработку в память и может незаметно загрузить много строк.

Подробно доступ к данным изучается в уроке 12. Сейчас важно не считать LINQ to Objects и LINQ provider одним и тем же исполнением.

## Частые ошибки

- `ToList` ставится после каждого оператора;
- один deferred query перечисляется много раз;
- `First` используется там, где требуется доказать уникальность;
- `Single` применяется к данным, где дубликаты допустимы;
- `Count() > 0` полностью перечисляет поток вместо `Any`;
- `OrderBy(...).OrderBy(...)` случайно сбрасывает первый порядок;
- side effects внутри `Select`/`Where` делают повторное выполнение неожиданным;
- пустые последовательности не учитываются в агрегатах;
- LINQ-запрос маскирует квадратичный вложенный поиск;
- ожидается, что любой C#-метод переведётся `IQueryable` provider-ом.

## Итоговая карта

```text
Источник IEnumerable<T>
  ↓ Where        — какие элементы
  ↓ Select       — какая форма
  ↓ Group/Join   — какие связи
  ↓ OrderBy      — какой порядок
  ↓ ToList/Count — когда выполнить и получить результат
```

Вы готовы дальше, если можете прочитать цепочку слева направо, предсказать момент выполнения, выбрать `First` или `Single` по ожиданию и заметить повторное перечисление.
